# You Don't Know JS: *this* & Prototipos de Objetos
# Capítulo 2: ¡`this` Ahora Todo Tiene Sentido!

En el Capítulo 1, descartamos varios conceptos erróneos sobre `this` y aprendimos en su lugar que `this` es un enlace hecho para cada invocación de función, basado enteramente en su **sitio de llamada** (cómo se llama a la función).

## Sitio de Llamada

Para entender el enlace de `this`, tenemos que entender el sitio de llamada: la ubicación en el código donde se llama a una función (**no donde se declara**). Debemos inspeccionar el sitio de llamada para responder la pregunta: ¿a qué es una referencia *este* `this`?

Encontrar el sitio de llamada es generalmente: "ve a localizar desde dónde se llama a una función", pero no siempre es tan fácil, ya que ciertos patrones de código pueden obscurecer el *verdadero* sitio de llamada.

Lo importante es pensar en la **pila de llamadas** (la pila de funciones que han sido llamadas para llegar al momento actual en la ejecución). El sitio de llamada que nos importa está *en* la invocación *antes* de la función que se está ejecutando actualmente.

Demostremos la pila de llamadas y el sitio de llamada:

```js
function baz() {
    // la pila de llamadas es: `baz`
    // así que, nuestro sitio de llamada está en el ámbito global

    console.log( "baz" );
    bar(); // <-- sitio de llamada para `bar`
}

function bar() {
    // la pila de llamadas es: `baz` -> `bar`
    // así que, nuestro sitio de llamada está en `baz`

    console.log( "bar" );
    foo(); // <-- sitio de llamada para `foo`
}

function foo() {
    // la pila de llamadas es: `baz` -> `bar` -> `foo`
    // así que, nuestro sitio de llamada está en `bar`

    console.log( "foo" );
}

baz(); // <-- sitio de llamada para `baz`
```

Ten cuidado al analizar código para encontrar el sitio de llamada real (desde la pila de llamadas), porque es lo único que importa para el enlace de `this`.

**Nota:** Puedes visualizar una pila de llamadas en tu mente mirando la cadena de llamadas a funciones en orden, como hicimos con los comentarios en el fragmento anterior. Pero esto es laborioso y propenso a errores. Otra forma de ver la pila de llamadas es usar una herramienta de depuración en tu navegador. La mayoría de los navegadores de escritorio modernos tienen herramientas de desarrollo integradas, que incluyen un depurador JS. En el fragmento anterior, podrías haber establecido un punto de interrupción en las herramientas para la primera línea de la función `foo()`, o simplemente insertado la sentencia `debugger;` en esa primera línea. Cuando ejecutas la página, el depurador pausará en esa ubicación, y te mostrará una lista de las funciones que han sido llamadas para llegar a esa línea, que será tu pila de llamadas. Así que, si estás tratando de diagnosticar el enlace de `this`, usa las herramientas de desarrollo para obtener la pila de llamadas, luego encuentra el segundo elemento desde arriba, y eso te mostrará el sitio de llamada real.

## Nada Más Que Reglas

Dirigimos ahora nuestra atención a *cómo* el sitio de llamada determina dónde apuntará `this` durante la ejecución de una función.

Debes inspeccionar el sitio de llamada y determinar cuál de 4 reglas aplica. Primero explicaremos cada una de estas 4 reglas independientemente, y luego ilustraremos su orden de precedencia, si múltiples reglas *pudieran* aplicar al sitio de llamada.

### Enlace por Defecto

La primera regla que examinaremos viene del caso más común de llamadas a funciones: invocación de función independiente. Piensa en *esta* regla de `this` como la regla catch-all por defecto cuando ninguna de las otras reglas aplica.

Considera este código:

```js
function foo() {
	console.log( this.a );
}

var a = 2;

foo(); // 2
```

Lo primero a notar, si no lo sabías ya, es que las variables declaradas en el ámbito global, como `var a = 2`, son sinónimos con propiedades del objeto global del mismo nombre. No son copias una de otra, *son* la misma cosa. Piensa en ello como dos caras de la misma moneda.

Segundo, vemos que cuando se llama a `foo()`, `this.a` se resuelve a nuestra variable global `a`. ¿Por qué? Porque en este caso, el *enlace por defecto* para `this` aplica a la llamada de función, y así apunta `this` al objeto global.

¿Cómo sabemos que la regla de *enlace por defecto* aplica aquí? Examinamos el sitio de llamada para ver cómo se llama a `foo()`. En nuestro fragmento, `foo()` se llama con una referencia de función simple, sin decoración. Ninguna de las otras reglas que demostraremos aplicará aquí, así que el *enlace por defecto* aplica en su lugar.

Si el `strict mode` está en efecto, el objeto global no es elegible para el *enlace por defecto*, así que `this` se establece a `undefined` en su lugar.

```js
function foo() {
	"use strict";

	console.log( this.a );
}

var a = 2;

foo(); // TypeError: `this` is `undefined`
```

Un detalle sutil pero importante es: aunque las reglas generales de enlace de `this` están enteramente basadas en el sitio de llamada, el objeto global **solo** es elegible para el *enlace por defecto* si los **contenidos** de `foo()` **no** están ejecutándose en `strict mode`; el estado de `strict mode` del sitio de llamada de `foo()` es irrelevante.

```js
function foo() {
	console.log( this.a );
}

var a = 2;

(function(){
	"use strict";

	foo(); // 2
})();
```

**Nota:** Mezclar intencionalmente `strict mode` y no-`strict mode` juntos en tu propio código es generalmente mal visto. Tu programa completo debería probablemente ser **Strict** o **no-Strict**. Sin embargo, a veces incluyes una biblioteca de terceros que tiene diferente modo **Strict** que tu propio código, así que se debe tener cuidado con estos detalles sutiles de compatibilidad.

### Enlace Implícito

Otra regla a considerar es: ¿tiene el sitio de llamada un objeto de contexto, también referido como un objeto propietario o contenedor, aunque *estos* términos alternativos podrían ser ligeramente engañosos?

Considera:

```js
function foo() {
	console.log( this.a );
}

var obj = {
	a: 2,
	foo: foo
};

obj.foo(); // 2
```

Primero, nota la manera en que `foo()` se declara y luego se añade como una propiedad de referencia en `obj`. Independientemente de si `foo()` se declara inicialmente *en* `obj`, o se añade como referencia después (como muestra este fragmento), en ningún caso la **función** es realmente "poseída" o "contenida" por el objeto `obj`.

Sin embargo, el sitio de llamada *usa* el contexto de `obj` para **referenciar** la función, así que *podrías* decir que el objeto `obj` "posee" o "contiene" la **referencia a la función** en el momento en que la función es llamada.

Como sea que elijas llamar a este patrón, en el punto en que `foo()` es llamada, está precedida por una referencia de objeto a `obj`. Cuando hay un objeto de contexto para una referencia de función, la regla de *enlace implícito* dice que es *ese* objeto el que debería usarse para el enlace de `this` de la llamada a función.

Porque `obj` es el `this` para la llamada a `foo()`, `this.a` es sinónimo de `obj.a`.

Solo el nivel superior/último de una cadena de referencia de propiedad de objeto importa para el sitio de llamada. Por ejemplo:

```js
function foo() {
	console.log( this.a );
}

var obj2 = {
	a: 42,
	foo: foo
};

var obj1 = {
	a: 2,
	obj2: obj2
};

obj1.obj2.foo(); // 42
```

#### Pérdida Implícita

Una de las frustraciones más comunes que crea el enlace de `this` es cuando una función con *enlace implícito* pierde ese enlace, lo que usualmente significa que recurre al *enlace por defecto*, ya sea del objeto global o `undefined`, dependiendo del `strict mode`.

Considera:

```js
function foo() {
	console.log( this.a );
}

var obj = {
	a: 2,
	foo: foo
};

var bar = obj.foo; // ¡referencia/alias de función!

var a = "oops, global"; // `a` también es propiedad del objeto global

bar(); // "oops, global"
```

Aunque `bar` parece ser una referencia a `obj.foo`, de hecho, es realmente solo otra referencia a `foo` misma. Además, el sitio de llamada es lo que importa, y el sitio de llamada es `bar()`, que es una llamada simple, sin decoración y por lo tanto el *enlace por defecto* aplica.

La forma más sutil, más común, y más inesperada en que esto ocurre es cuando consideramos pasar una función callback:

```js
function foo() {
	console.log( this.a );
}

function doFoo(fn) {
	// `fn` es solo otra referencia a `foo`

	fn(); // <-- ¡sitio de llamada!
}

var obj = {
	a: 2,
	foo: foo
};

var a = "oops, global"; // `a` también es propiedad del objeto global

doFoo( obj.foo ); // "oops, global"
```

El paso de parámetros es solo una asignación implícita, y como estamos pasando una función, es una asignación implícita de referencia, así que el resultado final es el mismo que el fragmento anterior.

¿Qué pasa si la función a la que le estás pasando tu callback no es tuya, sino integrada en el lenguaje? Sin diferencia, mismo resultado.

```js
function foo() {
	console.log( this.a );
}

var obj = {
	a: 2,
	foo: foo
};

var a = "oops, global"; // `a` también es propiedad del objeto global

setTimeout( obj.foo, 100 ); // "oops, global"
```

Piensa en esta pseudo-implementación teórica cruda de `setTimeout()` proporcionada como integrada del entorno JavaScript:

```js
function setTimeout(fn,delay) {
	// esperar (de alguna manera) por `delay` milisegundos
	fn(); // <-- ¡sitio de llamada!
}
```

Es bastante común que nuestros callbacks de función *pierdan* su enlace de `this`, como acabamos de ver. Pero otra forma en que `this` puede sorprendernos es cuando la función a la que hemos pasado nuestro callback intencionalmente cambia el `this` para la llamada. Los manejadores de eventos en bibliotecas JavaScript populares son bastante aficionados a forzar que tu callback tenga un `this` que apunte, por ejemplo, al elemento DOM que disparó el evento. Aunque eso puede ser útil a veces, otras veces puede ser francamente exasperante. Desafortunadamente, estas herramientas rara vez te dejan elegir.

De cualquier manera que el `this` se cambie inesperadamente, realmente no estás en control de cómo tu referencia de función callback será ejecutada, así que no tienes forma (aún) de controlar el sitio de llamada para dar tu enlace pretendido. Veremos en breve una forma de "arreglar" ese problema *fijando* el `this`.

### Enlace Explícito

Con el *enlace implícito* como acabamos de ver, tuvimos que mutar el objeto en cuestión para incluir una referencia en sí mismo a la función, y usar esta referencia de propiedad de función para indirectamente (implícitamente) enlazar `this` al objeto.

Pero, ¿qué pasa si quieres forzar a una llamada de función a usar un objeto particular para el enlace de `this`, sin poner una referencia de propiedad de función en el objeto?

"Todas" las funciones en el lenguaje tienen algunas utilidades disponibles para ellas (vía su `[[Prototype]]` -- más sobre eso después) que pueden ser útiles para esta tarea. Específicamente, las funciones tienen los métodos `call(..)` y `apply(..)`. Técnicamente, los entornos host de JavaScript a veces proporcionan funciones que son lo suficientemente especiales (¡una forma amable de decirlo!) que no tienen tal funcionalidad. Pero esas son pocas. La gran mayoría de funciones proporcionadas, y ciertamente todas las funciones que crearás, sí tienen acceso a `call(..)` y `apply(..)`.

¿Cómo funcionan estas utilidades? Ambas toman, como su primer parámetro, un objeto para usar como `this`, y luego invocan la función con ese `this` especificado. Como estás declarando directamente lo que quieres que sea el `this`, lo llamamos *enlace explícito*.

Considera:

```js
function foo() {
	console.log( this.a );
}

var obj = {
	a: 2
};

foo.call( obj ); // 2
```

Invocar `foo` con *enlace explícito* mediante `foo.call(..)` nos permite forzar que su `this` sea `obj`.

Si pasas un valor primitivo simple (de tipo `string`, `boolean`, o `number`) como el enlace de `this`, el valor primitivo se envuelve en su forma de objeto (`new String(..)`, `new Boolean(..)`, o `new Number(..)`, respectivamente). Esto a menudo se conoce como "boxing".

**Nota:** Con respecto al enlace de `this`, `call(..)` y `apply(..)` son idénticos. *Sí* se comportan de manera diferente con sus parámetros adicionales, pero eso no es algo que nos preocupe de momento.

Desafortunadamente, el *enlace explícito* solo aún no ofrece ninguna solución al problema mencionado anteriormente, de una función "perdiendo" su enlace de `this` pretendido, o simplemente siendo sobrescrito por un framework, etc.

#### Enlace Duro

Pero un patrón de variación alrededor del *enlace explícito* realmente hace el truco. Considera:

```js
function foo() {
	console.log( this.a );
}

var obj = {
	a: 2
};

var bar = function() {
	foo.call( obj );
};

bar(); // 2
setTimeout( bar, 100 ); // 2

// `bar` enlaza duramente el `this` de `foo` a `obj`
// para que no pueda ser anulado
bar.call( window ); // 2
```

Examinemos cómo funciona esta variación. Creamos una función `bar()` que, internamente, llama manualmente a `foo.call(obj)`, invocando así forzosamente a `foo` con el enlace de `obj` para `this`. No importa cómo invoques después la función `bar`, siempre invocará manualmente a `foo` con `obj`. Este enlace es tanto explícito como fuerte, así que lo llamamos *enlace duro*.

La forma más típica de envolver una función con un *enlace duro* crea un paso-a-través de cualquier argumento pasado y cualquier valor de retorno recibido:

```js
function foo(something) {
	console.log( this.a, something );
	return this.a + something;
}

var obj = {
	a: 2
};

var bar = function() {
	return foo.apply( obj, arguments );
};

var b = bar( 3 ); // 2 3
console.log( b ); // 5
```

Otra forma de expresar este patrón es crear un helper reutilizable:

```js
function foo(something) {
	console.log( this.a, something );
	return this.a + something;
}

// helper simple de `bind`
function bind(fn, obj) {
	return function() {
		return fn.apply( obj, arguments );
	};
}

var obj = {
	a: 2
};

var bar = bind( foo, obj );

var b = bar( 3 ); // 2 3
console.log( b ); // 5
```

Como el *enlace duro* es un patrón tan común, se proporciona con una utilidad integrada a partir de ES5: `Function.prototype.bind`, y se usa así:

```js
function foo(something) {
	console.log( this.a, something );
	return this.a + something;
}

var obj = {
	a: 2
};

var bar = foo.bind( obj );

var b = bar( 3 ); // 2 3
console.log( b ); // 5
```

`bind(..)` devuelve una nueva función que está codificada para llamar a la función original con el contexto `this` establecido como especificaste.

**Nota:** A partir de ES6, la función con enlace duro producida por `bind(..)` tiene una propiedad `.name` que se deriva de la *función objetivo* original. Por ejemplo: `bar = foo.bind(..)` debería tener un valor de `bar.name` de `"bound foo"`, que es el nombre de llamada de función que debería aparecer en un stack trace.

#### "Contextos" de Llamadas API

Muchas funciones de bibliotecas, y de hecho muchas nuevas funciones integradas en el lenguaje JavaScript y el entorno host, proporcionan un parámetro opcional, usualmente llamado "context", que está diseñado como una solución alternativa para que no tengas que usar `bind(..)` para asegurar que tu función callback use un `this` particular.

Por ejemplo:

```js
function foo(el) {
	console.log( el, this.id );
}

var obj = {
	id: "awesome"
};

// usa `obj` como `this` para las llamadas de `foo(..)`
[1, 2, 3].forEach( foo, obj ); // 1 awesome  2 awesome  3 awesome
```

Internamente, estas varias funciones casi con certeza usan *enlace explícito* vía `call(..)` o `apply(..)`, ahorrándote el trabajo.

### Enlace `new`

La cuarta y última regla para el enlace de `this` requiere que repensemos un concepto erróneo muy común sobre funciones y objetos en JavaScript.

En los lenguajes tradicionales orientados a clases, los "constructores" son métodos especiales adjuntos a clases, que cuando la clase es instanciada con un operador `new`, se llama al constructor de esa clase. Esto usualmente se ve algo como:

```js
something = new MyClass(..);
```

JavaScript tiene un operador `new`, y el patrón de código para usarlo se ve básicamente idéntico a lo que vemos en esos lenguajes orientados a clases; la mayoría de los desarrolladores asumen que el mecanismo de JavaScript está haciendo algo similar. Sin embargo, realmente *no hay conexión* con la funcionalidad orientada a clases implicada por el uso de `new` en JS.

Primero, redefinamos lo que es un "constructor" en JavaScript. En JS, los constructores son **simplemente funciones** que casualmente son llamadas con el operador `new` delante de ellas. No están adjuntas a clases, ni están instanciando una clase. Ni siquiera son tipos especiales de funciones. Son simplemente funciones regulares que son, en esencia, secuestradas por el uso de `new` en su invocación.

Por ejemplo, la función `Number(..)` actuando como constructor, citando de la especificación ES5.1:

> 15.7.2 El Constructor Number
>
> Cuando Number se llama como parte de una expresión new es un constructor: inicializa el objeto recién creado.

Así que, prácticamente cualquier función, incluyendo las funciones de objeto integradas como `Number(..)` (ver Capítulo 3) puede ser llamada con `new` delante, y eso convierte esa llamada de función en una *llamada de constructor*. Esta es una distinción importante pero sutil: realmente no existe tal cosa como "funciones constructoras", sino más bien llamadas de construcción *de* funciones.

Cuando una función se invoca con `new` delante de ella, también conocida como una llamada de constructor, las siguientes cosas se hacen automáticamente:

1. un objeto completamente nuevo se crea (es decir, se construye) de la nada
2. *el objeto recién construido se vincula por `[[Prototype]]`*
3. el objeto recién construido se establece como el enlace de `this` para esa llamada de función
4. a menos que la función devuelva su propio **objeto** alternativo, la llamada de función invocada con `new` *automáticamente* devolverá el objeto recién construido.

Los pasos 1, 3 y 4 aplican a nuestra discusión actual. Saltaremos el paso 2 por ahora y volveremos a él en el Capítulo 5.

Considera este código:

```js
function foo(a) {
	this.a = a;
}

var bar = new foo( 2 );
console.log( bar.a ); // 2
```

Al llamar a `foo(..)` con `new` delante, hemos construido un nuevo objeto y establecido ese nuevo objeto como el `this` para la llamada de `foo(..)`. **Así que `new` es la última forma en que el `this` de una llamada a función puede ser enlazado.** Llamaremos a esto *enlace new*.

## Todo En Orden

Así que, ahora hemos descubierto las 4 reglas para enlazar `this` en llamadas a funciones. *Todo* lo que necesitas hacer es encontrar el sitio de llamada e inspeccionar cuál regla aplica. Pero, ¿qué pasa si el sitio de llamada tiene múltiples reglas elegibles? Debe haber un orden de precedencia para estas reglas, y así demostraremos a continuación en qué orden aplicar las reglas.

Debería estar claro que el *enlace por defecto* es la regla de menor prioridad de las 4. Así que simplemente la dejaremos de lado.

¿Cuál tiene más precedencia, el *enlace implícito* o el *enlace explícito*? Probémoslo:

```js
function foo() {
	console.log( this.a );
}

var obj1 = {
	a: 2,
	foo: foo
};

var obj2 = {
	a: 3,
	foo: foo
};

obj1.foo(); // 2
obj2.foo(); // 3

obj1.foo.call( obj2 ); // 3
obj2.foo.call( obj1 ); // 2
```

Así que, el *enlace explícito* tiene precedencia sobre el *enlace implícito*, lo que significa que deberías preguntar **primero** si el *enlace explícito* aplica antes de verificar el *enlace implícito*.

Ahora, solo necesitamos averiguar dónde encaja el *enlace new* en la precedencia.

```js
function foo(something) {
	this.a = something;
}

var obj1 = {
	foo: foo
};

var obj2 = {};

obj1.foo( 2 );
console.log( obj1.a ); // 2

obj1.foo.call( obj2, 3 );
console.log( obj2.a ); // 3

var bar = new obj1.foo( 4 );
console.log( obj1.a ); // 2
console.log( bar.a ); // 4
```

OK, el *enlace new* tiene más precedencia que el *enlace implícito*. Pero ¿crees que el *enlace new* tiene más o menos precedencia que el *enlace explícito*?

**Nota:** `new` y `call`/`apply` no pueden usarse juntos, así que `new foo.call(obj1)` no está permitido, para probar el *enlace new* directamente contra el *enlace explícito*. Pero aún podemos usar un *enlace duro* para probar la precedencia de las dos reglas.

Antes de explorar eso en un listado de código, recuerda cómo funciona físicamente el *enlace duro*, que es que `Function.prototype.bind(..)` crea una nueva función envoltorio que está codificada para ignorar su propio enlace de `this` (sea lo que sea), y usar uno manual que proporcionamos.

Por ese razonamiento, parecería obvio asumir que el *enlace duro* (que es una forma de *enlace explícito*) tiene más precedencia que el *enlace new*, y por lo tanto no puede ser anulado con `new`.

Verifiquemos:

```js
function foo(something) {
	this.a = something;
}

var obj1 = {};

var bar = foo.bind( obj1 );
bar( 2 );
console.log( obj1.a ); // 2

var baz = new bar( 3 );
console.log( obj1.a ); // 2
console.log( baz.a ); // 3
```

¡Vaya! `bar` está enlazado duramente contra `obj1`, pero `new bar(3)` **no** cambió `obj1.a` a `3` como habríamos esperado. En su lugar, la llamada con *enlace duro* (a `obj1`) de `bar(..)` ***es*** capaz de ser anulada con `new`. Como se aplicó `new`, obtuvimos el objeto recién creado de vuelta, que nombramos `baz`, y vemos de hecho que `baz.a` tiene el valor `3`.

Esto debería ser sorprendente si vuelves a nuestro helper "falso" de bind:

```js
function bind(fn, obj) {
	return function() {
		fn.apply( obj, arguments );
	};
}
```

Si razonas sobre cómo funciona el código del helper, no tiene forma de que una llamada con el operador `new` anule el enlace duro a `obj` como acabamos de observar.

Pero el integrado `Function.prototype.bind(..)` a partir de ES5 es más sofisticado, bastante más de hecho. Aquí está el polyfill (ligeramente reformateado) proporcionado por la página MDN para `bind(..)`:

```js
if (!Function.prototype.bind) {
	Function.prototype.bind = function(oThis) {
		if (typeof this !== "function") {
			// lo más cercano posible a la función interna
			// IsCallable de ECMAScript 5
			throw new TypeError( "Function.prototype.bind - what " +
				"is trying to be bound is not callable"
			);
		}

		var aArgs = Array.prototype.slice.call( arguments, 1 ),
			fToBind = this,
			fNOP = function(){},
			fBound = function(){
				return fToBind.apply(
					(
						this instanceof fNOP &&
						oThis ? this : oThis
					),
					aArgs.concat( Array.prototype.slice.call( arguments ) )
				);
			}
		;

		fNOP.prototype = this.prototype;
		fBound.prototype = new fNOP();

		return fBound;
	};
}
```

**Nota:** El polyfill de `bind(..)` mostrado anteriormente difiere del `bind(..)` integrado en ES5 con respecto a funciones con enlace duro que se usarán con `new` (ver más abajo para saber por qué eso es útil). Debido a que el polyfill no puede crear una función sin un `.prototype` como la utilidad integrada lo hace, hay alguna indirección matizada para aproximar el mismo comportamiento. Procede con cuidado si planeas usar `new` con una función con enlace duro y dependes de este polyfill.

La parte que permite la anulación con `new` es:

```js
this instanceof fNOP &&
oThis ? this : oThis

// ... y:

fNOP.prototype = this.prototype;
fBound.prototype = new fNOP();
```

Realmente no vamos a sumergirnos en explicar cómo funciona este truco (es complicado y está más allá de nuestro alcance aquí), pero esencialmente la utilidad determina si la función con enlace duro ha sido llamada con `new` (resultando en un objeto recién construido siendo su `this`), y si es así, usa *ese* `this` recién creado en lugar del *enlace duro* previamente especificado para `this`.

¿Por qué es útil que `new` pueda anular el *enlace duro*?

La razón principal para este comportamiento es crear una función (que puede usarse con `new` para construir objetos) que esencialmente ignora el *enlace duro* de `this` pero que preestablece algunos o todos los argumentos de la función. Una de las capacidades de `bind(..)` es que cualquier argumento pasado después del primer argumento de enlace de `this` se establece como argumentos estándar para la función subyacente (técnicamente llamado "aplicación parcial", que es un subconjunto de "currying").

Por ejemplo:

```js
function foo(p1,p2) {
	this.val = p1 + p2;
}

// usando `null` aquí porque no nos importa
// el enlace duro de `this` en este escenario, y
// ¡será anulado por la llamada `new` de todos modos!
var bar = foo.bind( null, "p1" );

var baz = new bar( "p2" );

baz.val; // p1p2
```

### Determinando `this`

Ahora, podemos resumir las reglas para determinar `this` desde el sitio de llamada de una función, en su orden de precedencia. Haz estas preguntas en este orden, y detente cuando la primera regla aplique.

1. ¿Se llama a la función con `new` (**enlace new**)? Si es así, `this` es el objeto recién construido.

    `var bar = new foo()`

2. ¿Se llama a la función con `call` o `apply` (**enlace explícito**), incluso oculto dentro de un *enlace duro* `bind`? Si es así, `this` es el objeto explícitamente especificado.

    `var bar = foo.call( obj2 )`

3. ¿Se llama a la función con un contexto (**enlace implícito**), también conocido como un objeto propietario o contenedor? Si es así, `this` es *ese* objeto de contexto.

    `var bar = obj1.foo()`

4. De lo contrario, usar el `this` por defecto (**enlace por defecto**). Si está en `strict mode`, elegir `undefined`, de lo contrario elegir el objeto `global`.

    `var bar = foo()`

Eso es todo. Eso es *todo lo que se necesita* para entender las reglas del enlace de `this` para llamadas a funciones normales. Bueno... casi.

## Excepciones de Enlace

Como es usual, hay algunas *excepciones* a las "reglas".

El comportamiento del enlace de `this` puede en algunos escenarios ser sorprendente, donde pretendías un enlace diferente pero terminas con el comportamiento de enlace de la regla de *enlace por defecto* (ver anterior).

### `this` Ignorado

Si pasas `null` o `undefined` como parámetro de enlace de `this` a `call`, `apply`, o `bind`, esos valores son efectivamente ignorados, y en su lugar la regla de *enlace por defecto* aplica a la invocación.

```js
function foo() {
	console.log( this.a );
}

var a = 2;

foo.call( null ); // 2
```

¿Por qué pasarías intencionalmente algo como `null` para un enlace de `this`?

Es bastante común usar `apply(..)` para expandir arrays de valores como parámetros a una llamada de función. Similarmente, `bind(..)` puede hacer currying de parámetros (pre-establecer valores), lo cual puede ser muy útil.

```js
function foo(a,b) {
	console.log( "a:" + a + ", b:" + b );
}

// expandiendo array como parámetros
foo.apply( null, [2, 3] ); // a:2, b:3

// currying con `bind(..)`
var bar = foo.bind( null, 2 );
bar( 3 ); // a:2, b:3
```

Ambas utilidades requieren un enlace de `this` para el primer parámetro. Si las funciones en cuestión no se preocupan por `this`, necesitas un valor marcador de posición, y `null` podría parecer una elección razonable como se muestra en este fragmento.

**Nota:** No lo cubrimos en este libro, pero ES6 tiene el operador spread `...` que te permitirá sintácticamente "expandir" un array como parámetros sin necesitar `apply(..)`, como `foo(...[1,2])`, que equivale a `foo(1,2)` -- evitando sintácticamente un enlace de `this` si es innecesario. Desafortunadamente, no hay sustituto sintáctico en ES6 para currying, así que el parámetro `this` de la llamada `bind(..)` aún necesita atención.

Sin embargo, hay un ligero "peligro" oculto en siempre usar `null` cuando no te importa el enlace de `this`. Si alguna vez usas eso contra una llamada de función (por ejemplo, una función de biblioteca de terceros que no controlas), y esa función *sí* hace una referencia a `this`, la regla de *enlace por defecto* significa que podría inadvertidamente referenciar (¡o peor, mutar!) el objeto `global` (`window` en el navegador).

Obviamente, tal trampa puede llevar a una variedad de bugs *muy difíciles* de diagnosticar/rastrear.

#### `this` Más Seguro

Quizás una práctica algo "más segura" es pasar un objeto específicamente configurado para `this` que esté garantizado que no sea un objeto que pueda crear efectos secundarios problemáticos en tu programa. Tomando prestada la terminología de redes (y lo militar), podemos crear un objeto "DMZ" (zona desmilitarizada) -- nada más especial que un objeto completamente vacío, sin delegación (ver Capítulos 5 y 6).

Si siempre pasamos un objeto DMZ para enlaces de `this` ignorados que no creemos necesitar, estamos seguros de que cualquier uso oculto/inesperado de `this` estará restringido al objeto vacío, que aísla el objeto `global` de nuestro programa de efectos secundarios.

Como este objeto está totalmente vacío, personalmente me gusta darle el nombre de variable `ø` (el símbolo matemático en minúscula para el conjunto vacío). En muchos teclados (como el de distribución US en Mac), este símbolo se escribe fácilmente con `⌥`+`o` (option+`o`). Algunos sistemas también te permiten configurar teclas rápidas para símbolos específicos. Si no te gusta el símbolo `ø`, o tu teclado no hace tan fácil escribirlo, por supuesto puedes llamarlo como quieras.

Como sea que lo llames, la forma más fácil de configurarlo como **totalmente vacío** es `Object.create(null)` (ver Capítulo 5). `Object.create(null)` es similar a `{ }`, pero sin la delegación a `Object.prototype`, así que es "más vacío" que solo `{ }`.

```js
function foo(a,b) {
	console.log( "a:" + a + ", b:" + b );
}

// nuestro objeto DMZ vacío
var ø = Object.create( null );

// expandiendo array como parámetros
foo.apply( ø, [2, 3] ); // a:2, b:3

// currying con `bind(..)`
var bar = foo.bind( ø, 2 );
bar( 3 ); // a:2, b:3
```

No solo es funcionalmente "más seguro", hay una especie de beneficio estilístico con `ø`, en que semánticamente transmite "quiero que el `this` esté vacío" un poco más claramente de lo que `null` podría. Pero de nuevo, nombra tu objeto DMZ como prefieras.

### Indirección

Otra cosa a tener en cuenta es que puedes (¡intencional o no!) crear "referencias indirectas" a funciones, y en esos casos, cuando esa referencia de función es invocada, la regla de *enlace por defecto* también aplica.

Una de las formas más comunes en que ocurren las *referencias indirectas* es desde una asignación:

```js
function foo() {
	console.log( this.a );
}

var a = 2;
var o = { a: 3, foo: foo };
var p = { a: 4 };

o.foo(); // 3
(p.foo = o.foo)(); // 2
```

El *valor resultado* de la expresión de asignación `p.foo = o.foo` es una referencia solo al objeto función subyacente. Como tal, el sitio de llamada efectivo es simplemente `foo()`, no `p.foo()` o `o.foo()` como podrías esperar. Según las reglas anteriores, la regla de *enlace por defecto* aplica.

Recordatorio: sin importar cómo llegues a una invocación de función usando la regla de *enlace por defecto*, el estado de `strict mode` de los **contenidos** de la función invocada que hace la referencia a `this` -- no el sitio de llamada de la función -- determina el valor del *enlace por defecto*: ya sea el objeto `global` si está en modo no-`strict` o `undefined` si está en modo `strict`.

### Suavizando el Enlace

Vimos antes que el *enlace duro* era una estrategia para prevenir que una llamada a función recurriera inadvertidamente a la regla de *enlace por defecto*, forzándolo a estar enlazado a un `this` específico (¡a menos que uses `new` para anularlo!). El problema es, el *enlace duro* reduce enormemente la flexibilidad de una función, previniendo la anulación manual de `this` con el *enlace implícito* o incluso intentos subsecuentes de *enlace explícito*.

Sería bueno si hubiera una forma de proporcionar un valor por defecto diferente para el *enlace por defecto* (ni `global` ni `undefined`), mientras se deja la función capaz de ser enlazada manualmente a `this` vía técnicas de *enlace implícito* o *enlace explícito*.

Podemos construir una utilidad llamada *enlace suave* que emula nuestro comportamiento deseado.

```js
if (!Function.prototype.softBind) {
	Function.prototype.softBind = function(obj) {
		var fn = this,
			curried = [].slice.call( arguments, 1 ),
			bound = function bound() {
				return fn.apply(
					(!this ||
						(typeof window !== "undefined" &&
							this === window) ||
						(typeof global !== "undefined" &&
							this === global)
					) ? obj : this,
					curried.concat.apply( curried, arguments )
				);
			};
		bound.prototype = Object.create( fn.prototype );
		return bound;
	};
}
```

La utilidad `softBind(..)` proporcionada aquí funciona de manera similar a la utilidad integrada `bind(..)` de ES5, excepto con nuestro comportamiento de *enlace suave*. Envuelve la función especificada en lógica que verifica el `this` en el momento de la llamada y si es `global` o `undefined`, usa un *valor por defecto* alternativo pre-especificado (`obj`). De lo contrario, el `this` se deja intacto. También proporciona currying opcional (ver la discusión de `bind(..)` anterior).

Demostremos su uso:

```js
function foo() {
   console.log("name: " + this.name);
}

var obj = { name: "obj" },
    obj2 = { name: "obj2" },
    obj3 = { name: "obj3" };

var fooOBJ = foo.softBind( obj );

fooOBJ(); // name: obj

obj2.foo = foo.softBind(obj);
obj2.foo(); // name: obj2   <---- ¡mira!

fooOBJ.call( obj3 ); // name: obj3   <---- ¡mira!

setTimeout( obj2.foo, 10 ); // name: obj   <---- recurre al enlace suave
```

La versión con enlace suave de la función `foo()` puede ser enlazada manualmente a `this` a `obj2` o `obj3` como se muestra, pero recurre a `obj` si el *enlace por defecto* se aplicaría de lo contrario.
