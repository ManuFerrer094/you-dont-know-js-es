## Introspección

Si has pasado mucho tiempo con programación orientada a clases (ya sea en JS u otros lenguajes), probablemente estás familiarizado con la *introspección de tipo*: inspeccionar una instancia para averiguar qué *tipo* de objeto es. El objetivo principal de la *introspección de tipo* con instancias de clase es razonar sobre la estructura/capacidades del objeto basándose en *cómo fue creado*.

Considera este código que usa `instanceof` (ver Capítulo 5) para hacer introspección en un objeto `a1` para inferir su capacidad:

```js
function Foo() {
	// ...
}
Foo.prototype.something = function(){
	// ...
}

var a1 = new Foo();

// later

if (a1 instanceof Foo) {
	a1.something();
}
```

Porque `Foo.prototype` (¡no `Foo`!) está en la cadena `[[Prototype]]` (ver Capítulo 5) de `a1`, el operador `instanceof` (confusamente) pretende decirnos que `a1` es una instancia de la "clase" `Foo`. Con este conocimiento, luego asumimos que `a1` tiene las capacidades descritas por la "clase" `Foo`.

Por supuesto, no hay clase `Foo`, solo una función simple, vieja y normal `Foo`, que resulta tener una referencia a un objeto arbitrario (`Foo.prototype`) al que `a1` resulta estar vinculado por delegación. Por su sintaxis, `instanceof` pretende estar inspeccionando la relación entre `a1` y `Foo`, pero realmente nos está diciendo si `a1` y (el objeto arbitrario referenciado por) `Foo.prototype` están relacionados.

La confusión semántica (e indirección) de la sintaxis de `instanceof` significa que para usar introspección basada en `instanceof` para preguntar si el objeto `a1` está relacionado con el objeto de capacidades en cuestión, *tienes que* tener una función que mantenga una referencia a ese objeto -- no puedes simplemente preguntar directamente si los dos objetos están relacionados.

Recuerda el ejemplo abstracto `Foo` / `Bar` / `b1` de antes en este capítulo, que abreviaremos aquí:

```js
function Foo() { /* .. */ }
Foo.prototype...

function Bar() { /* .. */ }
Bar.prototype = Object.create( Foo.prototype );

var b1 = new Bar( "b1" );
```

Para propósitos de *introspección de tipo* en las entidades de ese ejemplo, usando semánticas de `instanceof` y `.prototype`, aquí están las diversas verificaciones que podrías necesitar realizar:

```js
// relating `Foo` and `Bar` to each other
Bar.prototype instanceof Foo; // true
Object.getPrototypeOf( Bar.prototype ) === Foo.prototype; // true
Foo.prototype.isPrototypeOf( Bar.prototype ); // true

// relating `b1` to both `Foo` and `Bar`
b1 instanceof Foo; // true
b1 instanceof Bar; // true
Object.getPrototypeOf( b1 ) === Bar.prototype; // true
Foo.prototype.isPrototypeOf( b1 ); // true
Bar.prototype.isPrototypeOf( b1 ); // true
```

Es justo decir que algo de eso como que apesta. Por ejemplo, intuitivamente (con clases) quizás querrías poder decir algo como `Bar instanceof Foo` (porque es fácil confundir lo que "instancia" significa y pensar que incluye "herencia"), pero esa no es una comparación sensata en JS. Tienes que hacer `Bar.prototype instanceof Foo` en su lugar.

Otro patrón común, pero quizás menos robusto, para la *introspección de tipo*, que muchos desarrolladores parecen preferir sobre `instanceof`, se llama "duck typing". Este término viene del dicho, "si parece un pato, y hace cuac como un pato, debe ser un pato".

Ejemplo:

```js
if (a1.something) {
	a1.something();
}
```

En lugar de inspeccionar una relación entre `a1` y un objeto que contiene la función delegable `something()`, asumimos que la prueba de `a1.something` pasando significa que `a1` tiene la capacidad de llamar `.something()` (sin importar si encontró el método directamente en `a1` o delegó a algún otro objeto). En sí misma, esa suposición no es tan riesgosa.

Pero el "duck typing" a menudo se extiende para hacer **otras suposiciones sobre las capacidades del objeto** más allá de lo que se está probando, lo que por supuesto introduce más riesgo (también conocido como diseño frágil) en la prueba.

Un ejemplo notable de "duck typing" viene con las Promises de ES6 (que como una nota anterior explicó no se cubren en este libro).

Por varias razones, hay una necesidad de determinar si cualquier referencia de objeto arbitraria *es una Promise*, pero la forma en que se hace esa prueba es verificar si el objeto resulta tener un método `then()` presente en él. En otras palabras, **si cualquier objeto** resulta tener un método `then()`, las Promises de ES6 asumirán incondicionalmente que el objeto **es un "thenable"** y por lo tanto esperarán que se comporte conformemente a todos los comportamientos estándar de Promises.

Si tienes cualquier objeto que no es una Promise que resulta por cualquier razón tener un método `then()` en él, se te aconseja fuertemente mantenerlo lejos del mecanismo de Promises de ES6 para evitar suposiciones rotas.

Ese ejemplo ilustra claramente los peligros del "duck typing". Solo deberías usar tales enfoques con moderación y en condiciones controladas.

Volviendo nuestra atención una vez más de vuelta al código estilo OLOO como se presenta aquí en este capítulo, la *introspección de tipo* resulta ser mucho más limpia. Recordemos (y abreviemos) el ejemplo OLOO de `Foo` / `Bar` / `b1` de antes en el capítulo:

```js
var Foo = { /* .. */ };

var Bar = Object.create( Foo );
Bar...

var b1 = Object.create( Bar );
```

Usando este enfoque OLOO, donde todo lo que tenemos son objetos simples relacionados vía delegación `[[Prototype]]`, aquí está la *introspección de tipo* bastante simplificada que podríamos usar:

```js
// relating `Foo` and `Bar` to each other
Foo.isPrototypeOf( Bar ); // true
Object.getPrototypeOf( Bar ) === Foo; // true

// relating `b1` to both `Foo` and `Bar`
Foo.isPrototypeOf( b1 ); // true
Bar.isPrototypeOf( b1 ); // true
Object.getPrototypeOf( b1 ) === Bar; // true
```

No estamos usando `instanceof` más, porque confusamente pretende tener algo que ver con clases. Ahora, solo hacemos la pregunta (informalmente planteada), "¿eres *un* prototipo mío?" No hay más indirección necesaria con cosas como `Foo.prototype` o el dolorosamente verboso `Foo.prototype.isPrototypeOf(..)`.

Creo que es justo decir que estas verificaciones son significativamente menos complicadas/confusas que el conjunto anterior de verificaciones de introspección. **Una vez más, vemos que OLOO es más simple que (pero con todo el mismo poder de) la codificación estilo clase en JavaScript.**

## Revisión (TL;DR)

Las clases y la herencia son un patrón de diseño que puedes *elegir*, o *no elegir*, en tu arquitectura de software. La mayoría de los desarrolladores dan por sentado que las clases son la única forma (correcta) de organizar código, pero aquí hemos visto que hay otro patrón menos comúnmente hablado que es realmente bastante poderoso: **la delegación de comportamiento**.

La delegación de comportamiento sugiere objetos como pares entre sí, que delegan entre ellos, en lugar de relaciones de clase padre e hijo. El mecanismo `[[Prototype]]` de JavaScript es, por su misma naturaleza de diseño, un mecanismo de delegación de comportamiento. Eso significa que podemos elegir luchar para implementar mecánicas de clases encima de JS (ver Capítulos 4 y 5), o podemos simplemente abrazar el estado natural de `[[Prototype]]` como un mecanismo de delegación.

Cuando diseñas código con solo objetos, no solo simplifica la sintaxis que usas, sino que realmente puede llevar a un diseño de arquitectura de código más simple.

**OLOO** (objects-linked-to-other-objects, objetos-vinculados-a-otros-objetos) es un estilo de código que crea y relaciona objetos directamente sin la abstracción de clases. OLOO implementa bastante naturalmente la delegación de comportamiento basada en `[[Prototype]]`.
