# Capítulo 6: Delegación de Comportamiento

En el Capítulo 5, abordamos el mecanismo `[[Prototype]]` en detalle, y *por qué* es confuso e inapropiado (a pesar de innumerables intentos por casi dos décadas) describirlo como "clase" o "herencia". Recorrimos trabajosamente no solo la sintaxis bastante verbosa (`.prototype` ensuciando el código), sino las diversas trampas (como la sorprendente resolución de `.constructor` o la fea sintaxis de pseudo-polimorfismo explícito). Exploramos variaciones del enfoque "mixin", que muchas personas usan para intentar suavizar tales áreas ásperas.

Es una reacción común en este punto preguntarse por qué tiene que ser tan complejo hacer algo aparentemente tan simple. Ahora que hemos descorrido la cortina y visto cuán sucio se pone todo, no es una sorpresa que la mayoría de los desarrolladores JS nunca profundicen tanto, y en su lugar releguen tal desorden a una librería de "clases" para que lo maneje por ellos.

Espero que a estas alturas no estés contento con simplemente pasar por alto y dejar tales detalles a una librería "caja negra". Ahora profundicemos en cómo *podríamos y deberíamos* estar pensando sobre el mecanismo `[[Prototype]]` de objetos en JS, de una **forma mucho más simple y directa** que la confusión de las clases.

Como breve repaso de nuestras conclusiones del Capítulo 5, el mecanismo `[[Prototype]]` es un enlace interno que existe en un objeto que referencia otro objeto.

Este enlace se ejercita cuando una referencia de propiedad/método se hace contra el primer objeto, y no existe tal propiedad/método. En ese caso, el enlace `[[Prototype]]` le dice al motor que busque la propiedad/método en el objeto vinculado. A su vez, si ese objeto no puede cumplir la búsqueda, su `[[Prototype]]` se sigue, y así sucesivamente. Esta serie de enlaces entre objetos forma lo que se llama la "cadena de prototipos".

En otras palabras, el mecanismo real, la esencia de lo que es importante para la funcionalidad que podemos aprovechar en JavaScript, **tiene que ver completamente con objetos vinculados a otros objetos.**

Esa observación singular es fundamental y crítica para entender las motivaciones y enfoques para el resto de este capítulo.

## Hacia el Diseño Orientado a Delegación

Para enfocar adecuadamente nuestros pensamientos en cómo usar `[[Prototype]]` de la forma más directa, debemos reconocer que representa un patrón de diseño fundamentalmente diferente de las clases (ver Capítulo 4).

**Nota:** *Algunos* principios del diseño orientado a clases siguen siendo muy válidos, así que no deseches todo lo que sabes (¡solo la mayoría!). Por ejemplo, la *encapsulación* es bastante poderosa, y es compatible (aunque no tan común) con la delegación.

Necesitamos intentar cambiar nuestro pensamiento del patrón de diseño de clase/herencia al patrón de diseño de delegación de comportamiento. Si has hecho la mayor parte o toda tu programación en tu educación/carrera pensando en clases, esto puede ser incómodo o sentirse antinatural. Puede que necesites intentar este ejercicio mental bastantes veces para dominar esta forma tan diferente de pensar.

Voy a guiarte a través de algunos ejercicios teóricos primero, luego veremos lado a lado un ejemplo más concreto para darte contexto práctico para tu propio código.

### Teoría de Clases

Digamos que tenemos varias tareas similares ("XYZ", "ABC", etc.) que necesitamos modelar en nuestro software.

Con clases, la forma en que diseñas el escenario es: definir una clase padre general (base) como `Task`, definiendo comportamiento compartido para todas las tareas "parecidas". Luego, defines clases hijas `XYZ` y `ABC`, ambas heredando de `Task`, y cada una agregando comportamiento especializado para manejar sus tareas respectivas.

**Importante,** el patrón de diseño de clases te animará a que para obtener lo máximo de la herencia, querrás emplear sobreescritura de métodos (y polimorfismo), donde sobreescribes la definición de algún método general de `Task` en tu tarea `XYZ`, quizás incluso haciendo uso de `super` para llamar a la versión base de ese método mientras le agregas más comportamiento. **Probablemente encontrarás bastantes lugares** donde puedes "abstraer" comportamiento general a la clase padre y especializarlo (sobreescribirlo) en tus clases hijas.

Aquí hay algo de pseudocódigo suelto para ese escenario:

```js
class Task {
	id;

	// constructor `Task()`
	Task(ID) { id = ID; }
	outputTask() { output( id ); }
}

class XYZ inherits Task {
	label;

	// constructor `XYZ()`
	XYZ(ID,Label) { super( ID ); label = Label; }
	outputTask() { super(); output( label ); }
}

class ABC inherits Task {
	// ...
}
```

Ahora, puedes instanciar una o más **copias** de la clase hija `XYZ`, y usar esa(s) instancia(s) para realizar la tarea "XYZ". Estas instancias tienen **copias tanto** del comportamiento general definido en `Task` como del comportamiento específico definido en `XYZ`. De igual forma, las instancias de la clase `ABC` tendrían copias del comportamiento de `Task` y el comportamiento específico de `ABC`. Después de la construcción, generalmente solo interactúas con estas instancias (y no con las clases), ya que las instancias cada una tienen copias de todo el comportamiento que necesitas para hacer la tarea prevista.

### Teoría de Delegación

Pero ahora intentemos pensar sobre el mismo dominio de problema, pero usando *delegación de comportamiento* en lugar de *clases*.

Primero definirás un **objeto** (no una clase, ni una `function` como la mayoría de los desarrolladores JS te harían creer) llamado `Task`, y tendrá comportamiento concreto en él que incluye métodos utilitarios que varias tareas pueden usar (léase: ¡*delegar a*!). Luego, para cada tarea ("XYZ", "ABC"), defines un **objeto** para contener los datos/comportamiento específicos de esa tarea. **Vinculas** tu(s) objeto(s) específicos de tarea al objeto utilitario `Task`, permitiéndoles delegar a él cuando necesiten.

Básicamente, piensas en realizar la tarea "XYZ" como necesitando comportamientos de dos objetos hermanos/pares (`XYZ` y `Task`) para lograrla. Pero en lugar de necesitar componerlos juntos, vía copias de clase, podemos mantenerlos en sus objetos separados, y podemos permitir que el objeto `XYZ` **delegue a** `Task` cuando sea necesario.

Aquí hay algo de código simple para sugerir cómo logras eso:

```js
var Task = {
	setID: function(ID) { this.id = ID; },
	outputID: function() { console.log( this.id ); }
};

// make `XYZ` delegate to `Task`
var XYZ = Object.create( Task );

XYZ.prepareTask = function(ID,Label) {
	this.setID( ID );
	this.label = Label;
};

XYZ.outputTaskDetails = function() {
	this.outputID();
	console.log( this.label );
};

// ABC = Object.create( Task );
// ABC ... = ...
```

En este código, `Task` y `XYZ` no son clases (o funciones), son **solo objetos**. `XYZ` se establece vía `Object.create(..)` para `[[Prototype]]` delegar al objeto `Task` (ver Capítulo 5).

Comparado con la orientación a clases (también conocida como OO -- orientado a objetos), llamo a este estilo de código **"OLOO"** (objects-linked-to-other-objects, objetos-vinculados-a-otros-objetos). Todo lo que *realmente* nos importa es que el objeto `XYZ` delega al objeto `Task` (al igual que el objeto `ABC`).

En JavaScript, el mecanismo `[[Prototype]]` vincula **objetos** a otros **objetos**. No hay mecanismos abstractos como "clases", no importa cuánto intentes convencerte de lo contrario. Es como remar una canoa contra la corriente: *puedes* hacerlo, pero estás *eligiendo* ir contra la corriente natural, así que obviamente **va a ser más difícil llegar a donde vas.**

Algunas otras diferencias a notar con el **código estilo OLOO**:

1. Tanto los miembros de datos `id` como `label` del ejemplo de clase anterior son propiedades de datos directamente en `XYZ` (ninguno está en `Task`). En general, con delegación `[[Prototype]]` involucrada, **quieres que el estado esté en los delegadores** (`XYZ`, `ABC`), no en el delegado (`Task`).
2. Con el patrón de diseño de clases, intencionalmente nombramos `outputTask` igual tanto en el padre (`Task`) como en el hijo (`XYZ`), para poder aprovechar la sobreescritura (polimorfismo). En delegación de comportamiento, hacemos lo opuesto: **evitamos si es posible nombrar cosas igual** en diferentes niveles de la cadena `[[Prototype]]` (llamado ensombrecimiento -- ver Capítulo 5), porque tener esas colisiones de nombres crea sintaxis incómoda/frágil para desambiguar referencias (ver Capítulo 4), y queremos evitar eso si podemos.

   Este patrón de diseño llama a menos de nombres de métodos generales que son propensos a sobreescritura y en su lugar más nombres de métodos descriptivos, *específicos* al tipo de comportamiento que cada objeto está haciendo. **Esto realmente puede crear código más fácil de entender/mantener**, porque los nombres de los métodos (no solo en la ubicación de definición sino esparcidos por otro código) son más obvios (auto-documentados).
3. `this.setID(ID);` dentro de un método en el objeto `XYZ` primero busca en `XYZ` por `setID(..)`, pero como no encuentra un método de ese nombre en `XYZ`, la *delegación* `[[Prototype]]` significa que puede seguir el enlace a `Task` para buscar `setID(..)`, que por supuesto encuentra. Además, por las reglas implícitas de enlace de `this` del call-site (ver Capítulo 2), cuando `setID(..)` se ejecuta, aunque el método fue encontrado en `Task`, el enlace de `this` para esa llamada de función es `XYZ` exactamente como esperaríamos y querríamos. Vemos lo mismo con `this.outputID()` más adelante en la lista de código.

   En otras palabras, los métodos utilitarios generales que existen en `Task` están disponibles para nosotros mientras interactuamos con `XYZ`, porque `XYZ` puede delegar a `Task`.

**Delegación de Comportamiento** significa: dejar que algún objeto (`XYZ`) proporcione una delegación (a `Task`) para referencias de propiedad o método si no se encuentran en el objeto (`XYZ`).

Este es un patrón de diseño *extremadamente poderoso*, muy distinto de la idea de clases padre e hijo, herencia, polimorfismo, etc. En lugar de organizar los objetos en tu mente verticalmente, con Padres fluyendo hacia abajo a Hijos, piensa en objetos lado a lado, como pares, con cualquier dirección de enlaces de delegación entre los objetos según sea necesario.

**Nota:** La delegación se usa más apropiadamente como un detalle de implementación interna en lugar de estar expuesta directamente en el diseño de la API. En el ejemplo anterior, no necesariamente *pretendemos* con nuestro diseño de API que los desarrolladores llamen `XYZ.setID()` (¡aunque podemos, por supuesto!). Más o menos *ocultamos* la delegación como un detalle interno de nuestra API, donde `XYZ.prepareTask(..)` delega a `Task.setID(..)`. Ver la discusión "¿Enlaces como Respaldos?" en el Capítulo 5 para más detalle.

#### Delegación Mutua (No Permitida)

No puedes crear un *ciclo* donde dos o más objetos estén mutuamente delegados (bidireccionalmente) entre sí. Si haces que `B` se vincule a `A`, y luego intentas vincular `A` a `B`, obtendrás un error.

Es una pena (no terriblemente sorprendente, pero levemente molesto) que esto no esté permitido. Si hicieras una referencia a una propiedad/método que no existiera en ninguno de los dos lugares, tendrías una recursión infinita en el bucle `[[Prototype]]`. Pero si todas las referencias estuvieran estrictamente presentes, entonces `B` podría delegar a `A`, y viceversa, y *podría* funcionar. Esto significaría que podrías usar cualquiera de los objetos para delegar al otro, para varias tareas. Hay algunos casos de uso de nicho donde esto podría ser útil.

Pero no está permitido porque los implementadores de motores han observado que es más eficiente verificar (¡y rechazar!) la referencia circular infinita una vez al momento de establecerla en lugar de necesitar tener el impacto de rendimiento de esa verificación de guardia cada vez que buscas una propiedad en un objeto.

#### Depuración

Cubriremos brevemente un detalle sutil que puede ser confuso para los desarrolladores. En general, la especificación JS no controla cómo las herramientas de desarrollo de los navegadores deberían representar valores/estructuras específicos a un desarrollador, así que cada navegador/motor es libre de interpretar tales cosas como vea conveniente. Como tal, los navegadores/herramientas *no siempre están de acuerdo*. Específicamente, el comportamiento que ahora examinaremos se observa actualmente solo en las Herramientas de Desarrollo de Chrome.

Considera este código JS tradicional de estilo "constructor de clase", como aparecería en la *consola* de las Herramientas de Desarrollo de Chrome:

```js
function Foo() {}

var a1 = new Foo();

a1; // Foo {}
```

Veamos la última línea de ese fragmento: la salida de evaluar la expresión `a1`, que imprime `Foo {}`. Si intentas este mismo código en Firefox, probablemente verás `Object {}`. ¿Por qué la diferencia? ¿Qué significan estas salidas?

Chrome esencialmente está diciendo "{} es un objeto vacío que fue construido por una función con nombre 'Foo'". Firefox está diciendo "{} es un objeto vacío de construcción general desde Object". La diferencia sutil es que Chrome está rastreando activamente, como una propiedad *interna*, el nombre de la función real que hizo la construcción, mientras que otros navegadores no rastrean esa información adicional.

Sería tentador intentar explicar esto con mecanismos de JavaScript:

```js
function Foo() {}

var a1 = new Foo();

a1.constructor; // Foo(){}
a1.constructor.name; // "Foo"
```

Entonces, ¿es así como Chrome está mostrando "Foo", simplemente examinando el `.constructor.name` del objeto? Confusamente, la respuesta es tanto "sí" como "no".

Considera este código:

```js
function Foo() {}

var a1 = new Foo();

Foo.prototype.constructor = function Gotcha(){};

a1.constructor; // Gotcha(){}
a1.constructor.name; // "Gotcha"

a1; // Foo {}
```

Aunque cambiamos `a1.constructor.name` para que legítimamente sea algo diferente ("Gotcha"), la consola de Chrome aún usa el nombre "Foo".

Entonces, parecería que la respuesta a la pregunta anterior (¿usa `.constructor.name`?) es **no**, debe rastrearlo en algún otro lugar, internamente.

Pero, ¡no tan rápido! Veamos cómo este tipo de comportamiento funciona con código estilo OLOO:

```js
var Foo = {};

var a1 = Object.create( Foo );

a1; // Object {}

Object.defineProperty( Foo, "constructor", {
	enumerable: false,
	value: function Gotcha(){}
});

a1; // Gotcha {}
```

¡Ajá! **¡Gotcha!** Aquí, la consola de Chrome **sí** encontró y usó `.constructor.name`. En realidad, mientras escribía este libro, este comportamiento exacto fue identificado como un bug en Chrome, y para cuando estés leyendo esto, puede que ya haya sido corregido. Así que podrías haber visto en su lugar el corregido `a1; // Object {}`.

Aparte de ese bug, el rastreo interno (aparentemente solo para propósitos de salida de depuración) del "nombre del constructor" que Chrome hace (mostrado en los fragmentos anteriores) es una extensión de comportamiento intencional solo de Chrome más allá de lo que la especificación JS pide.

Si no usas un "constructor" para hacer tus objetos, como hemos desalentado con el código estilo OLOO aquí en este capítulo, entonces obtendrás objetos que Chrome *no* rastrea un "nombre de constructor" interno, y tales objetos serán correctamente mostrados solo como "Object {}", significando "objeto generado desde la construcción de Object()".

**No pienses** que esto representa una desventaja del estilo de codificación OLOO. Cuando codificas con OLOO y delegación de comportamiento como tu patrón de diseño, *quién* "construyó" (es decir, *qué función* fue llamada con `new`?) algún objeto es un detalle irrelevante. El rastreo interno específico de Chrome del "nombre del constructor" solo es realmente útil si estás abrazando completamente la codificación "estilo clase", pero es irrelevante si estás abrazando la delegación OLOO.

### Modelos Mentales Comparados

Ahora que puedes ver una diferencia entre los patrones de diseño de "clase" y "delegación", al menos teóricamente, veamos las implicaciones que estos patrones de diseño tienen en los modelos mentales que usamos para razonar sobre nuestro código.

Examinaremos algo más de código teórico ("Foo", "Bar"), y compararemos ambas formas (OO vs. OLOO) de implementar el código. El primer fragmento usa el estilo OO clásico ("prototípico"):

```js
function Foo(who) {
	this.me = who;
}
Foo.prototype.identify = function() {
	return "I am " + this.me;
};

function Bar(who) {
	Foo.call( this, who );
}
Bar.prototype = Object.create( Foo.prototype );

Bar.prototype.speak = function() {
	alert( "Hello, " + this.identify() + "." );
};

var b1 = new Bar( "b1" );
var b2 = new Bar( "b2" );

b1.speak();
b2.speak();
```

Clase padre `Foo`, heredada por la clase hija `Bar`, que luego es instanciada dos veces como `b1` y `b2`. Lo que tenemos es `b1` delegando a `Bar.prototype` que delega a `Foo.prototype`. Esto debería lucir bastante familiar a estas alturas. Nada demasiado revolucionario.

Ahora, implementemos **la misma funcionalidad exacta** usando código estilo *OLOO*:

```js
var Foo = {
	init: function(who) {
		this.me = who;
	},
	identify: function() {
		return "I am " + this.me;
	}
};

var Bar = Object.create( Foo );

Bar.speak = function() {
	alert( "Hello, " + this.identify() + "." );
};

var b1 = Object.create( Bar );
b1.init( "b1" );
var b2 = Object.create( Bar );
b2.init( "b2" );

b1.speak();
b2.speak();
```

Aprovechamos exactamente la misma ventaja de la delegación `[[Prototype]]` de `b1` a `Bar` a `Foo` como hicimos en el fragmento anterior entre `b1`, `Bar.prototype` y `Foo.prototype`. **Aún tenemos los mismos 3 objetos vinculados juntos**.

Pero, importantemente, hemos simplificado enormemente *todo lo demás* que estaba pasando, porque ahora simplemente establecemos **objetos** vinculados entre sí, sin necesitar toda la basura y confusión de cosas que parecen (¡pero no se comportan como!) clases, con constructores y prototipos y llamadas `new`.

Pregúntate: si puedo obtener la misma funcionalidad con código estilo OLOO que con código estilo "clase", pero OLOO es más simple y tiene menos cosas en qué pensar, **¿no es OLOO mejor?**

## Clases vs. Objetos

Acabamos de ver varias exploraciones teóricas y modelos mentales de "clases" vs. "delegación de comportamiento". Pero, ahora veamos escenarios de código más concretos para mostrar cómo usarías realmente estas ideas.

Primero examinaremos un escenario típico en el desarrollo web front-end: crear widgets de UI (botones, drop-downs, etc.).

### "Clases" de Widget

Porque probablemente aún estás tan acostumbrado al patrón de diseño OO, probablemente pensarás inmediatamente en este dominio de problema en términos de una clase padre (quizás llamada `Widget`) con todo el comportamiento base común de widget, y luego clases derivadas hijas para tipos de widget específicos (como `Button`).

**Nota:** Vamos a usar jQuery aquí para manipulación de DOM y CSS, solo porque es un detalle que realmente no nos importa para los propósitos de nuestra discusión actual. Nada de este código se preocupa por qué framework JS (jQuery, Dojo, YUI, etc.), si alguno, podrías usar para resolver tales tareas mundanas.

Examinemos cómo implementaríamos el diseño "clase" en JS puro estilo clásico sin ninguna librería o sintaxis helper de "clase":

```js
// Parent class
function Widget(width,height) {
	this.width = width || 50;
	this.height = height || 50;
	this.$elem = null;
}

Widget.prototype.render = function($where){
	if (this.$elem) {
		this.$elem.css( {
			width: this.width + "px",
			height: this.height + "px"
		} ).appendTo( $where );
	}
};

// Child class
function Button(width,height,label) {
	// "super" constructor call
	Widget.call( this, width, height );
	this.label = label || "Default";

	this.$elem = $( "<button>" ).text( this.label );
}

// make `Button` "inherit" from `Widget`
Button.prototype = Object.create( Widget.prototype );

// override base "inherited" `render(..)`
Button.prototype.render = function($where) {
	// "super" call
	Widget.prototype.render.call( this, $where );
	this.$elem.click( this.onClick.bind( this ) );
};

Button.prototype.onClick = function(evt) {
	console.log( "Button '" + this.label + "' clicked!" );
};

$( document ).ready( function(){
	var $body = $( document.body );
	var btn1 = new Button( 125, 30, "Hello" );
	var btn2 = new Button( 150, 40, "World" );

	btn1.render( $body );
	btn2.render( $body );
} );
```

Los patrones de diseño OO nos dicen que declaremos un `render(..)` base en la clase padre, luego lo sobreescribamos en nuestra clase hija, pero no reemplazarlo per se, sino aumentar la funcionalidad base con comportamiento específico de botón.

Nota la fealdad del *pseudo-polimorfismo explícito* (ver Capítulo 4) con las referencias `Widget.call` y `Widget.prototype.render.call` para falsificar llamadas "super" desde los métodos de la "clase" hija de vuelta a los métodos base de la "clase" padre. Puaj.

#### Azúcar sintáctica de `class` en ES6

Cubrimos la sintaxis de azúcar `class` de ES6 en detalle en el Apéndice A, pero demostremos brevemente cómo implementaríamos el mismo código usando `class`:

```js
class Widget {
	constructor(width,height) {
		this.width = width || 50;
		this.height = height || 50;
		this.$elem = null;
	}
	render($where){
		if (this.$elem) {
			this.$elem.css( {
				width: this.width + "px",
				height: this.height + "px"
			} ).appendTo( $where );
		}
	}
}

class Button extends Widget {
	constructor(width,height,label) {
		super( width, height );
		this.label = label || "Default";
		this.$elem = $( "<button>" ).text( this.label );
	}
	render($where) {
		super.render( $where );
		this.$elem.click( this.onClick.bind( this ) );
	}
	onClick(evt) {
		console.log( "Button '" + this.label + "' clicked!" );
	}
}

$( document ).ready( function(){
	var $body = $( document.body );
	var btn1 = new Button( 125, 30, "Hello" );
	var btn2 = new Button( 150, 40, "World" );

	btn1.render( $body );
	btn2.render( $body );
} );
```

Sin duda, varias de las fealdades sintácticas del enfoque clásico anterior se han suavizado con el `class` de ES6. La presencia de un `super(..)` en particular parece bastante agradable (¡aunque cuando profundizas en ello, no todo son rosas!).

A pesar de las mejoras sintácticas, **estas no son clases *reales***, ya que aún operan sobre el mecanismo `[[Prototype]]`. Sufren de todas las mismas discrepancias de modelo mental que exploramos en los Capítulos 4, 5 y hasta ahora en este capítulo. El Apéndice A expandirá sobre la sintaxis de `class` de ES6 y sus implicaciones en detalle. Veremos por qué resolver inconvenientes de sintaxis no resuelve sustancialmente nuestras confusiones de clases en JS, aunque hace un esfuerzo valiente enmascarándose como una solución.

Ya sea que uses la sintaxis prototípica clásica o el nuevo azúcar de ES6, has hecho una *elección* de modelar el dominio del problema (widgets de UI) con "clases". Y como los capítulos anteriores intentan demostrar, esta *elección* en JavaScript te está optando por dolores de cabeza extra y carga mental.

### Delegando Objetos Widget

Aquí está nuestro ejemplo más simple de `Widget` / `Button`, usando **delegación estilo OLOO**:

```js
var Widget = {
	init: function(width,height){
		this.width = width || 50;
		this.height = height || 50;
		this.$elem = null;
	},
	insert: function($where){
		if (this.$elem) {
			this.$elem.css( {
				width: this.width + "px",
				height: this.height + "px"
			} ).appendTo( $where );
		}
	}
};

var Button = Object.create( Widget );

Button.setup = function(width,height,label){
	// delegated call
	this.init( width, height );
	this.label = label || "Default";

	this.$elem = $( "<button>" ).text( this.label );
};
Button.build = function($where) {
	// delegated call
	this.insert( $where );
	this.$elem.click( this.onClick.bind( this ) );
};
Button.onClick = function(evt) {
	console.log( "Button '" + this.label + "' clicked!" );
};

$( document ).ready( function(){
	var $body = $( document.body );

	var btn1 = Object.create( Button );
	btn1.setup( 125, 30, "Hello" );

	var btn2 = Object.create( Button );
	btn2.setup( 150, 40, "World" );

	btn1.build( $body );
	btn2.build( $body );
} );
```

Con este enfoque estilo OLOO, no pensamos en `Widget` como padre y `Button` como hijo. Más bien, `Widget` **es simplemente un objeto** y es una especie de colección de utilidades a la que cualquier tipo específico de widget podría querer delegar, y `Button` **también es simplemente un objeto independiente** (¡con un enlace de delegación a `Widget`, por supuesto!).

Desde la perspectiva del patrón de diseño, **no** compartimos el mismo nombre de método `render(..)` en ambos objetos, como sugieren las clases, sino que elegimos nombres diferentes (`insert(..)` y `build(..)`) que eran más descriptivos de qué tarea hace cada uno específicamente. Los métodos de *inicialización* se llaman `init(..)` y `setup(..)`, respectivamente, por las mismas razones.

No solo este patrón de diseño de delegación sugiere nombres diferentes y más descriptivos (en lugar de nombres compartidos y más genéricos), sino que hacerlo con OLOO resulta en evitar la fealdad de las llamadas de pseudo-polimorfismo explícito (`Widget.call` y `Widget.prototype.render.call`), como puedes ver por las simples, relativas, llamadas delegadas a `this.init(..)` y `this.insert(..)`.

Sintácticamente, tampoco tenemos constructores, `.prototype` o `new` presentes, ya que, de hecho, son solo basura innecesaria.

Ahora, si estás prestando atención, puedes notar que lo que anteriormente era solo una llamada (`var btn1 = new Button(..)`) ahora son dos llamadas (`var btn1 = Object.create(Button)` y `btn1.setup(..)`). Inicialmente esto puede parecer una desventaja (más código).

Sin embargo, incluso esto es algo que es **un pro del código estilo OLOO** comparado con el código estilo prototipo clásico. ¿Cómo?

Con constructores de clase, estás "forzado" (no realmente, pero fuertemente sugerido) a hacer tanto la construcción como la inicialización en el mismo paso. Sin embargo, hay muchos casos donde poder hacer estos dos pasos por separado (¡como haces con OLOO!) es más flexible.

Por ejemplo, digamos que creas todas tus instancias en un pool al inicio de tu programa, pero esperas para inicializarlas con configuración específica hasta que son sacadas del pool y usadas. Mostramos las dos llamadas sucediendo justo una al lado de la otra, pero por supuesto pueden suceder en tiempos muy diferentes y en partes muy diferentes de nuestro código, según sea necesario.

**OLOO** soporta *mejor* el principio de separación de preocupaciones, donde la creación y la inicialización no están necesariamente combinadas en la misma operación.

## Diseño Más Simple

Además de que OLOO proporciona código ostensiblemente más simple (¡y más flexible!), la delegación de comportamiento como patrón puede realmente llevar a una arquitectura de código más simple. Examinemos un último ejemplo que ilustra cómo OLOO simplifica tu diseño general.

El escenario que examinaremos son dos objetos controladores, uno para manejar el formulario de login de una página web, y otro para realmente manejar la autenticación (comunicación) con el servidor.

Necesitaremos un helper de utilidad para hacer la comunicación Ajax con el servidor. Usaremos jQuery (aunque cualquier framework serviría bien), ya que no solo maneja el Ajax por nosotros, sino que retorna una respuesta tipo promise para que podamos escuchar la respuesta en nuestro código de llamada con `.then(..)`.

**Nota:** No cubrimos Promises aquí, pero los cubriremos en un título futuro de la serie *"You Don't Know JS"*.

Siguiendo el patrón de diseño típico de clases, dividiremos la tarea en funcionalidad base en una clase llamada `Controller`, y luego derivaremos dos clases hijas, `LoginController` y `AuthController`, que ambas heredan de `Controller` y especializan algunos de esos comportamientos base.

```js
// Parent class
function Controller() {
	this.errors = [];
}
Controller.prototype.showDialog = function(title,msg) {
	// display title & message to user in dialog
};
Controller.prototype.success = function(msg) {
	this.showDialog( "Success", msg );
};
Controller.prototype.failure = function(err) {
	this.errors.push( err );
	this.showDialog( "Error", err );
};
```

```js
// Child class
function LoginController() {
	Controller.call( this );
}
// Link child class to parent
LoginController.prototype = Object.create( Controller.prototype );
LoginController.prototype.getUser = function() {
	return document.getElementById( "login_username" ).value;
};
LoginController.prototype.getPassword = function() {
	return document.getElementById( "login_password" ).value;
};
LoginController.prototype.validateEntry = function(user,pw) {
	user = user || this.getUser();
	pw = pw || this.getPassword();

	if (!(user && pw)) {
		return this.failure( "Please enter a username & password!" );
	}
	else if (pw.length < 5) {
		return this.failure( "Password must be 5+ characters!" );
	}

	// got here? validated!
	return true;
};
// Override to extend base `failure()`
LoginController.prototype.failure = function(err) {
	// "super" call
	Controller.prototype.failure.call( this, "Login invalid: " + err );
};
```

```js
// Child class
function AuthController(login) {
	Controller.call( this );
	// in addition to inheritance, we also need composition
	this.login = login;
}
// Link child class to parent
AuthController.prototype = Object.create( Controller.prototype );
AuthController.prototype.server = function(url,data) {
	return $.ajax( {
		url: url,
		data: data
	} );
};
AuthController.prototype.checkAuth = function() {
	var user = this.login.getUser();
	var pw = this.login.getPassword();

	if (this.login.validateEntry( user, pw )) {
		this.server( "/check-auth",{
			user: user,
			pw: pw
		} )
		.then( this.success.bind( this ) )
		.fail( this.failure.bind( this ) );
	}
};
// Override to extend base `success()`
AuthController.prototype.success = function() {
	// "super" call
	Controller.prototype.success.call( this, "Authenticated!" );
};
// Override to extend base `failure()`
AuthController.prototype.failure = function(err) {
	// "super" call
	Controller.prototype.failure.call( this, "Auth Failed: " + err );
};
```

```js
var auth = new AuthController(
	// in addition to inheritance, we also need composition
	new LoginController()
);
auth.checkAuth();
```

Tenemos comportamientos base que todos los controladores comparten, que son `success(..)`, `failure(..)` y `showDialog(..)`. Nuestras clases hijas `LoginController` y `AuthController` sobreescriben `failure(..)` y `success(..)` para aumentar el comportamiento por defecto de la clase base. También nota que `AuthController` necesita una instancia de `LoginController` para interactuar con el formulario de login, así que eso se convierte en una propiedad de datos miembro.

La otra cosa a mencionar es que elegimos algo de *composición* para esparcir encima de la herencia. `AuthController` necesita saber sobre `LoginController`, así que lo instanciamos (`new LoginController()`) y mantenemos una propiedad miembro de clase llamada `this.login` para referenciarlo, de modo que `AuthController` pueda invocar comportamiento en `LoginController`.

**Nota:** *Podría* haber habido una ligera tentación de hacer que `AuthController` herede de `LoginController`, o viceversa, de modo que tuviéramos *composición virtual* a través de la cadena de herencia. Pero este es un ejemplo fuertemente claro de lo que está mal con la herencia de clases como *el* modelo para el dominio del problema, porque ni `AuthController` ni `LoginController` están especializando comportamiento base del otro, así que la herencia entre ellos tiene poco sentido excepto si las clases son tu único patrón de diseño. En su lugar, intercalamos algo de *composición* simple y ahora pueden cooperar, mientras ambos siguen beneficiándose de la herencia de la clase padre base `Controller`.

Si estás familiarizado con el diseño orientado a clases (OO), todo esto debería lucir bastante familiar y natural.

### Des-clasificado

Pero, **¿realmente necesitamos modelar este problema** con una clase padre `Controller`, dos clases hijas, **y algo de composición**? ¿Hay una forma de aprovechar la delegación de comportamiento estilo OLOO y tener un diseño *mucho* más simple? **¡Sí!**

```js
var LoginController = {
	errors: [],
	getUser: function() {
		return document.getElementById( "login_username" ).value;
	},
	getPassword: function() {
		return document.getElementById( "login_password" ).value;
	},
	validateEntry: function(user,pw) {
		user = user || this.getUser();
		pw = pw || this.getPassword();

		if (!(user && pw)) {
			return this.failure( "Please enter a username & password!" );
		}
		else if (pw.length < 5) {
			return this.failure( "Password must be 5+ characters!" );
		}

		// got here? validated!
		return true;
	},
	showDialog: function(title,msg) {
		// display success message to user in dialog
	},
	failure: function(err) {
		this.errors.push( err );
		this.showDialog( "Error", "Login invalid: " + err );
	}
};
```

```js
// Link `AuthController` to delegate to `LoginController`
var AuthController = Object.create( LoginController );

AuthController.errors = [];
AuthController.checkAuth = function() {
	var user = this.getUser();
	var pw = this.getPassword();

	if (this.validateEntry( user, pw )) {
		this.server( "/check-auth",{
			user: user,
			pw: pw
		} )
		.then( this.accepted.bind( this ) )
		.fail( this.rejected.bind( this ) );
	}
};
AuthController.server = function(url,data) {
	return $.ajax( {
		url: url,
		data: data
	} );
};
AuthController.accepted = function() {
	this.showDialog( "Success", "Authenticated!" )
};
AuthController.rejected = function(err) {
	this.failure( "Auth Failed: " + err );
};
```

Como `AuthController` es solo un objeto (también lo es `LoginController`), no necesitamos instanciar (como `new AuthController()`) para realizar nuestra tarea. Todo lo que necesitamos hacer es:

```js
AuthController.checkAuth();
```

Por supuesto, con OLOO, si necesitas crear uno o más objetos adicionales en la cadena de delegación, eso es fácil, y aún no requiere nada como instanciación de clase:

```js
var controller1 = Object.create( AuthController );
var controller2 = Object.create( AuthController );
```

Con delegación de comportamiento, `AuthController` y `LoginController` son **solo objetos**, pares *horizontales* entre sí, y no están arreglados o relacionados como padres e hijos en la orientación a clases. De forma algo arbitraria elegimos que `AuthController` delegue a `LoginController` -- habría sido igual de válido que la delegación fuera en la dirección inversa.

La conclusión principal de esta segunda lista de código es que solo tenemos dos entidades (`LoginController` y `AuthController`), **no tres** como antes.

No necesitamos una clase base `Controller` para "compartir" comportamiento entre las dos, porque la delegación es un mecanismo lo suficientemente poderoso para darnos la funcionalidad que necesitamos. Tampoco, como se señaló antes, necesitamos instanciar nuestras clases para trabajar con ellas, porque no hay clases, **solo los objetos mismos**. Además, no hay necesidad de *composición* ya que la delegación les da a los dos objetos la capacidad de cooperar *diferencialmente* según sea necesario.

Por último, evitamos las trampas del polimorfismo del diseño orientado a clases al no tener los nombres `success(..)` y `failure(..)` iguales en ambos objetos, lo que habría requerido feo pseudo-polimorfismo explícito. En su lugar, los llamamos `accepted()` y `rejected(..)` en `AuthController` -- nombres ligeramente más descriptivos para sus tareas específicas.

**En resumen**: terminamos con la misma capacidad, pero un diseño (significativamente) más simple. Ese es el poder del código estilo OLOO y el poder del patrón de diseño de *delegación de comportamiento*.

## Sintaxis Más Agradable

Una de las cosas más agradables que hace al `class` de ES6 tan engañosamente atractivo (¡ver Apéndice A sobre por qué evitarlo!) es la sintaxis abreviada para declarar métodos de clase:

```js
class Foo {
	methodName() { /* .. */ }
}
```

¡Podemos dejar caer la palabra `function` de la declaración, lo que hace que los desarrolladores JS en todas partes celebren!

Y puede que hayas notado y te hayas frustrado porque la sintaxis OLOO sugerida arriba tiene muchas apariciones de `function`, lo que parece ser un poco un detractor del objetivo de simplificación de OLOO. **¡Pero no tiene que ser así!**

A partir de ES6, podemos usar *declaraciones de método concisas* en cualquier literal de objeto, así que un objeto en estilo OLOO puede ser declarado de esta forma (mismo azúcar abreviado que con la sintaxis del cuerpo de `class`):

```js
var LoginController = {
	errors: [],
	getUser() { // Look ma, no `function`!
		// ...
	},
	getPassword() {
		// ...
	}
	// ...
};
```

La prácticamente única diferencia es que los literales de objeto aún requerirán separadores de coma `,` entre elementos mientras que la sintaxis `class` no. Una concesión bastante menor en el esquema general de las cosas.

Además, a partir de ES6, la sintaxis más torpe que usas (como para la definición de `AuthController`), donde asignas propiedades individualmente y no usas un literal de objeto, puede ser reescrita usando un literal de objeto (para poder usar métodos concisos), y solo puedes modificar el `[[Prototype]]` de ese objeto con `Object.setPrototypeOf(..)`, así:

```js
// use nicer object literal syntax w/ concise methods!
var AuthController = {
	errors: [],
	checkAuth() {
		// ...
	},
	server(url,data) {
		// ...
	}
	// ...
};

// NOW, link `AuthController` to delegate to `LoginController`
Object.setPrototypeOf( AuthController, LoginController );
```

El estilo OLOO a partir de ES6, con métodos concisos, **es mucho más amigable** de lo que era antes (e incluso entonces, era mucho más simple y agradable que el código estilo prototipo clásico). **No tienes que optar por class** (complejidad) para obtener sintaxis de objetos limpia y agradable.

### Sin Léxico

*Hay* una desventaja de los métodos concisos que es sutil pero importante a notar. Considera este código:

```js
var Foo = {
	bar() { /*..*/ },
	baz: function baz() { /*..*/ }
};
```

Aquí está el des-azucarado sintáctico que expresa cómo operará ese código:

```js
var Foo = {
	bar: function() { /*..*/ },
	baz: function baz() { /*..*/ }
};
```

¿Ves la diferencia? La abreviación `bar()` se convirtió en una *expresión de función anónima* (`function()..`) adjunta a la propiedad `bar`, porque el objeto función mismo no tiene un identificador de nombre. Compara eso con la *expresión de función nombrada* (`function baz()..`) especificada manualmente que tiene un identificador de nombre léxico `baz` además de estar adjunta a una propiedad `.baz`.

¿Y qué? En el título *"Scope & Closures"* de esta serie de libros *"You Don't Know JS"*, cubrimos las tres principales desventajas de las *expresiones de función anónimas* en detalle. Solo las repetiremos brevemente para que podamos comparar con la abreviación de método conciso.

La falta de un identificador `name` en una función anónima:

1. hace la depuración de stack traces más difícil
2. hace la auto-referencia (recursión, (des)enlace de eventos, etc.) más difícil
3. hace el código (un poco) más difícil de entender

Los puntos 1 y 3 no aplican a los métodos concisos.

Aunque el des-azucarado usa una *expresión de función anónima* que normalmente no tendría `name` en los stack traces, los métodos concisos se especifican para establecer la propiedad `name` interna del objeto función correspondientemente, así que los stack traces deberían poder usarlo (aunque eso depende de la implementación así que no está garantizado).

El punto 2 es, desafortunadamente, **aún una desventaja de los métodos concisos**. No tendrán un identificador léxico para usar como auto-referencia. Considera:

```js
var Foo = {
	bar: function(x) {
		if (x < 10) {
			return Foo.bar( x * 2 );
		}
		return x;
	},
	baz: function baz(x) {
		if (x < 10) {
			return baz( x * 2 );
		}
		return x;
	}
};
```

La referencia manual `Foo.bar(x*2)` anterior más o menos alcanza en este ejemplo, pero hay muchos casos donde una función no necesariamente podría hacer eso, como casos donde la función se comparte en delegación a través de diferentes objetos, usando enlace de `this`, etc. Querrías usar una verdadera auto-referencia, y el identificador `name` del objeto función es la mejor forma de lograr eso.

Solo ten presente esta advertencia para los métodos concisos, y si te encuentras con tales problemas con la falta de auto-referencia, asegúrate de renunciar a la sintaxis de método conciso **solo para esa declaración** en favor de la forma de declaración manual de *expresión de función nombrada*: `baz: function baz(){..}`.
