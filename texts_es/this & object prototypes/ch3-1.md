# You Don't Know JS: *this* & Prototipos de Objetos
# Capítulo 3: Objetos

En los Capítulos 1 y 2, explicamos cómo el enlace de `this` apunta a varios objetos dependiendo del sitio de llamada de la invocación de la función. Pero ¿qué son exactamente los objetos, y por qué necesitamos apuntar a ellos? Exploraremos los objetos en detalle en este capítulo.

## Sintaxis

Los objetos vienen en dos formas: la forma declarativa (literal), y la forma construida.

La sintaxis literal para un objeto se ve así:

```js
var myObj = {
	key: value
	// ...
};
```

La forma construida se ve así:

```js
var myObj = new Object();
myObj.key = value;
```

La forma construida y la forma literal resultan en exactamente el mismo tipo de objeto. La única diferencia realmente es que puedes agregar uno o más pares clave/valor a la declaración literal, mientras que con objetos de forma construida, debes agregar las propiedades una por una.

**Nota:** Es extremadamente poco común usar la "forma construida" para crear objetos como se mostró. Casi siempre querrás usar la forma de sintaxis literal. Lo mismo será cierto para la mayoría de los objetos integrados (ver abajo).

## Tipo

Los objetos son el bloque de construcción general sobre el cual mucho de JS está construido. Son uno de los 6 tipos primarios (llamados "tipos de lenguaje" en la especificación) en JS:

* `string`
* `number`
* `boolean`
* `null`
* `undefined`
* `object`

Nota que los *primitivos simples* (`string`, `number`, `boolean`, `null`, y `undefined`) **no** son en sí mismos `objects`. `null` a veces se refiere como un tipo objeto, pero esta concepción errónea proviene de un bug en el lenguaje que causa que `typeof null` retorne el string `"object"` incorrectamente (y confusamente). De hecho, `null` es su propio tipo primitivo.

**Es una declaración errónea común que "todo en JavaScript es un objeto". Esto claramente no es cierto.**

En contraste, *hay* algunos sub-tipos especiales de objetos, a los que podemos referirnos como *primitivos complejos*.

`function` es un sub-tipo de objeto (técnicamente, un "objeto invocable"). Se dice que las funciones en JS son de "primera clase" en que son básicamente solo objetos normales (con semánticas de comportamiento invocable adjuntas), y así pueden ser manejadas como cualquier otro objeto simple.

Los Arrays también son una forma de objetos, con comportamiento extra. La organización del contenido en arrays está ligeramente más estructurada que la de objetos generales.

### Objetos Integrados

Hay varios otros sub-tipos de objetos, usualmente referidos como objetos integrados. Para algunos de ellos, sus nombres parecen implicar que están directamente relacionados con sus contrapartes de primitivos simples, pero de hecho, su relación es más complicada, lo cual exploraremos en breve.

* `String`
* `Number`
* `Boolean`
* `Object`
* `Function`
* `Array`
* `Date`
* `RegExp`
* `Error`

Estos integrados tienen la apariencia de ser tipos reales, incluso clases, si te basas en la similitud con otros lenguajes como la clase `String` de Java.

Pero en JS, estos son en realidad solo funciones integradas. Cada una de estas funciones integradas puede usarse como constructor (es decir, una llamada de función con el operador `new` -- ver Capítulo 2), con el resultado siendo un objeto recién *construido* del sub-tipo en cuestión. Por ejemplo:

```js
var strPrimitive = "I am a string";
typeof strPrimitive;							// "string"
strPrimitive instanceof String;					// false

var strObject = new String( "I am a string" );
typeof strObject; 								// "object"
strObject instanceof String;					// true

// inspeccionar el sub-tipo del objeto
Object.prototype.toString.call( strObject );	// [object String]
```

Veremos en detalle en un capítulo posterior exactamente cómo funciona la parte de `Object.prototype.toString...`, pero brevemente, podemos inspeccionar el sub-tipo interno tomando prestado el método base `toString()` por defecto, y puedes ver que revela que `strObject` es un objeto que fue de hecho creado por el constructor `String`.

El valor primitivo `"I am a string"` no es un objeto, es un literal primitivo e inmutable. Para realizar operaciones en él, como verificar su longitud, acceder al contenido de caracteres individuales, etc., se requiere un objeto `String`.

Afortunadamente, el lenguaje automáticamente coacciona un primitivo `"string"` a un objeto `String` cuando es necesario, lo que significa que casi nunca necesitas crear explícitamente la forma de Objeto. Es **fuertemente preferido** por la mayoría de la comunidad JS usar la forma literal para un valor, donde sea posible, en lugar de la forma de objeto construido.

Considera:

```js
var strPrimitive = "I am a string";

console.log( strPrimitive.length );			// 13

console.log( strPrimitive.charAt( 3 ) );	// "m"
```

En ambos casos, llamamos a una propiedad o método en un primitivo string, y el motor automáticamente lo coacciona a un objeto `String`, para que el acceso a la propiedad/método funcione.

El mismo tipo de coerción ocurre entre el primitivo literal numérico `42` y el envoltorio de objeto `new Number(42)`, cuando se usan métodos como `42.359.toFixed(2)`. De igual manera para objetos `Boolean` desde primitivos `"boolean"`.

`null` y `undefined` no tienen forma de envoltorio de objeto, solo sus valores primitivos. En contraste, los valores `Date` *solo* pueden ser creados con su forma de objeto construido, ya que no tienen contraparte en forma literal.

Los `Object`s, `Array`s, `Function`s, y `RegExp`s (expresiones regulares) son todos objetos independientemente de si se usa la forma literal o construida. La forma construida ofrece, en algunos casos, más opciones en la creación que la contraparte en forma literal. Como los objetos se crean de cualquier manera, la forma literal más simple es casi universalmente preferida. **Solo usa la forma construida si necesitas las opciones extras.**

Los objetos `Error` rara vez se crean explícitamente en código, pero usualmente se crean automáticamente cuando se lanzan excepciones. Pueden ser creados con la forma construida `new Error(..)`, pero a menudo es innecesario.

## Contenidos

Como se mencionó anteriormente, los contenidos de un objeto consisten en valores (cualquier tipo) almacenados en *ubicaciones* con nombre específico, que llamamos propiedades.

Es importante notar que aunque decimos "contenidos" lo cual implica que estos valores están *realmente* almacenados dentro del objeto, eso es meramente una apariencia. El motor almacena valores de formas dependientes de la implementación, y puede muy bien no almacenarlos *en* algún contenedor de objeto. Lo que *sí* se almacena en el contenedor son estos nombres de propiedades, que actúan como punteros (técnicamente, *referencias*) a donde los valores están almacenados.

Considera:

```js
var myObject = {
	a: 2
};

myObject.a;		// 2

myObject["a"];	// 2
```

Para acceder al valor en la *ubicación* `a` en `myObject`, necesitamos usar ya sea el operador `.` o el operador `[ ]`. La sintaxis `.a` se refiere usualmente como acceso de "propiedad", mientras que la sintaxis `["a"]` se refiere usualmente como acceso de "clave". En realidad, ambos acceden a la misma *ubicación*, y extraerán el mismo valor, `2`, así que los términos pueden usarse intercambiablemente. Usaremos el término más común, "acceso de propiedad" de aquí en adelante.

La diferencia principal entre las dos sintaxis es que el operador `.` requiere un nombre de propiedad compatible con `Identifier` después de él, mientras que la sintaxis `[".."]` puede tomar básicamente cualquier string compatible con UTF-8/unicode como nombre para la propiedad. Para referenciar una propiedad con el nombre "Super-Fun!", por ejemplo, tendrías que usar la sintaxis de acceso `["Super-Fun!"]`, ya que `Super-Fun!` no es un nombre de propiedad `Identifier` válido.

También, como la sintaxis `[".."]` usa el **valor** de un string para especificar la ubicación, esto significa que el programa puede construir programáticamente el valor del string, como:

```js
var wantA = true;
var myObject = {
	a: 2
};

var idx;

if (wantA) {
	idx = "a";
}

// después

console.log( myObject[idx] ); // 2
```

En objetos, los nombres de propiedades son **siempre** strings. Si usas cualquier otro valor además de un `string` (primitivo) como la propiedad, primero será convertido a string. Esto incluso incluye números, que son comúnmente usados como índices de array, así que ten cuidado de no confundir el uso de números entre objetos y arrays.

```js
var myObject = { };

myObject[true] = "foo";
myObject[3] = "bar";
myObject[myObject] = "baz";

myObject["true"];				// "foo"
myObject["3"];					// "bar"
myObject["[object Object]"];	// "baz"
```

### Nombres de Propiedades Computados

La sintaxis de acceso a propiedad `myObject[..]` que acabamos de describir es útil si necesitas usar un valor de expresión computada *como* nombre de clave, como `myObject[prefix + name]`. Pero eso no es realmente útil cuando declaras objetos usando la sintaxis de objeto-literal.

ES6 añade *nombres de propiedades computados*, donde puedes especificar una expresión, rodeada por un par `[ ]`, en la posición de nombre-clave de una declaración de objeto-literal:

```js
var prefix = "foo";

var myObject = {
	[prefix + "bar"]: "hello",
	[prefix + "baz"]: "world"
};

myObject["foobar"]; // hello
myObject["foobaz"]; // world
```

El uso más común de *nombres de propiedades computados* probablemente será para `Symbol`s de ES6, que no cubriremos en detalle en este libro. En resumen, son un nuevo tipo de dato primitivo que tiene un valor opaco e imposible de adivinar (técnicamente un valor `string`). Serás fuertemente desalentado de trabajar con el *valor real* de un `Symbol` (que teóricamente puede ser diferente entre diferentes motores JS), así que el nombre del `Symbol`, como `Symbol.Something` (¡solo un nombre inventado!), será lo que uses:

```js
var myObject = {
	[Symbol.Something]: "hello world"
};
```

### Propiedad vs. Método

Algunos desarrolladores les gusta hacer una distinción cuando hablan de un acceso de propiedad en un objeto, si el valor que se accede resulta ser una función. Porque es tentador pensar que la función *pertenece* al objeto, y en otros lenguajes, las funciones que pertenecen a objetos (también conocidos como "clases") se refieren como "métodos", no es poco común escuchar "acceso de método" en oposición a "acceso de propiedad".

**La especificación hace esta misma distinción**, curiosamente.

Técnicamente, las funciones nunca "pertenecen" a objetos, así que decir que una función que casualmente se accede en una referencia de objeto es automáticamente un "método" parece un poco exagerado semánticamente.

*Es* cierto que algunas funciones tienen referencias a `this` en ellas, y que *a veces* estas referencias a `this` se refieren a la referencia del objeto en el sitio de llamada. Pero este uso realmente no hace que esa función sea más un "método" que cualquier otra función, ya que `this` se enlaza dinámicamente en tiempo de ejecución, en el sitio de llamada, y por lo tanto su relación con el objeto es indirecta, en el mejor de los casos.

Cada vez que accedes a una propiedad en un objeto, eso es un **acceso de propiedad**, independientemente del tipo de valor que obtengas de vuelta. Si *sucede* que obtienes una función de ese acceso de propiedad, no es mágicamente un "método" en ese punto. No hay nada especial (aparte del posible enlace implícito de `this` como se explicó anteriormente) sobre una función que viene de un acceso de propiedad.

Por ejemplo:

```js
function foo() {
	console.log( "foo" );
}

var someFoo = foo;	// referencia de variable a `foo`

var myObject = {
	someFoo: foo
};

foo;				// function foo(){..}

someFoo;			// function foo(){..}

myObject.someFoo;	// function foo(){..}
```

`someFoo` y `myObject.someFoo` son solo dos referencias separadas a la misma función, y ninguna implica nada sobre la función siendo especial o "poseída" por algún otro objeto. Si `foo()` arriba fue definida para tener una referencia a `this` dentro de ella, ese *enlace implícito* de `myObject.someFoo` sería la **única** diferencia observable entre las dos referencias. Ninguna referencia realmente tiene sentido ser llamada "método".

**Quizás uno podría argumentar** que una función *se convierte en un método*, no en el momento de definición, sino durante tiempo de ejecución solo para esa invocación, dependiendo de cómo se llame en su sitio de llamada (con un contexto de referencia de objeto o no -- ver Capítulo 2 para más detalles). Incluso esta interpretación es un poco exagerada.

La conclusión más segura es probablemente que "función" y "método" son intercambiables en JavaScript.

**Nota:** ES6 añade una referencia `super`, que típicamente se usará con `class` (ver Apéndice A). La forma en que `super` se comporta (enlace estático en lugar de enlace tardío como `this`) da más peso a la idea de que una función que está enlazada por `super` en algún lugar es más un "método" que "función". Pero de nuevo, estos son solo matices semánticos (y mecánicos) sutiles.

Incluso cuando declaras una expresión de función como parte del objeto-literal, esa función no *pertenece* mágicamente más al objeto -- sigue siendo solo múltiples referencias al mismo objeto función:

```js
var myObject = {
	foo: function foo() {
		console.log( "foo" );
	}
};

var someFoo = myObject.foo;

someFoo;		// function foo(){..}

myObject.foo;	// function foo(){..}
```

**Nota:** En el Capítulo 6, cubriremos una forma abreviada de ES6 para la sintaxis de declaración `foo: function foo(){ .. }` en nuestro objeto-literal.

### Arrays

Los Arrays también usan la forma de acceso `[ ]`, pero como se mencionó arriba, tienen una organización ligeramente más estructurada para cómo y dónde los valores son almacenados (aunque aún sin restricción en qué *tipo* de valores se almacenan). Los Arrays asumen *indexación numérica*, lo que significa que los valores se almacenan en ubicaciones, usualmente llamadas *índices*, en enteros no negativos, como `0` y `42`.

```js
var myArray = [ "foo", 42, "bar" ];

myArray.length;		// 3

myArray[0];			// "foo"

myArray[2];			// "bar"
```

Los Arrays *son* objetos, así que aunque cada índice es un entero positivo, también *puedes* agregar propiedades al array:

```js
var myArray = [ "foo", 42, "bar" ];

myArray.baz = "baz";

myArray.length;	// 3

myArray.baz;	// "baz"
```

Nota que agregar propiedades con nombre (independientemente de la sintaxis de operador `.` o `[ ]`) no cambia el `length` reportado del array.

*Podrías* usar un array como un objeto simple de clave/valor, y nunca agregar ningún índice numérico, pero esto es una mala idea porque los arrays tienen comportamiento y optimizaciones específicas para su uso previsto, e igualmente con los objetos simples. Usa objetos para almacenar pares clave/valor, y arrays para almacenar valores en índices numéricos.

**Ten cuidado:** Si intentas agregar una propiedad a un array, pero el nombre de la propiedad *parece* un número, terminará en su lugar como un índice numérico (modificando así los contenidos del array):

```js
var myArray = [ "foo", 42, "bar" ];

myArray["3"] = "baz";

myArray.length;	// 4

myArray[3];		// "baz"
```

### Duplicando Objetos

Una de las características más comúnmente solicitadas cuando los desarrolladores recién adoptan el lenguaje JavaScript es cómo duplicar un objeto. Parecería como que debería haber simplemente un método integrado `copy()`, ¿verdad? Resulta que es un poco más complicado que eso, porque no está completamente claro cuál debería ser, por defecto, el algoritmo para la duplicación.

Por ejemplo, considera este objeto:

```js
function anotherFunction() { /*..*/ }

var anotherObject = {
	c: true
};

var anotherArray = [];

var myObject = {
	a: 2,
	b: anotherObject,	// ¡referencia, no una copia!
	c: anotherArray,	// ¡otra referencia!
	d: anotherFunction
};

anotherArray.push( anotherObject, myObject );
```

¿Cuál debería ser exactamente la representación de una *copia* de `myObject`?

Primero, deberíamos responder si debería ser una copia *superficial* o *profunda*. Una *copia superficial* terminaría con `a` en el nuevo objeto como una copia del valor `2`, pero las propiedades `b`, `c`, y `d` como solo referencias a los mismos lugares que las referencias en el objeto original. Una *copia profunda* duplicaría no solo `myObject`, sino `anotherObject` y `anotherArray`. Pero entonces tenemos problemas porque `anotherArray` tiene referencias a `anotherObject` y `myObject` en él, así que *esos* también deberían ser duplicados en lugar de preservar las referencias. Ahora tenemos un problema de duplicación circular infinita por la referencia circular.

¿Deberíamos detectar una referencia circular y simplemente interrumpir el recorrido circular (dejando el elemento profundo no completamente duplicado)? ¿Deberíamos dar un error completamente? ¿Algo intermedio?

Además, no está realmente claro qué significaría "duplicar" una función. Hay algunos hacks como extraer la serialización `toString()` del código fuente de una función (que varía entre implementaciones y ni siquiera es confiable en todos los motores dependiendo del tipo de función siendo inspeccionada).

Entonces, ¿cómo resolvemos todas estas preguntas complicadas? Varios frameworks JS han elegido sus propias interpretaciones y tomado sus propias decisiones. Pero, ¿cuál de estos (si alguno) debería JS adoptar como *el* estándar? Por mucho tiempo, no hubo una respuesta clara.

Un subconjunto de solución es que los objetos que son seguros para JSON (es decir, pueden ser serializados a un string JSON y luego re-parseados a un objeto con la misma estructura y valores) pueden fácilmente ser *duplicados* con:

```js
var newObj = JSON.parse( JSON.stringify( someObj ) );
```

Por supuesto, eso requiere que te asegures de que tu objeto sea seguro para JSON. Para algunas situaciones, eso es trivial. Para otras, es insuficiente.

Al mismo tiempo, una copia superficial es bastante comprensible y tiene muchos menos problemas, así que ES6 ahora ha definido `Object.assign(..)` para esta tarea. `Object.assign(..)` toma un objeto *destino* como su primer parámetro, y uno o más objetos *fuente* como sus parámetros subsecuentes. Itera sobre todas las claves *enumerables* (ver abajo), *propias* (**inmediatamente presentes**) en el/los objeto(s) *fuente* y los copia (vía asignación `=` solamente) al *destino*. También, útilmente, retorna *destino*, como puedes ver abajo:

```js
var newObj = Object.assign( {}, myObject );

newObj.a;						// 2
newObj.b === anotherObject;		// true
newObj.c === anotherArray;		// true
newObj.d === anotherFunction;	// true
```

**Nota:** En la siguiente sección, describimos los "descriptores de propiedad" (características de propiedad) y mostramos el uso de `Object.defineProperty(..)`. La duplicación que ocurre con `Object.assign(..)` sin embargo es puramente asignación estilo `=`, así que cualquier característica especial de una propiedad (como `writable`) en un objeto fuente **no se preserva** en el objeto destino.

### Descriptores de Propiedad

Antes de ES5, el lenguaje JavaScript no daba forma directa para que tu código inspeccionara o hiciera alguna distinción entre las características de las propiedades, como si la propiedad era de solo lectura o no.

Pero a partir de ES5, todas las propiedades se describen en términos de un **descriptor de propiedad**.

Considera este código:

```js
var myObject = {
	a: 2
};

Object.getOwnPropertyDescriptor( myObject, "a" );
// {
//    value: 2,
//    writable: true,
//    enumerable: true,
//    configurable: true
// }
```

Como puedes ver, el descriptor de propiedad (llamado "descriptor de datos" ya que es solo para mantener un valor de datos) para nuestra propiedad de objeto normal `a` es mucho más que solo su `value` de `2`. Incluye 3 otras características: `writable`, `enumerable`, y `configurable`.

Mientras que podemos ver cuáles son los valores por defecto para las características del descriptor de propiedad cuando creamos una propiedad normal, podemos usar `Object.defineProperty(..)` para agregar una nueva propiedad, o modificar una existente (¡si es `configurable`!), con las características deseadas.

Por ejemplo:

```js
var myObject = {};

Object.defineProperty( myObject, "a", {
	value: 2,
	writable: true,
	configurable: true,
	enumerable: true
} );

myObject.a; // 2
```

Usando `defineProperty(..)`, agregamos la propiedad simple, normal `a` a `myObject` de una forma manualmente explícita. Sin embargo, generalmente no usarías este enfoque manual a menos que quisieras modificar una de las características del descriptor de su comportamiento normal.

#### Writable

La capacidad de cambiar el valor de una propiedad es controlada por `writable`.

Considera:

```js
var myObject = {};

Object.defineProperty( myObject, "a", {
	value: 2,
	writable: false, // ¡no escribible!
	configurable: true,
	enumerable: true
} );

myObject.a = 3;

myObject.a; // 2
```

Como puedes ver, nuestra modificación del `value` falló silenciosamente. Si intentamos en `strict mode`, obtenemos un error:

```js
"use strict";

var myObject = {};

Object.defineProperty( myObject, "a", {
	value: 2,
	writable: false, // ¡no escribible!
	configurable: true,
	enumerable: true
} );

myObject.a = 3; // TypeError
```

El `TypeError` nos dice que no podemos cambiar una propiedad no escribible.

**Nota:** Discutiremos getters/setters en breve, pero brevemente, puedes observar que `writable:false` significa que un valor no puede ser cambiado, lo cual es algo equivalente a si definieras un setter sin operación. En realidad, tu setter sin operación necesitaría lanzar un `TypeError` cuando se llama, para ser verdaderamente conforme con `writable:false`.

#### Configurable

Mientras una propiedad sea actualmente configurable, podemos modificar su definición de descriptor, usando la misma utilidad `defineProperty(..)`.

```js
var myObject = {
	a: 2
};

myObject.a = 3;
myObject.a;					// 3

Object.defineProperty( myObject, "a", {
	value: 4,
	writable: true,
	configurable: false,	// ¡no configurable!
	enumerable: true
} );

myObject.a;					// 4
myObject.a = 5;
myObject.a;					// 5

Object.defineProperty( myObject, "a", {
	value: 6,
	writable: true,
	configurable: true,
	enumerable: true
} ); // TypeError
```

La llamada final a `defineProperty(..)` resulta en un TypeError, independientemente del `strict mode`, si intentas cambiar la definición del descriptor de una propiedad no configurable. ¡Ten cuidado: como puedes ver, cambiar `configurable` a `false` es una **acción de un solo sentido, y no puede deshacerse!**

**Nota:** Hay una excepción matizada a tener en cuenta: incluso si la propiedad ya es `configurable:false`, `writable` siempre puede ser cambiado de `true` a `false` sin error, pero no de vuelta a `true` si ya está en `false`.

Otra cosa que `configurable:false` previene es la capacidad de usar el operador `delete` para eliminar una propiedad existente.

```js
var myObject = {
	a: 2
};

myObject.a;				// 2
delete myObject.a;
myObject.a;				// undefined

Object.defineProperty( myObject, "a", {
	value: 2,
	writable: true,
	configurable: false,
	enumerable: true
} );

myObject.a;				// 2
delete myObject.a;
myObject.a;				// 2
```

Como puedes ver, la última llamada a `delete` falló (silenciosamente) porque hicimos la propiedad `a` no configurable.

`delete` solo se usa para eliminar propiedades de objeto (que pueden ser eliminadas) directamente del objeto en cuestión. Si una propiedad de objeto es la última *referencia* restante a algún objeto/función, y la eliminas con `delete`, eso elimina la referencia y ahora ese objeto/función sin referencia puede ser recolectado por el garbage collector. Pero, **no** es apropiado pensar en `delete` como una herramienta para liberar memoria asignada como lo hace en otros lenguajes (como C/C++). `delete` es solo una operación de eliminación de propiedad de objeto -- nada más.

#### Enumerable

La última característica del descriptor que mencionaremos aquí (hay otras dos, que trataremos en breve cuando discutamos getter/setters) es `enumerable`.

El nombre probablemente lo hace obvio, pero esta característica controla si una propiedad aparecerá en ciertas enumeraciones de propiedades de objeto, como el bucle `for..in`. Establécelo a `false` para evitar que aparezca en tales enumeraciones, aunque sigue siendo completamente accesible. Establécelo a `true` para mantenerlo presente.

Todas las propiedades normales definidas por el usuario tienen por defecto `enumerable`, ya que esto es lo más comúnmente deseado. Pero si tienes una propiedad especial que quieres ocultar de la enumeración, establécela a `enumerable:false`.

Demostraremos la enumerabilidad con mucho más detalle en breve, así que mantén un marcador mental sobre este tema.

### Inmutabilidad

A veces se desea crear propiedades u objetos que no puedan ser cambiados (ya sea por accidente o intencionalmente). ES5 agrega soporte para manejar eso de varias formas matizadas diferentes.

Es importante notar que **todos** estos enfoques crean inmutabilidad superficial. Es decir, afectan solo al objeto y las características de sus propiedades directas. Si un objeto tiene una referencia a otro objeto (array, objeto, función, etc.), los *contenidos* de ese objeto no se ven afectados, y permanecen mutables.

```js
myImmutableObject.foo; // [1,2,3]
myImmutableObject.foo.push( 4 );
myImmutableObject.foo; // [1,2,3,4]
```

Asumimos en este fragmento que `myImmutableObject` ya está creado y protegido como inmutable. Pero, para también proteger los contenidos de `myImmutableObject.foo` (que es su propio objeto -- array), necesitarías también hacer `foo` inmutable, usando una o más de las siguientes funcionalidades.

**Nota:** No es terriblemente común crear objetos con inmutabilidad profundamente arraigada en programas JS. Casos especiales ciertamente pueden requerirlo, pero como patrón de diseño general, si te encuentras queriendo *sellar* o *congelar* todos tus objetos, puedes querer dar un paso atrás y reconsiderar el diseño de tu programa para ser más robusto ante cambios potenciales en los valores de los objetos.

#### Constante de Objeto

Combinando `writable:false` y `configurable:false`, puedes esencialmente crear una *constante* (no puede ser cambiada, redefinida o eliminada) como una propiedad de objeto, como:

```js
var myObject = {};

Object.defineProperty( myObject, "FAVORITE_NUMBER", {
	value: 42,
	writable: false,
	configurable: false
} );
```

#### Prevenir Extensiones

Si quieres prevenir que un objeto tenga nuevas propiedades agregadas, pero de otro modo dejar las propiedades restantes del objeto en paz, llama a `Object.preventExtensions(..)`:

```js
var myObject = {
	a: 2
};

Object.preventExtensions( myObject );

myObject.b = 3;
myObject.b; // undefined
```

En modo `non-strict`, la creación de `b` falla silenciosamente. En `strict mode`, lanza un `TypeError`.

#### Seal

`Object.seal(..)` crea un objeto "sellado", lo que significa que toma un objeto existente y esencialmente llama a `Object.preventExtensions(..)` en él, pero también marca todas sus propiedades existentes como `configurable:false`.

Así que, no solo no puedes agregar más propiedades, sino que tampoco puedes reconfigurar o eliminar propiedades existentes (aunque *sí puedes* modificar sus valores).

#### Freeze

`Object.freeze(..)` crea un objeto congelado, lo que significa que toma un objeto existente y esencialmente llama a `Object.seal(..)` en él, pero también marca todas las propiedades de "accessor de datos" como `writable:false`, para que sus valores no puedan ser cambiados.

Este enfoque es el nivel más alto de inmutabilidad que puedes alcanzar para un objeto en sí mismo, ya que previene cualquier cambio al objeto o a cualquiera de sus propiedades directas (aunque, como se mencionó arriba, los contenidos de cualquier otro objeto referenciado no se ven afectados).

Podrías "congelar profundamente" un objeto llamando a `Object.freeze(..)` en el objeto, y luego iterando recursivamente sobre todos los objetos que referencia (que no habrían sido afectados hasta ahora), y llamando a `Object.freeze(..)` en ellos también. Ten cuidado sin embargo, ya que eso podría afectar otros objetos (compartidos) que no pretendías afectar.

### `[[Get]]`

Hay un detalle sutil, pero importante, sobre cómo se realizan los accesos a propiedades.

Considera:

```js
var myObject = {
	a: 2
};

myObject.a; // 2
```

`myObject.a` es un acceso de propiedad, pero no *solo* busca en `myObject` una propiedad con el nombre `a`, como podría parecer.

De acuerdo con la especificación, el código anterior realmente realiza una operación `[[Get]]` (algo como una llamada de función: `[[Get]]()`) en `myObject`. La operación `[[Get]]` integrada por defecto para un objeto *primero* inspecciona el objeto buscando una propiedad con el nombre solicitado, y si la encuentra, retornará el valor correspondiente.

Sin embargo, el algoritmo `[[Get]]` define otro comportamiento importante si *no* encuentra una propiedad con el nombre solicitado. Examinaremos en el Capítulo 5 qué pasa *después* (recorrido de la cadena `[[Prototype]]`, si la hay).

Pero un resultado importante de esta operación `[[Get]]` es que si no puede por ningún medio obtener un valor para la propiedad solicitada, en su lugar retorna el valor `undefined`.

```js
var myObject = {
	a: 2
};

myObject.b; // undefined
```

Este comportamiento es diferente de cuando referencias *variables* por sus nombres de identificador. Si referencias una variable que no puede ser resuelta dentro de la búsqueda de ámbito léxico aplicable, el resultado no es `undefined` como lo es para propiedades de objeto, sino que se lanza un `ReferenceError`.

```js
var myObject = {
	a: undefined
};

myObject.a; // undefined

myObject.b; // undefined
```

Desde una perspectiva de *valor*, no hay diferencia entre estas dos referencias -- ambas resultan en `undefined`. Sin embargo, la operación `[[Get]]` por debajo, aunque sutil a primera vista, potencialmente realizó un poco más de "trabajo" para la referencia `myObject.b` que para la referencia `myObject.a`.

Inspeccionando solo los resultados de valor, no puedes distinguir si una propiedad existe y mantiene el valor explícito `undefined`, o si la propiedad *no* existe y `undefined` fue el valor de retorno por defecto después de que `[[Get]]` falló en retornar algo explícitamente. Sin embargo, veremos en breve cómo *puedes* distinguir estos dos escenarios.

### `[[Put]]`

Como hay una operación `[[Get]]` definida internamente para obtener un valor de una propiedad, debería ser obvio que también hay una operación `[[Put]]` por defecto.

Puede ser tentador pensar que una asignación a una propiedad en un objeto simplemente invocaría `[[Put]]` para establecer o crear esa propiedad en el objeto en cuestión. Pero la situación es más matizada que eso.

Cuando se invoca `[[Put]]`, cómo se comporta difiere basándose en un número de factores, incluyendo (de mayor impacto) si la propiedad ya está presente en el objeto o no.

Si la propiedad está presente, el algoritmo `[[Put]]` revisará aproximadamente:

1. ¿Es la propiedad un descriptor accessor (ver sección "Getters y Setters" abajo)? **Si es así, llamar al setter, si hay alguno.**
2. ¿Es la propiedad un descriptor de datos con `writable` en `false`? **Si es así, fallar silenciosamente en modo `non-strict`, o lanzar `TypeError` en `strict mode`.**
3. De lo contrario, establecer el valor en la propiedad existente normalmente.

Si la propiedad aún no está presente en el objeto en cuestión, la operación `[[Put]]` es aún más matizada y compleja. Revisitaremos este escenario en el Capítulo 5 cuando discutamos `[[Prototype]]` para darle más claridad.

### Getters y Setters

Las operaciones `[[Put]]` y `[[Get]]` por defecto para objetos controlan completamente cómo los valores se establecen en propiedades existentes o nuevas, o se recuperan de propiedades existentes, respectivamente.

**Nota:** Usando capacidades futuras/avanzadas del lenguaje, puede ser posible anular las operaciones `[[Get]]` o `[[Put]]` por defecto para un objeto completo (no solo por propiedad). Esto está fuera del alcance de nuestra discusión en este libro, pero será cubierto más adelante en la serie "You Don't Know JS".

ES5 introdujo una forma de anular parte de estas operaciones por defecto, no a nivel de objeto sino a nivel por-propiedad, mediante el uso de getters y setters. Los Getters son propiedades que realmente llaman a una función oculta para recuperar un valor. Los Setters son propiedades que realmente llaman a una función oculta para establecer un valor.

Cuando defines una propiedad para tener ya sea un getter o un setter o ambos, su definición se convierte en un "descriptor accessor" (a diferencia de un "descriptor de datos"). Para los descriptores accessor, las características `value` y `writable` del descriptor son irrelevantes e ignoradas, y en su lugar JS considera las características `set` y `get` de la propiedad (así como `configurable` y `enumerable`).

Considera:

```js
var myObject = {
	// define un getter para `a`
	get a() {
		return 2;
	}
};

Object.defineProperty(
	myObject,	// destino
	"b",		// nombre de propiedad
	{			// descriptor
		// define un getter para `b`
		get: function(){ return this.a * 2 },

		// asegura que `b` aparezca como propiedad del objeto
		enumerable: true
	}
);

myObject.a; // 2

myObject.b; // 4
```

Ya sea a través de la sintaxis de objeto-literal con `get a() { .. }` o a través de definición explícita con `defineProperty(..)`, en ambos casos creamos una propiedad en el objeto que realmente no contiene un valor, pero cuyo acceso automáticamente resulta en una llamada de función oculta a la función getter, con cualquier valor que retorne siendo el resultado del acceso de propiedad.

```js
var myObject = {
	// define un getter para `a`
	get a() {
		return 2;
	}
};

myObject.a = 3;

myObject.a; // 2
```

Como solo definimos un getter para `a`, si intentamos establecer el valor de `a` luego, la operación de set no lanzará un error pero simplemente descartará la asignación silenciosamente. Incluso si hubiera un setter válido, nuestro getter personalizado está codificado para retornar solo `2`, así que la operación de set sería inútil.

Para hacer este escenario más sensato, las propiedades también deberían ser definidas con setters, que anulan la operación `[[Put]]` por defecto (también conocida como asignación), por-propiedad, justo como esperarías. Casi con certeza querrás siempre declarar tanto getter como setter (tener solo uno u otro a menudo lleva a comportamiento inesperado/sorprendente):

```js
var myObject = {
	// define un getter para `a`
	get a() {
		return this._a_;
	},

	// define un setter para `a`
	set a(val) {
		this._a_ = val * 2;
	}
};

myObject.a = 2;

myObject.a; // 4
```

**Nota:** En este ejemplo, realmente almacenamos el valor especificado `2` de la asignación (operación `[[Put]]`) en otra variable `_a_`. El nombre `_a_` es puramente por convención para este ejemplo y no implica nada especial sobre su comportamiento -- es una propiedad normal como cualquier otra.

### Existencia

Mostramos anteriormente que un acceso de propiedad como `myObject.a` puede resultar en un valor `undefined` ya sea que el `undefined` explícito esté almacenado allí o que la propiedad `a` no exista en absoluto. Entonces, si el valor es el mismo en ambos casos, ¿cómo los distinguimos de otra forma?

Podemos preguntar a un objeto si tiene una cierta propiedad *sin* pedir obtener el valor de esa propiedad:

```js
var myObject = {
	a: 2
};

("a" in myObject);				// true
("b" in myObject);				// false

myObject.hasOwnProperty( "a" );	// true
myObject.hasOwnProperty( "b" );	// false
```

El operador `in` verificará si la propiedad está *en* el objeto, o si existe en cualquier nivel superior del recorrido de la cadena de objetos `[[Prototype]]` (ver Capítulo 5). En contraste, `hasOwnProperty(..)` verifica si *solo* `myObject` tiene la propiedad o no, y *no* consultará la cadena `[[Prototype]]`. Volveremos a las diferencias importantes entre estas dos operaciones en el Capítulo 5 cuando exploremos los `[[Prototype]]`s en detalle.

`hasOwnProperty(..)` es accesible para todos los objetos normales vía delegación a `Object.prototype` (ver Capítulo 5). Pero es posible crear un objeto que no se vincule a `Object.prototype` (vía `Object.create(null)` -- ver Capítulo 5). En este caso, una llamada de método como `myObject.hasOwnProperty(..)` fallaría.

En ese escenario, una forma más robusta de realizar tal verificación es `Object.prototype.hasOwnProperty.call(myObject,"a")`, que toma prestado el método base `hasOwnProperty(..)` y usa *enlace explícito de `this`* (ver Capítulo 2) para aplicarlo contra nuestro `myObject`.

**Nota:** El operador `in` tiene la apariencia de que verificará la existencia de un *valor* dentro de un contenedor, pero realmente verifica la existencia de un nombre de propiedad. Esta diferencia es importante de notar con respecto a los arrays, ya que la tentación de intentar una verificación como `4 in [2, 4, 6]` es fuerte, pero esto no se comportará como se espera.

#### Enumeración

Anteriormente, explicamos brevemente la idea de "enumerabilidad" cuando vimos la característica del descriptor de propiedad `enumerable`. Revisitemos eso y examinémoslo con más detalle.

```js
var myObject = { };

Object.defineProperty(
	myObject,
	"a",
	// hace `a` enumerable, como es normal
	{ enumerable: true, value: 2 }
);

Object.defineProperty(
	myObject,
	"b",
	// hace `b` NO enumerable
	{ enumerable: false, value: 3 }
);

myObject.b; // 3
("b" in myObject); // true
myObject.hasOwnProperty( "b" ); // true

// .......

for (var k in myObject) {
	console.log( k, myObject[k] );
}
// "a" 2
```

Notarás que `myObject.b` de hecho **existe** y tiene un valor accesible, pero no aparece en un bucle `for..in` (aunque, sorprendentemente, **sí** es revelado por la verificación de existencia del operador `in`). Eso es porque "enumerable" básicamente significa "será incluido si las propiedades del objeto son iteradas".

**Nota:** Los bucles `for..in` aplicados a arrays pueden dar resultados algo inesperados, en que la enumeración de un array incluirá no solo todos los índices numéricos, sino también cualquier propiedad enumerable. Es buena idea usar bucles `for..in` *solo* en objetos, y bucles `for` tradicionales con iteración de índice numérico para los valores almacenados en arrays.

Otra forma en que las propiedades enumerables y no enumerables pueden ser distinguidas:

```js
var myObject = { };

Object.defineProperty(
	myObject,
	"a",
	// hace `a` enumerable, como es normal
	{ enumerable: true, value: 2 }
);

Object.defineProperty(
	myObject,
	"b",
	// hace `b` no enumerable
	{ enumerable: false, value: 3 }
);

myObject.propertyIsEnumerable( "a" ); // true
myObject.propertyIsEnumerable( "b" ); // false

Object.keys( myObject ); // ["a"]
Object.getOwnPropertyNames( myObject ); // ["a", "b"]
```

`propertyIsEnumerable(..)` verifica si el nombre de propiedad dado existe *directamente* en el objeto y si también es `enumerable:true`.

`Object.keys(..)` retorna un array de todas las propiedades enumerables, mientras que `Object.getOwnPropertyNames(..)` retorna un array de *todas* las propiedades, enumerables o no.

Mientras que `in` vs. `hasOwnProperty(..)` difieren en si consultan la cadena `[[Prototype]]` o no, `Object.keys(..)` y `Object.getOwnPropertyNames(..)` ambos inspeccionan *solo* el objeto directo especificado.

Actualmente no hay forma integrada de obtener una lista de **todas las propiedades** que sea equivalente a lo que la prueba del operador `in` consultaría (recorrer todas las propiedades en toda la cadena `[[Prototype]]`, como se explica en el Capítulo 5). Podrías aproximar tal utilidad recorriendo recursivamente la cadena `[[Prototype]]` de un objeto, y para cada nivel, capturando la lista de `Object.keys(..)` -- solo propiedades enumerables.
