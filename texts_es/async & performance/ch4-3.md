## Concurrencia con Generators

Como discutimos tanto en el Capítulo 1 como anteriormente en este capítulo, dos "procesos" ejecutándose simultáneamente pueden intercalar cooperativamente sus operaciones, y muchas veces esto puede *yield* (juego de palabras intencionado) expresiones de asincronía muy poderosas.

Francamente, nuestros ejemplos anteriores de intercalación concurrente de múltiples generators mostraron cómo hacerlo realmente confuso. Pero insinuamos que hay lugares donde esta capacidad es bastante útil.

Recuerda un escenario que vimos en el Capítulo 1, donde dos manejadores de respuesta Ajax simultáneos diferentes necesitaban coordinarse entre sí para asegurarse de que la comunicación de datos no fuera una condición de carrera. Colocamos las respuestas en el array `res` así:

```js
function response(data) {
	if (data.url == "http://some.url.1") {
		res[0] = data;
	}
	else if (data.url == "http://some.url.2") {
		res[1] = data;
	}
}
```

Pero ¿cómo podemos usar múltiples generators concurrentemente para este escenario?

```js
// `request(..)` es una utilidad Ajax consciente de Promises

var res = [];

function *reqData(url) {
	res.push(
		yield request( url )
	);
}
```

**Nota:** Vamos a usar dos instancias del generator `*reqData(..)` aquí, pero no hay diferencia con ejecutar una sola instancia de dos generators diferentes; ambos enfoques se razonan de manera idéntica. Veremos dos generators diferentes coordinándose en un momento.

En lugar de tener que ordenar manualmente las asignaciones `res[0]` y `res[1]`, usaremos ordenamiento coordinado para que `res.push(..)` coloque correctamente los valores en el orden esperado y predecible. La lógica expresada debería sentirse un poco más limpia.

Pero, ¿cómo orquestaremos realmente esta interacción? Primero, hagámoslo manualmente, con Promises:

```js
var it1 = reqData( "http://some.url.1" );
var it2 = reqData( "http://some.url.2" );

var p1 = it1.next().value;
var p2 = it2.next().value;

p1
.then( function(data){
	it1.next( data );
	return p2;
} )
.then( function(data){
	it2.next( data );
} );
```

Las dos instancias de `*reqData(..)` se inician ambas para hacer sus peticiones Ajax, luego se pausan con `yield`. Entonces elegimos reanudar la primera instancia cuando `p1` se resuelve, y luego la resolución de `p2` reiniciará la segunda instancia. De esta manera, usamos orquestación de Promises para asegurar que `res[0]` tendrá la primera respuesta y `res[1]` tendrá la segunda respuesta.

Pero francamente, esto es terriblemente manual, y realmente no permite que los generators se orquesten a sí mismos, que es donde reside el verdadero poder. Intentémoslo de una manera diferente:

```js
// `request(..)` es una utilidad Ajax consciente de Promises

var res = [];

function *reqData(url) {
	var data = yield request( url );

	// transferir control
	yield;

	res.push( data );
}

var it1 = reqData( "http://some.url.1" );
var it2 = reqData( "http://some.url.2" );

var p1 = it1.next().value;
var p2 = it2.next().value;

p1.then( function(data){
	it1.next( data );
} );

p2.then( function(data){
	it2.next( data );
} );

Promise.all( [p1,p2] )
.then( function(){
	it1.next();
	it2.next();
} );
```

OK, esto es un poco mejor (¡aunque todavía manual!), porque ahora las dos instancias de `*reqData(..)` se ejecutan verdaderamente de forma concurrente, y (al menos para la primera parte) independientemente.

En el fragmento anterior, la segunda instancia no recibía sus datos hasta después de que la primera instancia hubiera terminado completamente. Pero aquí, ambas instancias reciben sus datos tan pronto como sus respectivas respuestas regresan, y entonces cada instancia hace otro `yield` con propósitos de transferencia de control. Luego elegimos en qué orden reanudarlas en el manejador `Promise.all([ .. ])`.

Lo que puede no ser tan obvio es que este enfoque sugiere una forma más fácil para una utilidad reutilizable, debido a la simetría. Podemos hacerlo aún mejor. Imaginemos usar una utilidad llamada `runAll(..)`:

```js
// `request(..)` es una utilidad Ajax consciente de Promises

var res = [];

runAll(
	function*(){
		var p1 = request( "http://some.url.1" );

		// transferir control
		yield;

		res.push( yield p1 );
	},
	function*(){
		var p2 = request( "http://some.url.2" );

		// transferir control
		yield;

		res.push( yield p2 );
	}
);
```

**Nota:** No estamos incluyendo un listado de código para `runAll(..)` ya que no solo es lo suficientemente largo como para atascar el texto, sino que es una extensión de la lógica que ya implementamos en `run(..)` anteriormente. Así que, como buen ejercicio complementario para el lector, intenta evolucionar el código de `run(..)` para que funcione como el imaginado `runAll(..)`. Además, mi librería *asynquence* proporciona una utilidad `runner(..)` mencionada anteriormente con este tipo de capacidad ya incorporada, y se discutirá en el Apéndice A de este libro.

Así es como operaría el procesamiento dentro de `runAll(..)`:

1. El primer generator obtiene una promise para la primera respuesta Ajax de `"http://some.url.1"`, luego hace `yield` del control de vuelta a la utilidad `runAll(..)`.
2. El segundo generator se ejecuta y hace lo mismo para `"http://some.url.2"`, haciendo `yield` del control de vuelta a la utilidad `runAll(..)`.
3. El primer generator se reanuda, y luego hace `yield` de su promise `p1`. La utilidad `runAll(..)` hace lo mismo en este caso que nuestro `run(..)` anterior, en que espera a que esa promise se resuelva, luego reanuda el mismo generator (¡sin transferencia de control!). Cuando `p1` se resuelve, `runAll(..)` reanuda el primer generator de nuevo con ese valor de resolución, y entonces `res[0]` recibe su valor. Cuando el primer generator termina, eso es una transferencia implícita de control.
4. El segundo generator se reanuda, hace `yield` de su promise `p2`, y espera a que se resuelva. Una vez que lo hace, `runAll(..)` reanuda el segundo generator con ese valor, y `res[1]` queda establecido.

En este ejemplo en ejecución, usamos una variable externa llamada `res` para almacenar los resultados de las dos respuestas Ajax diferentes -- esa es nuestra coordinación de concurrencia haciéndolo posible.

Pero podría ser bastante útil extender aún más `runAll(..)` para proporcionar un espacio de variables interno para que las múltiples instancias de generators *compartan*, como un objeto vacío que llamaremos `data` a continuación. También, podría tomar valores que no son Promises que se hacen `yield` y pasarlos al siguiente generator.

Considera:

```js
// `request(..)` es una utilidad Ajax consciente de Promises

runAll(
	function*(data){
		data.res = [];

		// transferir control (y pasar mensaje)
		var url1 = yield "http://some.url.2";

		var p1 = request( url1 ); // "http://some.url.1"

		// transferir control
		yield;

		data.res.push( yield p1 );
	},
	function*(data){
		// transferir control (y pasar mensaje)
		var url2 = yield "http://some.url.1";

		var p2 = request( url2 ); // "http://some.url.2"

		// transferir control
		yield;

		data.res.push( yield p2 );
	}
);
```

En esta formulación, los dos generators no solo están coordinando la transferencia de control, sino que realmente se comunican entre sí, tanto a través de `data.res` como de los mensajes con `yield` que intercambian los valores `url1` y `url2`. ¡Eso es increíblemente poderoso!

Tal realización también sirve como base conceptual para una técnica de asincronía más sofisticada llamada CSP (Communicating Sequential Processes), que cubriremos en el Apéndice B de este libro.

## Thunks

Hasta ahora, hemos asumido que hacer `yield` de una Promise desde un generator -- y hacer que esa Promise reanude el generator mediante una utilidad auxiliar como `run(..)` -- era la mejor manera posible de gestionar la asincronía con generators. Para ser claros, lo es.

Pero omitimos otro patrón que tiene una adopción moderadamente amplia, así que en interés de la completitud le echaremos un breve vistazo.

En ciencias de la computación en general, hay un viejo concepto pre-JS llamado "thunk." Sin adentrarnos en la naturaleza histórica, una expresión estrecha de un thunk en JS es una función que -- sin ningún parámetro -- está conectada para llamar a otra función.

En otras palabras, envuelves una definición de función alrededor de una llamada a función -- con cualquier parámetro que necesite -- para *diferir* la ejecución de esa llamada, y esa función envolvente es un thunk. Cuando más tarde ejecutas el thunk, terminas llamando a la función original.

Por ejemplo:

```js
function foo(x,y) {
	return x + y;
}

function fooThunk() {
	return foo( 3, 4 );
}

// después

console.log( fooThunk() );	// 7
```

Así que, un thunk síncrono es bastante directo. ¿Pero qué hay de un thunk asíncrono? Podemos esencialmente extender la definición estrecha de thunk para incluir que reciba un callback.

Considera:

```js
function foo(x,y,cb) {
	setTimeout( function(){
		cb( x + y );
	}, 1000 );
}

function fooThunk(cb) {
	foo( 3, 4, cb );
}

// después

fooThunk( function(sum){
	console.log( sum );		// 7
} );
```

Como puedes ver, `fooThunk(..)` solo espera un parámetro `cb(..)`, ya que tiene los valores `3` y `4` (para `x` e `y`, respectivamente) pre-especificados y listos para pasar a `foo(..)`. Un thunk simplemente está esperando pacientemente la última pieza que necesita para hacer su trabajo: el callback.

Sin embargo, no querrás crear thunks manualmente. Así que, inventemos una utilidad que haga este envolvimiento por nosotros.

Considera:

```js
function thunkify(fn) {
	var args = [].slice.call( arguments, 1 );
	return function(cb) {
		args.push( cb );
		return fn.apply( null, args );
	};
}

var fooThunk = thunkify( foo, 3, 4 );

// después

fooThunk( function(sum) {
	console.log( sum );		// 7
} );
```

**Consejo:** Aquí asumimos que la firma de la función original (`foo(..)`) espera su callback en la última posición, con cualquier otro parámetro viniendo antes. Este es un "estándar" bastante ubicuo para estándares de funciones JS asíncronas. Podrías llamarlo estilo "callback-último". Si por alguna razón tuvieras la necesidad de manejar firmas de estilo "callback-primero", simplemente harías una utilidad que usara `args.unshift(..)` en lugar de `args.push(..)`.

La formulación anterior de `thunkify(..)` toma tanto la referencia de función `foo(..)`, como cualquier parámetro que necesite, y devuelve el thunk en sí (`fooThunk(..)`). Sin embargo, ese no es el enfoque típico que encontrarás para thunks en JS.

En lugar de que `thunkify(..)` cree el thunk en sí, típicamente -- si no desconcertantemente -- la utilidad `thunkify(..)` produciría una función que produce thunks.

Uhhhh... sí.

Considera:

```js
function thunkify(fn) {
	return function() {
		var args = [].slice.call( arguments );
		return function(cb) {
			args.push( cb );
			return fn.apply( null, args );
		};
	};
}
```

La diferencia principal aquí es la capa extra `return function() { .. }`. Así es como difiere su uso:

```js
var whatIsThis = thunkify( foo );

var fooThunk = whatIsThis( 3, 4 );

// después

fooThunk( function(sum) {
	console.log( sum );		// 7
} );
```

Obviamente, la gran pregunta que implica este fragmento es ¿cómo se llama propiamente `whatIsThis`? No es el thunk, es la cosa que producirá thunks a partir de llamadas a `foo(..)`. Es algo así como una "fábrica" para "thunks." No parece haber ningún tipo de acuerdo estándar para nombrar tal cosa.

Así que, mi propuesta es "thunkory" ("thunk" + "factory"). Entonces, `thunkify(..)` produce un thunkory, y un thunkory produce thunks. Ese razonamiento es simétrico a mi propuesta de "promisory" en el Capítulo 3:

```js
var fooThunkory = thunkify( foo );

var fooThunk1 = fooThunkory( 3, 4 );
var fooThunk2 = fooThunkory( 5, 6 );

// después

fooThunk1( function(sum) {
	console.log( sum );		// 7
} );

fooThunk2( function(sum) {
	console.log( sum );		// 11
} );
```

**Nota:** El ejemplo continuo de `foo(..)` espera un estilo de callback que no es "estilo error-primero". Por supuesto, "estilo error-primero" es mucho más común. Si `foo(..)` tuviera algún tipo de expectativa legítima de producción de errores, podríamos cambiarlo para esperar y usar un callback error-primero. Nada de la maquinaria subsiguiente de `thunkify(..)` se preocupa por qué estilo de callback se asume. La única diferencia en el uso sería `fooThunk1(function(err,sum){..`.

Exponer el método thunkory -- en lugar de cómo el `thunkify(..)` anterior oculta este paso intermedio -- puede parecer una complicación innecesaria. Pero en general, es bastante útil crear thunkories al principio de tu programa para envolver métodos de API existentes, y luego poder pasar y llamar esos thunkories cuando necesites thunks. Los dos pasos distintos preservan una separación más limpia de capacidades.

Para ilustrar:

```js
// más limpio:
var fooThunkory = thunkify( foo );

var fooThunk1 = fooThunkory( 3, 4 );
var fooThunk2 = fooThunkory( 5, 6 );

// en lugar de:
var fooThunk1 = thunkify( foo, 3, 4 );
var fooThunk2 = thunkify( foo, 5, 6 );
```

Independientemente de si te gusta tratar con los thunkories explícitamente o no, el uso de los thunks `fooThunk1(..)` y `fooThunk2(..)` permanece igual.

### s/promise/thunk/

Entonces, ¿qué tiene que ver todo esto de los thunks con los generators?

Comparando thunks con promises en general: no son directamente intercambiables ya que no son equivalentes en comportamiento. Las Promises son vastamente más capaces y confiables que los thunks simples.

Pero en otro sentido, ambos pueden verse como una solicitud de un valor, que puede ser asíncrono en su respuesta.

Recuerda del Capítulo 3 que definimos una utilidad para promisificar una función, que llamamos `Promise.wrap(..)` -- ¡también podríamos haberla llamado `promisify(..)`! Esta utilidad de envolvimiento en Promise no produce Promises; produce promisories que a su vez producen Promises. Esto es completamente simétrico a los thunkories y thunks que se están discutiendo actualmente.

Para ilustrar la simetría, primero alteremos el ejemplo continuo de `foo(..)` de antes para asumir un callback "estilo error-primero":

```js
function foo(x,y,cb) {
	setTimeout( function(){
		// asumimos `cb(..)` como "estilo error-primero"
		cb( null, x + y );
	}, 1000 );
}
```

Ahora, compararemos usando `thunkify(..)` y `promisify(..)` (también conocido como `Promise.wrap(..)` del Capítulo 3):

```js
// simétrico: construyendo el formulador de preguntas
var fooThunkory = thunkify( foo );
var fooPromisory = promisify( foo );

// simétrico: haciendo la pregunta
var fooThunk = fooThunkory( 3, 4 );
var fooPromise = fooPromisory( 3, 4 );

// obtener la respuesta del thunk
fooThunk( function(err,sum){
	if (err) {
		console.error( err );
	}
	else {
		console.log( sum );		// 7
	}
} );

// obtener la respuesta de la promise
fooPromise
.then(
	function(sum){
		console.log( sum );		// 7
	},
	function(err){
		console.error( err );
	}
);
```

Tanto el thunkory como el promisory están esencialmente haciendo una pregunta (por un valor), y respectivamente el thunk `fooThunk` y la promise `fooPromise` representan las respuestas futuras a esa pregunta. Presentado bajo esa luz, la simetría es clara.

Con esa perspectiva en mente, podemos ver que generators que hacen `yield` de Promises para asincronía podrían en su lugar hacer `yield` de thunks para asincronía. Todo lo que necesitaríamos es una utilidad `run(..)` más inteligente (como la anterior) que pueda no solo buscar y conectarse a una Promise a la que se hizo `yield`, sino también proporcionar un callback a un thunk al que se hizo `yield`.

Considera:

```js
function *foo() {
	var val = yield request( "http://some.url.1" );
	console.log( val );
}

run( foo );
```

En este ejemplo, `request(..)` podría ser un promisory que devuelve una promise, o un thunkory que devuelve un thunk. Desde la perspectiva de lo que está sucediendo dentro de la lógica del código del generator, no nos importa ese detalle de implementación, ¡lo cual es bastante poderoso!

Así que, `request(..)` podría ser cualquiera de:

```js
// promisory `request(..)` (ver Capítulo 3)
var request = Promise.wrap( ajax );

// vs.

// thunkory `request(..)`
var request = thunkify( ajax );
```

Finalmente, como parche consciente de thunks a nuestra utilidad `run(..)` anterior, necesitaríamos lógica como esta:

```js
// ..
// ¿recibimos un thunk de vuelta?
else if (typeof next.value == "function") {
	return new Promise( function(resolve,reject){
		// llamar al thunk con un callback estilo error-primero
		next.value( function(err,msg) {
			if (err) {
				reject( err );
			}
			else {
				resolve( msg );
			}
		} );
	} )
	.then(
		handleNext,
		function handleErr(err) {
			return Promise.resolve(
				it.throw( err )
			)
			.then( handleResult );
		}
	);
}
```

Ahora, nuestros generators pueden llamar promisories para hacer `yield` de Promises, o llamar thunkories para hacer `yield` de thunks, y en cualquier caso, `run(..)` manejaría ese valor y lo usaría para esperar la completación y reanudar el generator.

En cuanto a simetría, estos dos enfoques se ven idénticos. Sin embargo, debemos señalar que eso es verdad solo desde la perspectiva de Promises o thunks representando la continuación de valor futuro de un generator.

Desde la perspectiva más amplia, los thunks no tienen en sí mismos casi ninguna de las garantías de confiabilidad o composabilidad con las que las Promises están diseñadas. Usar un thunk como sustituto de una Promise en este patrón particular de asincronía con generators es factible pero debería verse como menos que ideal cuando se compara con todos los beneficios que las Promises ofrecen (ver Capítulo 3).

Si tienes la opción, prefiere `yield pr` en lugar de `yield th`. Pero no hay nada malo en tener una utilidad `run(..)` que pueda manejar ambos tipos de valor.

**Nota:** La utilidad `runner(..)` en mi librería *asynquence*, que se discutirá en el Apéndice A, maneja `yield`s de Promises, thunks y secuencias *asynquence*.

## Generators Pre-ES6

Esperamos que estés convencido ahora de que los generators son una adición muy importante a la caja de herramientas de programación asíncrona. Pero es una nueva sintaxis en ES6, lo que significa que no puedes simplemente hacer polyfill de generators como puedes con Promises (que son solo una nueva API). Entonces, ¿qué podemos hacer para llevar generators a nuestro JS del navegador si no tenemos el lujo de ignorar los navegadores pre-ES6?

Para todas las nuevas extensiones de sintaxis en ES6, hay herramientas -- el término más común para ellas es transpiladores, por trans-compiladores -- que pueden tomar tu sintaxis ES6 y transformarla en código pre-ES6 equivalente (¡pero obviamente más feo!). Así que, los generators pueden ser transpilados en código que tendrá el mismo comportamiento pero funcionará en ES5 y versiones anteriores.

Pero, ¿cómo? La "magia" de `yield` no suena obviamente como código que sea fácil de transpilar. En realidad insinuamos una solución en nuestra discusión anterior sobre *iterators* basados en closures.

### Transformación Manual

Antes de discutir los transpiladores, derivemos cómo funcionaría la transpilación manual en el caso de los generators. Esto no es solo un ejercicio académico, porque hacerlo realmente ayudará a reforzar aún más cómo funcionan.

Considera:

```js
// `request(..)` es una utilidad Ajax consciente de Promises

function *foo(url) {
	try {
		console.log( "requesting:", url );
		var val = yield request( url );
		console.log( val );
	}
	catch (err) {
		console.log( "Oops:", err );
		return false;
	}
}

var it = foo( "http://some.url.1" );
```

Lo primero que hay que observar es que todavía necesitaremos una función normal `foo()` que pueda ser llamada, y todavía necesitará devolver un *iterator*. Así que, esbocemos la transformación sin generator:

```js
function foo(url) {

	// ..

	// crear y devolver un iterator
	return {
		next: function(v) {
			// ..
		},
		throw: function(e) {
			// ..
		}
	};
}

var it = foo( "http://some.url.1" );
```

Lo siguiente que hay que observar es que un generator hace su "magia" suspendiendo su alcance/estado, pero podemos emular eso con closure de funciones (ver el título *Scope & Closures* de esta serie). Para entender cómo escribir tal código, primero anotaremos diferentes partes de nuestro generator con valores de estado:

```js
// `request(..)` es una utilidad Ajax consciente de Promises

function *foo(url) {
	// ESTADO *1*

	try {
		console.log( "requesting:", url );
		var TMP1 = request( url );

		// ESTADO *2*
		var val = yield TMP1;
		console.log( val );
	}
	catch (err) {
		// ESTADO *3*
		console.log( "Oops:", err );
		return false;
	}
}
```

**Nota:** Para una ilustración más precisa, separamos la sentencia `val = yield request..` en dos partes, usando la variable temporal `TMP1`. `request(..)` ocurre en el estado `*1*`, y la asignación de su valor de completación a `val` ocurre en el estado `*2*`. Nos desharemos de ese `TMP1` intermedio cuando convirtamos el código a su equivalente sin generator.

En otras palabras, `*1*` es el estado inicial, `*2*` es el estado si `request(..)` tiene éxito, y `*3*` es el estado si `request(..)` falla. Probablemente puedas imaginar cómo cualquier paso extra de `yield` simplemente se codificaría como estados adicionales.

Volviendo a nuestro generator transpilado, definamos una variable `state` en el closure que podemos usar para rastrear el estado:

```js
function foo(url) {
	// gestionar estado del generator
	var state;

	// ..
}
```

Ahora, definamos una función interna llamada `process(..)` dentro del closure que maneja cada estado, usando una sentencia `switch`:

```js
// `request(..)` es una utilidad Ajax consciente de Promises

function foo(url) {
	// gestionar estado del generator
	var state;

	// declaraciones de variables de todo el generator
	var val;

	function process(v) {
		switch (state) {
			case 1:
				console.log( "requesting:", url );
				return request( url );
			case 2:
				val = v;
				console.log( val );
				return;
			case 3:
				var err = v;
				console.log( "Oops:", err );
				return false;
		}
	}

	// ..
}
```

Cada estado en nuestro generator está representado por su propio `case` en la sentencia `switch`. `process(..)` será llamado cada vez que necesitemos procesar un nuevo estado. Volveremos a cómo funciona esto en un momento.

Para cualquier declaración de variable de todo el generator (`val`), las movemos a una declaración `var` fuera de `process(..)` para que puedan sobrevivir a múltiples llamadas a `process(..)`. Pero la variable `err` con "alcance de bloque" solo se necesita para el estado `*3*`, así que la dejamos en su lugar.

En el estado `*1*`, en lugar de `yield request(..)`, hicimos `return request(..)`. En el estado terminal `*2*`, no había un `return` explícito, así que simplemente hacemos un `return;` que es lo mismo que `return undefined`. En el estado terminal `*3*`, había un `return false`, así que lo preservamos.

Ahora necesitamos definir el código en las funciones del *iterator* para que llamen a `process(..)` apropiadamente:

```js
function foo(url) {
	// gestionar estado del generator
	var state;

	// declaraciones de variables de todo el generator
	var val;

	function process(v) {
		switch (state) {
			case 1:
				console.log( "requesting:", url );
				return request( url );
			case 2:
				val = v;
				console.log( val );
				return;
			case 3:
				var err = v;
				console.log( "Oops:", err );
				return false;
		}
	}

	// crear y devolver un iterator
	return {
		next: function(v) {
			// estado inicial
			if (!state) {
				state = 1;
				return {
					done: false,
					value: process()
				};
			}
			// yield reanudado exitosamente
			else if (state == 1) {
				state = 2;
				return {
					done: true,
					value: process( v )
				};
			}
			// generator ya completado
			else {
				return {
					done: true,
					value: undefined
				};
			}
		},
		"throw": function(e) {
			// el único manejo de error explícito está en
			// el estado *1*
			if (state == 1) {
				state = 3;
				return {
					done: true,
					value: process( e )
				};
			}
			// de lo contrario, un error no será manejado,
			// así que simplemente lo lanzamos de vuelta
			else {
				throw e;
			}
		}
	};
}
```

¿Cómo funciona este código?

1. La primera llamada al `next()` del *iterator* movería el generator del estado no inicializado al estado `1`, y luego llamaría a `process()` para manejar ese estado. El valor de retorno de `request(..)`, que es la promise para la respuesta Ajax, se devuelve como la propiedad `value` de la llamada a `next()`.
2. Si la petición Ajax tiene éxito, la segunda llamada a `next(..)` debería enviar el valor de respuesta Ajax, lo que mueve nuestro estado a `2`. `process(..)` se llama de nuevo (esta vez con el valor de respuesta Ajax pasado), y la propiedad `value` devuelta por `next(..)` será `undefined`.
3. Sin embargo, si la petición Ajax falla, `throw(..)` debería ser llamado con el error, lo que movería el estado de `1` a `3` (en lugar de `2`). De nuevo se llama a `process(..)`, esta vez con el valor de error. Ese `case` devuelve `false`, que se establece como la propiedad `value` devuelta de la llamada a `throw(..)`.

Desde el exterior -- es decir, interactuando solo con el *iterator* -- esta función normal `foo(..)` funciona prácticamente igual que el generator `*foo(..)` habría funcionado. ¡Así que hemos efectivamente "transpilado" nuestro generator ES6 a compatibilidad pre-ES6!

Entonces podríamos instanciar manualmente nuestro generator y controlar su iterator -- llamando `var it = foo("..")` e `it.next(..)` y demás -- o mejor, podríamos pasarlo a nuestra utilidad `run(..)` previamente definida como `run(foo,"..")`.

### Transpilación Automática

El ejercicio anterior de derivar manualmente una transformación de nuestro generator ES6 a equivalente pre-ES6 nos enseña cómo funcionan los generators conceptualmente. Pero esa transformación fue realmente intrincada y muy poco portable a otros generators en nuestro código. Sería bastante impráctico hacer este trabajo a mano, y anularía completamente todo el beneficio de los generators.

Pero afortunadamente, ya existen varias herramientas que pueden convertir automáticamente los generators ES6 en cosas como las que derivamos en la sección anterior. No solo hacen el trabajo pesado por nosotros, sino que también manejan varias complicaciones que pasamos por alto.

Una de esas herramientas es regenerator (https://facebook.github.io/regenerator/), de la gente inteligente de Facebook.

Si usamos regenerator para transpilar nuestro generator anterior, aquí está el código producido (al momento de escribir esto):

```js
// `request(..)` es una utilidad Ajax consciente de Promises

var foo = regeneratorRuntime.mark(function foo(url) {
    var val;

    return regeneratorRuntime.wrap(function foo$(context$1$0) {
        while (1) switch (context$1$0.prev = context$1$0.next) {
        case 0:
            context$1$0.prev = 0;
            console.log( "requesting:", url );
            context$1$0.next = 4;
            return request( url );
        case 4:
            val = context$1$0.sent;
            console.log( val );
            context$1$0.next = 12;
            break;
        case 8:
            context$1$0.prev = 8;
            context$1$0.t0 = context$1$0.catch(0);
            console.log("Oops:", context$1$0.t0);
            return context$1$0.abrupt("return", false);
        case 12:
        case "end":
            return context$1$0.stop();
        }
    }, foo, this, [[0, 8]]);
});
```

Hay algunas similitudes obvias aquí con nuestra derivación manual, como las sentencias `switch` / `case`, e incluso vemos `val` sacado del closure tal como hicimos nosotros.

Por supuesto, una compensación es que la transpilación de regenerator requiere una librería auxiliar `regeneratorRuntime` que contiene toda la lógica reutilizable para gestionar un generator / *iterator* general. Mucho de ese código repetitivo se ve diferente de nuestra versión, pero aún así, los conceptos se pueden ver, como con `context$1$0.next = 4` rastreando el siguiente estado para el generator.

La conclusión principal es que los generators no están restringidos a ser útiles solo en entornos ES6+. Una vez que entiendas los conceptos, puedes emplearlos en todo tu código, y usar herramientas para transformar el código para que sea compatible con entornos más antiguos.

Esto es más trabajo que simplemente usar un polyfill de API `Promise` para Promises pre-ES6, pero el esfuerzo vale totalmente la pena, porque los generators son mucho mejores para expresar el control de flujo asíncrono de una manera razonable, sensata, de aspecto síncrono y secuencial.

¡Una vez que te enganches a los generators, nunca querrás volver al infierno de callbacks de espagueti asíncrono!

## Repaso

Los generators son un nuevo tipo de función ES6 que no se ejecuta hasta completarse como las funciones normales. En su lugar, el generator puede ser pausado a mitad de completación (preservando enteramente su estado), y puede ser reanudado más tarde desde donde lo dejó.

Este intercambio de pausa/reanudación es cooperativo en lugar de preventivo, lo que significa que el generator tiene la capacidad exclusiva de pausarse a sí mismo, usando la palabra clave `yield`, y sin embargo el *iterator* que controla el generator tiene la capacidad exclusiva (vía `next(..)`) de reanudar el generator.

La dualidad `yield` / `next(..)` no es solo un mecanismo de control, es en realidad un mecanismo de paso de mensajes bidireccional. Una expresión `yield ..` esencialmente pausa esperando un valor, y la siguiente llamada a `next(..)` pasa un valor (o `undefined` implícito) de vuelta a esa expresión `yield` pausada.

El beneficio clave de los generators relacionado con el control de flujo asíncrono es que el código dentro de un generator expresa una secuencia de pasos para la tarea de una manera naturalmente síncrona/secuencial. El truco es que esencialmente ocultamos la asincronía potencial detrás de la palabra clave `yield` -- moviendo la asincronía al código donde el *iterator* del generator es controlado.

En otras palabras, los generators preservan un patrón de código secuencial, síncrono y bloqueante para código asíncrono, lo que permite a nuestros cerebros razonar sobre el código de manera mucho más natural, abordando uno de los dos inconvenientes clave de la asincronía basada en callbacks.
