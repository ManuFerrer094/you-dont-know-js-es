## Flujo de Cadena

Ya hemos insinuado esto un par de veces, pero las Promesas no son solo un mecanismo para una operación de un solo paso tipo *esto-luego-aquello*. Ese es el bloque de construcción, por supuesto, pero resulta que podemos encadenar múltiples Promesas juntas para representar una secuencia de pasos asíncronos.

La clave para que esto funcione se basa en dos comportamientos intrínsecos de las Promesas:

* Cada vez que llamas a `then(..)` en una Promesa, crea y devuelve una nueva Promesa, que podemos *encadenar*.
* Cualquier valor que devuelvas del callback de cumplimiento de la llamada `then(..)` (el primer parámetro) se establece automáticamente como el cumplimiento de la Promesa *encadenada* (del primer punto).

Primero ilustremos qué significa eso, y *luego* derivaremos cómo eso nos ayuda a crear secuencias asíncronas de control de flujo. Considera lo siguiente:

```js
var p = Promise.resolve( 21 );

var p2 = p.then( function(v){
	console.log( v );	// 21

	// cumplir `p2` con el valor `42`
	return v * 2;
} );

// encadenar de `p2`
p2.then( function(v){
	console.log( v );	// 42
} );
```

Al devolver `v * 2` (es decir, `42`), cumplimos la promesa `p2` que la primera llamada `then(..)` creó y devolvió. Cuando se ejecuta el `then(..)` de `p2`, está recibiendo el cumplimiento de la sentencia `return v * 2`. Por supuesto, `p2.then(..)` crea otra promesa más, que podríamos haber almacenado en una variable `p3`.

Pero es un poco molesto tener que crear una variable intermedia `p2` (o `p3`, etc.). Afortunadamente, podemos simplemente encadenarlas juntas fácilmente:

```js
var p = Promise.resolve( 21 );

p
.then( function(v){
	console.log( v );	// 21

	// cumplir la promesa encadenada con el valor `42`
	return v * 2;
} )
// aquí está la promesa encadenada
.then( function(v){
	console.log( v );	// 42
} );
```

Así que ahora el primer `then(..)` es el primer paso en una secuencia asíncrona, y el segundo `then(..)` es el segundo paso. Esto podría continuar tanto como necesitaras extenderlo. Solo sigue encadenando del `then(..)` anterior con cada Promesa creada automáticamente.

Pero falta algo aquí. ¿Qué pasa si queremos que el paso 2 espere a que el paso 1 haga algo asíncrono? Estamos usando una sentencia `return` inmediata, que cumple inmediatamente la promesa encadenada.

La clave para hacer una secuencia de Promesas verdaderamente capaz de asincronía en cada paso es recordar cómo opera `Promise.resolve(..)` cuando lo que le pasas es una Promesa o thenable en lugar de un valor final. `Promise.resolve(..)` devuelve directamente una Promesa genuina recibida, o desenvuelve el valor de un thenable recibido -- y sigue recursivamente mientras siga desenvolviendo thenables.

El mismo tipo de desenvolvimiento ocurre si devuelves con `return` un thenable o Promesa desde el manejador de cumplimiento (o rechazo). Considera:

```js
var p = Promise.resolve( 21 );

p.then( function(v){
	console.log( v );	// 21

	// crear una promesa y devolverla
	return new Promise( function(resolve,reject){
		// cumplir con el valor `42`
		resolve( v * 2 );
	} );
} )
.then( function(v){
	console.log( v );	// 42
} );
```

Aunque envolvimos `42` en una promesa que devolvimos, aún así fue desenvuelta y terminó como la resolución de la promesa encadenada, de modo que el segundo `then(..)` aún recibió `42`. Si introducimos asincronía en esa promesa envolvente, todo funciona igual de bien:

```js
var p = Promise.resolve( 21 );

p.then( function(v){
	console.log( v );	// 21

	// crear una promesa para devolver
	return new Promise( function(resolve,reject){
		// ¡introducir asincronía!
		setTimeout( function(){
			// cumplir con el valor `42`
			resolve( v * 2 );
		}, 100 );
	} );
} )
.then( function(v){
	// se ejecuta después del retraso de 100ms en el paso anterior
	console.log( v );	// 42
} );
```

¡Eso es increíblemente poderoso! Ahora podemos construir una secuencia de tantos pasos asíncronos como queramos, y cada paso puede retrasar el siguiente paso (¡o no!), según sea necesario.

Por supuesto, el paso de valores de paso a paso en estos ejemplos es opcional. Si no devuelves un valor explícito, se asume un `undefined` implícito, y las promesas aún se encadenan de la misma manera. Cada resolución de Promesa es simplemente una señal para proceder al siguiente paso.

Para ampliar la ilustración de la cadena, generalicemos una creación de Promesa con retraso (sin mensajes de resolución) en una utilidad que podemos reutilizar para múltiples pasos:

```js
function delay(time) {
	return new Promise( function(resolve,reject){
		setTimeout( resolve, time );
	} );
}

delay( 100 ) // paso 1
.then( function STEP2(){
	console.log( "paso 2 (después de 100ms)" );
	return delay( 200 );
} )
.then( function STEP3(){
	console.log( "paso 3 (después de otros 200ms)" );
} )
.then( function STEP4(){
	console.log( "paso 4 (siguiente Job)" );
	return delay( 50 );
} )
.then( function STEP5(){
	console.log( "paso 5 (después de otros 50ms)" );
} )
...
```

Llamar a `delay(200)` crea una promesa que se cumplirá en 200ms, y luego la devolvemos del callback de cumplimiento del primer `then(..)`, lo que causa que la promesa del segundo `then(..)` espere esa promesa de 200ms.

**Nota:** Como se describió, técnicamente hay dos promesas en ese intercambio: la promesa de retraso de 200ms y la promesa encadenada de la que el segundo `then(..)` encadena. Pero puede resultarte más fácil combinar mentalmente estas dos promesas, porque el mecanismo de Promesas automáticamente fusiona sus estados por ti. En ese sentido, podrías pensar en `return delay(200)` como creando una promesa que reemplaza la promesa encadenada devuelta anteriormente.

Para ser honesto, sin embargo, secuencias de retrasos sin paso de mensajes no son un ejemplo terriblemente útil de control de flujo con Promesas. Veamos un escenario un poco más práctico.

En lugar de temporizadores, consideremos hacer peticiones Ajax:

```js
// asumiendo una utilidad `ajax( {url}, {callback} )`

// Ajax consciente de Promesas
function request(url) {
	return new Promise( function(resolve,reject){
		// el callback de `ajax(..)` debería ser la
		// función `resolve(..)` de nuestra promesa
		ajax( url, resolve );
	} );
}
```

Primero definimos una utilidad `request(..)` que construye una promesa para representar la completación de la llamada `ajax(..)`:

```js
request( "http://some.url.1/" )
.then( function(response1){
	return request( "http://some.url.2/?v=" + response1 );
} )
.then( function(response2){
	console.log( response2 );
} );
```

**Nota:** Los desarrolladores comúnmente se encuentran con situaciones en las que quieren hacer control de flujo asíncrono consciente de Promesas con utilidades que no están habilitadas para Promesas (como `ajax(..)` aquí, que espera un callback). Aunque el mecanismo nativo ES6 `Promise` no resuelve automáticamente este patrón por nosotros, prácticamente todas las bibliotecas de Promesas *sí* lo hacen. Usualmente llaman a este proceso "lifting" o "promisifying" o alguna variación. Volveremos a esta técnica más adelante.

Usando el `request(..)` que devuelve Promesas, creamos el primer paso en nuestra cadena implícitamente al llamarlo con la primera URL, y encadenamos de esa promesa devuelta con el primer `then(..)`.

Una vez que `response1` vuelve, usamos ese valor para construir una segunda URL, y hacemos una segunda llamada `request(..)`. Esa segunda promesa de `request(..)` se devuelve con `return` para que el tercer paso en nuestro control de flujo asíncrono espere a que esa llamada Ajax se complete. Finalmente, imprimimos `response2` una vez que vuelve.

La cadena de Promesas que construimos no es solo un control de flujo que expresa una secuencia asíncrona de múltiples pasos, sino que también actúa como un canal de mensajes para propagar mensajes de paso a paso.

¿Qué pasa si algo sale mal en uno de los pasos de la cadena de Promesas? Un error/excepción es por Promesa, lo que significa que es posible capturar tal error en cualquier punto de la cadena, y esa captura actúa para "reiniciar" la cadena de vuelta a la operación normal en ese punto:

```js
// paso 1:
request( "http://some.url.1/" )

// paso 2:
.then( function(response1){
	foo.bar(); // undefined, ¡error!

	// nunca llega aquí
	return request( "http://some.url.2/?v=" + response1 );
} )

// paso 3:
.then(
	function fulfilled(response2){
		// nunca llega aquí
	},
	// manejador de rechazo para capturar el error
	function rejected(err){
		console.log( err );	// `TypeError` del error `foo.bar()`
		return 42;
	}
)

// paso 4:
.then( function(msg){
	console.log( msg );		// 42
} );
```

Cuando el error ocurre en el paso 2, el manejador de rechazo en el paso 3 lo captura. El valor de retorno (`42` en este fragmento), si existe, de ese manejador de rechazo cumple la promesa para el siguiente paso (4), de modo que la cadena está de vuelta en un estado de cumplimiento.

**Nota:** Como discutimos antes, cuando devuelves una promesa desde un manejador de cumplimiento, se desenvuelve y puede retrasar el siguiente paso. Eso también es cierto para devolver promesas desde manejadores de rechazo, de modo que si el `return 42` en el paso 3 en su lugar devolviera una promesa, esa promesa podría retrasar el paso 4. Una excepción lanzada dentro de cualquiera de los manejadores de cumplimiento o rechazo de una llamada `then(..)` causa que la siguiente promesa (encadenada) sea inmediatamente rechazada con esa excepción.

Si llamas a `then(..)` en una promesa, y solo pasas un manejador de cumplimiento, un manejador de rechazo supuesto se sustituye:

```js
var p = new Promise( function(resolve,reject){
	reject( "Oops" );
} );

var p2 = p.then(
	function fulfilled(){
		// nunca llega aquí
	}
	// manejador de rechazo supuesto, si se omite o
	// se pasa cualquier otro valor no-función
	// function(err) {
	//     throw err;
	// }
);
```

Como puedes ver, el manejador de rechazo supuesto simplemente relanza el error, lo que termina forzando a `p2` (la promesa encadenada) a rechazar con la misma razón de error. En esencia, esto permite que el error continúe propagándose a lo largo de una cadena de Promesas hasta que se encuentre un manejador de rechazo explícitamente definido.

**Nota:** Cubriremos más detalles del manejo de errores con Promesas un poco más adelante, porque hay otros detalles con matices de los que preocuparse.

Si una función válida apropiada no se pasa como el parámetro manejador de cumplimiento a `then(..)`, también hay un manejador por defecto sustituido:

```js
var p = Promise.resolve( 42 );

p.then(
	// manejador de cumplimiento supuesto, si se omite o
	// se pasa cualquier otro valor no-función
	// function(v) {
	//     return v;
	// }
	null,
	function rejected(err){
		// nunca llega aquí
	}
);
```

Como puedes ver, el manejador de cumplimiento por defecto simplemente pasa cualquier valor que recibe al siguiente paso (Promesa).

**Nota:** El patrón `then(null,function(err){ .. })` -- solo manejar rechazos (si los hay) pero dejar pasar los cumplimientos -- tiene un atajo en la API: `catch(function(err){ .. })`. Cubriremos `catch(..)` más completamente en la siguiente sección.

Revisemos brevemente los comportamientos intrínsecos de las Promesas que permiten el control de flujo encadenado:

* Una llamada `then(..)` contra una Promesa automáticamente produce una nueva Promesa para devolver de la llamada.
* Dentro de los manejadores de cumplimiento/rechazo, si devuelves un valor o se lanza una excepción, la nueva Promesa devuelta (encadenable) se resuelve en consecuencia.
* Si el manejador de cumplimiento o rechazo devuelve una Promesa, se desenvuelve, de modo que cualquiera que sea su resolución se convertirá en la resolución de la Promesa encadenada devuelta del `then(..)` actual.

Aunque el control de flujo encadenado es útil, es probablemente más preciso pensarlo como un beneficio colateral de cómo las Promesas se componen (combinan) juntas, en lugar de la intención principal. Como hemos discutido en detalle varias veces ya, las Promesas normalizan la asincronía y encapsulan el estado de valor dependiente del tiempo, y *eso* es lo que nos permite encadenarlas juntas de esta manera útil.

Ciertamente, la expresividad secuencial de la cadena (esto-luego-esto-luego-esto...) es una gran mejora sobre el lío enredado de callbacks que identificamos en el Capítulo 2. Pero todavía hay una buena cantidad de código repetitivo (`then(..)` y `function(){ .. }`) por el que navegar. En el próximo capítulo, veremos un patrón significativamente más agradable para la expresividad del control de flujo secuencial, con generadores.

### Terminología: Resolver, Cumplir y Rechazar

Hay una ligera confusión alrededor de los términos "resolver" (resolve), "cumplir" (fulfill) y "rechazar" (reject) que necesitamos aclarar, antes de que profundices demasiado en aprender sobre Promesas. Primero consideremos el constructor `Promise(..)`:

```js
var p = new Promise( function(X,Y){
	// X() para cumplimiento
	// Y() para rechazo
} );
```

Como puedes ver, se proporcionan dos callbacks (aquí etiquetados `X` e `Y`). El primero *usualmente* se usa para marcar la Promesa como cumplida, y el segundo *siempre* marca la Promesa como rechazada. ¿Pero qué hay del "usualmente", y qué implica eso sobre nombrar esos parámetros con precisión?

En última instancia, es solo tu código de usuario y los nombres de identificadores no son interpretados por el motor para significar nada, así que *técnicamente* no importa; `foo(..)` y `bar(..)` son igualmente funcionales. Pero las palabras que usas pueden afectar no solo cómo estás pensando sobre el código, sino cómo otros desarrolladores en tu equipo pensarán sobre él. Pensar erróneamente sobre código asíncrono cuidadosamente orquestado es casi seguramente peor que las alternativas de callbacks-espagueti.

Así que en realidad sí importa cómo los llamas.

El segundo parámetro es fácil de decidir. Casi toda la literatura usa `reject(..)` como su nombre, y porque eso es exactamente (¡y únicamente!) lo que hace, es una muy buena elección para el nombre. Recomiendo fuertemente que siempre uses `reject(..)`.

Pero hay un poco más de ambigüedad alrededor del primer parámetro, que en la literatura de Promesas frecuentemente se etiqueta como `resolve(..)`. Esa palabra obviamente está relacionada con "resolución", que es lo que se usa en la literatura (incluyendo este libro) para describir establecer un valor/estado final para una Promesa. Ya hemos usado "resolver la Promesa" varias veces para significar ya sea cumplir o rechazar la Promesa.

Pero si este parámetro parece usarse específicamente para cumplir la Promesa, ¿por qué no deberíamos llamarlo `fulfill(..)` en lugar de `resolve(..)` para ser más precisos? Para responder esa pregunta, veamos también dos de los métodos de la API de `Promise`:

```js
var fulfilledPr = Promise.resolve( 42 );

var rejectedPr = Promise.reject( "Oops" );
```

`Promise.resolve(..)` crea una Promesa que está resuelta al valor que se le da. En este ejemplo, `42` es un valor normal, no-Promesa, no-thenable, así que la promesa cumplida `fulfilledPr` se crea para el valor `42`. `Promise.reject("Oops")` crea la promesa rechazada `rejectedPr` por la razón `"Oops"`.

Ahora ilustremos por qué la palabra "resolve" (como en `Promise.resolve(..)`) es inequívoca y de hecho más precisa, si se usa explícitamente en un contexto que podría resultar en cumplimiento o rechazo:

```js
var rejectedTh = {
	then: function(resolved,rejected) {
		rejected( "Oops" );
	}
};

var rejectedPr = Promise.resolve( rejectedTh );
```

Como discutimos anteriormente en este capítulo, `Promise.resolve(..)` devolverá una Promesa genuina recibida directamente, o desenvolverá un thenable recibido. Si ese desenvolvimiento del thenable revela un estado rechazado, la Promesa devuelta por `Promise.resolve(..)` está de hecho en ese mismo estado rechazado.

Así que `Promise.resolve(..)` es un nombre bueno y preciso para el método de la API, porque en realidad puede resultar en cumplimiento o rechazo.

El primer parámetro callback del constructor `Promise(..)` desenvolverá ya sea un thenable (idénticamente a `Promise.resolve(..)`) o una Promesa genuina:

```js
var rejectedPr = new Promise( function(resolve,reject){
	// resolver esta promesa con una promesa rechazada
	resolve( Promise.reject( "Oops" ) );
} );

rejectedPr.then(
	function fulfilled(){
		// nunca llega aquí
	},
	function rejected(err){
		console.log( err );	// "Oops"
	}
);
```

Debería estar claro ahora que `resolve(..)` es el nombre apropiado para el primer parámetro callback del constructor `Promise(..)`.

**Advertencia:** El mencionado anteriormente `reject(..)` **no** hace el desenvolvimiento que `resolve(..)` hace. Si pasas un valor Promesa/thenable a `reject(..)`, ese valor intacto será establecido como la razón de rechazo. Un manejador de rechazo subsiguiente recibiría la Promesa/thenable real que pasaste a `reject(..)`, no su valor inmediato subyacente.

Pero ahora dirijamos nuestra atención a los callbacks proporcionados a `then(..)`. ¿Cómo deberían llamarse (tanto en la literatura como en el código)? Yo sugeriría `fulfilled(..)` y `rejected(..)`:

```js
function fulfilled(msg) {
	console.log( msg );
}

function rejected(err) {
	console.error( err );
}

p.then(
	fulfilled,
	rejected
);
```

En el caso del primer parámetro de `then(..)`, inequívocamente siempre es el caso de cumplimiento, así que no hay necesidad de la dualidad de la terminología "resolve". Como nota al margen, la especificación ES6 usa `onFulfilled(..)` y `onRejected(..)` para etiquetar estos dos callbacks, así que son términos precisos.

## Manejo de Errores

Ya hemos visto varios ejemplos de cómo el rechazo de Promesas -- ya sea intencional a través de llamar a `reject(..)` o accidental a través de excepciones JS -- permite un manejo de errores más cuerdo en la programación asíncrona. Volvamos sin embargo y seamos explícitos sobre algunos de los detalles que pasamos por alto.

La forma más natural de manejo de errores para la mayoría de los desarrolladores es la construcción síncrona `try..catch`. Desafortunadamente, es solo síncrona, así que falla en ayudar en patrones de código asíncrono:

```js
function foo() {
	setTimeout( function(){
		baz.bar();
	}, 100 );
}

try {
	foo();
	// después lanza un error global de `baz.bar()`
}
catch (err) {
	// nunca llega aquí
}
```

`try..catch` ciertamente sería agradable tener, pero no funciona a través de operaciones asíncronas. Es decir, a menos que haya algún soporte ambiental adicional, al cual volveremos con generadores en el Capítulo 4.

En callbacks, algunos estándares han emergido para el manejo de errores con patrón, más notablemente el estilo de callback "error-primero":

```js
function foo(cb) {
	setTimeout( function(){
		try {
			var x = baz.bar();
			cb( null, x ); // ¡éxito!
		}
		catch (err) {
			cb( err );
		}
	}, 100 );
}

foo( function(err,val){
	if (err) {
		console.error( err ); // qué lástima :(
	}
	else {
		console.log( val );
	}
} );
```

**Nota:** El `try..catch` aquí funciona solo desde la perspectiva de que la llamada `baz.bar()` tendrá éxito o fallará inmediatamente, síncronamente. Si `baz.bar()` fuera a su vez su propia función que completa asíncronamente, cualquier error asíncrono dentro de ella no sería capturable.

El callback que pasamos a `foo(..)` espera recibir una señal de error por el primer parámetro reservado `err`. Si está presente, se asume error. Si no, se asume éxito.

Este tipo de manejo de errores es técnicamente *capaz de asincronía*, pero no compone bien en absoluto. Múltiples niveles de callbacks error-primero entretejidos con estas ubicuas verificaciones `if` inevitablemente te llevarán a los peligros del infierno de callbacks (ver Capítulo 2).

Así que volvemos al manejo de errores en Promesas, con el manejador de rechazo pasado a `then(..)`. Las Promesas no usan el popular estilo de diseño de callback "error-primero", sino que usan el estilo de "callbacks divididos"; hay un callback para cumplimiento y otro para rechazo:

```js
var p = Promise.reject( "Oops" );

p.then(
	function fulfilled(){
		// nunca llega aquí
	},
	function rejected(err){
		console.log( err ); // "Oops"
	}
);
```

Aunque este patrón de manejo de errores tiene buen sentido en la superficie, los matices del manejo de errores de Promesas son frecuentemente bastante más difíciles de comprender completamente.

Considera:

```js
var p = Promise.resolve( 42 );

p.then(
	function fulfilled(msg){
		// los números no tienen funciones de string,
		// así que lanzará un error
		console.log( msg.toLowerCase() );
	},
	function rejected(err){
		// nunca llega aquí
	}
);
```

Si el `msg.toLowerCase()` legítimamente lanza un error (¡lo hace!), ¿por qué nuestro manejador de error no es notificado? Como explicamos antes, es porque *ese* manejador de error es para la promesa `p`, que ya ha sido cumplida con el valor `42`. La promesa `p` es inmutable, así que la única promesa que puede ser notificada del error es la devuelta por `p.then(..)`, que en este caso no capturamos.

Eso debería pintar una imagen clara de por qué el manejo de errores con Promesas es propenso a errores (juego de palabras intencional). Es demasiado fácil tener errores tragados, ya que esto es muy raramente lo que pretenderías.

**Advertencia:** Si usas la API de Promesas de una manera inválida y ocurre un error que previene la construcción apropiada de la Promesa, el resultado será una excepción lanzada inmediatamente, **no una Promesa rechazada**. Algunos ejemplos de uso incorrecto que fallan la construcción de Promesa: `new Promise(null)`, `Promise.all()`, `Promise.race(42)`, y así sucesivamente. ¡No puedes obtener una Promesa rechazada si no usas la API de Promesas lo suficientemente válido como para realmente construir una Promesa en primer lugar!

### Pozo de la Desesperación

Jeff Atwood notó hace años: los lenguajes de programación frecuentemente están configurados de tal manera que por defecto, los desarrolladores caen en el "pozo de la desesperación" (http://blog.codinghorror.com/falling-into-the-pit-of-success/) -- donde los accidentes son castigados -- y que tienes que esforzarte más para hacerlo bien. Nos imploró en su lugar crear un "pozo del éxito", donde por defecto caes en la acción esperada (exitosa), y por lo tanto tendrías que esforzarte mucho para fallar.

El manejo de errores de Promesas es incuestionablemente un diseño de "pozo de la desesperación". Por defecto, asume que quieres que cualquier error sea tragado por el estado de la Promesa, y si olvidas observar ese estado, el error languidece/muere silenciosamente en la obscuridad -- usualmente desesperación.

Para evitar perder un error en el silencio de una Promesa olvidada/descartada, algunos desarrolladores han afirmado que una "mejor práctica" para cadenas de Promesas es siempre terminar tu cadena con un `catch(..)` final, como:

```js
var p = Promise.resolve( 42 );

p.then(
	function fulfilled(msg){
		// los números no tienen funciones de string,
		// así que lanzará un error
		console.log( msg.toLowerCase() );
	}
)
.catch( handleErrors );
```

Porque no pasamos un manejador de rechazo al `then(..)`, el manejador por defecto fue sustituido, que simplemente propaga el error a la siguiente promesa en la cadena. Como tal, tanto los errores que llegan a `p`, como los errores que vienen *después* de `p` en su resolución (como el de `msg.toLowerCase()`) se filtrarán hasta el `handleErrors(..)` final.

¿Problema resuelto, verdad? ¡No tan rápido!

¿Qué pasa si `handleErrors(..)` mismo también tiene un error? ¿Quién captura eso? Todavía hay otra promesa no atendida: la que `catch(..)` devuelve, que no capturamos y no registramos un manejador de rechazo para ella.

No puedes simplemente meter otro `catch(..)` al final de esa cadena, porque también podría fallar. El último paso en cualquier cadena de Promesas, sea lo que sea, siempre tiene la posibilidad, aunque disminuyendo, de quedar colgado con un error no capturado atrapado dentro de una Promesa no observada.

¿Suena como un dilema imposible todavía?

### Manejo de No Capturados

No es exactamente un problema fácil de resolver completamente. Hay otras formas de abordarlo que muchos dirían que son *mejores*.

Algunas bibliotecas de Promesas han añadido métodos para registrar algo como un manejador de "rechazo global no manejado", que sería llamado en lugar de un error lanzado globalmente. Pero su solución para cómo identificar un error como "no capturado" es tener un temporizador de duración arbitraria, digamos 3 segundos, ejecutándose desde el momento del rechazo. Si una Promesa es rechazada pero ningún manejador de error se registra antes de que el temporizador se dispare, entonces se asume que nunca registrarás un manejador, así que es "no capturado".

En la práctica, esto ha funcionado bien para muchas bibliotecas, ya que la mayoría de los patrones de uso no suelen requerir un retraso significativo entre el rechazo de la Promesa y la observación de ese rechazo. Pero este patrón es problemático porque 3 segundos es tan arbitrario (incluso si es empírico), y también porque de hecho hay algunos casos donde quieres que una Promesa retenga su estado de rechazo por un período indefinido de tiempo, y realmente no quieres que tu manejador de "no capturados" sea llamado para todos esos falsos positivos (errores "no capturados" que aún no han sido manejados).

Otra sugerencia más común es que las Promesas deberían tener un `done(..)` añadido a ellas, que esencialmente marca la cadena de Promesas como "terminada". `done(..)` no crea ni devuelve una Promesa, así que los callbacks pasados a `done(..)` obviamente no están conectados para reportar problemas a una Promesa encadenada que no existe.

Entonces, ¿qué pasa en su lugar? Se trata como normalmente esperarías en condiciones de error no capturado: cualquier excepción dentro de un manejador de rechazo de `done(..)` sería lanzada como un error global no capturado (en la consola del desarrollador, básicamente):

```js
var p = Promise.resolve( 42 );

p.then(
	function fulfilled(msg){
		// los números no tienen funciones de string,
		// así que lanzará un error
		console.log( msg.toLowerCase() );
	}
)
.done( null, handleErrors );

// si `handleErrors(..)` causara su propia excepción, sería
// lanzada globalmente aquí
```

Esto podría sonar más atractivo que la cadena sin fin o los timeouts arbitrarios. Pero el mayor problema es que no es parte del estándar ES6, así que no importa qué tan bien suene, en el mejor de los casos es un camino mucho más largo para ser una solución confiable y ubicua.

¿Estamos atrapados entonces? No del todo.

Los navegadores tienen una capacidad única que nuestro código no tiene: pueden rastrear y saber con seguridad cuándo cualquier objeto es descartado y recolectado por el garbage collector. Así que los navegadores pueden rastrear objetos Promesa, y cuando son recolectados por garbage collection, si tienen un rechazo en ellos, el navegador sabe con seguridad que fue un legítimo "error no capturado", y por lo tanto puede con confianza reportarlo a la consola del desarrollador.

**Nota:** Al momento de escribir esto, tanto Chrome como Firefox tienen intentos tempranos de ese tipo de capacidad de "rechazo no capturado", aunque el soporte es incompleto en el mejor de los casos.

Sin embargo, si una Promesa no es recolectada por garbage collection -- es excesivamente fácil que eso ocurra accidentalmente a través de muchos patrones de codificación diferentes -- la detección de garbage collection del navegador no te ayudará a saber y diagnosticar que tienes una Promesa silenciosamente rechazada por ahí.

¿Hay alguna otra alternativa? Sí.

### Pozo del Éxito

Lo siguiente es solo teórico, cómo las Promesas *podrían* cambiar algún día para comportarse. Creo que sería muy superior a lo que tenemos actualmente. Y creo que este cambio sería posible incluso post-ES6 porque no creo que rompería la compatibilidad web con las Promesas ES6. Además, puede ser polyfilled/prollyfilled, si tienes cuidado. Echemos un vistazo:

* Las Promesas podrían por defecto reportar (a la consola del desarrollador) cualquier rechazo, en el siguiente tick de Job o bucle de eventos, si en ese momento exacto no se ha registrado ningún manejador de error para la Promesa.
* Para los casos donde quieres que una Promesa rechazada retenga su estado rechazado por una cantidad indefinida de tiempo antes de observarla, podrías llamar a `defer()`, que suprime el reporte automático de errores en esa Promesa.

Si una Promesa es rechazada, por defecto reporta ruidosamente ese hecho a la consola del desarrollador (en lugar de callar por defecto). Puedes optar fuera de ese reporte ya sea implícitamente (registrando un manejador de error antes del rechazo), o explícitamente (con `defer()`). En cualquier caso, *tú* controlas los falsos positivos.

Considera:

```js
var p = Promise.reject( "Oops" ).defer();

// `foo(..)` es consciente de Promesas
foo( 42 )
.then(
	function fulfilled(){
		return p;
	},
	function rejected(err){
		// manejar el error de `foo(..)`
	}
);
...
```

Cuando creamos `p`, sabemos que vamos a esperar un rato para usar/observar su rechazo, así que llamamos a `defer()` -- por lo tanto no hay reporte global. `defer()` simplemente devuelve la misma promesa, para propósitos de encadenamiento.

La promesa devuelta por `foo(..)` tiene un manejador de error adjuntado *inmediatamente*, así que está implícitamente optada fuera y no ocurre reporte global para ella tampoco.

Pero la promesa devuelta por la llamada `then(..)` no tiene `defer()` ni manejador de error adjuntado, así que si se rechaza (desde dentro de cualquier manejador de resolución), entonces *será* reportada a la consola del desarrollador como un error no capturado.

**Este diseño es un pozo del éxito.** Por defecto, todos los errores son manejados o reportados -- lo que casi todos los desarrolladores en casi todos los casos esperarían. Tienes que registrar un manejador o tienes que intencionalmente optar fuera, e indicar que pretendes diferir el manejo de errores hasta *después*; estás optando por la responsabilidad extra en solo ese caso específico.

El único peligro real en este enfoque es si haces `defer()` de una Promesa pero luego fallas en realmente observar/manejar su rechazo.

Pero tenías que intencionalmente llamar a `defer()` para optar por ese pozo de la desesperación -- el defecto era el pozo del éxito -- así que no hay mucho más que podamos hacer para salvarte de tus propios errores.

Creo que todavía hay esperanza para el manejo de errores de Promesas (post-ES6). Espero que los poderes que sean reconsideren la situación y consideren esta alternativa. Mientras tanto, ¡puedes implementar esto tú mismo (un ejercicio desafiante para el lector!), o usar una biblioteca de Promesas más *inteligente* que lo haga por ti!

**Nota:** Este modelo exacto para manejo/reporte de errores está implementado en mi biblioteca de abstracción de Promesas *asynquence*, que se discutirá en el Apéndice A de este libro.

## Patrones de Promesas

Ya hemos visto implícitamente el patrón de secuencia con cadenas de Promesas (control de flujo esto-luego-esto-luego-aquello) pero hay muchas variaciones de patrones asíncronos que podemos construir como abstracciones sobre las Promesas. Estos patrones sirven para simplificar la expresión del control de flujo asíncrono -- lo que ayuda a hacer nuestro código más razonable y más mantenible -- incluso en las partes más complejas de nuestros programas.

Dos de esos patrones están codificados directamente en la implementación nativa ES6 `Promise`, así que los obtenemos gratis, para usar como bloques de construcción para otros patrones.

### Promise.all([ .. ])

En una secuencia asíncrona (cadena de Promesas), solo una tarea asíncrona está siendo coordinada en un momento dado -- el paso 2 sigue estrictamente al paso 1, y el paso 3 sigue estrictamente al paso 2. ¿Pero qué hay de hacer dos o más pasos concurrentemente (es decir, "en paralelo")?

En la terminología clásica de programación, una "compuerta" es un mecanismo que espera a que dos o más tareas paralelas/concurrentes se completen antes de continuar. No importa en qué orden terminen, solo que todas tienen que completarse para que la compuerta se abra y deje pasar el control de flujo.

En la API de Promesas, llamamos a este patrón `all([ .. ])`.

Di que quisieras hacer dos peticiones Ajax al mismo tiempo, y esperar a que ambas terminen, independientemente de su orden, antes de hacer una tercera petición Ajax. Considera:

```js
// `request(..)` es una utilidad Ajax consciente de Promesas,
// como la que definimos anteriormente en el capítulo

var p1 = request( "http://some.url.1/" );
var p2 = request( "http://some.url.2/" );

Promise.all( [p1,p2] )
.then( function(msgs){
	// tanto `p1` como `p2` se cumplen y pasan
	// sus mensajes aquí
	return request(
		"http://some.url.3/?v=" + msgs.join(",")
	);
} )
.then( function(msg){
	console.log( msg );
} );
```

`Promise.all([ .. ])` espera un solo argumento, un `array`, que consiste generalmente en instancias de Promesa. La promesa devuelta por la llamada `Promise.all([ .. ])` recibirá un mensaje de cumplimiento (`msgs` en este fragmento) que es un `array` de todos los mensajes de cumplimiento de las promesas pasadas, en el mismo orden especificado (independientemente del orden de cumplimiento).

**Nota:** Técnicamente, el `array` de valores pasado a `Promise.all([ .. ])` puede incluir Promesas, thenables, o incluso valores inmediatos. Cada valor en la lista se pasa esencialmente a través de `Promise.resolve(..)` para asegurar que sea una Promesa genuina a esperar, así que un valor inmediato se normalizará en una Promesa para ese valor. Si el `array` está vacío, la Promesa principal se cumple inmediatamente.

La promesa principal devuelta por `Promise.all([ .. ])` solo será cumplida si y cuando todas sus promesas constituyentes son cumplidas. Si alguna de esas promesas en su lugar es rechazada, la promesa principal de `Promise.all([ .. ])` es rechazada inmediatamente, descartando todos los resultados de cualquier otra promesa.

Recuerda siempre adjuntar un manejador de rechazo/error a cada promesa, incluso y especialmente la que viene de `Promise.all([ .. ])`.

### Promise.race([ .. ])

Mientras `Promise.all([ .. ])` coordina múltiples Promesas concurrentemente y asume que todas son necesarias para el cumplimiento, a veces solo quieres responder a la "primera Promesa en cruzar la línea de meta", dejando que las otras Promesas se descarten.

Este patrón se llama clásicamente un "latch" (pestillo), pero en Promesas se llama una "carrera" (race).

**Advertencia:** Aunque la metáfora de "solo el primero en cruzar la línea de meta gana" encaja bien con el comportamiento, desafortunadamente "race" es un término un poco cargado, porque "condiciones de carrera" (race conditions) generalmente se toman como bugs en los programas (ver Capítulo 1). No confundas `Promise.race([ .. ])` con "condición de carrera".

`Promise.race([ .. ])` también espera un solo argumento `array`, conteniendo una o más Promesas, thenables, o valores inmediatos. No tiene mucho sentido práctico tener una carrera con valores inmediatos, porque el primero listado obviamente ganará -- ¡como una carrera de pies donde un corredor empieza en la línea de meta!

Similar a `Promise.all([ .. ])`, `Promise.race([ .. ])` se cumplirá si y cuando cualquier resolución de Promesa sea un cumplimiento, y se rechazará si y cuando cualquier resolución de Promesa sea un rechazo.

**Advertencia:** Una "carrera" requiere al menos un "corredor", así que si pasas un `array` vacío, en lugar de resolverse inmediatamente, la Promesa principal de `race([..])` nunca se resolverá. ¡Esto es un footgun! ES6 debería haber especificado que o se cumple, se rechaza, o simplemente lanza algún tipo de error síncrono. Desafortunadamente, debido a la precedencia en bibliotecas de Promesas que preceden a ES6 `Promise`, tuvieron que dejar esta trampa ahí, así que ten cuidado de nunca enviar un `array` vacío.

Revisitemos nuestro ejemplo anterior de Ajax concurrente, pero en el contexto de una carrera entre `p1` y `p2`:

```js
// `request(..)` es una utilidad Ajax consciente de Promesas,
// como la que definimos anteriormente en el capítulo

var p1 = request( "http://some.url.1/" );
var p2 = request( "http://some.url.2/" );

Promise.race( [p1,p2] )
.then( function(msg){
	// ya sea `p1` o `p2` ganará la carrera
	return request(
		"http://some.url.3/?v=" + msg
	);
} )
.then( function(msg){
	console.log( msg );
} );
```

Porque solo una promesa gana, el valor de cumplimiento es un solo mensaje, no un `array` como lo fue para `Promise.all([ .. ])`.

#### Carrera de Timeout

Vimos este ejemplo antes, ilustrando cómo `Promise.race([ .. ])` puede usarse para expresar el patrón de "timeout de promesa":

```js
// `foo()` es una función consciente de Promesas

// `timeoutPromise(..)`, definida antes, devuelve
// una Promesa que se rechaza después de un retraso especificado

// configurar un timeout para `foo()`
Promise.race( [
	foo(),					// intentar `foo()`
	timeoutPromise( 3000 )	// darle 3 segundos
] )
.then(
	function(){
		// ¡`foo(..)` se cumplió a tiempo!
	},
	function(err){
		// o `foo()` fue rechazado, o simplemente
		// no terminó a tiempo, así que inspecciona
		// `err` para saber cuál
	}
);
```

Este patrón de timeout funciona bien en la mayoría de los casos. Pero hay algunos matices a considerar, y francamente aplican tanto a `Promise.race([ .. ])` como a `Promise.all([ .. ])` por igual.

#### "Finally"

La pregunta clave es, "¿Qué pasa con las promesas que se descartan/ignoran?" No estamos preguntando esa cuestión desde la perspectiva de rendimiento -- típicamente terminarían siendo elegibles para garbage collection -- sino desde la perspectiva de comportamiento (efectos secundarios, etc.). Las Promesas no pueden ser canceladas -- y no deberían serlo ya que eso destruiría la confianza de inmutabilidad externa discutida en la sección "Promesa No Cancelable" más adelante en este capítulo -- así que solo pueden ser silenciosamente ignoradas.

¿Pero qué pasa si `foo()` en el ejemplo anterior está reservando algún tipo de recurso para su uso, pero el timeout se dispara primero y causa que esa promesa sea ignorada? ¿Hay algo en este patrón que proactivamente libere el recurso reservado después del timeout, o de otra manera cancele cualquier efecto secundario que pueda haber tenido? ¿Qué pasaría si todo lo que quisieras fuera registrar el hecho de que `foo()` excedió el tiempo?

Algunos desarrolladores han propuesto que las Promesas necesitan un registro de callback `finally(..)`, que siempre se llama cuando una Promesa se resuelve, y te permite especificar cualquier limpieza que pueda ser necesaria. Esto no existe en la especificación en este momento, pero podría venir en ES7+. Tendremos que esperar y ver.

Podría verse así:

```js
var p = Promise.resolve( 42 );

p.then( something )
.finally( cleanup )
.then( another )
.finally( cleanup );
```

**Nota:** En varias bibliotecas de Promesas, `finally(..)` todavía crea y devuelve una nueva Promesa (para mantener la cadena). Si la función `cleanup(..)` devolviera una Promesa, se enlazaría a la cadena, lo que significa que aún podrías tener los problemas de rechazo no manejado que discutimos antes.

Mientras tanto, podríamos hacer una utilidad helper estática que nos permita observar (sin interferir) la resolución de una Promesa:

```js
// verificación guardia segura para polyfill
if (!Promise.observe) {
	Promise.observe = function(pr,cb) {
		// observar lateralmente la resolución de `pr`
		pr.then(
			function fulfilled(msg){
				// programar callback asíncronamente (como Job)
				Promise.resolve( msg ).then( cb );
			},
			function rejected(err){
				// programar callback asíncronamente (como Job)
				Promise.resolve( err ).then( cb );
			}
		);

		// devolver la promesa original
		return pr;
	};
}
```

Así es como la usaríamos en el ejemplo de timeout de antes:

```js
Promise.race( [
	Promise.observe(
		foo(),					// intentar `foo()`
		function cleanup(msg){
			// limpiar después de `foo()`, incluso si
			// no terminó antes del timeout
		}
	),
	timeoutPromise( 3000 )	// darle 3 segundos
] )
```

Este helper `Promise.observe(..)` es solo una ilustración de cómo podrías observar las completaciones de Promesas sin interferir con ellas. Otras bibliotecas de Promesas tienen sus propias soluciones. Independientemente de cómo lo hagas, probablemente tendrás lugares donde quieras asegurarte de que tus Promesas no estén siendo *solo* silenciosamente ignoradas por accidente.

### Variaciones sobre all([ .. ]) y race([ .. ])

Mientras las Promesas nativas ES6 vienen con `Promise.all([ .. ])` y `Promise.race([ .. ])` incorporados, hay varios otros patrones comúnmente usados con variaciones sobre esas semánticas:

* `none([ .. ])` es como `all([ .. ])`, pero cumplimientos y rechazos se transponen. Todas las Promesas necesitan ser rechazadas -- los rechazos se convierten en los valores de cumplimiento y viceversa.
* `any([ .. ])` es como `all([ .. ])`, pero ignora cualquier rechazo, así que solo una necesita cumplirse en lugar de *todas* ellas.
* `first([ .. ])` es como una carrera con `any([ .. ])`, que es que ignora cualquier rechazo y se cumple tan pronto como la primera Promesa se cumple.
* `last([ .. ])` es como `first([ .. ])`, pero solo el último cumplimiento gana.

Algunas bibliotecas de abstracción de Promesas proporcionan estas, pero también podrías definirlas tú mismo usando la mecánica de Promesas, `race([ .. ])` y `all([ .. ])`.

Por ejemplo, así es como podríamos definir `first([ .. ])`:

```js
// verificación guardia segura para polyfill
if (!Promise.first) {
	Promise.first = function(prs) {
		return new Promise( function(resolve,reject){
			// recorrer todas las promesas
			prs.forEach( function(pr){
				// normalizar el valor
				Promise.resolve( pr )
				// la primera que se cumpla gana, y
				// resuelve la promesa principal
				.then( resolve );
			} );
		} );
	};
}
```

**Nota:** Esta implementación de `first(..)` no rechaza si todas sus promesas rechazan; simplemente se cuelga, muy parecido a lo que hace `Promise.race([])`. Si se desea, podrías añadir lógica adicional para rastrear cada rechazo de promesa y si todas rechazan, llamar a `reject()` en la promesa principal. Lo dejaremos como ejercicio para el lector.

### Iteraciones Concurrentes

A veces quieres iterar sobre una lista de Promesas y realizar alguna tarea contra todas ellas, muy parecido a como puedes hacer con `array`s síncronos (por ejemplo, `forEach(..)`, `map(..)`, `some(..)`, y `every(..)`). Si la tarea a realizar contra cada Promesa es fundamentalmente síncrona, estas funcionan bien, justo como usamos `forEach(..)` en el fragmento anterior.

Pero si las tareas son fundamentalmente asíncronas, o pueden/deberían de otra manera realizarse concurrentemente, puedes usar versiones asíncronas de estas utilidades como las que proporcionan muchas bibliotecas.

Por ejemplo, consideremos una utilidad asíncrona `map(..)` que toma un `array` de valores (podrían ser Promesas o cualquier otra cosa), más una función (tarea) para realizar contra cada uno. `map(..)` mismo devuelve una promesa cuyo valor de cumplimiento es un `array` que contiene (en el mismo orden de mapeo) el valor de cumplimiento asíncrono de cada tarea:

```js
if (!Promise.map) {
	Promise.map = function(vals,cb) {
		// nueva promesa que espera a todas las promesas mapeadas
		return Promise.all(
			// nota: `map(..)` de array regular, convierte
			// el array de valores en un array de
			// promesas
			vals.map( function(val){
				// reemplazar `val` con una nueva promesa que
				// se resuelve después de que `val` se mapea asíncronamente
				return new Promise( function(resolve){
					cb( val, resolve );
				} );
			} )
		);
	};
}
```

**Nota:** En esta implementación de `map(..)`, no puedes señalar un rechazo asíncrono, pero si ocurre una excepción/error síncrono dentro del callback de mapeo (`cb(..)`), la promesa principal devuelta por `Promise.map(..)` rechazaría.

Ilustremos el uso de `map(..)` con una lista de Promesas (en lugar de valores simples):

```js
var p1 = Promise.resolve( 21 );
var p2 = Promise.resolve( 42 );
var p3 = Promise.reject( "Oops" );

// duplicar valores en la lista incluso si están en
// Promesas
Promise.map( [p1,p2,p3], function(pr,done){
	// asegurar que el elemento mismo sea una Promesa
	Promise.resolve( pr )
	.then(
		// extraer el valor como `v`
		function(v){
			// mapear cumplimiento `v` a nuevo valor
			done( v * 2 );
		},
		// o, mapear al mensaje de rechazo de la promesa
		done
	);
} )
.then( function(vals){
	console.log( vals );	// [42,84,"Oops"]
} );
```
