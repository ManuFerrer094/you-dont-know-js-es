## Generadores + Promesas

En nuestra discusión anterior, mostramos cómo los generadores pueden iterarse asíncronamente, lo cual es un enorme paso adelante en razonabilidad secuencial sobre el lío de espagueti de callbacks. ¡Pero perdimos algo muy importante: la confiabilidad y componibilidad de las Promesas (ver Capítulo 3)!

No te preocupes -- podemos recuperar eso. Lo mejor de todos los mundos en ES6 es combinar generadores (código asíncrono con apariencia síncrona) con Promesas (confiables y componibles).

¿Pero cómo?

Recuerda del Capítulo 3 el enfoque basado en Promesas de nuestro ejemplo continuo de Ajax:

```js
function foo(x,y) {
	return request(
		"http://some.url.1/?x=" + x + "&y=" + y
	);
}

foo( 11, 31 )
.then(
	function(text){
		console.log( text );
	},
	function(err){
		console.error( err );
	}
);
```

En nuestro código de generador anterior para el ejemplo continuo de Ajax, `foo(..)` no devolvía nada (`undefined`), y nuestro código de control del *iterador* no se preocupaba por ese valor emitido con `yield`.

Pero aquí el `foo(..)` consciente de Promesas devuelve una promesa después de hacer la llamada Ajax. Eso sugiere que podríamos construir una promesa con `foo(..)` y luego emitirla con `yield` desde el generador, y entonces el código de control del *iterador* recibiría esa promesa.

¿Pero qué debería hacer el *iterador* con la promesa?

Debería escuchar a que la promesa se resuelva (cumplimiento o rechazo), y entonces o reanudar el generador con el mensaje de cumplimiento o lanzar un error dentro del generador con la razón de rechazo.

Déjame repetir eso, porque es muy importante. La forma natural de obtener el máximo de Promesas y generadores es **emitir con `yield` una Promesa**, y conectar esa Promesa para controlar el *iterador* del generador.

¡Intentémoslo! Primero, juntaremos el `foo(..)` consciente de Promesas con el generador `*main()`:

```js
function foo(x,y) {
	return request(
		"http://some.url.1/?x=" + x + "&y=" + y
	);
}

function *main() {
	try {
		var text = yield foo( 11, 31 );
		console.log( text );
	}
	catch (err) {
		console.error( err );
	}
}
```

¡La revelación más poderosa en esta refactorización es que el código dentro de `*main()` **no tuvo que cambiar en absoluto**! Dentro del generador, cualesquiera valores que se emitan con `yield` son solo un detalle opaco de implementación, así que ni siquiera somos conscientes de que está pasando, ni necesitamos preocuparnos por ello.

¿Pero cómo vamos a ejecutar `*main()` ahora? Todavía tenemos algo del trabajo de fontanería de implementación que hacer, para recibir y conectar la promesa emitida con `yield` de modo que reanude el generador tras la resolución. Empezaremos intentando eso manualmente:

```js
var it = main();

var p = it.next().value;

// esperar a que la promesa `p` se resuelva
p.then(
	function(text){
		it.next( text );
	},
	function(err){
		it.throw( err );
	}
);
```

En realidad, eso no fue tan doloroso, ¿verdad?

Este fragmento debería verse muy similar a lo que hicimos antes con el generador controlado manualmente por el callback error-primero. En lugar de un `if (err) { it.throw..`, la promesa ya divide cumplimiento (éxito) y rechazo (fallo) por nosotros, pero por lo demás el control del *iterador* es idéntico.

Ahora, hemos pasado por alto algunos detalles importantes.

Lo más importante, aprovechamos el hecho de que sabíamos que `*main()` solo tenía un paso consciente de Promesas en él. ¿Y si quisiéramos poder impulsar un generador con Promesas sin importar cuántos pasos tenga? ¡Ciertamente no queremos escribir manualmente la cadena de Promesas de forma diferente para cada generador! Lo que sería mucho más agradable es si hubiera una forma de repetir (es decir, "recorrer") el control de iteración, y cada vez que una Promesa salga, esperar su resolución antes de continuar.

Además, ¿qué pasa si el generador lanza un error (intencional o accidentalmente) durante la llamada `it.next(..)`? ¿Deberíamos parar, o deberíamos hacer `catch` y enviarlo de vuelta? Similarmente, ¿qué pasa si hacemos `it.throw(..)` de un rechazo de Promesa dentro del generador, pero no se maneja, y sale de vuelta?

### Ejecutor de Generadores Consciente de Promesas

Cuanto más empiezas a explorar este camino, más te das cuenta, "wow, sería genial si hubiera alguna utilidad que lo hiciera por mí." Y tienes toda la razón. Este es un patrón tan importante, y no quieres equivocarte (o agotarte repitiéndolo una y otra vez), así que tu mejor opción es usar una utilidad que esté específicamente diseñada para *ejecutar* generadores que emiten Promesas con `yield` de la manera que hemos ilustrado.

Varias bibliotecas de abstracción de Promesas proporcionan justo tal utilidad, incluyendo mi biblioteca *asynquence* y su `runner(..)`, que se discutirá en el Apéndice A de este libro.

Pero por el bien del aprendizaje e ilustración, simplemente definamos nuestra propia utilidad independiente que llamaremos `run(..)`:

```js
// ¡gracias a Benjamin Gruenbaum (@benjamingr en GitHub) por
// grandes mejoras aquí!
function run(gen) {
	var args = [].slice.call( arguments, 1), it;

	// inicializar el generador en el contexto actual
	it = gen.apply( this, args );

	// devolver una promesa para la completación del generador
	return Promise.resolve()
		.then( function handleNext(value){
			// ejecutar hasta el siguiente valor emitido con yield
			var next = it.next( value );

			return (function handleResult(next){
				// ¿el generador ha completado su ejecución?
				if (next.done) {
					return next.value;
				}
				// de lo contrario seguir adelante
				else {
					return Promise.resolve( next.value )
						.then(
							// reanudar el bucle asíncrono en
							// éxito, enviando el valor resuelto
							// de vuelta al generador
							handleNext,

							// si `value` es una promesa
							// rechazada, propagar el error de
							// vuelta al generador para su propio
							// manejo de errores
							function handleErr(err) {
								return Promise.resolve(
									it.throw( err )
								)
								.then( handleResult );
							}
						);
				}
			})(next);
		} );
}
```

Como puedes ver, es bastante más complejo de lo que probablemente querrías escribir tú mismo, y especialmente no querrías repetir este código para cada generador que uses. Así que, una utilidad/helper de biblioteca es definitivamente el camino a seguir. Sin embargo, te animo a dedicar unos minutos a estudiar ese listado de código para obtener un mejor sentido de cómo manejar la negociación generador+Promesa.

¿Cómo usarías `run(..)` con `*main()` en nuestro ejemplo *continuo* de Ajax?

```js
function *main() {
	// ..
}

run( main );
```

¡Eso es todo! La forma en que conectamos `run(..)`, avanzará automáticamente el generador que le pases, asíncronamente hasta completarse.

**Nota:** El `run(..)` que definimos devuelve una promesa que está conectada para resolverse una vez que el generador esté completo, o recibir una excepción no capturada si el generador no la maneja. No mostramos esa capacidad aquí, pero volveremos a ella más adelante en el capítulo.

#### ES7: ¿`async` y `await`?

El patrón anterior -- generadores emitiendo Promesas con yield que luego controlan el *iterador* del generador para avanzarlo hasta su completación -- es un enfoque tan poderoso y útil, que sería más agradable si pudiéramos hacerlo sin el desorden de la utilidad helper de biblioteca (es decir, `run(..)`).

Probablemente hay buenas noticias en ese frente. Al momento de escribir esto, hay soporte temprano pero fuerte para una propuesta de más adiciones sintácticas en este ámbito para el marco temporal post-ES6, ES7-ish. Obviamente, es demasiado temprano para garantizar los detalles, pero hay una probabilidad bastante decente de que se concrete de manera similar a lo siguiente:

```js
function foo(x,y) {
	return request(
		"http://some.url.1/?x=" + x + "&y=" + y
	);
}

async function main() {
	try {
		var text = await foo( 11, 31 );
		console.log( text );
	}
	catch (err) {
		console.error( err );
	}
}

main();
```

Como puedes ver, no hay llamada a `run(..)` (¡lo que significa que no hay necesidad de una utilidad de biblioteca!) para invocar e impulsar `main()` -- simplemente se llama como una función normal. Además, `main()` ya no se declara como una función generadora; es un nuevo tipo de función: `async function`. Y finalmente, en lugar de emitir una Promesa con `yield`, la esperamos (`await`) a que se resuelva.

La `async function` automáticamente sabe qué hacer si haces `await` de una Promesa -- pausará la función (igual que con generadores) hasta que la Promesa se resuelva. No lo ilustramos en este fragmento, pero llamar a una función async como `main()` automáticamente devuelve una promesa que se resuelve cuando la función termina completamente.

**Consejo:** La sintaxis `async` / `await` debería lucir muy familiar para lectores con experiencia en C#, porque es básicamente idéntica.

La propuesta esencialmente codifica soporte para el patrón que ya hemos derivado, en un mecanismo sintáctico: combinar Promesas con código de control de flujo con apariencia síncrona. Eso es lo mejor de ambos mundos combinado, para abordar efectivamente prácticamente todas las preocupaciones principales que esbozamos con los callbacks.

El mero hecho de que tal propuesta de ES7-ish ya exista y tenga soporte temprano y entusiasmo es un gran voto de confianza en la importancia futura de este patrón asíncrono.

### Concurrencia de Promesas en Generadores

Hasta ahora, todo lo que hemos demostrado es un flujo asíncrono de un solo paso con Promesas+generadores. Pero el código del mundo real frecuentemente tendrá muchos pasos asíncronos.

Si no tienes cuidado, el estilo con apariencia síncrona de los generadores puede adormecerte en la complacencia con cómo estructuras tu concurrencia asíncrona, llevando a patrones de rendimiento subóptimos. Así que queremos dedicar un poco de tiempo explorando las opciones.

Imagina un escenario donde necesitas obtener datos de dos fuentes diferentes, luego combinar esas respuestas para hacer una tercera petición, y finalmente imprimir la última respuesta. Exploramos un escenario similar con Promesas en el Capítulo 3, pero reconsiderémoslo en el contexto de generadores.

Tu primer instinto podría ser algo como:

```js
function *foo() {
	var r1 = yield request( "http://some.url.1" );
	var r2 = yield request( "http://some.url.2" );

	var r3 = yield request(
		"http://some.url.3/?v=" + r1 + "," + r2
	);

	console.log( r3 );
}

// usar la utilidad `run(..)` definida previamente
run( foo );
```

Este código funcionará, pero en los detalles de nuestro escenario, no es óptimo. ¿Puedes detectar por qué?

Porque las peticiones `r1` y `r2` pueden -- y por razones de rendimiento, *deberían* -- ejecutarse concurrentemente, pero en este código se ejecutarán secuencialmente; la URL `"http://some.url.2"` no se obtiene con Ajax hasta después de que la petición `"http://some.url.1"` haya terminado. Estas dos peticiones son independientes, así que el mejor enfoque de rendimiento probablemente sería que se ejecutaran al mismo tiempo.

¿Pero cómo exactamente harías eso con un generador y `yield`? Sabemos que `yield` es solo un punto de pausa único en el código, así que realmente no puedes hacer dos pausas al mismo tiempo.

La respuesta más natural y efectiva es basar el flujo asíncrono en Promesas, específicamente en su capacidad de manejar estado de manera independiente del tiempo (ver "Valor Futuro" en el Capítulo 3).

El enfoque más simple:

```js
function *foo() {
	// hacer ambas peticiones "en paralelo"
	var p1 = request( "http://some.url.1" );
	var p2 = request( "http://some.url.2" );

	// esperar hasta que ambas promesas se resuelvan
	var r1 = yield p1;
	var r2 = yield p2;

	var r3 = yield request(
		"http://some.url.3/?v=" + r1 + "," + r2
	);

	console.log( r3 );
}

// usar la utilidad `run(..)` definida previamente
run( foo );
```

¿Por qué es esto diferente del fragmento anterior? Mira dónde está y dónde no está el `yield`. `p1` y `p2` son promesas para peticiones Ajax hechas concurrentemente (es decir, "en paralelo"). No importa cuál termine primero, porque las promesas retendrán su estado resuelto durante todo el tiempo necesario.

Luego usamos dos sentencias `yield` subsiguientes para esperar y obtener las resoluciones de las promesas (en `r1` y `r2`, respectivamente). Si `p1` se resuelve primero, el `yield p1` se reanuda primero y luego espera a que `yield p2` se reanude. Si `p2` se resuelve primero, pacientemente retendrá ese valor de resolución hasta que se le pida, pero el `yield p1` esperará primero, hasta que `p1` se resuelva.

De cualquier forma, tanto `p1` como `p2` se ejecutarán concurrentemente, y ambas tienen que terminar, en cualquier orden, antes de que se haga la petición Ajax `r3 = yield request..`.

Si ese modelo de procesamiento de control de flujo suena familiar, es básicamente lo mismo que lo que identificamos en el Capítulo 3 como el patrón de "compuerta", habilitado por la utilidad `Promise.all([ .. ])`. Así que, también podríamos expresar el control de flujo así:

```js
function *foo() {
	// hacer ambas peticiones "en paralelo," y
	// esperar hasta que ambas promesas se resuelvan
	var results = yield Promise.all( [
		request( "http://some.url.1" ),
		request( "http://some.url.2" )
	] );

	var r1 = results[0];
	var r2 = results[1];

	var r3 = yield request(
		"http://some.url.3/?v=" + r1 + "," + r2
	);

	console.log( r3 );
}

// usar la utilidad `run(..)` definida previamente
run( foo );
```

**Nota:** Como discutimos en el Capítulo 3, incluso podemos usar la asignación desestructurante de ES6 para simplificar las asignaciones `var r1 = .. var r2 = ..`, con `var [r1,r2] = results`.

En otras palabras, todas las capacidades de concurrencia de las Promesas están disponibles para nosotros en el enfoque generador+Promesa. Así que en cualquier lugar donde necesites más que pasos de control de flujo asíncronos secuenciales esto-luego-aquello, las Promesas son probablemente tu mejor opción.

#### Promesas Ocultas

Como una palabra de precaución estilística, ten cuidado con cuánta lógica de Promesas incluyes **dentro de tus generadores**. El punto completo de usar generadores para asincronía de la manera que hemos descrito es crear código simple, secuencial, con apariencia síncrona, y ocultar tantos detalles de asincronía de ese código como sea posible.

Por ejemplo, este podría ser un enfoque más limpio:

```js
// nota: función normal, no generador
function bar(url1,url2) {
	return Promise.all( [
		request( url1 ),
		request( url2 )
	] );
}

function *foo() {
	// ocultar los detalles de concurrencia basados en Promesas
	// dentro de `bar(..)`
	var results = yield bar(
		"http://some.url.1",
		"http://some.url.2"
	);

	var r1 = results[0];
	var r2 = results[1];

	var r3 = yield request(
		"http://some.url.3/?v=" + r1 + "," + r2
	);

	console.log( r3 );
}

// usar la utilidad `run(..)` definida previamente
run( foo );
```

Dentro de `*foo()`, es más limpio y claro que todo lo que estamos haciendo es simplemente pedir a `bar(..)` que nos consiga unos `results`, y esperaremos con `yield` a que eso suceda. No tenemos que preocuparnos de que bajo las sábanas se usará una composición de Promesas `Promise.all([ .. ])` para hacer que suceda.

**Tratamos la asincronía, y de hecho las Promesas, como un detalle de implementación.**

Ocultar tu lógica de Promesas dentro de una función que simplemente llamas desde tu generador es especialmente útil si vas a hacer una serie sofisticada de control de flujo. Por ejemplo:

```js
function bar() {
	return	Promise.all( [
		  baz( .. )
		  .then( .. ),
		  Promise.race( [ .. ] )
		] )
		.then( .. )
}
```

Ese tipo de lógica a veces es necesaria, y si la vuelcas directamente dentro de tu(s) generador(es), has derrotado la mayoría de la razón por la que querrías usar generadores en primer lugar. *Deberíamos* intencionalmente abstraer tales detalles fuera de nuestro código de generador para que no ensucien la expresión de tarea de nivel superior.

Más allá de crear código que sea tanto funcional como eficiente, también deberías esforzarte por hacer código que sea tan razonable y mantenible como sea posible.

**Nota:** La abstracción no es *siempre* algo saludable para la programación -- muchas veces puede aumentar la complejidad a cambio de brevedad. Pero en este caso, creo que es mucho más saludable para tu código asíncrono de generador+Promesa que las alternativas. Sin embargo, como con todos estos consejos, presta atención a tus situaciones específicas y toma decisiones adecuadas para ti y tu equipo.

## Delegación de Generadores

En la sección anterior, mostramos llamar funciones regulares desde dentro de un generador, y cómo eso sigue siendo una técnica útil para abstraer detalles de implementación (como flujo asíncrono de Promesas). Pero la principal desventaja de usar una función normal para esta tarea es que tiene que comportarse según las reglas de funciones normales, lo que significa que no puede pausarse a sí misma con `yield` como puede hacerlo un generador.

Entonces puede ocurrírsete que podrías intentar llamar a un generador desde otro generador, usando nuestro helper `run(..)`, como:

```js
function *foo() {
	var r2 = yield request( "http://some.url.2" );
	var r3 = yield request( "http://some.url.3/?v=" + r2 );

	return r3;
}

function *bar() {
	var r1 = yield request( "http://some.url.1" );

	// "delegando" a `*foo()` a través de `run(..)`
	var r3 = yield run( foo );

	console.log( r3 );
}

run( bar );
```

Ejecutamos `*foo()` dentro de `*bar()` usando nuestra utilidad `run(..)` de nuevo. Aprovechamos aquí el hecho de que el `run(..)` que definimos antes devuelve una promesa que se resuelve cuando su generador se ejecuta hasta completarse (o da error), así que si emitimos con `yield` a una instancia de `run(..)` la promesa de otra llamada `run(..)`, automáticamente pausa `*bar()` hasta que `*foo()` termine.

Pero hay una forma aún mejor de integrar la llamada a `*foo()` dentro de `*bar()`, y se llama delegación con `yield`. La sintaxis especial para delegación con `yield` es: `yield * __` (nota el `*` extra). Antes de verla funcionar en nuestro ejemplo anterior, veamos un escenario más simple:

```js
function *foo() {
	console.log( "`*foo()` starting" );
	yield 3;
	yield 4;
	console.log( "`*foo()` finished" );
}

function *bar() {
	yield 1;
	yield 2;
	yield *foo();	// ¡delegación con `yield`!
	yield 5;
}

var it = bar();

it.next().value;	// 1
it.next().value;	// 2
it.next().value;	// `*foo()` starting
					// 3
it.next().value;	// 4
it.next().value;	// `*foo()` finished
					// 5
```

**Nota:** Similar a una nota anterior en el capítulo donde expliqué por qué prefiero `function *foo() ..` en lugar de `function* foo() ..`, también prefiero -- difiriendo de la mayoría de otra documentación sobre el tema -- decir `yield *foo()` en lugar de `yield* foo()`. La posición del `*` es puramente estilística y depende de tu mejor juicio. Pero encuentro atractiva la consistencia de estilo.

¿Cómo funciona la delegación `yield *foo()`?

Primero, llamar a `foo()` crea un *iterador* exactamente como ya hemos visto. Luego, `yield *` delega/transfiere el control de la instancia del *iterador* (del generador `*bar()` presente) a este otro *iterador* de `*foo()`.

Así que, las primeras dos llamadas `it.next()` están controlando `*bar()`, pero cuando hacemos la tercera llamada `it.next()`, ahora `*foo()` se inicia, y ahora estamos controlando `*foo()` en lugar de `*bar()`. Por eso se llama delegación -- `*bar()` delegó su control de iteración a `*foo()`.

Tan pronto como el control del *iterador* `it` agota el *iterador* completo de `*foo()`, automáticamente vuelve a controlar `*bar()`.

Así que ahora de vuelta al ejemplo anterior con las tres peticiones Ajax secuenciales:

```js
function *foo() {
	var r2 = yield request( "http://some.url.2" );
	var r3 = yield request( "http://some.url.3/?v=" + r2 );

	return r3;
}

function *bar() {
	var r1 = yield request( "http://some.url.1" );

	// "delegando" a `*foo()` a través de `yield*`
	var r3 = yield *foo();

	console.log( r3 );
}

run( bar );
```

La única diferencia entre este fragmento y la versión usada antes es el uso de `yield *foo()` en lugar del previo `yield run(foo)`.

**Nota:** `yield *` cede el control de iteración, no el control del generador; cuando invocas el generador `*foo()`, ahora estás delegando con `yield` a su *iterador*. Pero en realidad puedes delegar con `yield` a cualquier *iterable*; `yield *[1,2,3]` consumiría el *iterador* por defecto del valor array `[1,2,3]`.

### ¿Por Qué Delegación?

El propósito de la delegación con `yield` es mayormente organización de código, y de esa manera es simétrica con las llamadas normales a funciones.

Imagina dos módulos que respectivamente proporcionan métodos `foo()` y `bar()`, donde `bar()` llama a `foo()`. La razón por la que los dos están separados es generalmente porque la organización apropiada del código para el programa requiere que estén en funciones separadas. Por ejemplo, puede haber casos donde `foo()` se llama de forma independiente, y otros lugares donde `bar()` llama a `foo()`.

Por todas estas mismas razones, mantener generadores separados ayuda a la legibilidad, mantenibilidad y depurabilidad del programa. En ese sentido, `yield *` es un atajo sintáctico para iterar manualmente sobre los pasos de `*foo()` mientras estás dentro de `*bar()`.

Tal enfoque manual sería especialmente complejo si los pasos en `*foo()` fueran asíncronos, razón por la cual probablemente necesitarías usar esa utilidad `run(..)` para hacerlo. Y como hemos mostrado, `yield *foo()` elimina la necesidad de una sub-instancia de la utilidad `run(..)` (como `run(foo)`).

### Delegando Mensajes

Puede que te preguntes cómo funciona esta delegación con `yield` no solo con el control del *iterador* sino con el paso de mensajes bidireccional. Sigue cuidadosamente el flujo de mensajes entrando y saliendo, a través de la delegación con `yield`:

```js
function *foo() {
	console.log( "inside `*foo()`:", yield "B" );

	console.log( "inside `*foo()`:", yield "C" );

	return "D";
}

function *bar() {
	console.log( "inside `*bar()`:", yield "A" );

	// ¡delegación con `yield`!
	console.log( "inside `*bar()`:", yield *foo() );

	console.log( "inside `*bar()`:", yield "E" );

	return "F";
}

var it = bar();

console.log( "outside:", it.next().value );
// outside: A

console.log( "outside:", it.next( 1 ).value );
// inside `*bar()`: 1
// outside: B

console.log( "outside:", it.next( 2 ).value );
// inside `*foo()`: 2
// outside: C

console.log( "outside:", it.next( 3 ).value );
// inside `*foo()`: 3
// inside `*bar()`: D
// outside: E

console.log( "outside:", it.next( 4 ).value );
// inside `*bar()`: 4
// outside: F
```

Presta particular atención a los pasos de procesamiento después de la llamada `it.next(3)`:

1. El valor `3` se pasa (a través de la delegación con `yield` en `*bar()`) a la expresión `yield "C"` en espera dentro de `*foo()`.
2. `*foo()` luego llama a `return "D"`, pero este valor no se devuelve directamente a la llamada exterior `it.next(3)`.
3. En cambio, el valor `"D"` se envía como resultado de la expresión `yield *foo()` en espera dentro de `*bar()` -- esta expresión de delegación con `yield` se ha pausado esencialmente mientras todo `*foo()` fue agotado. Así que `"D"` termina dentro de `*bar()` para que lo imprima.
4. `yield "E"` se llama dentro de `*bar()`, y el valor `"E"` se emite hacia afuera como resultado de la llamada `it.next(3)`.

Desde la perspectiva del *iterador* externo (`it`), no parece diferente entre controlar el generador inicial o uno delegado.

De hecho, la delegación con `yield` ni siquiera tiene que dirigirse a otro generador; simplemente puede dirigirse a un *iterable* general no-generador. Por ejemplo:

```js
function *bar() {
	console.log( "inside `*bar()`:", yield "A" );

	// ¡delegación con `yield` a un no-generador!
	console.log( "inside `*bar()`:", yield *[ "B", "C", "D" ] );

	console.log( "inside `*bar()`:", yield "E" );

	return "F";
}

var it = bar();

console.log( "outside:", it.next().value );
// outside: A

console.log( "outside:", it.next( 1 ).value );
// inside `*bar()`: 1
// outside: B

console.log( "outside:", it.next( 2 ).value );
// outside: C

console.log( "outside:", it.next( 3 ).value );
// outside: D

console.log( "outside:", it.next( 4 ).value );
// inside `*bar()`: undefined
// outside: E

console.log( "outside:", it.next( 5 ).value );
// inside `*bar()`: 5
// outside: F
```

Nota las diferencias en dónde se recibieron/reportaron los mensajes entre este ejemplo y el anterior.

Lo más llamativo, el *iterador* por defecto del `array` no se preocupa por ningún mensaje enviado a través de llamadas `next(..)`, así que los valores `2`, `3`, y `4` son esencialmente ignorados. Además, porque ese *iterador* no tiene un valor `return` explícito (a diferencia del `*foo()` usado previamente), la expresión `yield *` obtiene un `undefined` cuando termina.

#### ¡Excepciones Delegadas También!

De la misma manera que la delegación con `yield` pasa mensajes transparentemente en ambas direcciones, los errores/excepciones también pasan en ambas direcciones:

```js
function *foo() {
	try {
		yield "B";
	}
	catch (err) {
		console.log( "error caught inside `*foo()`:", err );
	}

	yield "C";

	throw "D";
}

function *bar() {
	yield "A";

	try {
		yield *foo();
	}
	catch (err) {
		console.log( "error caught inside `*bar()`:", err );
	}

	yield "E";

	yield *baz();

	// nota: ¡no puede llegar aquí!
	yield "G";
}

function *baz() {
	throw "F";
}

var it = bar();

console.log( "outside:", it.next().value );
// outside: A

console.log( "outside:", it.next( 1 ).value );
// outside: B

console.log( "outside:", it.throw( 2 ).value );
// error caught inside `*foo()`: 2
// outside: C

console.log( "outside:", it.next( 3 ).value );
// error caught inside `*bar()`: D
// outside: E

try {
	console.log( "outside:", it.next( 4 ).value );
}
catch (err) {
	console.log( "error caught outside:", err );
}
// error caught outside: F
```

Algunas cosas a notar de este fragmento:

1. Cuando llamamos a `it.throw(2)`, envía el mensaje de error `2` dentro de `*bar()`, que lo delega a `*foo()`, que luego lo captura con `catch` y lo maneja elegantemente. Luego, el `yield "C"` envía `"C"` de vuelta como el `value` de retorno de la llamada `it.throw(2)`.
2. El valor `"D"` que luego se lanza con `throw` desde dentro de `*foo()` se propaga hacia `*bar()`, que lo captura con `catch` y lo maneja elegantemente. Luego el `yield "E"` envía `"E"` de vuelta como el `value` de retorno de la llamada `it.next(3)`.
3. A continuación, la excepción lanzada con `throw` desde `*baz()` no es capturada en `*bar()` -- aunque sí la capturamos afuera -- así que tanto `*baz()` como `*bar()` son establecidos en estado completado. Después de este fragmento, no serías capaz de obtener el valor `"G"` con ninguna llamada `next(..)` subsiguiente -- simplemente devolverían `undefined` para `value`.

### Delegando Asincronía

Volvamos finalmente a nuestro ejemplo anterior de delegación con `yield` con las múltiples peticiones Ajax secuenciales:

```js
function *foo() {
	var r2 = yield request( "http://some.url.2" );
	var r3 = yield request( "http://some.url.3/?v=" + r2 );

	return r3;
}

function *bar() {
	var r1 = yield request( "http://some.url.1" );

	var r3 = yield *foo();

	console.log( r3 );
}

run( bar );
```

En lugar de llamar a `yield run(foo)` dentro de `*bar()`, simplemente llamamos a `yield *foo()`.

En la versión anterior de este ejemplo, el mecanismo de Promesas (controlado por `run(..)`) fue usado para transportar el valor del `return r3` en `*foo()` a la variable local `r3` dentro de `*bar()`. Ahora, ese valor simplemente se devuelve directamente a través de la mecánica de `yield *`.

De lo contrario, el comportamiento es prácticamente idéntico.

### "Recursión" Delegada

Por supuesto, la delegación con `yield` puede seguir tantos pasos de delegación como conectes. Incluso podrías usar delegación con `yield` para "recursión" de generadores con capacidad asíncrona -- un generador delegando con `yield` a sí mismo:

```js
function *foo(val) {
	if (val > 1) {
		// recursión de generadores
		val = yield *foo( val - 1 );
	}

	return yield request( "http://some.url/?v=" + val );
}

function *bar() {
	var r1 = yield *foo( 3 );
	console.log( r1 );
}

run( bar );
```

**Nota:** Nuestra utilidad `run(..)` podría haberse llamado con `run( foo, 3 )`, porque soporta parámetros adicionales siendo pasados a la inicialización del generador. Sin embargo, usamos aquí un `*bar()` sin parámetros para resaltar la flexibilidad de `yield *`.

¿Qué pasos de procesamiento sigue ese código? Aguanta, esto va a ser bastante intrincado de describir en detalle:

1. `run(bar)` inicia el generador `*bar()`.
2. `foo(3)` crea un *iterador* para `*foo(..)` y pasa `3` como su parámetro `val`.
3. Porque `3 > 1`, `foo(2)` crea otro *iterador* y pasa `2` como su parámetro `val`.
4. Porque `2 > 1`, `foo(1)` crea otro *iterador* más y pasa `1` como su parámetro `val`.
5. `1 > 1` es `false`, así que luego llamamos a `request(..)` con el valor `1`, y obtenemos una promesa de vuelta para esa primera llamada Ajax.
6. Esa promesa se emite con `yield`, que vuelve a la instancia del generador `*foo(2)`.
7. El `yield *` pasa esa promesa de vuelta a la instancia del generador `*foo(3)`. Otro `yield *` pasa la promesa a la instancia del generador `*bar()`. Y aún otro `yield *` más pasa la promesa a la utilidad `run(..)`, que esperará en esa promesa (para la primera petición Ajax) para proceder.
8. Cuando la promesa se resuelve, su mensaje de cumplimiento se envía para reanudar `*bar()`, que pasa a través del `yield *` a la instancia `*foo(3)`, que luego pasa a través del `yield *` a la instancia del generador `*foo(2)`, que luego pasa a través del `yield *` al `yield` normal que está esperando en la instancia del generador `*foo(3)`.
9. La respuesta Ajax de esa primera llamada se devuelve inmediatamente con `return` de la instancia del generador `*foo(3)`, que envía ese valor de vuelta como resultado de la expresión `yield *` en la instancia `*foo(2)`, y se asigna a su variable local `val`.
10. Dentro de `*foo(2)`, una segunda petición Ajax se hace con `request(..)`, cuya promesa se emite con `yield` de vuelta a la instancia `*foo(1)`, y luego `yield *` se propaga todo el camino hasta `run(..)` (paso 7 de nuevo). Cuando la promesa se resuelve, la segunda respuesta Ajax se propaga todo el camino de vuelta a la instancia del generador `*foo(2)`, y se asigna a su variable local `val`.
11. Finalmente, la tercera petición Ajax se hace con `request(..)`, su promesa sale a `run(..)`, y luego su valor de resolución vuelve todo el camino, que luego se devuelve con `return` para que vuelva a la expresión `yield *` en espera en `*bar()`.
