## Resumen de la API de Promesas

Revisemos la API ES6 `Promise` que ya hemos visto desplegarse en fragmentos a lo largo de este capítulo.

**Nota:** La siguiente API es nativa solo a partir de ES6, pero existen polyfills conformes con la especificación (no solo bibliotecas extendidas de Promesas) que pueden definir `Promise` y todo su comportamiento asociado para que puedas usar Promesas nativas incluso en navegadores pre-ES6. Uno de estos polyfills es "Native Promise Only" (http://github.com/getify/native-promise-only), ¡que escribí yo!

### Constructor new Promise(..)

El *constructor revelador* `Promise(..)` debe usarse con `new`, y debe proporcionársele un callback de función que es llamado síncrona/inmediatamente. A esta función se le pasan dos callbacks de función que actúan como capacidades de resolución para la promesa. Comúnmente los etiquetamos `resolve(..)` y `reject(..)`:

```js
var p = new Promise( function(resolve,reject){
	// `resolve(..)` para resolver/cumplir la promesa
	// `reject(..)` para rechazar la promesa
} );
```

`reject(..)` simplemente rechaza la promesa, pero `resolve(..)` puede cumplir o rechazar la promesa, dependiendo de lo que se le pase. Si a `resolve(..)` se le pasa un valor inmediato, no-Promesa, no-thenable, entonces la promesa se cumple con ese valor.

Pero si a `resolve(..)` se le pasa una Promesa genuina o un valor thenable, ese valor se desenvuelve recursivamente, y cualquiera que sea su resolución/estado final será adoptado por la promesa.

### Promise.resolve(..) y Promise.reject(..)

Un atajo para crear una Promesa ya rechazada es `Promise.reject(..)`, así que estas dos promesas son equivalentes:

```js
var p1 = new Promise( function(resolve,reject){
	reject( "Oops" );
} );

var p2 = Promise.reject( "Oops" );
```

`Promise.resolve(..)` usualmente se usa para crear una Promesa ya cumplida de manera similar a `Promise.reject(..)`. Sin embargo, `Promise.resolve(..)` también desenvuelve valores thenable (como ya se ha discutido varias veces). En ese caso, la Promesa devuelta adopta la resolución final del thenable que pasaste, que podría ser cumplimiento o rechazo:

```js
var fulfilledTh = {
	then: function(cb) { cb( 42 ); }
};
var rejectedTh = {
	then: function(cb,errCb) {
		errCb( "Oops" );
	}
};

var p1 = Promise.resolve( fulfilledTh );
var p2 = Promise.resolve( rejectedTh );

// `p1` será una promesa cumplida
// `p2` será una promesa rechazada
```

Y recuerda, `Promise.resolve(..)` no hace nada si lo que pasas ya es una Promesa genuina; simplemente devuelve el valor directamente. Así que no hay sobrecarga en llamar a `Promise.resolve(..)` sobre valores de los que no conoces la naturaleza, si uno resulta ser ya una Promesa genuina.

### then(..) y catch(..)

Cada instancia de Promesa (**no** el namespace de la API `Promise`) tiene métodos `then(..)` y `catch(..)`, que permiten registrar manejadores de cumplimiento y rechazo para la Promesa. Una vez que la Promesa está resuelta, uno u otro de estos manejadores será llamado, pero no ambos, y siempre será llamado asíncronamente (ver "Jobs" en el Capítulo 1).

`then(..)` toma uno o dos parámetros, el primero para el callback de cumplimiento, y el segundo para el callback de rechazo. Si cualquiera se omite o se pasa como un valor no-función, un callback por defecto se sustituye respectivamente. El callback de cumplimiento por defecto simplemente pasa el mensaje, mientras que el callback de rechazo por defecto simplemente relanza (propaga) la razón de error que recibe.

`catch(..)` toma solo el callback de rechazo como parámetro, y automáticamente sustituye el callback de cumplimiento por defecto, como se acaba de discutir. En otras palabras, es equivalente a `then(null,..)`:

```js
p.then( fulfilled );

p.then( fulfilled, rejected );

p.catch( rejected ); // o `p.then( null, rejected )`
```

`then(..)` y `catch(..)` también crean y devuelven una nueva promesa, que puede usarse para expresar control de flujo de cadena de Promesas. Si los callbacks de cumplimiento o rechazo lanzan una excepción, la promesa devuelta es rechazada. Si cualquier callback devuelve un valor inmediato, no-Promesa, no-thenable, ese valor se establece como el cumplimiento de la promesa devuelta. Si el manejador de cumplimiento específicamente devuelve una promesa o valor thenable, ese valor se desenvuelve y se convierte en la resolución de la promesa devuelta.

### Promise.all([ .. ]) y Promise.race([ .. ])

Los helpers estáticos `Promise.all([ .. ])` y `Promise.race([ .. ])` en la API ES6 `Promise` ambos crean una Promesa como su valor de retorno. La resolución de esa promesa es controlada enteramente por el array de promesas que pases.

Para `Promise.all([ .. ])`, todas las promesas que pases deben cumplirse para que la promesa devuelta se cumpla. Si cualquier promesa es rechazada, la promesa principal devuelta es rechazada inmediatamente también (descartando los resultados de cualquiera de las otras promesas). Para cumplimiento, recibes un `array` de todos los valores de cumplimiento de las promesas pasadas. Para rechazo, recibes solo la primera razón de rechazo de promesa. Este patrón se llama clásicamente una "compuerta": todas deben llegar antes de que la compuerta se abra.

Para `Promise.race([ .. ])`, solo la primera promesa en resolverse (cumplimiento o rechazo) "gana", y cualquiera que sea esa resolución se convierte en la resolución de la promesa devuelta. Este patrón se llama clásicamente un "pestillo" (latch): el primero en abrir el pestillo pasa. Considera:

```js
var p1 = Promise.resolve( 42 );
var p2 = Promise.resolve( "Hello World" );
var p3 = Promise.reject( "Oops" );

Promise.race( [p1,p2,p3] )
.then( function(msg){
	console.log( msg );		// 42
} );

Promise.all( [p1,p2,p3] )
.catch( function(err){
	console.error( err );	// "Oops"
} );

Promise.all( [p1,p2] )
.then( function(msgs){
	console.log( msgs );	// [42,"Hello World"]
} );
```

**Advertencia:** ¡Ten cuidado! Si un `array` vacío se pasa a `Promise.all([ .. ])`, se cumplirá inmediatamente, pero `Promise.race([ .. ])` se colgará para siempre y nunca se resolverá.

La API ES6 `Promise` es bastante simple y directa. Es al menos lo suficientemente buena para servir los casos asíncronos más básicos, y es un buen lugar para empezar cuando reorganizas tu código del infierno de callbacks a algo mejor.

Pero hay una cantidad enorme de sofisticación asíncrona que las aplicaciones frecuentemente demandan y que las Promesas por sí mismas serán limitadas para abordar. En la siguiente sección, profundizaremos en esas limitaciones como motivaciones para el beneficio de las bibliotecas de Promesas.

## Limitaciones de las Promesas

Muchos de los detalles que discutiremos en esta sección ya han sido aludidos en este capítulo, pero solo nos aseguraremos de revisar estas limitaciones específicamente.

### Manejo de Errores en Secuencia

Cubrimos el manejo de errores con sabor a Promesas en detalle anteriormente en este capítulo. Las limitaciones de cómo las Promesas están diseñadas -- cómo se encadenan, específicamente -- crean una trampa muy fácil donde un error en una cadena de Promesas puede ser silenciosamente ignorado accidentalmente.

Pero hay algo más a considerar con los errores de Promesas. Porque una cadena de Promesas no es más que sus Promesas constituyentes conectadas entre sí, no hay ninguna entidad para referirse a la cadena entera como una sola *cosa*, lo que significa que no hay manera externa de observar cualquier error que pueda ocurrir.

Si construyes una cadena de Promesas que no tiene manejo de errores en ella, cualquier error en cualquier parte de la cadena se propagará indefinidamente por la cadena, hasta ser observado (registrando un manejador de rechazo en algún paso). Así que, en ese caso específico, tener una referencia a la *última* promesa en la cadena es suficiente (`p` en el siguiente fragmento), porque puedes registrar un manejador de rechazo ahí, y será notificado de cualquier error propagado:

```js
// `foo(..)`, `STEP2(..)` y `STEP3(..)` son
// todas utilidades conscientes de promesas

var p = foo( 42 )
.then( STEP2 )
.then( STEP3 );
```

Aunque pueda parecer confusamente furtivo, `p` aquí no apunta a la primera promesa en la cadena (la de la llamada `foo(42)`), sino a la última promesa, la que viene de la llamada `then(STEP3)`.

Además, ningún paso en la cadena de promesas está observablemente haciendo su propio manejo de errores. Eso significa que podrías entonces registrar un manejador de error de rechazo en `p`, y sería notificado si ocurre cualquier error en cualquier parte de la cadena:

```
p.catch( handleErrors );
```

Pero si cualquier paso de la cadena de hecho hace su propio manejo de errores (quizás oculto/abstraído fuera de lo que puedes ver), tu `handleErrors(..)` no será notificado. Esto puede ser lo que quieres -- era, después de todo, un "rechazo manejado" -- pero también puede *no* ser lo que quieres. La completa falta de capacidad para ser notificado (de errores de rechazo "ya manejados") es una limitación que restringe capacidades en algunos casos de uso.

Es básicamente la misma limitación que existe con un `try..catch` que puede capturar una excepción y simplemente tragarla. Así que esto no es una limitación **única de las Promesas**, pero *es* algo para lo que podríamos desear tener una solución alternativa.

Desafortunadamente, muchas veces no se mantiene ninguna referencia para los pasos intermedios en una secuencia de cadena de Promesas, así que sin tales referencias, no puedes adjuntar manejadores de error para observar confiablemente los errores.

### Valor Único

Las Promesas por definición solo tienen un único valor de cumplimiento o una única razón de rechazo. En ejemplos simples, esto no es tan gran cosa, pero en escenarios más sofisticados, puedes encontrar esto limitante.

El consejo típico es construir un envoltorio de valores (como un `object` o `array`) para contener estos múltiples mensajes. Esta solución funciona, pero puede ser bastante incómodo y tedioso envolver y desenvolver tus mensajes con cada paso individual de tu cadena de Promesas.

#### Dividiendo Valores

A veces puedes tomar esto como una señal de que podrías/deberías descomponer el problema en dos o más Promesas.

Imagina que tienes una utilidad `foo(..)` que produce dos valores (`x` e `y`) asíncronamente:

```js
function getY(x) {
	return new Promise( function(resolve,reject){
		setTimeout( function(){
			resolve( (3 * x) - 1 );
		}, 100 );
	} );
}

function foo(bar,baz) {
	var x = bar * baz;

	return getY( x )
	.then( function(y){
		// envolver ambos valores en un contenedor
		return [x,y];
	} );
}

foo( 10, 20 )
.then( function(msgs){
	var x = msgs[0];
	var y = msgs[1];

	console.log( x, y );	// 200 599
} );
```

Primero, reorganicemos lo que `foo(..)` devuelve para que no tengamos que envolver `x` e `y` en un único valor `array` para transportar a través de una Promesa. En su lugar, podemos envolver cada valor en su propia promesa:

```js
function foo(bar,baz) {
	var x = bar * baz;

	// devolver ambas promesas
	return [
		Promise.resolve( x ),
		getY( x )
	];
}

Promise.all(
	foo( 10, 20 )
)
.then( function(msgs){
	var x = msgs[0];
	var y = msgs[1];

	console.log( x, y );
} );
```

¿Es un `array` de promesas realmente mejor que un `array` de valores pasados a través de una sola promesa? Sintácticamente, no es una gran mejora.

Pero este enfoque abraza más de cerca la teoría de diseño de Promesas. Es ahora más fácil en el futuro refactorizar para dividir el cálculo de `x` e `y` en funciones separadas. Es más limpio y flexible dejar que el código que llama decida cómo orquestar las dos promesas -- usando `Promise.all([ .. ])` aquí, pero ciertamente no la única opción -- en lugar de abstraer tales detalles dentro de `foo(..)`.

#### Desenvolver/Expandir Argumentos

Las asignaciones `var x = ..` y `var y = ..` son todavía sobrecarga incómoda. Podemos emplear algo de triquiñuela funcional (sombrero en alto a Reginald Braithwaite, @raganwald en Twitter) en una utilidad helper:

```js
function spread(fn) {
	return Function.apply.bind( fn, null );
}

Promise.all(
	foo( 10, 20 )
)
.then(
	spread( function(x,y){
		console.log( x, y );	// 200 599
	} )
)
```

¡Eso es un poco más agradable! Por supuesto, podrías poner en línea la magia funcional para evitar el helper extra:

```js
Promise.all(
	foo( 10, 20 )
)
.then( Function.apply.bind(
	function(x,y){
		console.log( x, y );	// 200 599
	},
	null
) );
```

Estos trucos pueden ser ingeniosos, pero ES6 tiene una respuesta aún mejor para nosotros: desestructuración. La forma de asignación desestructuradora de array luce así:

```js
Promise.all(
	foo( 10, 20 )
)
.then( function(msgs){
	var [x,y] = msgs;

	console.log( x, y );	// 200 599
} );
```

Pero lo mejor de todo, ES6 ofrece la forma de parámetro desestructurante de array:

```js
Promise.all(
	foo( 10, 20 )
)
.then( function([x,y]){
	console.log( x, y );	// 200 599
} );
```

¡Ahora hemos adoptado el mantra de un-valor-por-Promesa, pero mantenido nuestro código repetitivo de soporte al mínimo!

**Nota:** Para más información sobre formas de desestructuración ES6, ve el título *ES6 & Beyond* de esta serie.

### Resolución Única

Uno de los comportamientos más intrínsecos de las Promesas es que una Promesa solo puede resolverse una vez (cumplimiento o rechazo). Para muchos casos de uso asíncronos, solo estás recuperando un valor una vez, así que esto funciona bien.

Pero también hay muchos casos asíncronos que encajan en un modelo diferente -- uno que es más parecido a eventos y/o flujos de datos. No está claro en la superficie qué tan bien las Promesas pueden encajar en tales casos de uso, si es que lo hacen. Sin una abstracción significativa encima de las Promesas, se quedarán completamente cortas para manejar resolución de múltiples valores.

Imagina un escenario donde podrías querer disparar una secuencia de pasos asíncronos en respuesta a un estímulo (como un evento) que de hecho puede ocurrir múltiples veces, como un clic de botón.

Esto probablemente no funcionará de la manera que quieres:

```js
// `click(..)` vincula el evento `"click"` a un elemento DOM
// `request(..)` es la utilidad Ajax consciente de Promesas definida antes

var p = new Promise( function(resolve,reject){
	click( "#mybtn", resolve );
} );

p.then( function(evt){
	var btnID = evt.currentTarget.id;
	return request( "http://some.url.1/?id=" + btnID );
} )
.then( function(text){
	console.log( text );
} );
```

El comportamiento aquí solo funciona si tu aplicación requiere que el botón sea clicado solo una vez. Si el botón es clicado una segunda vez, la promesa `p` ya ha sido resuelta, así que la segunda llamada a `resolve(..)` sería ignorada.

En su lugar, probablemente necesitarías invertir el paradigma, creando una cadena de Promesas completamente nueva para cada disparo de evento:

```js
click( "#mybtn", function(evt){
	var btnID = evt.currentTarget.id;

	request( "http://some.url.1/?id=" + btnID )
	.then( function(text){
		console.log( text );
	} );
} );
```

Este enfoque *funcionará* en que una secuencia de Promesas completamente nueva será disparada para cada evento `"click"` en el botón.

Pero más allá de solo la fealdad de tener que definir toda la cadena de Promesas dentro del manejador de eventos, este diseño en algunos aspectos viola la idea de separación de preocupaciones/capacidades (SoC). Podrías muy bien querer definir tu manejador de eventos en un lugar diferente en tu código de donde defines la *respuesta* al evento (la cadena de Promesas). Eso es bastante incómodo de hacer en este patrón, sin mecanismos helper.

**Nota:** Otra forma de articular esta limitación es que sería agradable si pudiéramos construir algún tipo de "observable" al que podamos suscribir una cadena de Promesas. Hay bibliotecas que han creado estas abstracciones (como RxJS -- http://rxjs.codeplex.com/), pero las abstracciones pueden parecer tan pesadas que ya no puedes ver la naturaleza de las Promesas. Tal abstracción pesada trae preguntas importantes a la mente como si (sin Promesas) estos mecanismos son tan *confiables* como las Promesas mismas han sido diseñadas para ser. Revisitaremos el patrón "Observable" en el Apéndice B.

### Inercia

Una barrera concreta para comenzar a usar Promesas en tu propio código es todo el código que actualmente existe y que no es consciente de Promesas. Si tienes mucho código basado en callbacks, es mucho más fácil simplemente seguir codificando en ese mismo estilo.

"Una base de código en movimiento (con callbacks) permanecerá en movimiento (con callbacks) a menos que una fuerza actúe sobre ella por un desarrollador inteligente y consciente de Promesas."

Las Promesas ofrecen un paradigma diferente, y como tal, el enfoque del código puede ser desde solo un poco diferente hasta, en algunos casos, radicalmente diferente. Tienes que ser intencional al respecto, porque las Promesas no saldrán naturalmente de las mismas viejas formas de hacer código que te han servido bien hasta ahora.

Considera un escenario basado en callbacks como el siguiente:

```js
function foo(x,y,cb) {
	ajax(
		"http://some.url.1/?x=" + x + "&y=" + y,
		cb
	);
}

foo( 11, 31, function(err,text) {
	if (err) {
		console.error( err );
	}
	else {
		console.log( text );
	}
} );
```

¿Es inmediatamente obvio cuáles son los primeros pasos para convertir este código basado en callbacks a código consciente de Promesas? Depende de tu experiencia. Cuanta más práctica tengas con ello, más natural se sentirá. Pero ciertamente, las Promesas no anuncian en la etiqueta exactamente cómo hacerlo -- no hay una respuesta única para todo -- así que la responsabilidad depende de ti.

Como hemos cubierto antes, definitivamente necesitamos una utilidad Ajax que sea consciente de Promesas en lugar de basada en callbacks, que podríamos llamar `request(..)`. Puedes hacer la tuya propia, como ya lo hemos hecho. Pero la sobrecarga de tener que definir manualmente envoltorios conscientes de Promesas para cada utilidad basada en callbacks hace menos probable que elijas refactorizar a código consciente de Promesas en absoluto.

Las Promesas no ofrecen una respuesta directa a esa limitación. La mayoría de las bibliotecas de Promesas sí ofrecen un helper, sin embargo. Pero incluso sin una biblioteca, imagina un helper como este:

```js
// verificación guardia segura para polyfill
if (!Promise.wrap) {
	Promise.wrap = function(fn) {
		return function() {
			var args = [].slice.call( arguments );

			return new Promise( function(resolve,reject){
				fn.apply(
					null,
					args.concat( function(err,v){
						if (err) {
							reject( err );
						}
						else {
							resolve( v );
						}
					} )
				);
			} );
		};
	};
}
```

OK, eso es más que solo una pequeña utilidad trivial. Sin embargo, aunque pueda parecer un poco intimidante, no es tan malo como pensarías. Toma una función que espera un callback estilo error-primero como su último parámetro, y devuelve una nueva que automáticamente crea una Promesa para devolver, y sustituye el callback por ti, conectado al cumplimiento/rechazo de la Promesa.

En lugar de perder demasiado tiempo hablando de *cómo* funciona este helper `Promise.wrap(..)`, veamos cómo lo usamos:

```js
var request = Promise.wrap( ajax );

request( "http://some.url.1/" )
.then( .. )
..
```

¡Wow, eso fue bastante fácil!

`Promise.wrap(..)` **no** produce una Promesa. Produce una función que producirá Promesas. En cierto sentido, una función productora de Promesas podría verse como una "fábrica de Promesas". Propongo "promisory" como el nombre para tal cosa ("Promise" + "factory").

El acto de envolver una función que espera callbacks para que sea una función consciente de Promesas se refiere a veces como "lifting" o "promisifying". Pero no parece haber un término estándar para cómo llamar a la función resultante aparte de "función lifted", así que me gusta más "promisory" ya que creo que es más descriptivo.

**Nota:** Promisory no es un término inventado. Es una palabra real, y su definición significa contener o transmitir una promesa. ¡Eso es exactamente lo que estas funciones hacen, así que resulta ser una coincidencia terminológica bastante perfecta!

Así que, `Promise.wrap(ajax)` produce un `ajax(..)` promisory que llamamos `request(..)`, y ese promisory produce Promesas para respuestas Ajax.

Si todas las funciones fueran ya promisories, no necesitaríamos hacerlas nosotros mismos, así que el paso extra es un poco una lástima. Pero al menos el patrón de envolvimiento es (usualmente) repetible así que podemos ponerlo en un helper `Promise.wrap(..)` como se muestra para ayudar a nuestro código con promesas.

Así que volviendo a nuestro ejemplo anterior, necesitamos un promisory tanto para `ajax(..)` como para `foo(..)`:

```js
// hacer un promisory para `ajax(..)`
var request = Promise.wrap( ajax );

// refactorizar `foo(..)`, pero mantenerlo externamente
// basado en callbacks para compatibilidad con otras
// partes del código por ahora -- solo usar
// la promesa de `request(..)` internamente.
function foo(x,y,cb) {
	request(
		"http://some.url.1/?x=" + x + "&y=" + y
	)
	.then(
		function fulfilled(text){
			cb( null, text );
		},
		cb
	);
}

// ahora, para los propósitos de este código, hacer un
// promisory para `foo(..)`
var betterFoo = Promise.wrap( foo );

// y usar el promisory
betterFoo( 11, 31 )
.then(
	function fulfilled(text){
		console.log( text );
	},
	function rejected(err){
		console.error( err );
	}
);
```

Por supuesto, mientras estamos refactorizando `foo(..)` para usar nuestro nuevo promisory `request(..)`, podríamos simplemente hacer a `foo(..)` un promisory en sí mismo, en lugar de permanecer basado en callbacks y necesitar hacer y usar el subsiguiente promisory `betterFoo(..)`. Esta decisión solo depende de si `foo(..)` necesita permanecer compatible basado en callbacks con otras partes de la base de código o no.

Considera:

```js
// `foo(..)` es ahora también un promisory porque
// delega al promisory `request(..)`
function foo(x,y) {
	return request(
		"http://some.url.1/?x=" + x + "&y=" + y
	);
}

foo( 11, 31 )
.then( .. )
..
```

Aunque las Promesas ES6 no vienen nativamente con helpers para tal envolvimiento promisory, la mayoría de las bibliotecas los proporcionan, o puedes hacer el tuyo propio. De cualquier forma, esta limitación particular de las Promesas es abordable sin demasiado dolor (¡ciertamente comparado con el dolor del infierno de callbacks!).

### Promesa No Cancelable

Una vez que creas una Promesa y registras un manejador de cumplimiento y/o rechazo para ella, no hay nada externo que puedas hacer para detener esa progresión si algo más sucede que haga que esa tarea sea irrelevante.

**Nota:** Muchas bibliotecas de abstracción de Promesas proporcionan facilidades para cancelar Promesas, ¡pero esta es una idea terrible! Muchos desarrolladores desearían que las Promesas hubieran sido diseñadas nativamente con capacidad de cancelación externa, pero el problema es que eso dejaría que un consumidor/observador de una Promesa afecte la capacidad de algún otro consumidor de observar esa misma Promesa. Esto viola la confiabilidad del valor-futuro (inmutabilidad externa), pero además es la encarnación del anti-patrón "acción a distancia" (http://en.wikipedia.org/wiki/Action_at_a_distance_%28computer_programming%29). Independientemente de lo útil que parezca, en realidad te llevará directamente de vuelta a las mismas pesadillas que los callbacks.

Considera nuestro escenario de timeout de Promesa de antes:

```js
var p = foo( 42 );

Promise.race( [
	p,
	timeoutPromise( 3000 )
] )
.then(
	doSomething,
	handleError
);

p.then( function(){
	// aún sucede incluso en el caso del timeout :(
} );
```

El "timeout" era externo a la promesa `p`, así que `p` misma sigue adelante, lo cual probablemente no queremos.

Una opción es definir invasivamente tus callbacks de resolución:

```js
var OK = true;

var p = foo( 42 );

Promise.race( [
	p,
	timeoutPromise( 3000 )
	.catch( function(err){
		OK = false;
		throw err;
	} )
] )
.then(
	doSomething,
	handleError
);

p.then( function(){
	if (OK) {
		// ¡solo sucede si no hay timeout! :)
	}
} );
```

Esto es feo. Funciona, pero está lejos de ser ideal. Generalmente, deberías intentar evitar tales escenarios.

Pero si no puedes, la fealdad de esta solución debería ser una pista de que la *cancelación* es una funcionalidad que pertenece a un nivel más alto de abstracción por encima de las Promesas. Te recomendaría que busques en bibliotecas de abstracción de Promesas para asistencia en lugar de hacerlo tú mismo.

**Nota:** Mi biblioteca de abstracción de Promesas *asynquence* proporciona justo tal abstracción y una capacidad de `abort()` para la secuencia, todo lo cual se discutirá en el Apéndice A.

Una sola Promesa no es realmente un mecanismo de control de flujo (al menos no en un sentido muy significativo), que es exactamente a lo que se refiere la *cancelación*; por eso la cancelación de Promesas se sentiría incómoda.

Por contraste, una cadena de Promesas tomadas colectivamente juntas -- lo que me gusta llamar una "secuencia" -- *es* una expresión de control de flujo, y por lo tanto es apropiado que la cancelación sea definida en ese nivel de abstracción.

Ninguna Promesa individual debería ser cancelable, pero tiene sentido que una *secuencia* sea cancelable, porque no pasas una secuencia como un único valor inmutable como sí lo haces con una Promesa.

### Rendimiento de Promesas

Esta limitación particular es tanto simple como compleja.

Comparando cuántas piezas se mueven con una cadena de tareas asíncronas basadas en callbacks básicas versus una cadena de Promesas, está claro que las Promesas tienen bastante más sucediendo, lo que significa que son naturalmente al menos un poquito más lentas. Piensa de vuelta en solo la simple lista de garantías de confianza que las Promesas ofrecen, comparada con el código de solución ad hoc que tendrías que agregar encima de los callbacks para lograr las mismas protecciones.

Más trabajo que hacer, más guardas que proteger, significa que las Promesas *son* más lentas comparadas con callbacks desnudos y no confiables. Eso es obvio, y probablemente simple de envolver con tu cerebro.

Pero ¿cuánto más lentas? Bueno... eso en realidad está demostrando ser una pregunta increíblemente difícil de responder absolutamente, en todos los escenarios.

Francamente, es como una comparación de manzanas con naranjas, así que probablemente es la pregunta equivocada que hacer. Deberías en realidad comparar si un sistema de callbacks ad hoc con todas las mismas protecciones agregadas manualmente es más rápido que una implementación de Promesas.

Si las Promesas tienen una limitación de rendimiento legítima, es más que realmente no ofrecen una opción item por item sobre qué protecciones de confiabilidad quieres/necesitas o no -- las obtienes todas, siempre.

Sin embargo, si concedemos que una Promesa es generalmente un *poco más lenta* que su equivalente de callback no-Promesa, no confiable -- asumiendo que hay lugares donde sientes que puedes justificar la falta de confiabilidad -- ¿significa eso que las Promesas deberían evitarse en todos los casos, como si tu aplicación entera estuviera impulsada por nada más que código que debe-ser-absolutamente-lo-más-rápido-posible?

Verificación de cordura: si tu código es legítimamente así, **¿es JavaScript siquiera el lenguaje correcto para tales tareas?** JavaScript puede ser optimizado para ejecutar aplicaciones muy eficientemente (ver Capítulo 5 y Capítulo 6). Pero ¿es obsesionarse con pequeños intercambios de rendimiento con Promesas, a la luz de todos los beneficios que ofrecen, *realmente* apropiado?

Otro problema sutil es que las Promesas hacen *todo* asíncrono, lo que significa que algunos pasos que se completan inmediatamente (síncronamente) aún difieren el avance del siguiente paso a un Job (ver Capítulo 1). Eso significa que es posible que una secuencia de tareas de Promesa podría completarse ligeramente más lento que la misma secuencia conectada con callbacks.

Por supuesto, la pregunta aquí es esta: ¿valen estos deslices potenciales en pequeñas fracciones de rendimiento *todas* las otras ventajas articuladas de las Promesas que hemos expuesto a lo largo de este capítulo?

Mi postura es que en virtualmente todos los casos donde podrías pensar que el rendimiento de Promesas es lo suficientemente lento como para preocuparte, es en realidad un anti-patrón optimizar alejándose de los beneficios de confiabilidad y composición de Promesas evitándolas por completo.

En su lugar, deberías por defecto usarlas a lo largo de la base de código, y luego perfilar y analizar los caminos calientes (críticos) de tu aplicación. ¿Son las Promesas *realmente* un cuello de botella, o son solo una desaceleración teórica? Solo *entonces*, armado con benchmarks válidos reales (ver Capítulo 6) es responsable y prudente eliminar las Promesas en solo esas áreas críticas identificadas.

Las Promesas son un poco más lentas, pero a cambio obtienes mucha confiabilidad, previsibilidad no-Zalgo, y composición incorporada. Quizás la limitación no es realmente su rendimiento, sino tu falta de percepción de sus beneficios.

## Revisión

Las Promesas son increíbles. Úsalas. Resuelven los problemas de *inversión de control* que nos aquejan con código de solo callbacks.

No se deshacen de los callbacks, solo redirigen la orquestación de esos callbacks a un mecanismo intermediario confiable que se sitúa entre nosotros y otra utilidad.

Las cadenas de Promesas también comienzan a abordar (aunque ciertamente no perfectamente) una mejor manera de expresar flujo asíncrono de forma secuencial, lo que ayuda a nuestros cerebros a planificar y mantener código JS asíncrono mejor. ¡Veremos una solución aún mejor para *ese* problema en el próximo capítulo!
