# You Don't Know JS: Async & Performance
# Capítulo 4: Generadores

En el Capítulo 2, identificamos dos desventajas clave de expresar control de flujo asíncrono con callbacks:

* La asincronía basada en callbacks no encaja con cómo nuestro cerebro planifica los pasos de una tarea.
* Los callbacks no son confiables ni componibles debido a la *inversión de control*.

En el Capítulo 3, detallamos cómo las Promesas desinvierten la *inversión de control* de los callbacks, restaurando la confiabilidad/componibilidad.

Ahora dirigimos nuestra atención a expresar control de flujo asíncrono de una manera secuencial, con apariencia síncrona. La "magia" que lo hace posible son los **generadores** de ES6.

## Rompiendo la Ejecución-Hasta-Completar

En el Capítulo 1, explicamos una expectativa en la que los desarrolladores JS casi universalmente confían en su código: una vez que una función comienza a ejecutarse, se ejecuta hasta que se completa, y ningún otro código puede interrumpir y ejecutarse en medio.

Por extraño que pueda parecer, ES6 introduce un nuevo tipo de función que no se comporta con el comportamiento de ejecución-hasta-completar. Este nuevo tipo de función se llama un "generador".

Para entender las implicaciones, consideremos este ejemplo:

```js
var x = 1;

function foo() {
	x++;
	bar();				// <-- ¿qué hay de esta línea?
	console.log( "x:", x );
}

function bar() {
	x++;
}

foo();					// x: 3
```

En este ejemplo, sabemos con seguridad que `bar()` se ejecuta entre `x++` y `console.log(x)`. ¿Pero qué pasa si `bar()` no estuviera ahí? Obviamente, el resultado sería `2` en lugar de `3`.

Ahora retorcemos tu cerebro. ¿Qué pasa si `bar()` no estuviera presente, pero aún pudiera de algún modo ejecutarse entre las sentencias `x++` y `console.log(x)`? ¿Cómo sería eso posible?

En lenguajes multihilo **preemptivos**, sería esencialmente posible para `bar()` "interrumpir" y ejecutarse en exactamente el momento correcto entre esas dos sentencias. Pero JS no es preemptivo, ni es (actualmente) multihilo. Y sin embargo, una forma **cooperativa** de esta "interrupción" (concurrencia) es posible, si `foo()` mismo pudiera de algún modo indicar una "pausa" en esa parte del código.

**Nota:** Uso la palabra "cooperativa" no solo por la conexión con la terminología clásica de concurrencia (ver Capítulo 1), sino porque como verás en el siguiente fragmento, la sintaxis ES6 para indicar un punto de pausa en el código es `yield` -- sugiriendo una *cooperativa* y cortés cesión de control.

Aquí está el código ES6 para lograr tal concurrencia cooperativa:

```js
var x = 1;

function *foo() {
	x++;
	yield; // ¡pausa!
	console.log( "x:", x );
}

function bar() {
	x++;
}
```

**Nota:** Probablemente verás que la mayoría de otra documentación/código JS formateará una declaración de generador como `function* foo() { .. }` en lugar de como lo he hecho aquí con `function *foo() { .. }` -- la única diferencia siendo la posición estilística del `*`. Las dos formas son funcional/sintácticamente idénticas, como lo es una tercera forma `function*foo() { .. }` (sin espacio). Hay argumentos para ambos estilos, pero yo básicamente prefiero `function *foo..` porque entonces coincide cuando referencio un generador al escribir con `*foo()`. Si dijera solo `foo()`, no sabrías tan claramente si estoy hablando de un generador o una función regular. Es puramente una preferencia estilística.

Ahora, ¿cómo podemos ejecutar el código en el fragmento anterior de modo que `bar()` se ejecute en el punto del `yield` dentro de `*foo()`?

```js
// construir un iterador `it` para controlar el generador
var it = foo();

// ¡empezar `foo()` aquí!
it.next();
x;						// 2
bar();
x;						// 3
it.next();				// x: 3
```

OK, hay bastante contenido nuevo y potencialmente confuso en esos dos fragmentos de código, así que tenemos mucho que recorrer. Pero antes de explicar las diferentes mecánicas/sintaxis con los generadores ES6, recorramos el flujo de comportamiento:

1. La operación `it = foo()` *no* ejecuta el generador `*foo()` todavía, sino que simplemente construye un *iterador* que controlará su ejecución. Más sobre *iteradores* en un momento.
2. El primer `it.next()` inicia el generador `*foo()`, y ejecuta el `x++` en la primera línea de `*foo()`.
3. `*foo()` se pausa en la sentencia `yield`, en cuyo punto la primera llamada `it.next()` termina. En este momento, `*foo()` sigue ejecutándose y activo, pero está en un estado pausado.
4. Inspeccionamos el valor de `x`, y ahora es `2`.
5. Llamamos a `bar()`, que incrementa `x` de nuevo con `x++`.
6. Inspeccionamos el valor de `x` otra vez, y ahora es `3`.
7. La última llamada `it.next()` reanuda el generador `*foo()` desde donde estaba pausado, y ejecuta la sentencia `console.log(..)`, que usa el valor actual de `x` de `3`.

Claramente, `*foo()` empezó, pero *no* se ejecutó-hasta-completar -- se pausó en el `yield`. Reanudamos `*foo()` después, y lo dejamos terminar, pero eso ni siquiera era requerido.

Así que, un generador es un tipo especial de función que puede iniciar y detenerse una o más veces, y no necesariamente tiene que terminar nunca. Aunque aún no será terriblemente obvio por qué eso es tan poderoso, a medida que avancemos por el resto de este capítulo, ese será uno de los bloques fundamentales de construcción que usaremos para construir generadores-como-control-de-flujo-asíncrono como patrón para nuestro código.

### Entrada y Salida

Una función generadora es una función especial con el nuevo modelo de procesamiento al que acabamos de aludir. Pero sigue siendo una función, lo que significa que todavía tiene algunos principios básicos que no han cambiado -- a saber, que todavía acepta argumentos (es decir, "entrada"), y que todavía puede devolver un valor (es decir, "salida"):

```js
function *foo(x,y) {
	return x * y;
}

var it = foo( 6, 7 );

var res = it.next();

res.value;		// 42
```

Pasamos los argumentos `6` y `7` a `*foo(..)` como los parámetros `x` e `y`, respectivamente. Y `*foo(..)` devuelve el valor `42` al código que lo llamó.

Ahora vemos una diferencia con cómo el generador se invoca comparado con una función normal. `foo(6,7)` obviamente parece familiar. Pero sutilmente, el generador `*foo(..)` no se ha ejecutado realmente aún como lo habría hecho con una función.

En cambio, estamos simplemente creando un objeto *iterador*, que asignamos a la variable `it`, para controlar el generador `*foo(..)`. Luego llamamos a `it.next()`, que instruye al generador `*foo(..)` a avanzar desde su ubicación actual, deteniéndose ya sea en el siguiente `yield` o al final del generador.

El resultado de esa llamada `next(..)` es un objeto con una propiedad `value` que contiene cualquier valor (si existe) que fue devuelto por `*foo(..)`. En otras palabras, `yield` causó que un valor fuera enviado fuera del generador durante la mitad de su ejecución, como un `return` intermedio.

De nuevo, aún no será obvio por qué necesitamos todo este objeto *iterador* indirecto para controlar el generador. Llegaremos ahí, lo *prometo*.

#### Mensajería por Iteración

Además de que los generadores aceptan argumentos y tienen valores de retorno, hay una capacidad de mensajería de entrada/salida aún más poderosa y convincente incorporada en ellos, a través de `yield` y `next(..)`.

Considera:

```js
function *foo(x) {
	var y = x * (yield);
	return y;
}

var it = foo( 6 );

// iniciar `foo(..)`
it.next();

var res = it.next( 7 );

res.value;		// 42
```

Primero, pasamos `6` como el parámetro `x`. Luego llamamos a `it.next()`, y eso inicia `*foo(..)`.

Dentro de `*foo(..)`, la sentencia `var y = x ..` comienza a procesarse, pero entonces se encuentra con una expresión `yield`. En ese punto, pausa `*foo(..)` (¡en la mitad de la sentencia de asignación!), y esencialmente solicita al código que lo llama que proporcione un valor resultado para la expresión `yield`. A continuación, llamamos a `it.next( 7 )`, que está pasando el valor `7` de vuelta para *ser* ese resultado de la expresión `yield` pausada.

Así que, en este punto, la sentencia de asignación es esencialmente `var y = 6 * 7`. Ahora, `return y` devuelve ese valor `42` como el resultado de la llamada `it.next( 7 )`.

Nota algo muy importante pero también fácilmente confuso, incluso para desarrolladores JS experimentados: dependiendo de tu perspectiva, hay un desajuste entre el `yield` y la llamada `next(..)`. En general, vas a tener una llamada `next(..)` más de las que tienes sentencias `yield` -- el fragmento anterior tiene un `yield` y dos llamadas `next(..)`.

¿Por qué el desajuste?

Porque el primer `next(..)` siempre inicia un generador, y se ejecuta hasta el primer `yield`. Pero es la segunda llamada `next(..)` la que cumple la primera expresión `yield` pausada, y el tercer `next(..)` cumpliría el segundo `yield`, y así sucesivamente.

##### Cuento de Dos Preguntas

En realidad, qué código estás pensando principalmente afectará si hay un desajuste percibido o no.

Considera solo el código del generador:

```js
var y = x * (yield);
return y;
```

Este **primer** `yield` básicamente está *haciendo una pregunta*: "¿Qué valor debería insertar aquí?"

¿Quién va a responder esa pregunta? Bueno, el **primer** `next()` ya se ha ejecutado para llevar al generador hasta este punto, así que obviamente *él* no puede responder la pregunta. Así que, la **segunda** llamada `next(..)` debe responder la pregunta *planteada* por el **primer** `yield`.

¿Ves el desajuste -- segundo-a-primero?

Pero cambiemos nuestra perspectiva. Veámoslo no desde el punto de vista del generador, sino desde el punto de vista del iterador.

Para ilustrar apropiadamente esta perspectiva, también necesitamos explicar que los mensajes pueden ir en ambas direcciones -- `yield ..` como expresión puede enviar mensajes en respuesta a llamadas `next(..)`, y `next(..)` puede enviar valores a una expresión `yield` pausada. Considera este código ligeramente ajustado:

```js
function *foo(x) {
	var y = x * (yield "Hello");	// <-- ¡yield un valor!
	return y;
}

var it = foo( 6 );

var res = it.next();	// primer `next()`, no pasar nada
res.value;				// "Hello"

res = it.next( 7 );		// pasar `7` al `yield` en espera
res.value;				// 42
```

`yield ..` y `next(..)` se emparejan como un sistema de paso de mensajes bidireccional **durante la ejecución del generador**.

Así que, mirando solo al código del *iterador*:

```js
var res = it.next();	// primer `next()`, no pasar nada
res.value;				// "Hello"

res = it.next( 7 );		// pasar `7` al `yield` en espera
res.value;				// 42
```

**Nota:** No pasamos un valor al primer `next()`, y eso es a propósito. Solo un `yield` pausado podría aceptar tal valor pasado por un `next(..)`, y al comienzo del generador cuando llamamos al primer `next()`, **no hay un `yield` pausado** para aceptar tal valor. La especificación y todos los navegadores conformes simplemente **descartan** silenciosamente cualquier cosa pasada al primer `next()`. Sigue siendo una mala idea pasar un valor, ya que estás creando código silenciosamente "fallido" que es confuso. Así que, siempre inicia un generador con un `next()` sin argumentos.

La primera llamada `next()` (sin nada pasado) básicamente está *haciendo una pregunta*: "¿Qué *siguiente* valor tiene el generador `*foo(..)` para darme?" Y quién responde esta pregunta? La primera expresión `yield "hello"`.

¿Ves? Sin desajuste ahí.

Dependiendo de *quién* pienses que hace la pregunta, hay o no hay un desajuste entre las llamadas `yield` y `next(..)`.

¡Pero espera! Todavía hay un `next()` extra comparado con el número de sentencias `yield`. Así que, esa última llamada `it.next(7)` está de nuevo preguntando qué *siguiente* valor producirá el generador. Pero no quedan más sentencias `yield` para responder, ¿verdad? Entonces, ¿quién responde?

¡La sentencia `return` responde la pregunta!

Y si **no hay `return`** en tu generador -- `return` ciertamente no es más requerido en generadores que en funciones regulares -- siempre hay un `return;` asumido/implícito (es decir, `return undefined;`), que sirve al propósito de responder por defecto la pregunta *planteada* por la última llamada `it.next(7)`.

Estas preguntas y respuestas -- el paso de mensajes bidireccional con `yield` y `next(..)` -- son bastante poderosas, pero no es para nada obvio cómo estos mecanismos se conectan con el control de flujo asíncrono. ¡Estamos llegando ahí!

### Múltiples Iteradores

Puede parecer por el uso sintáctico que cuando usas un *iterador* para controlar un generador, estás controlando la función generadora declarada en sí. Pero hay una sutileza que es fácil pasar por alto: cada vez que construyes un *iterador*, estás implícitamente construyendo una instancia del generador que ese *iterador* controlará.

Puedes tener múltiples instancias del mismo generador ejecutándose al mismo tiempo, e incluso pueden interactuar:

```js
function *foo() {
	var x = yield 2;
	z++;
	var y = yield (x * z);
	console.log( x, y, z );
}

var z = 1;

var it1 = foo();
var it2 = foo();

var val1 = it1.next().value;			// 2 <-- yield 2
var val2 = it2.next().value;			// 2 <-- yield 2

val1 = it1.next( val2 * 10 ).value;		// 40  <-- x:20,  z:2
val2 = it2.next( val1 * 5 ).value;		// 600 <-- x:200, z:3

it1.next( val2 / 2 );					// y:300
										// 20 300 3
it2.next( val1 / 4 );					// y:10
										// 200 10 3
```

**Advertencia:** El uso más común de múltiples instancias del mismo generador ejecutándose concurrentemente no es tales interacciones, sino cuando el generador está produciendo sus propios valores sin entrada, quizás desde algún recurso conectado independientemente. Hablaremos más sobre producción de valores en la siguiente sección.

Recorramos brevemente el procesamiento:

1. Ambas instancias de `*foo()` son iniciadas al mismo tiempo, y ambas llamadas `next()` revelan un `value` de `2` de las sentencias `yield 2`, respectivamente.
2. `val2 * 10` es `2 * 10`, que se envía a la primera instancia del generador `it1`, de modo que `x` obtiene el valor `20`. `z` se incrementa de `1` a `2`, y luego `20 * 2` se emite con `yield`, estableciendo `val1` en `40`.
3. `val1 * 5` es `40 * 5`, que se envía a la segunda instancia del generador `it2`, de modo que `x` obtiene el valor `200`. `z` se incrementa de nuevo, de `2` a `3`, y luego `200 * 3` se emite con `yield`, estableciendo `val2` en `600`.
4. `val2 / 2` es `600 / 2`, que se envía a la primera instancia del generador `it1`, de modo que `y` obtiene el valor `300`, luego imprimiendo `20 300 3` para sus valores `x y z`, respectivamente.
5. `val1 / 4` es `40 / 4`, que se envía a la segunda instancia del generador `it2`, de modo que `y` obtiene el valor `10`, luego imprimiendo `200 10 3` para sus valores `x y z`, respectivamente.

Ese es un ejemplo "divertido" para recorrer en tu mente. ¿Lo seguiste bien?

#### Intercalado

Recuerda este escenario de la sección "Ejecución-hasta-completar" del Capítulo 1:

```js
var a = 1;
var b = 2;

function foo() {
	a++;
	b = b * a;
	a = b + 3;
}

function bar() {
	b--;
	a = 8 + b;
	b = a * 2;
}
```

Con funciones JS normales, por supuesto o `foo()` puede ejecutarse completamente primero, o `bar()` puede ejecutarse completamente primero, pero `foo()` no puede intercalar sus sentencias individuales con `bar()`. Así que, solo hay dos resultados posibles para el programa anterior.

Sin embargo, con generadores, claramente el intercalado (¡incluso en la mitad de las sentencias!) es posible:

```js
var a = 1;
var b = 2;

function *foo() {
	a++;
	yield;
	b = b * a;
	a = (yield b) + 3;
}

function *bar() {
	b--;
	yield;
	a = (yield 8) + b;
	b = a * (yield 2);
}
```

Dependiendo de en qué orden respectivo los *iteradores* que controlan `*foo()` y `*bar()` son llamados, el programa anterior podría producir varios resultados diferentes. En otras palabras, podemos realmente ilustrar (de una forma más o menos simulada) las circunstancias teóricas de "condiciones de carrera de hilos" discutidas en el Capítulo 1, intercalando las dos iteraciones del generador sobre las mismas variables compartidas.

Primero, hagamos un helper llamado `step(..)` que controla un *iterador*:

```js
function step(gen) {
	var it = gen();
	var last;

	return function() {
		// lo que sea que se emita con `yield`,
		// ¡simplemente enviarlo de vuelta la siguiente vez!
		last = it.next( last ).value;
	};
}
```

`step(..)` inicializa un generador para crear su *iterador* `it`, luego devuelve una función que, cuando se llama, avanza el *iterador* un paso. Adicionalmente, el valor previamente emitido con `yield` se envía de vuelta en el *siguiente* paso. Así que, `yield 8` simplemente se convertirá en `8` y `yield b` simplemente será `b` (lo que sea en el momento del `yield`).

Ahora, solo por diversión, experimentemos para ver los efectos de intercalar estos diferentes fragmentos de `*foo()` y `*bar()`. Empezaremos con el caso base aburrido, asegurándonos de que `*foo()` termine totalmente antes de `*bar()` (justo como hicimos en el Capítulo 1):

```js
// asegurar reiniciar `a` y `b`
a = 1;
b = 2;

var s1 = step( foo );
var s2 = step( bar );

// ejecutar `*foo()` completamente primero
s1();
s1();
s1();

// ahora ejecutar `*bar()`
s2();
s2();
s2();
s2();

console.log( a, b );	// 11 22
```

El resultado final es `11` y `22`, justo como fue en la versión del Capítulo 1. Ahora mezclemos el orden de intercalado y veamos cómo cambia los valores finales de `a` y `b`:

```js
// asegurar reiniciar `a` y `b`
a = 1;
b = 2;

var s1 = step( foo );
var s2 = step( bar );

s2();		// b--;
s2();		// yield 8
s1();		// a++;
s2();		// a = 8 + b;
			// yield 2
s1();		// b = b * a;
			// yield b
s1();		// a = b + 3;
s2();		// b = a * 2;
```

Antes de decirte los resultados, ¿puedes descifrar cuánto valen `a` y `b` después del programa anterior? ¡Sin hacer trampa!

```js
console.log( a, b );	// 12 18
```

**Nota:** Como ejercicio para el lector, intenta ver cuántas otras combinaciones de resultados puedes obtener reordenando las llamadas `s1()` y `s2()`. No olvides que siempre necesitarás tres llamadas `s1()` y cuatro llamadas `s2()`. Recuerda la discusión anterior sobre emparejar `next()` con `yield` para las razones de por qué.

Casi seguramente no querrás crear intencionalmente *este* nivel de confusión de intercalado, ya que crea código increíblemente difícil de entender. Pero el ejercicio es interesante e instructivo para entender más sobre cómo múltiples generadores pueden ejecutarse concurrentemente en el mismo ámbito compartido, porque habrá lugares donde esta capacidad es bastante útil.

Discutiremos la concurrencia de generadores con más detalle al final de este capítulo.

## Generando Valores

En la sección anterior, mencionamos un uso interesante para generadores, como una forma de producir valores. Esto **no** es el enfoque principal en este capítulo, pero seríamos negligentes si no cubriéramos lo básico, especialmente porque este caso de uso es esencialmente el origen del nombre: generadores.

Vamos a tomar un ligero desvío al tema de los *iteradores* por un momento, pero regresaremos a cómo se relacionan con los generadores y usar un generador para *generar* valores.

### Productores e Iteradores

Imagina que estás produciendo una serie de valores donde cada valor tiene una relación definible con el valor anterior. Para hacer esto, vas a necesitar un productor con estado que recuerde el último valor que entregó.

Puedes implementar algo así de manera directa usando un closure de función (ver el título *Scope & Closures* de esta serie):

```js
var gimmeSomething = (function(){
	var nextVal;

	return function(){
		if (nextVal === undefined) {
			nextVal = 1;
		}
		else {
			nextVal = (3 * nextVal) + 6;
		}

		return nextVal;
	};
})();

gimmeSomething();		// 1
gimmeSomething();		// 9
gimmeSomething();		// 33
gimmeSomething();		// 105
```

**Nota:** La lógica de cálculo de `nextVal` aquí podría haberse simplificado, pero conceptualmente, no queremos calcular el *siguiente valor* (es decir, `nextVal`) hasta que la *siguiente* llamada `gimmeSomething()` ocurra, porque en general eso podría ser un diseño con fugas de recursos para productores de valores más persistentes o limitados en recursos que simples `number`s.

Generar una serie de números arbitraria no es un ejemplo terriblemente realista. ¿Pero qué pasa si estuvieras generando registros desde una fuente de datos? Podrías imaginar prácticamente el mismo código.

De hecho, esta tarea es un patrón de diseño muy común, usualmente resuelto por iteradores. Un *iterador* es una interfaz bien definida para recorrer una serie de valores de un productor. La interfaz JS para iteradores, como en la mayoría de los lenguajes, es llamar a `next()` cada vez que quieres el siguiente valor del productor.

Podríamos implementar la interfaz estándar de *iterador* para nuestro productor de serie de números:

```js
var something = (function(){
	var nextVal;

	return {
		// necesario para bucles `for..of`
		[Symbol.iterator]: function(){ return this; },

		// método estándar de interfaz de iterador
		next: function(){
			if (nextVal === undefined) {
				nextVal = 1;
			}
			else {
				nextVal = (3 * nextVal) + 6;
			}

			return { done:false, value:nextVal };
		}
	};
})();

something.next().value;		// 1
something.next().value;		// 9
something.next().value;		// 33
something.next().value;		// 105
```

**Nota:** Explicaremos por qué necesitamos la parte `[Symbol.iterator]: ..` de este fragmento de código en la sección "Iterables". Sintácticamente sin embargo, dos características de ES6 están en juego. Primero, la sintaxis `[ .. ]` se llama un *nombre de propiedad computado* (ver el título *this & Object Prototypes* de esta serie). Es una forma en una definición literal de objeto de especificar una expresión y usar el resultado de esa expresión como el nombre de la propiedad. Luego, `Symbol.iterator` es uno de los valores especiales `Symbol` predefinidos de ES6 (ver el título *ES6 & Beyond* de esta serie de libros).

La llamada `next()` devuelve un objeto con dos propiedades: `done` es un valor `boolean` que señala el estado completo del *iterador*; `value` contiene el valor de iteración.

ES6 también añade el bucle `for..of`, que significa que un *iterador* estándar puede consumirse automáticamente con sintaxis de bucle nativa:

```js
for (var v of something) {
	console.log( v );

	// ¡no dejar que el bucle se ejecute para siempre!
	if (v > 500) {
		break;
	}
}
// 1 9 33 105 321 969
```

**Nota:** Porque nuestro *iterador* `something` siempre devuelve `done:false`, este bucle `for..of` se ejecutaría para siempre, razón por la cual pusimos la condición `break`. Está totalmente bien que los iteradores sean interminables, pero también hay casos donde el *iterador* recorrerá un conjunto finito de valores y eventualmente devolverá un `done:true`.

El bucle `for..of` automáticamente llama a `next()` para cada iteración -- no pasa ningún valor al `next()` -- y automáticamente terminará al recibir un `done:true`. Es bastante práctico para recorrer un conjunto de datos.

Por supuesto, podrías manualmente recorrer iteradores, llamando a `next()` y verificando la condición `done:true` para saber cuándo parar:

```js
for (
	var ret;
	(ret = something.next()) && !ret.done;
) {
	console.log( ret.value );

	// ¡no dejar que el bucle se ejecute para siempre!
	if (ret.value > 500) {
		break;
	}
}
// 1 9 33 105 321 969
```

**Nota:** Este enfoque manual con `for` es ciertamente más feo que la sintaxis del bucle `for..of` de ES6, pero su ventaja es que te da la oportunidad de pasar valores a las llamadas `next(..)` si es necesario.

Además de hacer tus propios *iteradores*, muchas estructuras de datos incorporadas en JS (a partir de ES6), como los `array`s, también tienen *iteradores* por defecto:

```js
var a = [1,3,5,7,9];

for (var v of a) {
	console.log( v );
}
// 1 3 5 7 9
```

El bucle `for..of` pide a `a` su *iterador*, y automáticamente lo usa para iterar sobre los valores de `a`.

**Nota:** Puede parecer una extraña omisión por ES6, pero los `object`s regulares intencionalmente no vienen con un *iterador* por defecto de la manera que lo hacen los `array`s. Las razones van más allá de lo que cubriremos aquí. Si todo lo que quieres es iterar sobre las propiedades de un objeto (sin garantía particular de orden), `Object.keys(..)` devuelve un `array`, que luego puede usarse como `for (var k of Object.keys(obj)) { ..`. Tal bucle `for..of` sobre las claves de un objeto sería similar a un bucle `for..in`, excepto que `Object.keys(..)` no incluye propiedades de la cadena `[[Prototype]]` mientras que `for..in` sí lo hace (ver el título *this & Object Prototypes* de esta serie).

### Iterables

El objeto `something` en nuestro ejemplo en curso se llama un *iterador*, ya que tiene el método `next()` en su interfaz. Pero un término estrechamente relacionado es *iterable*, que es un `object` que **contiene** un *iterador* que puede iterar sobre sus valores.

A partir de ES6, la forma de obtener un *iterador* de un *iterable* es que el *iterable* debe tener una función en él, con el nombre siendo el valor especial de símbolo ES6 `Symbol.iterator`. Cuando esta función se llama, devuelve un *iterador*. Aunque no es requerido, generalmente cada llamada debería devolver un nuevo *iterador* fresco.

`a` en el fragmento anterior es un *iterable*. El bucle `for..of` automáticamente llama a su función `Symbol.iterator` para construir un *iterador*. Pero por supuesto podríamos llamar a la función manualmente, y usar el *iterador* que devuelve:

```js
var a = [1,3,5,7,9];

var it = a[Symbol.iterator]();

it.next().value;	// 1
it.next().value;	// 3
it.next().value;	// 5
..
```

En el listado de código anterior que definió `something`, puede que hayas notado esta línea:

```js
[Symbol.iterator]: function(){ return this; }
```

Ese pedacito de código confuso está haciendo al valor `something` -- la interfaz del *iterador* de `something` -- también un *iterable*; ahora es tanto un *iterable* como un *iterador*. Luego, pasamos `something` al bucle `for..of`:

```js
for (var v of something) {
	..
}
```

El bucle `for..of` espera que `something` sea un *iterable*, así que busca y llama a su función `Symbol.iterator`. Definimos esa función para simplemente hacer `return this`, así que simplemente se devuelve a sí mismo, y el bucle `for..of` no se entera de nada.

### Iterador del Generador

Volvamos nuestra atención a los generadores, en el contexto de *iteradores*. Un generador puede tratarse como un productor de valores que extraemos uno a la vez a través de las llamadas `next()` de una interfaz de *iterador*.

Así que, un generador en sí no es técnicamente un *iterable*, aunque es muy similar -- cuando ejecutas el generador, obtienes un *iterador* de vuelta:

```js
function *foo(){ .. }

var it = foo();
```

Podemos implementar el productor de serie de números infinita `something` de antes con un generador, así:

```js
function *something() {
	var nextVal;

	while (true) {
		if (nextVal === undefined) {
			nextVal = 1;
		}
		else {
			nextVal = (3 * nextVal) + 6;
		}

		yield nextVal;
	}
}
```

**Nota:** Un bucle `while..true` normalmente sería algo muy malo incluir en un programa JS real, al menos si no tiene un `break` o `return` en él, ya que probablemente se ejecutaría para siempre, síncronamente, y bloquearía/trabaría la interfaz del navegador. Sin embargo, en un generador, tal bucle generalmente está totalmente bien si tiene un `yield` en él, ya que el generador se pausará en cada iteración, cediendo (con `yield`) de vuelta al programa principal y/o a la cola del bucle de eventos. Para decirlo coloquialmente, "¡los generadores ponen el `while..true` de vuelta en la programación JS!"

Eso es bastante más limpio y simple, ¿verdad? Porque el generador se pausa en cada `yield`, el estado (ámbito) de la función `*something()` se mantiene, lo que significa que no hay necesidad del código repetitivo del closure para preservar el estado de la variable entre llamadas.

No solo es código más simple -- no tenemos que hacer nuestra propia interfaz de *iterador* -- sino que realmente es código más razonable, porque expresa más claramente la intención. Por ejemplo, el bucle `while..true` nos dice que el generador pretende ejecutarse para siempre -- seguir *generando* valores mientras sigamos pidiéndolos.

Y ahora podemos usar nuestro flamante generador `*something()` con un bucle `for..of`, y verás que funciona básicamente de manera idéntica:

```js
for (var v of something()) {
	console.log( v );

	// ¡no dejar que el bucle se ejecute para siempre!
	if (v > 500) {
		break;
	}
}
// 1 9 33 105 321 969
```

¡Pero no pases por alto `for (var v of something()) ..`! No solo referenciamos `something` como un valor como en ejemplos anteriores, sino que llamamos al generador `*something()` para obtener su *iterador* para que el bucle `for..of` lo use.

Si estás prestando atención, dos preguntas pueden surgir de esta interacción entre el generador y el bucle:

* ¿Por qué no pudimos decir `for (var v of something) ..`? Porque `something` aquí es un generador, que no es un *iterable*. Tenemos que llamar a `something()` para construir un productor para que el bucle `for..of` itere sobre él.
* La llamada `something()` produce un *iterador*, pero el bucle `for..of` quiere un *iterable*, ¿correcto? Sí. El *iterador* del generador también tiene una función `Symbol.iterator` en él, que básicamente hace un `return this`, justo como el *iterable* `something` que definimos antes. En otras palabras, ¡el *iterador* de un generador también es un *iterable*!

#### Deteniendo el Generador

En el ejemplo anterior, parecería que la instancia del *iterador* para el generador `*something()` fue básicamente dejada en un estado suspendido para siempre después de que el `break` en el bucle fue llamado.

Pero hay un comportamiento oculto que se encarga de eso por ti. La "completación anormal" (es decir, "terminación temprana") del bucle `for..of` -- generalmente causada por un `break`, `return`, o una excepción no capturada -- envía una señal al *iterador* del generador para que termine.

**Nota:** Técnicamente, el bucle `for..of` también envía esta señal al *iterador* en la completación normal del bucle. Para un generador, eso es esencialmente una operación sin efecto, ya que el *iterador* del generador tuvo que completarse primero para que el bucle `for..of` se completara. Sin embargo, *iteradores* personalizados podrían desear recibir esta señal adicional de los consumidores del bucle `for..of`.

Mientras un bucle `for..of` enviará automáticamente esta señal, puedes desear enviar la señal manualmente a un *iterador*; lo haces llamando a `return(..)`.

Si especificas una cláusula `try..finally` dentro del generador, siempre se ejecutará incluso cuando el generador sea completado externamente. Esto es útil si necesitas limpiar recursos (conexiones a bases de datos, etc.):

```js
function *something() {
	try {
		var nextVal;

		while (true) {
			if (nextVal === undefined) {
				nextVal = 1;
			}
			else {
				nextVal = (3 * nextVal) + 6;
			}

			yield nextVal;
		}
	}
	// cláusula de limpieza
	finally {
		console.log( "cleaning up!" );
	}
}
```

El ejemplo anterior con `break` en el bucle `for..of` disparará la cláusula `finally`. Pero en su lugar podrías terminar manualmente la instancia del *iterador* del generador desde fuera con `return(..)`:

```js
var it = something();
for (var v of it) {
	console.log( v );

	// ¡no dejar que el bucle se ejecute para siempre!
	if (v > 500) {
		console.log(
			// completar el iterador del generador
			it.return( "Hello World" ).value
		);
		// no se necesita `break` aquí
	}
}
// 1 9 33 105 321 969
// cleaning up!
// Hello World
```

Cuando llamamos a `it.return(..)`, inmediatamente termina el generador, lo que por supuesto ejecuta la cláusula `finally`. También, establece el `value` devuelto a lo que sea que pasaste a `return(..)`, que es cómo `"Hello World"` sale de vuelta. Tampoco necesitamos incluir un `break` ahora porque el *iterador* del generador está establecido en `done:true`, así que el bucle `for..of` terminará en su siguiente iteración.

Los generadores deben su nombre mayormente a este uso de *consumir valores producidos*. Pero de nuevo, eso es solo uno de los usos para generadores, y francamente ni siquiera el principal que nos preocupa en el contexto de este libro.

Pero ahora que entendemos más completamente algo de la mecánica de cómo funcionan, podemos *a continuación* dirigir nuestra atención a cómo los generadores aplican a la concurrencia asíncrona.

## Iterando Generadores Asíncronamente

¿Qué tienen que ver los generadores con patrones de código asíncrono, arreglar problemas con callbacks, y similares? Vamos a responder esa pregunta importante.

Deberíamos revisitar uno de nuestros escenarios del Capítulo 3. Recordemos el enfoque con callbacks:

```js
function foo(x,y,cb) {
	ajax(
		"http://some.url.1/?x=" + x + "&y=" + y,
		cb
	);
}

foo( 11, 31, function(err,text) {
	if (err) {
		console.error( err );
	}
	else {
		console.log( text );
	}
} );
```

Si quisiéramos expresar este mismo control de flujo de tarea con un generador, podríamos hacer:

```js
function foo(x,y) {
	ajax(
		"http://some.url.1/?x=" + x + "&y=" + y,
		function(err,data){
			if (err) {
				// lanzar un error dentro de `*main()`
				it.throw( err );
			}
			else {
				// reanudar `*main()` con `data` recibida
				it.next( data );
			}
		}
	);
}

function *main() {
	try {
		var text = yield foo( 11, 31 );
		console.log( text );
	}
	catch (err) {
		console.error( err );
	}
}

var it = main();

// ¡arrancar todo!
it.next();
```

A primera vista, este fragmento es más largo, y quizás un poco más complejo que el fragmento con callbacks antes de él. Pero no dejes que esa impresión te desvíe. ¡El fragmento del generador es en realidad **mucho** mejor! Pero hay mucho que explicar.

Primero, veamos esta parte del código, que es la más importante:

```js
var text = yield foo( 11, 31 );
console.log( text );
```

Piensa en cómo funciona ese código por un momento. Estamos llamando a una función normal `foo(..)` y aparentemente somos capaces de obtener de vuelta el `text` de la llamada Ajax, incluso aunque es asíncrona.

¿Cómo es eso posible? Si recuerdas el comienzo del Capítulo 1, teníamos código casi idéntico:

```js
var data = ajax( "..url 1.." );
console.log( data );
```

¡Y ese código no funcionó! ¿Puedes detectar la diferencia? Es el `yield` usado en un generador.

¡Esa es la magia! Eso es lo que nos permite tener lo que parece ser código bloqueante, síncrono, pero realmente no bloquea todo el programa; solo pausa/bloquea el código en el generador mismo.

En `yield foo(11,31)`, primero se hace la llamada `foo(11,31)`, que no devuelve nada (es decir, `undefined`), así que estamos haciendo una llamada para solicitar datos, pero en realidad luego estamos haciendo `yield undefined`. Eso está bien, porque el código no está actualmente dependiendo de un valor emitido con `yield` para hacer algo interesante. Revisitaremos este punto más adelante en el capítulo.

No estamos usando `yield` en sentido de paso de mensajes aquí, solo en sentido de control de flujo para pausar/bloquear. En realidad, sí tendrá paso de mensajes, pero solo en una dirección, después de que el generador sea reanudado.

Así que, el generador se pausa en el `yield`, esencialmente haciendo la pregunta, "¿qué valor debería devolver para asignar a la variable `text`?" ¿Quién va a responder esa pregunta?

Mira `foo(..)`. Si la petición Ajax es exitosa, llamamos:

```js
it.next( data );
```

Eso reanuda el generador con los datos de respuesta, lo que significa que nuestra expresión `yield` pausada recibe ese valor directamente, y luego mientras reinicia el código del generador, ese valor se asigna a la variable local `text`.

Bastante genial, ¿eh?

Da un paso atrás y considera las implicaciones. Tenemos código con apariencia totalmente síncrona dentro del generador (aparte de la palabra clave `yield` en sí), pero detrás de escenas, dentro de `foo(..)`, las operaciones pueden completarse asíncronamente.

**¡Eso es enorme!** Esa es una solución casi perfecta a nuestro problema previamente declarado con los callbacks de no poder expresar asincronía de forma secuencial, síncrona, con la que nuestros cerebros puedan relacionarse.

En esencia, estamos abstrayendo la asincronía como un detalle de implementación, para que podamos razonar síncrona/secuencialmente sobre nuestro control de flujo: "Hacer una petición Ajax, y cuando termine imprimir la respuesta." Y por supuesto, solo expresamos dos pasos en el control de flujo, pero esta misma capacidad se extiende sin límites, para dejarnos expresar tantos pasos como necesitemos.

**Consejo:** ¡Esta es una realización tan importante, simplemente vuelve atrás y relee los últimos tres párrafos de nuevo para que cale!

### Manejo Síncrono de Errores

Pero el código del generador anterior tiene aún más bondades que cedernos con *yield*. Dirijamos nuestra atención al `try..catch` dentro del generador:

```js
try {
	var text = yield foo( 11, 31 );
	console.log( text );
}
catch (err) {
	console.error( err );
}
```

¿Cómo funciona esto? La llamada `foo(..)` se completa asíncronamente, ¿y no falla `try..catch` en capturar errores asíncronos, como vimos en el Capítulo 3?

Ya vimos cómo el `yield` permite que la sentencia de asignación se pause para esperar a que `foo(..)` termine, de modo que la respuesta completada pueda ser asignada a `text`. La parte increíble es que esta pausa con `yield` *también* permite al generador hacer `catch` de un error. Lanzamos ese error dentro del generador con esta parte del listado de código anterior:

```js
if (err) {
	// lanzar un error dentro de `*main()`
	it.throw( err );
}
```

La naturaleza de pausa-con-yield de los generadores significa que no solo obtenemos valores de retorno con apariencia síncrona de llamadas a funciones asíncronas, sino que también podemos capturar síncronamente con `catch` errores de esas llamadas a funciones asíncronas.

Así que hemos visto que podemos lanzar errores *dentro de* un generador, pero ¿qué hay de lanzar errores *fuera de* un generador? Exactamente como esperarías:

```js
function *main() {
	var x = yield "Hello World";

	yield x.toLowerCase();	// ¡causar una excepción!
}

var it = main();

it.next().value;			// Hello World

try {
	it.next( 42 );
}
catch (err) {
	console.error( err );	// TypeError
}
```

Por supuesto, podríamos haber lanzado manualmente un error con `throw ..` en lugar de causar una excepción.

Incluso podemos hacer `catch` del mismo error que lanzamos con `throw(..)` dentro del generador, esencialmente dando al generador una oportunidad de manejarlo pero si no lo hace, el código del *iterador* debe manejarlo:

```js
function *main() {
	var x = yield "Hello World";

	// nunca llega aquí
	console.log( x );
}

var it = main();

it.next();

try {
	// ¿manejará `*main()` este error? ¡Ya veremos!
	it.throw( "Oops" );
}
catch (err) {
	// ¡no, no lo manejó!
	console.error( err );			// Oops
}
```

El manejo de errores con apariencia síncrona (a través de `try..catch`) con código asíncrono es una enorme victoria para la legibilidad y razonabilidad.
