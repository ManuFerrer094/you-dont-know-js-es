## Comparación Relacional Abstracta

Aunque esta parte de la coerción *implícita* a menudo recibe mucha menos atención, es importante sin embargo pensar en lo que sucede con las comparaciones `a < b` (similar a cómo acabamos de examinar `a == b` en profundidad).

El algoritmo de "Comparación Relacional Abstracta" en la sección 11.8.5 de ES5 esencialmente se divide en dos partes: qué hacer si la comparación involucra ambos valores `string` (segunda mitad), o cualquier otra cosa (primera mitad).

**Nota:** El algoritmo solo está definido para `a < b`. Así que, `a > b` se maneja como `b < a`.

El algoritmo primero llama a la coerción `ToPrimitive` sobre ambos valores, y si el resultado de retorno de cualquiera de las llamadas no es un `string`, entonces ambos valores son coercionados a valores `number` usando las reglas de la operación `ToNumber`, y comparados numéricamente.

Por ejemplo:

```js
var a = [ 42 ];
var b = [ "43" ];

a < b;	// true
b < a;	// false
```

**Nota:** Advertencias similares para `-0` y `NaN` aplican aquí como lo hicieron en el algoritmo `==` discutido anteriormente.

Sin embargo, si ambos valores son `string`s para la comparación `<`, se realiza una simple comparación lexicográfica (alfabética natural) sobre los caracteres:

```js
var a = [ "42" ];
var b = [ "043" ];

a < b;	// false
```

`a` y `b` *no* son coercionados a `number`s, porque ambos terminan como `string`s después de la coerción `ToPrimitive` sobre los dos `array`s. Así que, `"42"` se compara carácter por carácter con `"043"`, empezando con los primeros caracteres `"4"` y `"0"`, respectivamente. Dado que `"0"` es lexicográficamente *menor que* `"4"`, la comparación devuelve `false`.

El exactamente mismo comportamiento y razonamiento aplica para:

```js
var a = [ 4, 2 ];
var b = [ 0, 4, 3 ];

a < b;	// false
```

Aquí, `a` se convierte en `"4,2"` y `b` se convierte en `"0,4,3"`, y esos se comparan lexicográficamente idénticamente al fragmento anterior.

¿Qué hay de:

```js
var a = { b: 42 };
var b = { b: 43 };

a < b;	// ??
```

`a < b` también es `false`, porque `a` se convierte en `[object Object]` y `b` se convierte en `[object Object]`, y claramente `a` no es lexicográficamente menor que `b`.

Pero extrañamente:

```js
var a = { b: 42 };
var b = { b: 43 };

a < b;	// false
a == b;	// false
a > b;	// false

a <= b;	// true
a >= b;	// true
```

¿Por qué `a == b` no es `true`? Son el mismo valor `string` (`"[object Object]"`), así que parece que deberían ser iguales, ¿verdad? No. Recuerda la discusión anterior sobre cómo `==` funciona con referencias de `object`.

¿Pero entonces cómo es que `a <= b` y `a >= b` resultan en `true`, si `a < b` **y** `a == b` **y** `a > b` son todos `false`?

Porque la especificación dice que para `a <= b`, en realidad evaluará `b < a` primero, y luego negará ese resultado. Dado que `b < a` *también* es `false`, el resultado de `a <= b` es `true`.

Eso es probablemente terriblemente contrario a cómo podrías haber explicado lo que `<=` hace hasta ahora, que probablemente habría sido lo literal: "menor que *o* igual a." JS más precisamente considera `<=` como "no mayor que" (`!(a > b)`, que JS trata como `!(b < a)`). Además, `a >= b` se explica considerándolo primero como `b <= a`, y luego aplicando el mismo razonamiento.

Desafortunadamente, no hay "comparación relacional strict" como la hay para la igualdad. En otras palabras, no hay forma de prevenir que la coerción *implícita* ocurra con las comparaciones relacionales como `a < b`, más que asegurar que `a` y `b` sean del mismo tipo explícitamente antes de hacer la comparación.

Usa el mismo razonamiento de nuestra discusión anterior de verificación de cordura de `==` vs. `===`. Si la coerción es útil y razonablemente segura, como en una comparación `42 < "43"`, **úsala**. Por otro lado, si necesitas estar seguro sobre una comparación relacional, *coerciona explícitamente* los valores primero, antes de usar `<` (o sus contrapartes).

```js
var a = [ 42 ];
var b = "043";

a < b;						// false -- ¡comparación de strings!
Number( a ) < Number( b );	// true -- ¡comparación de números!
```

## Revisión

En este capítulo, dirigimos nuestra atención a cómo suceden las conversiones de tipo en JavaScript, llamadas **coerción**, que pueden ser caracterizadas como *explícitas* o *implícitas*.

La coerción tiene mala fama, pero en realidad es bastante útil en muchos casos. Una tarea importante para el desarrollador JS responsable es tomarse el tiempo para aprender todos los pormenores de la coerción para decidir qué partes ayudarán a mejorar su código, y qué partes realmente deberían evitar.

La coerción *explícita* es código donde es obvio que la intención es convertir un valor de un tipo a otro. El beneficio es la mejora en la legibilidad y mantenibilidad del código al reducir la confusión.

La coerción *implícita* es coerción que está "oculta" como un efecto secundario de alguna otra operación, donde no es tan obvio que la conversión de tipo ocurrirá. Aunque pueda parecer que la coerción *implícita* es lo opuesto de la *explícita* y por lo tanto es mala (¡y de hecho, muchos piensan así!), en realidad la coerción *implícita* también trata de mejorar la legibilidad del código.

Especialmente para la *implícita*, la coerción debe usarse de manera responsable y consciente. Sabe por qué estás escribiendo el código que escribes, y cómo funciona. Esfuérzate por escribir código del que otros puedan aprender y entender fácilmente también.
