# You Don't Know JS: Tipos y Gramática
# Capítulo 2: Valores

Los `array`s, `string`s y `number`s son los bloques de construcción más básicos de cualquier programa, pero JavaScript tiene algunas características únicas con estos tipos que pueden deleitarte o confundirte.

Veamos varios de los tipos de valores incorporados en JS, y exploremos cómo podemos comprender más plenamente y aprovechar correctamente sus comportamientos.

## Arrays

En comparación con otros lenguajes con tipos forzados, los `array`s de JavaScript son simplemente contenedores para cualquier tipo de valor, desde `string` a `number` a `object` e incluso otro `array` (que es cómo obtienes `array`s multidimensionales).

```js
var a = [ 1, "2", [3] ];

a.length;		// 3
a[0] === 1;		// true
a[2][0] === 3;	// true
```

No necesitas predimensionar tus `array`s (ver "Arrays" en el Capítulo 3), puedes simplemente declararlos y agregar valores como te parezca:

```js
var a = [ ];

a.length;	// 0

a[0] = 1;
a[1] = "2";
a[2] = [ 3 ];

a.length;	// 3
```

**Advertencia:** Usar `delete` en un valor `array` eliminará esa ranura del `array`, pero incluso si eliminas el último elemento, **no** actualiza la propiedad `length`, ¡así que ten cuidado! Cubriremos el operador `delete` en más detalle en el Capítulo 5.

Ten cuidado de crear `array`s "dispersos" (dejando o creando ranuras vacías/faltantes):

```js
var a = [ ];

a[0] = 1;
// no se asignó ranura `a[1]` aquí
a[2] = [ 3 ];

a[1];		// undefined

a.length;	// 3
```

Aunque eso funciona, puede llevar a un comportamiento confuso con las "ranuras vacías" que dejas en medio. Aunque la ranura parece tener el valor `undefined`, no se comportará igual que si la ranura se establece explícitamente (`a[1] = undefined`). Ver "Arrays" en el Capítulo 3 para más información.

Los `array`s están indexados numéricamente (como esperarías), pero lo engañoso es que también son objetos que pueden tener claves/propiedades `string` agregadas (pero que no cuentan para el `length` del `array`):

```js
var a = [ ];

a[0] = 1;
a["foobar"] = 2;

a.length;		// 1
a["foobar"];	// 2
a.foobar;		// 2
```

Sin embargo, un truco a tener en cuenta es que si un valor `string` destinado como clave puede ser convertido a un `number` estándar en base 10, ¡entonces se asume que querías usarlo como índice `number` en lugar de como clave `string`!

```js
var a = [ ];

a["13"] = 42;

a.length; // 14
```

Generalmente, no es una gran idea agregar claves/propiedades `string` a `array`s. Usa `object`s para mantener valores en claves/propiedades, y reserva los `array`s para valores estrictamente indexados numéricamente.

### Array-Likes

Habrá ocasiones donde necesites convertir un valor `array`-like (una colección de valores indexada numéricamente) en un verdadero `array`, generalmente para poder llamar utilidades de array (como `indexOf(..)`, `concat(..)`, `forEach(..)`, etc.) contra la colección de valores.

Por ejemplo, varias operaciones de consulta del DOM devuelven listas de elementos DOM que no son verdaderos `array`s pero son suficientemente `array`-like para nuestros propósitos de conversión. Otro ejemplo común es cuando las funciones exponen el objeto `arguments` (`array`-like) (a partir de ES6, deprecado) para acceder a los argumentos como una lista.

Una forma muy común de hacer tal conversión es tomar prestada la utilidad `slice(..)` contra el valor:

```js
function foo() {
	var arr = Array.prototype.slice.call( arguments );
	arr.push( "bam" );
	console.log( arr );
}

foo( "bar", "baz" ); // ["bar","baz","bam"]
```

Si `slice()` se llama sin ningún otro parámetro, como efectivamente ocurre en el fragmento anterior, los valores predeterminados para sus parámetros tienen el efecto de duplicar el `array` (o, en este caso, `array`-like).

A partir de ES6, también hay una utilidad incorporada llamada `Array.from(..)` que puede hacer la misma tarea:

```js
...
var arr = Array.from( arguments );
...
```

**Nota:** `Array.from(..)` tiene varias capacidades poderosas, y será cubierto en detalle en el título *ES6 & Beyond* de esta serie.

## Strings

Es una creencia muy común que los `string`s son esencialmente solo `array`s de caracteres. Aunque la implementación bajo el capó puede o no usar `array`s, es importante darse cuenta de que los `string`s de JavaScript realmente no son lo mismo que `array`s de caracteres. La similitud es mayormente superficial.

Por ejemplo, consideremos estos dos valores:

```js
var a = "foo";
var b = ["f","o","o"];
```

Los Strings tienen una semejanza superficial con los `array`s -- `array`-likes, como arriba -- por ejemplo, ambos tienen una propiedad `length`, un método `indexOf(..)` (versión de `array` solo a partir de ES5), y un método `concat(..)`:

```js
a.length;							// 3
b.length;							// 3

a.indexOf( "o" );					// 1
b.indexOf( "o" );					// 1

var c = a.concat( "bar" );			// "foobar"
var d = b.concat( ["b","a","r"] );	// ["f","o","o","b","a","r"]

a === c;							// false
b === d;							// false

a;									// "foo"
b;									// ["f","o","o"]
```

Entonces, ambos son básicamente solo "arrays de caracteres", ¿verdad? **No exactamente**:

```js
a[1] = "O";
b[1] = "O";

a; // "foo"
b; // ["f","O","o"]
```

Los `string`s de JavaScript son inmutables, mientras que los `array`s son bastante mutables. Además, la forma de acceso a la posición del carácter `a[1]` no siempre fue JavaScript ampliamente válido. Las versiones anteriores de IE no permitían esa sintaxis (pero ahora sí). En su lugar, el enfoque *correcto* ha sido `a.charAt(1)`.

Una consecuencia adicional de los `string`s inmutables es que ninguno de los métodos de `string` que alteran su contenido puede modificar en el lugar, sino que deben crear y devolver nuevos `string`s. Por contraste, muchos de los métodos que cambian el contenido de los `array`s realmente *sí* modifican en el lugar.

```js
c = a.toUpperCase();
a === c;	// false
a;			// "foo"
c;			// "FOO"

b.push( "!" );
b;			// ["f","O","o","!"]
```

También, muchos de los métodos de `array` que podrían ser útiles al tratar con `string`s no están realmente disponibles para ellos, pero podemos "tomar prestados" métodos de `array` sin mutación contra nuestro `string`:

```js
a.join;			// undefined
a.map;			// undefined

var c = Array.prototype.join.call( a, "-" );
var d = Array.prototype.map.call( a, function(v){
	return v.toUpperCase() + ".";
} ).join( "" );

c;				// "f-o-o"
d;				// "F.O.O."
```

Tomemos otro ejemplo: invertir un `string` (por cierto, ¡una pregunta común de trivia de entrevistas de JavaScript!). Los `array`s tienen un método mutador in situ `reverse()`, pero los `string`s no:

```js
a.reverse;		// undefined

b.reverse();	// ["!","o","O","f"]
b;				// ["!","o","O","f"]
```

Desafortunadamente, este "préstamo" no funciona con mutadores de `array`, porque los `string`s son inmutables y por lo tanto no pueden ser modificados en el lugar:

```js
Array.prototype.reverse.call( a );
// todavía devuelve un objeto envoltorio String (ver Capítulo 3)
// para "foo" :(
```

Otra solución alternativa (también conocida como hack) es convertir el `string` en un `array`, realizar la operación deseada, y luego convertirlo de vuelta a un `string`.

```js
var c = a
	// dividir `a` en un array de caracteres
	.split( "" )
	// invertir el array de caracteres
	.reverse()
	// unir el array de caracteres de vuelta a un string
	.join( "" );

c; // "oof"
```

Si eso se siente feo, lo es. Sin embargo, *funciona* para `string`s simples, así que si necesitas algo rápido y sucio, a menudo tal enfoque hace el trabajo.

**Advertencia:** ¡Ten cuidado! Este enfoque **no funciona** para `string`s con caracteres complejos (unicode) en ellos (símbolos astrales, caracteres multibyte, etc.). Necesitas utilidades de biblioteca más sofisticadas que sean conscientes de unicode para que tales operaciones se manejen con precisión. Consulta el trabajo de Mathias Bynens sobre el tema: *Esrever* (https://github.com/mathiasbynens/esrever).

La otra forma de verlo es: si más comúnmente haces tareas con tus "strings" que los tratan básicamente como *arrays de caracteres*, quizás es mejor simplemente almacenarlos como `array`s en lugar de como `string`s. Probablemente te ahorrarás muchas molestias de convertir de `string` a `array` cada vez. Siempre puedes llamar a `join("")` en el `array` *de caracteres* cuando realmente necesites la representación `string`.

## Numbers

JavaScript tiene solo un tipo numérico: `number`. Este tipo incluye tanto valores "enteros" como números decimales fraccionarios. Digo "entero" entre comillas porque ha sido una crítica de larga data de JS que no hay verdaderos enteros, como los hay en otros lenguajes. Eso puede cambiar en algún momento en el futuro, pero por ahora, solo tenemos `number`s para todo.

Entonces, en JS, un "entero" es solo un valor que no tiene valor decimal fraccionario. Es decir, `42.0` es tanto un "entero" como `42`.

Como la mayoría de los lenguajes modernos, incluyendo prácticamente todos los lenguajes de scripting, la implementación de los `number`s de JavaScript está basada en el estándar "IEEE 754", a menudo llamado "punto flotante". JavaScript específicamente usa el formato de "doble precisión" (también conocido como "binario de 64 bits") del estándar.

Hay muchos escritos excelentes en la Web sobre los detalles minuciosos de cómo se almacenan los números binarios de punto flotante en memoria, y las implicaciones de esas elecciones. Porque entender patrones de bits en memoria no es estrictamente necesario para entender cómo usar correctamente los `number`s en JS, lo dejaremos como ejercicio para el lector interesado si quieres profundizar más en los detalles de IEEE 754.

### Sintaxis Numérica

Los literales numéricos se expresan en JavaScript generalmente como literales decimales en base 10. Por ejemplo:

```js
var a = 42;
var b = 42.3;
```

La porción inicial de un valor decimal, si es `0`, es opcional:

```js
var a = 0.42;
var b = .42;
```

De manera similar, la porción final (la fraccionaria) de un valor decimal después del `.`, si es `0`, es opcional:

```js
var a = 42.0;
var b = 42.;
```

**Advertencia:** `42.` es bastante poco común, y probablemente no es una gran idea si estás tratando de evitar confusión cuando otras personas lean tu código. Pero es, sin embargo, válido.

Por defecto, la mayoría de los `number`s se mostrarán como decimales en base 10, con los `0`s fraccionarios finales eliminados. Así:

```js
var a = 42.300;
var b = 42.0;

a; // 42.3
b; // 42
```

Los `number`s muy grandes o muy pequeños por defecto se mostrarán en forma exponencial, igual que la salida del método `toExponential()`, como:

```js
var a = 5E10;
a;					// 50000000000
a.toExponential();	// "5e+10"

var b = a * a;
b;					// 2.5e+21

var c = 1 / a;
c;					// 2e-11
```

Debido a que los valores `number` pueden ser envueltos con el objeto envoltorio `Number` (ver Capítulo 3), los valores `number` pueden acceder a métodos que están incorporados en `Number.prototype` (ver Capítulo 3). Por ejemplo, el método `toFixed(..)` te permite especificar cuántos lugares decimales fraccionarios te gustaría que el valor sea representado con:

```js
var a = 42.59;

a.toFixed( 0 ); // "43"
a.toFixed( 1 ); // "42.6"
a.toFixed( 2 ); // "42.59"
a.toFixed( 3 ); // "42.590"
a.toFixed( 4 ); // "42.5900"
```

Observa que la salida es realmente una representación `string` del `number`, y que el valor es rellenado con `0` en el lado derecho si pides más decimales de los que el valor contiene.

`toPrecision(..)` es similar, pero especifica cuántos *dígitos significativos* deben usarse para representar el valor:

```js
var a = 42.59;

a.toPrecision( 1 ); // "4e+1"
a.toPrecision( 2 ); // "43"
a.toPrecision( 3 ); // "42.6"
a.toPrecision( 4 ); // "42.59"
a.toPrecision( 5 ); // "42.590"
a.toPrecision( 6 ); // "42.5900"
```

No necesitas usar una variable con el valor para acceder a estos métodos; puedes acceder a estos métodos directamente en literales `number`. Pero tienes que tener cuidado con el operador `.`. Como `.` es un carácter numérico válido, primero será interpretado como parte del literal `number`, si es posible, en lugar de ser interpretado como un acceso a propiedad.

```js
// sintaxis inválida:
42.toFixed( 3 );	// SyntaxError

// estos son todos válidos:
(42).toFixed( 3 );	// "42.000"
0.42.toFixed( 3 );	// "0.420"
42..toFixed( 3 );	// "42.000"
```

`42.toFixed(3)` es sintaxis inválida, porque el `.` es absorbido como parte del literal `42.` (que es válido -- ¡ver arriba!), y entonces no hay operador de propiedad `.` presente para hacer el acceso `.toFixed`.

`42..toFixed(3)` funciona porque el primer `.` es parte del `number` y el segundo `.` es el operador de propiedad. Pero probablemente se ve extraño, y de hecho es muy raro ver algo así en código JavaScript real. De hecho, es bastante poco común acceder a métodos directamente en cualquiera de los valores primitivos. Poco común no significa *malo* o *incorrecto*.

**Nota:** Hay bibliotecas que extienden el `Number.prototype` incorporado (ver Capítulo 3) para proporcionar operaciones extra en/con `number`s, y en esos casos, es perfectamente válido usar algo como `10..makeItRain()` para activar una animación de lluvia de dinero de 10 segundos, o algo más tonto como eso.

Esto también es técnicamente válido (nota el espacio):

```js
42 .toFixed(3); // "42.000"
```

Sin embargo, con el literal `number` específicamente, **este es un estilo de codificación particularmente confuso** y no servirá para otro propósito que confundir a otros desarrolladores (y a tu yo futuro). Evítalo.

Los `number`s también pueden ser especificados en forma exponencial, que es común cuando se representan `number`s más grandes, como:

```js
var onethousand = 1E3;						// significa 1 * 10^3
var onemilliononehundredthousand = 1.1E6;	// significa 1.1 * 10^6
```

Los literales `number` también pueden ser expresados en otras bases, como binario, octal y hexadecimal.

Estos formatos funcionan en las versiones actuales de JavaScript:

```js
0xf3; // hexadecimal para: 243
0Xf3; // ídem

0363; // octal para: 243
```

**Nota:** A partir de ES6 + modo `strict`, la forma octal `0363` ya no es permitida (ver abajo para la nueva forma). La forma `0363` todavía es permitida en modo no-`strict`, pero deberías dejar de usarla de todas formas, para ser amigable con el futuro (¡y porque ya deberías estar usando modo `strict`!).

A partir de ES6, las siguientes nuevas formas también son válidas:

```js
0o363;		// octal para: 243
0O363;		// ídem

0b11110011;	// binario para: 243
0B11110011; // ídem
```

Por favor, hazle un favor a tus compañeros desarrolladores: nunca uses la forma `0O363`. `0` junto a `O` mayúscula solo está pidiendo confusión. Siempre usa los predicados en minúscula `0x`, `0b`, y `0o`.

### Valores Decimales Pequeños

El efecto secundario más (in)famoso de usar números de punto flotante binarios (que, recuerda, es cierto para **todos** los lenguajes que usan IEEE 754 -- no *solo* JavaScript como muchos asumen/pretenden) es:

```js
0.1 + 0.2 === 0.3; // false
```

Matemáticamente, sabemos que esa declaración debería ser `true`. ¿Por qué es `false`?

En pocas palabras, las representaciones de `0.1` y `0.2` en punto flotante binario no son exactas, así que cuando se suman, el resultado no es exactamente `0.3`. Es **realmente** cercano: `0.30000000000000004`, pero si tu comparación falla, "cercano" es irrelevante.

**Nota:** ¿Debería JavaScript cambiar a una implementación de `number` diferente que tenga representaciones exactas para todos los valores? Algunos piensan que sí. Han habido muchas alternativas presentadas a lo largo de los años. Ninguna ha sido aceptada todavía, y quizás nunca lo serán. Por fácil que pueda parecer simplemente agitar la mano y decir, "¡corrijan ese bug ya!", no es ni de cerca tan fácil. Si lo fuera, definitivamente habría sido cambiado hace mucho tiempo.

Ahora, la pregunta es, si algunos `number`s no pueden ser *confiables* como exactos, ¿eso significa que no podemos usar `number`s en absoluto? **Por supuesto que no.**

Hay algunas aplicaciones donde necesitas ser más cuidadoso, especialmente cuando tratas con valores decimales fraccionarios. También hay muchas (¿quizás la mayoría?) aplicaciones que solo tratan con números enteros ("integers"), y además, solo tratan con números en los millones o billones como máximo. Estas aplicaciones han sido, y siempre serán, **perfectamente seguras** para usar operaciones numéricas en JS.

¿Qué pasa si *necesitamos* comparar dos `number`s, como `0.1 + 0.2` con `0.3`, sabiendo que la prueba de igualdad simple falla?

La práctica más comúnmente aceptada es usar un pequeño valor de "error de redondeo" como la *tolerancia* para la comparación. Este pequeño valor a menudo se llama "machine epsilon", que es comúnmente `2^-52` (`2.220446049250313e-16`) para el tipo de `number`s en JavaScript.

A partir de ES6, `Number.EPSILON` está predefinido con este valor de tolerancia, así que querrías usarlo, pero puedes hacer un polyfill seguro de la definición para pre-ES6:

```js
if (!Number.EPSILON) {
	Number.EPSILON = Math.pow(2,-52);
}
```

Podemos usar este `Number.EPSILON` para comparar dos `number`s por "igualdad" (dentro de la tolerancia de error de redondeo):

```js
function numbersCloseEnoughToEqual(n1,n2) {
	return Math.abs( n1 - n2 ) < Number.EPSILON;
}

var a = 0.1 + 0.2;
var b = 0.3;

numbersCloseEnoughToEqual( a, b );					// true
numbersCloseEnoughToEqual( 0.0000001, 0.0000002 );	// false
```

El valor máximo de punto flotante que puede ser representado es aproximadamente `1.798e+308` (¡que es realmente, realmente, realmente enorme!), predefinido para ti como `Number.MAX_VALUE`. En el extremo pequeño, `Number.MIN_VALUE` es aproximadamente `5e-324`, que no es negativo pero está realmente cerca de cero.

### Rangos de Enteros Seguros

Debido a cómo se representan los `number`s, hay un rango de valores "seguros" para los "enteros" `number` completos, y es significativamente menor que `Number.MAX_VALUE`.

El entero máximo que puede ser representado "de forma segura" (es decir, hay garantía de que el valor solicitado es realmente representable sin ambigüedad) es `2^53 - 1`, que es `9007199254740991`. Si insertas tus comas, verás que esto es poco más de 9 cuadrillones. Así que es bastante grande para el rango de `number`s.

Este valor está realmente predefinido automáticamente en ES6, como `Number.MAX_SAFE_INTEGER`. Como era de esperar, hay un valor mínimo, `-9007199254740991`, y está definido en ES6 como `Number.MIN_SAFE_INTEGER`.

La forma principal en que los programas JS se enfrentan a tratar con números tan grandes es cuando manejan IDs de 64 bits de bases de datos, etc. Los números de 64 bits no pueden ser representados con precisión con el tipo `number`, así que deben ser almacenados en (y transmitidos a/desde) JavaScript usando representación `string`.

Las operaciones numéricas en valores `number` de ID tan grandes (además de la comparación, que funcionará bien con `string`s) no son tan comunes, afortunadamente. Pero si *necesitas* realizar matemáticas con estos valores muy grandes, por ahora necesitarás usar una utilidad de *big number*. Los big numbers pueden obtener soporte oficial en una futura versión de JavaScript.

### Probando Enteros

Para probar si un valor es un entero, puedes usar el `Number.isInteger(..)` especificado en ES6:

```js
Number.isInteger( 42 );		// true
Number.isInteger( 42.000 );	// true
Number.isInteger( 42.3 );	// false
```

Para hacer polyfill de `Number.isInteger(..)` para pre-ES6:

```js
if (!Number.isInteger) {
	Number.isInteger = function(num) {
		return typeof num == "number" && num % 1 == 0;
	};
}
```

Para probar si un valor es un *entero seguro*, usa el `Number.isSafeInteger(..)` especificado en ES6:

```js
Number.isSafeInteger( Number.MAX_SAFE_INTEGER );	// true
Number.isSafeInteger( Math.pow( 2, 53 ) );			// false
Number.isSafeInteger( Math.pow( 2, 53 ) - 1 );		// true
```

Para hacer polyfill de `Number.isSafeInteger(..)` en navegadores pre-ES6:

```js
if (!Number.isSafeInteger) {
	Number.isSafeInteger = function(num) {
		return Number.isInteger( num ) &&
			Math.abs( num ) <= Number.MAX_SAFE_INTEGER;
	};
}
```

### Enteros de 32 bits (Con Signo)

Mientras que los enteros pueden llegar de forma segura hasta aproximadamente 9 cuadrillones (53 bits), hay algunas operaciones numéricas (como los operadores bit a bit) que solo están definidas para `number`s de 32 bits, por lo que el "rango seguro" para `number`s usados de esa manera debe ser mucho menor.

El rango entonces es `Math.pow(-2,31)` (`-2147483648`, aproximadamente -2.1 mil millones) hasta `Math.pow(2,31)-1` (`2147483647`, aproximadamente +2.1 mil millones).

Para forzar un valor `number` en `a` a un valor entero de 32 bits con signo, usa `a | 0`. Esto funciona porque el operador bit a bit `|` solo funciona para valores enteros de 32 bits (lo que significa que solo puede prestar atención a 32 bits y cualquier otro bit se perderá). Entonces, hacer "or" con cero es esencialmente una no-operación en términos de bits.

**Nota:** Ciertos valores especiales (que cubriremos en la siguiente sección), como `NaN` e `Infinity`, no son "seguros para 32 bits", ya que esos valores cuando se pasan a un operador bit a bit pasarán por la operación abstracta `ToInt32` (ver Capítulo 4) y se convertirán simplemente en el valor `+0` para el propósito de esa operación bit a bit.

## Valores Especiales

Hay varios valores especiales distribuidos entre los varios tipos de los que el desarrollador JS *alerta* necesita estar consciente, y usar apropiadamente.

### Los Valores No-Valor

Para el tipo `undefined`, hay uno y solo un valor: `undefined`. Para el tipo `null`, hay uno y solo un valor: `null`. Así que para ambos, la etiqueta es tanto su tipo como su valor.

Tanto `undefined` como `null` son frecuentemente tomados como intercambiables como valores "vacíos" o "no" valores. Otros desarrolladores prefieren distinguir entre ellos con matices. Por ejemplo:

* `null` es un valor vacío
* `undefined` es un valor faltante

O:

* `undefined` no ha tenido un valor todavía
* `null` tuvo un valor y ya no lo tiene

Independientemente de cómo elijas "definir" y usar estos dos valores, `null` es una palabra clave especial, no un identificador, y por lo tanto no puedes tratarla como una variable para asignar (¡¿por qué lo harías?!). Sin embargo, `undefined` *es* (desafortunadamente) un identificador. Oh oh.

### Undefined

En modo no-`strict`, ¡es realmente posible (aunque increíblemente desaconsejado!) asignar un valor al identificador `undefined` proporcionado globalmente:

```js
function foo() {
	undefined = 2; // ¡realmente mala idea!
}

foo();
```

```js
function foo() {
	"use strict";
	undefined = 2; // TypeError!
}

foo();
```

En modo no-`strict` y modo `strict`, sin embargo, puedes crear una variable local con el nombre `undefined`. ¡Pero de nuevo, esta es una idea terrible!

```js
function foo() {
	"use strict";
	var undefined = 2;
	console.log( undefined ); // 2
}

foo();
```

**Los amigos no dejan que los amigos sobreescriban `undefined`.** Jamás.

#### Operador `void`

Mientras que `undefined` es un identificador incorporado que contiene (¡a menos que se modifique -- ver arriba!) el valor `undefined` incorporado, otra forma de obtener este valor es el operador `void`.

La expresión `void ___` "anula" cualquier valor, de modo que el resultado de la expresión siempre es el valor `undefined`. No modifica el valor existente; solo asegura que ningún valor regrese de la expresión del operador.

```js
var a = 42;

console.log( void a, a ); // undefined 42
```

Por convención (mayormente de la programación en lenguaje C), para representar el valor `undefined` por sí solo usando `void`, usarías `void 0` (aunque claramente incluso `void true` o cualquier otra expresión `void` hace lo mismo). No hay diferencia práctica entre `void 0`, `void 1`, y `undefined`.

Pero el operador `void` puede ser útil en algunas otras circunstancias, si necesitas asegurar que una expresión no tenga valor de resultado (incluso si tiene efectos secundarios).

Por ejemplo:

```js
function doSomething() {
	// nota: `APP.ready` es proporcionado por nuestra aplicación
	if (!APP.ready) {
		// intentar de nuevo más tarde
		return void setTimeout( doSomething, 100 );
	}

	var result;

	// hacer algunas otras cosas
	return result;
}

// ¿pudimos hacerlo de inmediato?
if (doSomething()) {
	// manejar las siguientes tareas de inmediato
}
```

Aquí, la función `setTimeout(..)` devuelve un valor numérico (el identificador único del intervalo del temporizador, si quisieras cancelarlo), pero queremos usar `void` para anularlo de modo que el valor de retorno de nuestra función no dé un falso positivo con la declaración `if`.

Muchos desarrolladores prefieren simplemente hacer estas acciones por separado, lo que funciona igual pero no usa el operador `void`:

```js
if (!APP.ready) {
	// intentar de nuevo más tarde
	setTimeout( doSomething, 100 );
	return;
}
```

En general, si alguna vez hay un lugar donde existe un valor (de alguna expresión) y encontrarías útil que el valor sea `undefined` en su lugar, usa el operador `void`. Eso probablemente no será terriblemente común en tus programas, pero en los raros casos que lo necesites, puede ser bastante útil.

### Números Especiales

El tipo `number` incluye varios valores especiales. Echaremos un vistazo a cada uno en detalle.

#### El No Número, Número

Cualquier operación matemática que realices sin que ambos operandos sean `number`s (o valores que puedan ser interpretados como `number`s regulares en base 10 o base 16) resultará en que la operación falle al producir un `number` válido, en cuyo caso obtendrás el valor `NaN`.

`NaN` literalmente significa "not a `number`" (no es un número), aunque esta etiqueta/descripción es muy pobre y engañosa, como veremos en breve. Sería mucho más preciso pensar en `NaN` como "número inválido", "número fallido", o incluso "número malo", que pensar en él como "no un número".

Por ejemplo:

```js
var a = 2 / "foo";		// NaN

typeof a === "number";	// true
```

En otras palabras: "¡el tipo de no-un-número es 'number'!" ¡Hurra por los nombres confusos y la semántica!

`NaN` es un tipo de "valor centinela" (un valor por lo demás normal al que se le asigna un significado especial) que representa un tipo especial de condición de error dentro del conjunto de `number`. La condición de error es, en esencia: "Intenté realizar una operación matemática pero fallé, así que aquí está el resultado `number` fallido en su lugar."

Entonces, si tienes un valor en alguna variable y quieres probar si es este `NaN` especial de número fallido, podrías pensar que podrías comparar directamente con `NaN` mismo, como puedes con cualquier otro valor, como `null` o `undefined`. Nop.

```js
var a = 2 / "foo";

a == NaN;	// false
a === NaN;	// false
```

`NaN` es un valor muy especial en que nunca es igual a otro valor `NaN` (es decir, nunca es igual a sí mismo). Es el único valor, de hecho, que no es reflexivo (sin la característica de Identidad `x === x`). Así que, `NaN !== NaN`. Un poco extraño, ¿eh?

Entonces, ¿cómo *lo* probamos, si no podemos comparar con `NaN` (ya que esa comparación siempre fallaría)?

```js
var a = 2 / "foo";

isNaN( a ); // true
```

Bastante fácil, ¿verdad? Usamos la utilidad global incorporada llamada `isNaN(..)` y nos dice si el valor es `NaN` o no. ¡Problema resuelto!

No tan rápido.

La utilidad `isNaN(..)` tiene un defecto fatal. Parece que intentó tomar el significado de `NaN` ("Not a Number") demasiado literalmente -- que su trabajo es básicamente: "probar si lo que se pasó no es un `number` o es un `number`." Pero eso no es bastante preciso.

```js
var a = 2 / "foo";
var b = "foo";

a; // NaN
b; // "foo"

window.isNaN( a ); // true
window.isNaN( b ); // true -- ¡ouch!
```

Claramente, `"foo"` literalmente *no es un `number`*, pero definitivamente tampoco es el valor `NaN`. Este bug ha estado en JS desde el principio (más de 19 años de *ouch*).

A partir de ES6, finalmente se ha proporcionado una utilidad de reemplazo: `Number.isNaN(..)`. Un polyfill simple para que puedas verificar valores `NaN` de forma segura *ahora* incluso en navegadores pre-ES6 es:

```js
if (!Number.isNaN) {
	Number.isNaN = function(n) {
		return (
			typeof n === "number" &&
			window.isNaN( n )
		);
	};
}

var a = 2 / "foo";
var b = "foo";

Number.isNaN( a ); // true
Number.isNaN( b ); // false -- ¡uf!
```

De hecho, podemos implementar un polyfill de `Number.isNaN(..)` aún más fácil, aprovechando ese hecho peculiar de que `NaN` no es igual a sí mismo. `NaN` es el *único* valor en todo el lenguaje donde eso es cierto; cada otro valor siempre es **igual a sí mismo**.

Así que:

```js
if (!Number.isNaN) {
	Number.isNaN = function(n) {
		return n !== n;
	};
}
```

¿Raro, eh? ¡Pero funciona!

Los `NaN`s son probablemente una realidad en muchos programas JS del mundo real, ya sea a propósito o por accidente. Es una muy buena idea usar una prueba confiable, como `Number.isNaN(..)` como se proporciona (o con polyfill), para reconocerlos apropiadamente.

Si actualmente estás usando solo `isNaN(..)` en un programa, ¡la triste realidad es que tu programa *tiene un bug*, incluso si aún no te ha mordido!

#### Infinities

Los desarrolladores de lenguajes compilados tradicionales como C probablemente están acostumbrados a ver ya sea un error de compilador o una excepción en tiempo de ejecución, como "Divide by zero", para una operación como:

```js
var a = 1 / 0;
```

Sin embargo, en JS, esta operación está bien definida y resulta en el valor `Infinity` (también conocido como `Number.POSITIVE_INFINITY`). Como era de esperar:

```js
var a = 1 / 0;	// Infinity
var b = -1 / 0;	// -Infinity
```

Como puedes ver, `-Infinity` (también conocido como `Number.NEGATIVE_INFINITY`) resulta de una división por cero donde uno (¡pero no ambos!) de los operandos de la división es negativo.

JS usa representaciones numéricas finitas (punto flotante IEEE 754, que cubrimos antes), así que contrario a las matemáticas puras, parece que *es* posible desbordarse incluso con una operación como suma o resta, en cuyo caso obtendrías `Infinity` o `-Infinity`.

Por ejemplo:

```js
var a = Number.MAX_VALUE;	// 1.7976931348623157e+308
a + a;						// Infinity
a + Math.pow( 2, 970 );		// Infinity
a + Math.pow( 2, 969 );		// 1.7976931348623157e+308
```

De acuerdo con la especificación, si una operación como la suma resulta en un valor demasiado grande para representar, el modo "redondeo al más cercano" de IEEE 754 especifica cuál debería ser el resultado. Así, en un sentido burdo, `Number.MAX_VALUE + Math.pow( 2, 969 )` está más cerca de `Number.MAX_VALUE` que de `Infinity`, así que se "redondea hacia abajo", mientras que `Number.MAX_VALUE + Math.pow( 2, 970 )` está más cerca de `Infinity` así que se "redondea hacia arriba".

Si piensas demasiado en eso, te va a doler la cabeza. Así que no lo hagas. En serio, ¡para!

Una vez que desbordas a cualquiera de los *infinitos*, sin embargo, no hay vuelta atrás. En otras palabras, en un sentido casi poético, puedes ir de finito a infinito pero no de infinito a finito.

Es casi filosófico preguntar: "¿Qué es infinito dividido por infinito?". Nuestros cerebros ingenuos probablemente dirían "1" o quizás "infinito". Resulta que ninguno es cierto. Tanto matemáticamente como en JavaScript, `Infinity / Infinity` no es una operación definida. En JS, esto resulta en `NaN`.

¿Pero qué hay de cualquier `number` finito positivo dividido por `Infinity`? ¡Eso es fácil! `0`. ¿Y qué hay de un `number` finito negativo dividido por `Infinity`? ¡Sigue leyendo!

#### Zeros

Aunque pueda confundir al lector con mente matemática, JavaScript tiene tanto un cero normal `0` (también conocido como cero positivo `+0`) *y* un cero negativo `-0`. Antes de explicar por qué existe el `-0`, deberíamos examinar cómo JS lo maneja, porque puede ser bastante confuso.

Además de ser especificado literalmente como `-0`, el cero negativo también resulta de ciertas operaciones matemáticas. Por ejemplo:

```js
var a = 0 / -3; // -0
var b = 0 * -3; // -0
```

La suma y la resta no pueden resultar en un cero negativo.

Un cero negativo cuando se examina en la consola del desarrollador generalmente revelará `-0`, aunque ese no era el caso común hasta bastante recientemente, así que algunos navegadores más antiguos que encuentres todavía pueden reportarlo como `0`.

Sin embargo, si intentas convertir a string un valor de cero negativo, siempre será reportado como `"0"`, de acuerdo con la especificación.

```js
var a = 0 / -3;

// (algunas) consolas de navegadores al menos lo hacen bien
a;							// -0

// ¡pero la especificación insiste en mentirte!
a.toString();				// "0"
a + "";						// "0"
String( a );				// "0"

// extrañamente, incluso JSON participa en el engaño
JSON.stringify( a );		// "0"
```

Interesantemente, las operaciones inversas (ir de `string` a `number`) no mienten:

```js
+"-0";				// -0
Number( "-0" );		// -0
JSON.parse( "-0" );	// -0
```

**Advertencia:** El comportamiento de `JSON.stringify( -0 )` de `"0"` es particularmente extraño cuando observas que es inconsistente con lo inverso: `JSON.parse( "-0" )` reporta `-0` como correctamente esperarías.

Además de la conversión a string del cero negativo siendo engañosa para ocultar su verdadero valor, los operadores de comparación también están (intencionalmente) configurados para *mentir*.

```js
var a = 0;
var b = 0 / -3;

a == b;		// true
-0 == 0;	// true

a === b;	// true
-0 === 0;	// true

0 > -0;		// false
a > b;		// false
```

Claramente, si quieres distinguir un `-0` de un `0` en tu código, no puedes simplemente confiar en lo que la consola del desarrollador muestra, así que vas a tener que ser un poco más astuto:

```js
function isNegZero(n) {
	n = Number( n );
	return (n === 0) && (1 / n === -Infinity);
}

isNegZero( -0 );		// true
isNegZero( 0 / -3 );	// true
isNegZero( 0 );			// false
```

Ahora, ¿por qué necesitamos un cero negativo, además de trivia académica?

Hay ciertas aplicaciones donde los desarrolladores usan la magnitud de un valor para representar una pieza de información (como la velocidad de movimiento por cuadro de animación) y el signo de ese `number` para representar otra pieza de información (como la dirección de ese movimiento).

En esas aplicaciones, como un ejemplo, si una variable llega a cero y pierde su signo, entonces perderías la información de en qué dirección se estaba moviendo antes de llegar a cero. Preservar el signo del cero previene una pérdida de información potencialmente no deseada.

### Igualdad Especial

Como vimos arriba, el valor `NaN` y el valor `-0` tienen un comportamiento especial cuando se trata de comparación de igualdad. `NaN` nunca es igual a sí mismo, así que tienes que usar `Number.isNaN(..)` de ES6 (o un polyfill). De manera similar, `-0` miente y pretende que es igual (incluso `===` igualdad estricta -- ver Capítulo 4) al `0` positivo regular, así que tienes que usar la utilidad algo hacky `isNegZero(..)` que sugerimos arriba.

A partir de ES6, hay una nueva utilidad que puede ser usada para probar dos valores por igualdad absoluta, sin ninguna de estas excepciones. Se llama `Object.is(..)`:

```js
var a = 2 / "foo";
var b = -3 * 0;

Object.is( a, NaN );	// true
Object.is( b, -0 );		// true

Object.is( b, 0 );		// false
```

Hay un polyfill bastante simple para `Object.is(..)` para entornos pre-ES6:

```js
if (!Object.is) {
	Object.is = function(v1, v2) {
		// prueba para `-0`
		if (v1 === 0 && v2 === 0) {
			return 1 / v1 === 1 / v2;
		}
		// prueba para `NaN`
		if (v1 !== v1) {
			return v2 !== v2;
		}
		// todo lo demás
		return v1 === v2;
	};
}
```

`Object.is(..)` probablemente no debería usarse en casos donde se sabe que `==` o `===` son *seguros* (ver Capítulo 4 "Coerción"), ya que los operadores son probablemente mucho más eficientes y ciertamente son más idiomáticos/comunes. `Object.is(..)` es mayormente para estos casos especiales de igualdad.
