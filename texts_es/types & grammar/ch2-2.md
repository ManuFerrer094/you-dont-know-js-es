## Valor vs. Referencia

En muchos otros lenguajes, los valores pueden ser asignados/pasados por copia de valor o por copia de referencia dependiendo de la sintaxis que uses.

Por ejemplo, en C++ si quieres pasar una variable `number` a una función y que el valor de esa variable sea actualizado, puedes declarar el parámetro de la función como `int& myNum`, y cuando pases una variable como `x`, `myNum` será una **referencia a `x`**; las referencias son como una forma especial de punteros, donde obtienes un puntero a otra variable (como un *alias*). Si no declaras un parámetro de referencia, el valor pasado *siempre* será copiado, incluso si es un objeto complejo.

En JavaScript, no hay punteros, y las referencias funcionan un poco diferente. No puedes tener una referencia de una variable JS a otra variable. Eso simplemente no es posible.

Una referencia en JS apunta a un **valor** (compartido), así que si tienes 10 referencias diferentes, todas son siempre referencias distintas a un único valor compartido; **ninguna de ellas son referencias/punteros entre sí.**

Además, en JavaScript, no hay indicios sintácticos que controlen la asignación/paso por valor vs. referencia. En su lugar, el *tipo* del valor *únicamente* controla si ese valor será asignado por copia de valor o por copia de referencia.

Ilustremos:

```js
var a = 2;
var b = a; // `b` es siempre una copia del valor en `a`
b++;
a; // 2
b; // 3

var c = [1,2,3];
var d = c; // `d` es una referencia al valor compartido `[1,2,3]`
d.push( 4 );
c; // [1,2,3,4]
d; // [1,2,3,4]
```

Los valores simples (también conocidos como primitivos escalares) *siempre* se asignan/pasan por copia de valor: `null`, `undefined`, `string`, `number`, `boolean`, y `symbol` de ES6.

Los valores compuestos -- `object`s (incluyendo `array`s, y todos los objetos envoltorio encapsulados -- ver Capítulo 3) y `function`s -- *siempre* crean una copia de la referencia en la asignación o paso.

En el fragmento anterior, porque `2` es un primitivo escalar, `a` contiene una copia inicial de ese valor, y a `b` se le asigna otra *copia* del valor. Cuando cambias `b`, de ninguna manera estás cambiando el valor en `a`.

Pero **tanto `c` como `d`** son referencias separadas al mismo valor compartido `[1,2,3]`, que es un valor compuesto. Es importante notar que ni `c` ni `d` "poseen" más el valor `[1,2,3]` -- ambos son solo referencias par iguales al valor. Entonces, cuando usas cualquier referencia para modificar (`.push(4)`) el valor `array` compartido real, está afectando solo al único valor compartido, y ambas referencias referenciarán el valor recién modificado `[1,2,3,4]`.

Dado que las referencias apuntan a los valores en sí y no a las variables, no puedes usar una referencia para cambiar a dónde apunta otra referencia:

```js
var a = [1,2,3];
var b = a;
a; // [1,2,3]
b; // [1,2,3]

// después
b = [4,5,6];
a; // [1,2,3]
b; // [4,5,6]
```

Cuando hacemos la asignación `b = [4,5,6]`, no estamos haciendo absolutamente nada para afectar *a dónde* `a` todavía está referenciando (`[1,2,3]`). Para hacer eso, `b` tendría que ser un puntero a `a` en lugar de una referencia al `array` -- ¡pero tal capacidad no existe en JS!

La forma más común en que ocurre tal confusión es con los parámetros de función:

```js
function foo(x) {
	x.push( 4 );
	x; // [1,2,3,4]

	// después
	x = [4,5,6];
	x.push( 7 );
	x; // [4,5,6,7]
}

var a = [1,2,3];

foo( a );

a; // [1,2,3,4]  no  [4,5,6,7]
```

Cuando pasamos el argumento `a`, asigna una copia de la referencia de `a` a `x`. `x` y `a` son referencias separadas apuntando al mismo valor `[1,2,3]`. Ahora, dentro de la función, podemos usar esa referencia para mutar el valor en sí (`push(4)`). Pero cuando hacemos la asignación `x = [4,5,6]`, esto de ninguna manera afecta a dónde apunta la referencia inicial `a` -- todavía apunta al valor (ahora modificado) `[1,2,3,4]`.

No hay forma de usar la referencia `x` para cambiar a dónde apunta `a`. Solo podríamos modificar el contenido del valor compartido al que tanto `a` como `x` están apuntando.

Para lograr cambiar `a` para que tenga el contenido de valor `[4,5,6,7]`, no puedes crear un nuevo `array` y asignar -- debes modificar el valor `array` existente:

```js
function foo(x) {
	x.push( 4 );
	x; // [1,2,3,4]

	// después
	x.length = 0; // vaciar el array existente in situ
	x.push( 4, 5, 6, 7 );
	x; // [4,5,6,7]
}

var a = [1,2,3];

foo( a );

a; // [4,5,6,7]  no  [1,2,3,4]
```

Como puedes ver, `x.length = 0` y `x.push(4,5,6,7)` no estaban creando un nuevo `array`, sino modificando el `array` compartido existente. Así que por supuesto, `a` referencia el nuevo contenido `[4,5,6,7]`.

Recuerda: no puedes controlar/sobreescribir directamente la copia de valor vs. referencia -- esa semántica es controlada enteramente por el tipo del valor subyacente.

Para pasar efectivamente un valor compuesto (como un `array`) por copia de valor, necesitas hacer manualmente una copia de él, de modo que la referencia pasada no siga apuntando al original. Por ejemplo:

```js
foo( a.slice() );
```

`slice(..)` sin parámetros por defecto hace una copia enteramente nueva (superficial) del `array`. Entonces, pasamos una referencia solo al `array` copiado, y así `foo(..)` no puede afectar el contenido de `a`.

Para hacer lo inverso -- pasar un valor primitivo escalar de una manera donde sus actualizaciones de valor puedan ser vistas, algo así como una referencia -- tienes que envolver el valor en otro valor compuesto (`object`, `array`, etc.) que *pueda* ser pasado por copia de referencia:

```js
function foo(wrapper) {
	wrapper.a = 42;
}

var obj = {
	a: 2
};

foo( obj );

obj.a; // 42
```

Aquí, `obj` actúa como un envoltorio para la propiedad primitiva escalar `a`. Cuando se pasa a `foo(..)`, una copia de la referencia de `obj` se pasa y se establece en el parámetro `wrapper`. Ahora podemos usar la referencia `wrapper` para acceder al objeto compartido, y actualizar su propiedad. Después de que la función termina, `obj.a` verá el valor actualizado `42`.

Podría ocurrírsete que si quisieras pasar una referencia a un valor primitivo escalar como `2`, podrías simplemente encapsular el valor en su objeto envoltorio `Number` (ver Capítulo 3).

*Es* cierto que una copia de la referencia a este objeto `Number` *será* pasada a la función, pero desafortunadamente, tener una referencia al objeto compartido no te va a dar la capacidad de modificar el valor primitivo compartido, como podrías esperar:

```js
function foo(x) {
	x = x + 1;
	x; // 3
}

var a = 2;
var b = new Number( a ); // o equivalentemente `Object(a)`

foo( b );
console.log( b ); // 2, no 3
```

El problema es que el valor primitivo escalar subyacente *no es mutable* (lo mismo aplica para `String` y `Boolean`). Si un objeto `Number` contiene el valor primitivo escalar `2`, ese objeto `Number` exacto nunca puede ser cambiado para contener otro valor; solo puedes crear un objeto `Number` completamente nuevo con un valor diferente.

Cuando `x` se usa en la expresión `x + 1`, el valor primitivo escalar subyacente `2` es desencapsulado (extraído) del objeto `Number` automáticamente, así que la línea `x = x + 1` muy sutilmente cambia `x` de ser una referencia compartida al objeto `Number`, a solo contener el valor primitivo escalar `3` como resultado de la operación de suma `2 + 1`. Por lo tanto, `b` en el exterior todavía referencia el objeto `Number` original no modificado/inmutable que contiene el valor `2`.

*Puedes* agregar propiedades encima del objeto `Number` (solo no cambiar su valor primitivo interno), así que podrías intercambiar información indirectamente a través de esas propiedades adicionales.

Sin embargo, esto no es tan común; probablemente no sería considerado una buena práctica por la mayoría de los desarrolladores.

En lugar de usar el objeto envoltorio `Number` de esta manera, probablemente es mucho mejor usar el enfoque de envoltorio de objeto manual (`obj`) del fragmento anterior. Eso no quiere decir que no hay usos ingeniosos para los objetos envoltorio encapsulados como `Number` -- solo que probablemente deberías preferir la forma de valor primitivo escalar en la mayoría de los casos.

Las referencias son bastante poderosas, pero a veces se interponen en tu camino, y a veces las necesitas donde no existen. El único control que tienes sobre el comportamiento de referencia vs. copia de valor es el tipo del valor en sí, así que debes influir indirectamente en el comportamiento de asignación/paso mediante los tipos de valor que elijas usar.

## Repaso

En JavaScript, los `array`s son simplemente colecciones indexadas numéricamente de cualquier tipo de valor. Los `string`s son algo "`array`-like", pero tienen comportamientos distintos y se debe tener cuidado si quieres tratarlos como `array`s. Los números en JavaScript incluyen tanto "enteros" como valores de punto flotante.

Varios valores especiales están definidos dentro de los tipos primitivos.

El tipo `null` tiene solo un valor: `null`, y de igual manera el tipo `undefined` tiene solo el valor `undefined`. `undefined` es básicamente el valor predeterminado en cualquier variable o propiedad si ningún otro valor está presente. El operador `void` te permite crear el valor `undefined` a partir de cualquier otro valor.

Los `number`s incluyen varios valores especiales, como `NaN` (supuestamente "Not a Number", pero realmente más apropiadamente "número inválido"); `+Infinity` y `-Infinity`; y `-0`.

Los primitivos escalares simples (`string`s, `number`s, etc.) se asignan/pasan por copia de valor, pero los valores compuestos (`object`s, etc.) se asignan/pasan por copia de referencia. Las referencias no son como las referencias/punteros en otros lenguajes -- nunca apuntan a otras variables/referencias, solo a los valores subyacentes.
