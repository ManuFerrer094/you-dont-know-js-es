# You Don't Know JS: ES6 & Beyond
# Capítulo 6: Adiciones a la API

Desde conversiones de valores hasta cálculos matemáticos, ES6 añade muchas propiedades estáticas y métodos a varios nativos y objetos integrados para ayudar con tareas comunes. Además, las instancias de algunos de los nativos tienen nuevas capacidades a través de varios nuevos métodos de prototipo.

**Nota:** La mayoría de estas características pueden ser fielmente polyfilled. No entraremos en tales detalles aquí, pero revisa "ES6 Shim" (https://github.com/paulmillr/es6-shim/) para shims/polyfills compatibles con los estándares.

## `Array`

Una de las características más comúnmente extendidas en JS por varias bibliotecas de usuario es el tipo Array. No debería ser ninguna sorpresa que ES6 añada un número de ayudantes a Array, tanto estáticos como de prototipo (instancia).

### Función Estática `Array.of(..)`

Hay un gotcha bien conocido con el constructor `Array(..)`, que es que si solo hay un argumento pasado, y ese argumento es un número, en lugar de hacer un array de un elemento con ese valor numérico, construye un array vacío con una propiedad `length` igual al número. Esta acción produce el desafortunado y peculiar comportamiento de "slots vacíos" que es repudiado sobre los arrays de JS.

`Array.of(..)` reemplaza a `Array(..)` como el constructor preferido en forma de función para arrays, porque `Array.of(..)` no tiene ese caso especial de argumento-único-numérico. Considera:

```js
var a = Array( 3 );
a.length;						// 3
a[0];							// undefined

var b = Array.of( 3 );
b.length;						// 1
b[0];							// 3

var c = Array.of( 1, 2, 3 );
c.length;						// 3
c;								// [1,2,3]
```

¿Bajo qué circunstancias querrías usar `Array.of(..)` en lugar de simplemente crear un array con sintaxis literal, como `c = [1,2,3]`? Hay dos casos posibles.

Si tienes un callback que se supone que debe envolver argumento(s) pasados a él en un array, `Array.of(..)` encaja perfectamente. Eso probablemente no es terriblemente común, pero puede rascarle una comezón.

El otro escenario es si creas una subclase de `Array` (ver "Clases" en el Capítulo 3) y quieres ser capaz de crear e inicializar elementos en una instancia de tu subclase, como:

```js
class MyCoolArray extends Array {
	sum() {
		return this.reduce( function reducer(acc,curr){
			return acc + curr;
		}, 0 );
	}
}

var x = new MyCoolArray( 3 );
x.length;						// 3 -- ¡oops!
x.sum();						// 0 -- ¡oops!

var y = [3];					// Array, no MyCoolArray
y.length;						// 1
y.sum();						// `sum` is not a function

var z = MyCoolArray.of( 3 );
z.length;						// 1
z.sum();						// 3
```

No puedes simplemente (fácilmente) crear un constructor para `MyCoolArray` que sobrescriba el comportamiento del constructor padre `Array`, porque ese constructor es necesario para realmente crear un valor de array bien comportado (inicializando el `this`). El método estático `of(..)` "heredado" en la subclase `MyCoolArray` proporciona una buena solución.

### Función Estática `Array.from(..)`

Un "objeto array-like" en JavaScript es un objeto que tiene una propiedad `length` en él, específicamente con un valor entero de cero o superior.

Estos valores han sido notoriamente frustrantes para trabajar en JS; ha sido bastante común necesitar transformarlos en un array real, para que los varios métodos de `Array.prototype` (`map(..)`, `indexOf(..)` etc.) estén disponibles para usar con él. Ese proceso usualmente se ve así:

```js
// array-like object
var arrLike = {
	length: 3,
	0: "foo",
	1: "bar"
};

var arr = Array.prototype.slice.call( arrLike );
```

Otra tarea común donde `slice(..)` se usa frecuentemente es en duplicar un array real:

```js
var arr2 = arr.slice();
```

En ambos casos, el nuevo `Array.from(..)` de ES6 puede ser un enfoque más comprensible y elegante — aunque también menos verboso:

```js
var arr = Array.from( arrLike );

var arrCopy = Array.from( arr );
```

`Array.from(..)` mira si el primer argumento es un iterable (ver "Iteradores" en el Capítulo 3), y si es así, usa el iterador para producir valores para "copiar" en el array devuelto. Porque los arrays reales tienen un iterador para esos valores, ese iterador se usa automáticamente.

Pero si pasas un objeto array-like como primer argumento a `Array.from(..)`, se comporta básicamente igual que `slice()` (sin argumentos!) o `apply(..)`, lo cual es que simplemente itera sobre el valor, accediendo propiedades nombradas numéricamente desde `0` hasta cualquier valor que tenga `length`.

Considera:

```js
var arrLike = {
	length: 4,
	2: "foo"
};

Array.from( arrLike );
// [ undefined, undefined, "foo", undefined ]
```

Porque las posiciones `0`, `1` y `3` no existían en `arrLike`, el resultado fue el valor `undefined` para cada uno de esos slots.

Podrías producir un resultado similar así:

```js
var emptySlotsArr = [];
emptySlotsArr.length = 4;
emptySlotsArr[2] = "foo";

Array.from( emptySlotsArr );
// [ undefined, undefined, "foo", undefined ]
```

#### Evitando Slots Vacíos

Hay una diferencia sutil pero importante en el fragmento anterior entre `emptySlotsArr` y el resultado de la llamada `Array.from(..)`. `Array.from(..)` nunca produce slots vacíos.

Antes de ES6, si querías producir un array inicializado a una cierta longitud con valores `undefined` reales en cada slot (¡sin slots vacíos!), tenías que hacer trabajo extra:

```js
var a = Array( 4 );								// ¡cuatro slots vacíos!

var b = Array.apply( null, { length: 4 } );		// cuatro valores `undefined`
```

Pero `Array.from(..)` ahora hace esto más fácil:

```js
var c = Array.from( { length: 4 } );			// cuatro valores `undefined`
```

**Advertencia:** Usar un array con slots vacíos como `a` en los fragmentos anteriores funcionaría con algunas funciones de array, pero otras ignoran los slots vacíos (como `map(..)`, etc.). Nunca deberías trabajar intencionalmente con slots vacíos, ya que casi con certeza llevará a comportamiento extraño/impredecible en tus programas.

#### Mapeo

La utilidad `Array.from(..)` tiene otro truco útil bajo la manga. El segundo argumento, si se proporciona, es un callback de mapeo (casi igual al que el regular `Array#map(..)` espera) que se llama para mapear/transformar cada valor de la fuente al objetivo devuelto. Considera:

```js
var arrLike = {
	length: 4,
	2: "foo"
};

Array.from( arrLike, function mapper(val,idx){
	if (typeof val == "string") {
		return val.toUpperCase();
	}
	else {
		return idx;
	}
} );
// [ 0, 1, "FOO", 3 ]
```

**Nota:** Como con otros métodos de array que toman callbacks, `Array.from(..)` toma un tercer argumento opcional que si se establece especificará el enlace `this` para el callback pasado como segundo argumento. De lo contrario, `this` será `undefined`.

Ver "TypedArrays" en el Capítulo 5 para un ejemplo de usar `Array.from(..)` en la traducción de valores de un array de valores de 8 bits a un array de valores de 16 bits.

### Creando Arrays y Subtipos

En las últimas secciones, hemos discutido `Array.of(..)` y `Array.from(..)`, ambos de los cuales crean un nuevo array de manera similar a un constructor. ¿Pero qué hacen en subclases? ¿Crean instancias del `Array` base o de la subclase derivada?

```js
class MyCoolArray extends Array {
	..
}

MyCoolArray.from( [1, 2] ) instanceof MyCoolArray;	// true

Array.from(
	MyCoolArray.from( [1, 2] )
) instanceof MyCoolArray;							// false
```

Tanto `of(..)` como `from(..)` usan el constructor desde el cual se acceden para construir el array. Así que si usas el `Array.of(..)` base obtendrás una instancia de `Array`, pero si usas `MyCoolArray.of(..)`, obtendrás una instancia de `MyCoolArray`.

En "Clases" en el Capítulo 3, cubrimos la configuración `@@species` que todas las clases integradas (como `Array`) tienen definida, que es usada por cualquier método de prototipo si crean una nueva instancia. `slice(..)` es un gran ejemplo:

```js
var x = new MyCoolArray( 1, 2, 3 );

x.slice( 1 ) instanceof MyCoolArray;				// true
```

Generalmente, ese comportamiento por defecto probablemente será el deseado, pero como discutimos en el Capítulo 3, *puedes* sobrescribirlo si quieres:

```js
class MyCoolArray extends Array {
	// forzar `species` a ser constructor padre
	static get [Symbol.species]() { return Array; }
}

var x = new MyCoolArray( 1, 2, 3 );

x.slice( 1 ) instanceof MyCoolArray;				// false
x.slice( 1 ) instanceof Array;						// true
```

Es importante notar que la configuración `@@species` solo se usa para los métodos de prototipo, como `slice(..)`. No se usa por `of(..)` ni `from(..)`; ambos simplemente usan el enlace `this` (cualquier constructor que se use para hacer la referencia). Considera:

```js
class MyCoolArray extends Array {
	// forzar `species` a ser constructor padre
	static get [Symbol.species]() { return Array; }
}

var x = new MyCoolArray( 1, 2, 3 );

MyCoolArray.from( x ) instanceof MyCoolArray;		// true
MyCoolArray.of( [2, 3] ) instanceof MyCoolArray;	// true
```

### Método de Prototipo `copyWithin(..)`

`Array#copyWithin(..)` es un nuevo método mutador disponible para todos los arrays (incluyendo Typed Arrays; ver Capítulo 5). `copyWithin(..)` copia una porción de un array a otra ubicación en el mismo array, sobrescribiendo lo que estaba allí antes.

Los argumentos son *target* (el índice al cual copiar), *start* (el índice inclusivo desde el cual empezar a copiar), y opcionalmente *end* (el índice exclusivo donde parar de copiar). Si alguno de los argumentos es negativo, se toma como relativo desde el final del array.

Considera:

```js
[1,2,3,4,5].copyWithin( 3, 0 );			// [1,2,3,1,2]

[1,2,3,4,5].copyWithin( 3, 0, 1 );		// [1,2,3,1,5]

[1,2,3,4,5].copyWithin( 0, -2 );		// [4,5,3,4,5]

[1,2,3,4,5].copyWithin( 0, -2, -1 );	// [4,2,3,4,5]
```

El método `copyWithin(..)` no extiende la longitud del array, como muestra el primer ejemplo en el fragmento anterior. La copia simplemente se detiene cuando se alcanza el final del array.

Contrario a lo que podrías pensar, la copia no siempre va en orden de izquierda a derecha (índice ascendente). Es posible que esto resultara en copiar repetidamente un valor ya copiado si los rangos de origen y destino se superponen, lo cual presumiblemente no es el comportamiento deseado.

Así que internamente, el algoritmo evita este caso copiando en orden inverso para evitar ese gotcha. Considera:

```js
[1,2,3,4,5].copyWithin( 2, 1 );		// ???
```

Si el algoritmo fuera estrictamente de izquierda a derecha, entonces el `2` debería copiarse para sobrescribir el `3`, luego *esa* copia del `2` debería copiarse para sobrescribir el `4`, luego *esa* copia del `2` debería copiarse para sobrescribir el `5`, y terminarías con `[1,2,2,2,2]`.

En su lugar, el algoritmo de copia invierte la dirección y copia `4` para sobrescribir `5`, luego copia `3` para sobrescribir `4`, luego copia `2` para sobrescribir `3`, y el resultado final es `[1,2,2,3,4]`. Eso es probablemente más "correcto" en términos de expectativa, pero puede ser confuso si solo estás pensando en el algoritmo de copia de manera ingenua de izquierda a derecha.

### Método de Prototipo `fill(..)`

Rellenar un array existente total o parcialmente con un valor especificado es soportado nativamente a partir de ES6 con el método `Array#fill(..)`:

```js
var a = Array( 4 ).fill( undefined );
a;
// [undefined,undefined,undefined,undefined]
```

`fill(..)` opcionalmente toma parámetros *start* y *end*, que indican una porción subconjunto del array a rellenar, como:

```js
var a = [ null, null, null, null ].fill( 42, 1, 3 );

a;									// [null,42,42,null]
```

### Método de Prototipo `find(..)`

La manera más común de buscar un valor en un array ha sido generalmente el método `indexOf(..)`, que devuelve el índice donde se encuentra el valor o `-1` si no se encuentra:

```js
var a = [1,2,3,4,5];

(a.indexOf( 3 ) != -1);				// true
(a.indexOf( 7 ) != -1);				// false

(a.indexOf( "2" ) != -1);			// false
```

La comparación de `indexOf(..)` requiere una coincidencia estricta `===`, así que una búsqueda de `"2"` no encuentra un valor de `2`, y viceversa. No hay manera de sobrescribir el algoritmo de coincidencia para `indexOf(..)`. También es desafortunado/poco elegante tener que hacer la comparación manual contra el valor `-1`.

**Consejo:** Ver el título *Types & Grammar* de esta serie para una técnica interesante (y controversialmente confusa) para sortear la fealdad del `-1` con el operador `~`.

Desde ES5, la solución alternativa más común para tener control sobre la lógica de coincidencia ha sido el método `some(..)`. Funciona llamando un callback de función para cada elemento, hasta que una de esas llamadas devuelve un valor `true`/truthy, y entonces se detiene. Porque defines la función callback, tienes control total sobre cómo se hace una coincidencia:

```js
var a = [1,2,3,4,5];

a.some( function matcher(v){
	return v == "2";
} );								// true

a.some( function matcher(v){
	return v == 7;
} );								// false
```

Pero la desventaja de este enfoque es que solo obtienes el `true`/`false` indicando si se encontró un valor que coincidió adecuadamente, pero no cuál fue el valor coincidente real.

El `find(..)` de ES6 aborda esto. Funciona básicamente igual que `some(..)`, excepto que una vez que el callback devuelve un valor `true`/truthy, el valor real del array se devuelve:

```js
var a = [1,2,3,4,5];

a.find( function matcher(v){
	return v == "2";
} );								// 2

a.find( function matcher(v){
	return v == 7;					// undefined
});
```

Usar una función `matcher(..)` personalizada también te permite coincidir contra valores complejos como objetos:

```js
var points = [
	{ x: 10, y: 20 },
	{ x: 20, y: 30 },
	{ x: 30, y: 40 },
	{ x: 40, y: 50 },
	{ x: 50, y: 60 }
];

points.find( function matcher(point) {
	return (
		point.x % 3 == 0 &&
		point.y % 4 == 0
	);
} );								// { x: 30, y: 40 }
```

**Nota:** Como con otros métodos de array que toman callbacks, `find(..)` toma un segundo argumento opcional que si se establece especificará el enlace `this` para el callback pasado como primer argumento. De lo contrario, `this` será `undefined`.

### Método de Prototipo `findIndex(..)`

Mientras que la sección anterior ilustra cómo `some(..)` produce un resultado booleano para una búsqueda de un array, y `find(..)` produce el valor coincidente mismo de la búsqueda del array, también hay una necesidad de encontrar el índice posicional del valor coincidente.

`indexOf(..)` hace eso, pero no hay control sobre su lógica de coincidencia; siempre usa igualdad estricta `===`. Así que el `findIndex(..)` de ES6 es la respuesta:

```js
var points = [
	{ x: 10, y: 20 },
	{ x: 20, y: 30 },
	{ x: 30, y: 40 },
	{ x: 40, y: 50 },
	{ x: 50, y: 60 }
];

points.findIndex( function matcher(point) {
	return (
		point.x % 3 == 0 &&
		point.y % 4 == 0
	);
} );								// 2

points.findIndex( function matcher(point) {
	return (
		point.x % 6 == 0 &&
		point.y % 7 == 0
	);
} );								// -1
```

No uses `findIndex(..) != -1` (de la manera que siempre se ha hecho con `indexOf(..)`) para obtener un booleano de la búsqueda, porque `some(..)` ya produce el `true`/`false` que quieres. Y no hagas `a[ a.findIndex(..) ]` para obtener el valor coincidente, porque para eso es `find(..)`. Y finalmente, usa `indexOf(..)` si necesitas el índice de una coincidencia estricta, o `findIndex(..)` si necesitas el índice de una coincidencia más personalizada.

**Nota:** Como con otros métodos de array que toman callbacks, `findIndex(..)` toma un segundo argumento opcional que si se establece especificará el enlace `this` para el callback pasado como primer argumento. De lo contrario, `this` será `undefined`.

### Métodos de Prototipo `entries()`, `values()`, `keys()`

En el Capítulo 3, ilustramos cómo las estructuras de datos pueden proporcionar una enumeración con patrón elemento por elemento de sus valores, a través de un iterador. Luego expandimos este enfoque en el Capítulo 5, mientras explorábamos cómo las nuevas colecciones de ES6 (Map, Set, etc.) proporcionan varios métodos para producir diferentes tipos de iteraciones.

Porque no es nuevo en ES6, `Array` podría no pensarse tradicionalmente como una "colección", pero lo es en el sentido de que proporciona estos mismos métodos de iterador: `entries()`, `values()`, y `keys()`. Considera:

```js
var a = [1,2,3];

[...a.values()];					// [1,2,3]
[...a.keys()];						// [0,1,2]
[...a.entries()];					// [ [0,1], [1,2], [2,3] ]

[...a[Symbol.iterator]()];			// [1,2,3]
```

Igual que con `Set`, el iterador por defecto de `Array` es el mismo que lo que `values()` devuelve.

En "Evitando Slots Vacíos" anteriormente en este capítulo, ilustramos cómo `Array.from(..)` trata los slots vacíos en un array como simplemente ser slots presentes con `undefined` en ellos. Eso es en realidad porque bajo las cubiertas, los iteradores de array se comportan de esa manera:

```js
var a = [];
a.length = 3;
a[1] = 2;

[...a.values()];		// [undefined,2,undefined]
[...a.keys()];			// [0,1,2]
[...a.entries()];		// [ [0,undefined], [1,2], [2,undefined] ]
```

## `Object`

Algunos ayudantes estáticos adicionales se han añadido a `Object`. Tradicionalmente, funciones de este tipo se han visto como enfocadas en los comportamientos/capacidades de valores de objeto.

Sin embargo, a partir de ES6, las funciones estáticas de `Object` también serán para APIs globales de propósito general de cualquier tipo que no pertenezcan más naturalmente en alguna otra ubicación (es decir, `Array.from(..)`).

### Función Estática `Object.is(..)`

La función estática `Object.is(..)` hace comparaciones de valor de una manera incluso más estricta que la comparación `===`.

`Object.is(..)` invoca el algoritmo subyacente `SameValue` (spec ES6, sección 7.2.9). El algoritmo `SameValue` es básicamente igual que el algoritmo de Comparación de Igualdad Estricta `===` (spec ES6, sección 7.2.13), con dos excepciones importantes.

Considera:

```js
var x = NaN, y = 0, z = -0;

x === x;							// false
y === z;							// true

Object.is( x, x );					// true
Object.is( y, z );					// false
```

Deberías continuar usando `===` para comparaciones de igualdad estricta; `Object.is(..)` no debería pensarse como un reemplazo del operador. Sin embargo, en casos donde estés tratando de identificar estrictamente un valor `NaN` o `-0`, `Object.is(..)` es ahora la opción preferida.

**Nota:** ES6 también añade una utilidad `Number.isNaN(..)` (discutida más adelante en este capítulo) que puede ser una prueba ligeramente más conveniente; podrías preferir `Number.isNaN(x)` sobre `Object.is(x,NaN)`. *Puedes* probar con precisión para `-0` con un torpe `x == 0 && 1 / x === -Infinity`, pero en este caso `Object.is(x,-0)` es mucho mejor.

### Función Estática `Object.getOwnPropertySymbols(..)`

La sección "Symbols" en el Capítulo 2 discute el nuevo tipo de valor primitivo Symbol en ES6.

Los Symbols probablemente van a ser usados mayormente como propiedades especiales (meta) en objetos. Así que la utilidad `Object.getOwnPropertySymbols(..)` fue introducida, que recupera solo las propiedades symbol directamente en un objeto:

```js
var o = {
	foo: 42,
	[ Symbol( "bar" ) ]: "hello world",
	baz: true
};

Object.getOwnPropertySymbols( o );	// [ Symbol(bar) ]
```

### Función Estática `Object.setPrototypeOf(..)`

También en el Capítulo 2, mencionamos la utilidad `Object.setPrototypeOf(..)`, que (sin sorprender) establece el `[[Prototype]]` de un objeto para los propósitos de *delegación de comportamiento* (ver el título *this & Object Prototypes* de esta serie). Considera:

```js
var o1 = {
	foo() { console.log( "foo" ); }
};
var o2 = {
	// .. definición de o2 ..
};

Object.setPrototypeOf( o2, o1 );

// delega a `o1.foo()`
o2.foo();							// foo
```

Alternativamente:

```js
var o1 = {
	foo() { console.log( "foo" ); }
};

var o2 = Object.setPrototypeOf( {
	// .. definición de o2 ..
}, o1 );

// delega a `o1.foo()`
o2.foo();							// foo
```

En ambos fragmentos anteriores, la relación entre `o2` y `o1` aparece al final de la definición de `o2`. Más comúnmente, la relación entre un `o2` y `o1` se especifica al tope de la definición de `o2`, como lo es con clases, y también con `__proto__` en literales de objeto (ver "Estableciendo `[[Prototype]]`" en el Capítulo 2).

**Advertencia:** Establecer un `[[Prototype]]` justo después de la creación del objeto es razonable, como se muestra. Pero cambiarlo mucho después generalmente no es una buena idea y usualmente llevará a más confusión que claridad.

### Función Estática `Object.assign(..)`

Muchas bibliotecas/frameworks de JavaScript proporcionan utilidades para copiar/mezclar las propiedades de un objeto en otro (por ejemplo, `extend(..)` de jQuery). Hay varias diferencias matizadas entre estas diferentes utilidades, como si una propiedad con valor `undefined` se ignora o no.

ES6 añade `Object.assign(..)`, que es una versión simplificada de estos algoritmos. El primer argumento es el *objetivo*, y cualquier otro argumento pasado son las *fuentes*, que serán procesadas en el orden listado. Para cada fuente, sus claves enumerables y propias (es decir, no "heredadas"), incluyendo symbols, se copian como si fuera por asignación simple `=`. `Object.assign(..)` devuelve el objeto objetivo.

Considera esta configuración de objeto:

```js
var target = {},
	o1 = { a: 1 }, o2 = { b: 2 },
	o3 = { c: 3 }, o4 = { d: 4 };

// configurar propiedad de solo lectura
Object.defineProperty( o3, "e", {
	value: 5,
	enumerable: true,
	writable: false,
	configurable: false
} );

// configurar propiedad no-enumerable
Object.defineProperty( o3, "f", {
	value: 6,
	enumerable: false
} );

o3[ Symbol( "g" ) ] = 7;

// configurar symbol no-enumerable
Object.defineProperty( o3, Symbol( "h" ), {
	value: 8,
	enumerable: false
} );

Object.setPrototypeOf( o3, o4 );
```

Solo las propiedades `a`, `b`, `c`, `e` y `Symbol("g")` serán copiadas a `target`:

```js
Object.assign( target, o1, o2, o3 );

target.a;							// 1
target.b;							// 2
target.c;							// 3

Object.getOwnPropertyDescriptor( target, "e" );
// { value: 5, writable: true, enumerable: true,
//   configurable: true }

Object.getOwnPropertySymbols( target );
// [Symbol("g")]
```

Las propiedades `d`, `f` y `Symbol("h")` se omiten de la copia; las propiedades no-enumerables y propiedades no-propias son todas excluidas de la asignación. También, `e` se copia como una asignación de propiedad normal, no duplicado como propiedad de solo lectura.

En una sección anterior, mostramos usar `setPrototypeOf(..)` para establecer una relación `[[Prototype]]` entre un objeto `o2` y `o1`. Hay otra forma que aprovecha `Object.assign(..)`:

```js
var o1 = {
	foo() { console.log( "foo" ); }
};

var o2 = Object.assign(
	Object.create( o1 ),
	{
		// .. definición de o2 ..
	}
);

// delega a `o1.foo()`
o2.foo();							// foo
```

**Nota:** `Object.create(..)` es la utilidad estándar de ES5 que crea un objeto vacío que está enlazado vía `[[Prototype]]`. Ver el título *this & Object Prototypes* de esta serie para más información.

## `Math`

ES6 añade varias utilidades matemáticas nuevas que rellenan huecos o ayudan con operaciones comunes. Todas estas pueden calcularse manualmente, pero la mayoría ahora están definidas nativamente para que en algunos casos el motor JS pueda realizar los cálculos más óptimamente, o realizarlos con mejor precisión decimal que sus contrapartes manuales.

Es probable que código asm.js/JS transpilado (ver el título *Async & Performance* de esta serie) sea el consumidor más probable de muchas de estas utilidades en lugar de desarrolladores directos.

Trigonometría:

* `cosh(..)` - Coseno hiperbólico
* `acosh(..)` - Arcocoseno hiperbólico
* `sinh(..)` - Seno hiperbólico
* `asinh(..)` - Arcoseno hiperbólico
* `tanh(..)` - Tangente hiperbólica
* `atanh(..)` - Arcotangente hiperbólica
* `hypot(..)` - La raíz cuadrada de la suma de los cuadrados (es decir, el teorema de Pitágoras generalizado)

Aritmética:

* `cbrt(..)` - Raíz cúbica
* `clz32(..)` - Contar ceros iniciales en representación binaria de 32 bits
* `expm1(..)` - Lo mismo que `exp(x) - 1`
* `log2(..)` - Logaritmo binario (log base 2)
* `log10(..)` - Log base 10
* `log1p(..)` - Lo mismo que `log(x + 1)`
* `imul(..)` - Multiplicación de enteros de 32 bits de dos números

Meta:

* `sign(..)` - Devuelve el signo del número
* `trunc(..)` - Devuelve solo la parte entera de un número
* `fround(..)` - Redondea al valor de punto flotante más cercano de 32 bits (precisión simple)

## `Number`

Importantemente, para que tu programa funcione adecuadamente, debe manejar números con precisión. ES6 añade algunas propiedades y funciones adicionales para asistir con operaciones numéricas comunes.

Dos adiciones a `Number` son simplemente referencias a los globales preexistentes: `Number.parseInt(..)` y `Number.parseFloat(..)`.

### Propiedades Estáticas

ES6 añade algunas constantes numéricas útiles como propiedades estáticas:

* `Number.EPSILON` - El valor mínimo entre dos números cualesquiera: `2^-52` (ver el Capítulo 2 del título *Types & Grammar* de esta serie sobre usar este valor como tolerancia para imprecisión en aritmética de punto flotante)
* `Number.MAX_SAFE_INTEGER` - El entero más alto que puede representarse "seguramente" de manera inequívoca en un valor numérico de JS: `2^53 - 1`
* `Number.MIN_SAFE_INTEGER` - El entero más bajo que puede representarse "seguramente" de manera inequívoca en un valor numérico de JS: `-(2^53 - 1)` o `(-2)^53 + 1`.

**Nota:** Ver el Capítulo 2 del título *Types & Grammar* de esta serie para más información sobre enteros "seguros".

### Función Estática `Number.isNaN(..)`

La utilidad global estándar `isNaN(..)` ha estado rota desde su inicio, en que devuelve `true` para cosas que no son números, no solo para el valor real `NaN`, porque convierte el argumento al tipo número (lo que puede resultar falsamente en un NaN). ES6 añade una utilidad arreglada `Number.isNaN(..)` que funciona como debería:

```js
var a = NaN, b = "NaN", c = 42;

isNaN( a );							// true
isNaN( b );							// true -- ¡oops!
isNaN( c );							// false

Number.isNaN( a );					// true
Number.isNaN( b );					// false -- ¡arreglado!
Number.isNaN( c );					// false
```

### Función Estática `Number.isFinite(..)`

Hay una tentación de mirar un nombre de función como `isFinite(..)` y asumir que simplemente es "no infinito". Eso no es del todo correcto, sin embargo. Hay más matiz en esta nueva utilidad de ES6. Considera:

```js
var a = NaN, b = Infinity, c = 42;

Number.isFinite( a );				// false
Number.isFinite( b );				// false

Number.isFinite( c );				// true
```

La `isFinite(..)` global estándar convierte su argumento, pero `Number.isFinite(..)` omite el comportamiento de coerción:

```js
var a = "42";

isFinite( a );						// true
Number.isFinite( a );				// false
```

Podrías preferir la coerción, en cuyo caso usar la `isFinite(..)` global es una opción válida. Alternativamente, y quizás más sensatamente, puedes usar `Number.isFinite(+x)`, que explícitamente convierte `x` a un número antes de pasarlo (ver el Capítulo 4 del título *Types & Grammar* de esta serie).

### Funciones Estáticas Relacionadas con Enteros

Los valores numéricos de JavaScript siempre son punto flotante (IEEE-754). Así que la noción de determinar si un número es un "entero" no se trata de verificar su tipo, porque JS no hace tal distinción.

En su lugar, necesitas verificar si hay alguna porción decimal distinta de cero en el valor. La manera más fácil de hacer eso ha sido comúnmente:

```js
x === Math.floor( x );
```

ES6 añade una utilidad ayudante `Number.isInteger(..)` que potencialmente puede determinar esta cualidad de manera ligeramente más eficiente:

```js
Number.isInteger( 4 );				// true
Number.isInteger( 4.2 );			// false
```

**Nota:** En JavaScript, no hay diferencia entre `4`, `4.`, `4.0`, o `4.0000`. Todos estos serían considerados un "entero", y por lo tanto producirían `true` de `Number.isInteger(..)`.

Además, `Number.isInteger(..)` filtra algunos valores claramente no-enteros que `x === Math.floor(x)` podría potencialmente confundir:

```js
Number.isInteger( NaN );			// false
Number.isInteger( Infinity );		// false
```

Trabajar con "enteros" es a veces un dato importante, ya que puede simplificar ciertos tipos de algoritmos. El código JS en sí no correrá más rápido solo por filtrar solo enteros, pero hay técnicas de optimización que el motor puede tomar (por ejemplo, asm.js) cuando solo se usan enteros.

Debido al manejo de `Number.isInteger(..)` de valores `NaN` e `Infinity`, definir una utilidad `isFloat(..)` no sería tan simple como `!Number.isInteger(..)`. Necesitarías hacer algo como:

```js
function isFloat(x) {
	return Number.isFinite( x ) && !Number.isInteger( x );
}

isFloat( 4.2 );						// true
isFloat( 4 );						// false

isFloat( NaN );						// false
isFloat( Infinity );				// false
```

**Nota:** Puede parecer extraño, pero Infinity no debería considerarse ni un entero ni un flotante.

ES6 también define una utilidad `Number.isSafeInteger(..)`, que verifica que el valor sea tanto un entero como dentro del rango de `Number.MIN_SAFE_INTEGER`-`Number.MAX_SAFE_INTEGER` (inclusivo).

```js
var x = Math.pow( 2, 53 ),
	y = Math.pow( -2, 53 );

Number.isSafeInteger( x - 1 );		// true
Number.isSafeInteger( y + 1 );		// true

Number.isSafeInteger( x );			// false
Number.isSafeInteger( y );			// false
```

## `String`

Las cadenas ya tienen bastantes ayudantes antes de ES6, pero aún más se han añadido a la mezcla.

### Funciones Unicode

"Operaciones de Cadena Conscientes de Unicode" en el Capítulo 2 discute `String.fromCodePoint(..)`, `String#codePointAt(..)`, y `String#normalize(..)` en detalle. Se han añadido para mejorar el soporte de Unicode en valores de cadena JS.

```js
String.fromCodePoint( 0x1d49e );			// "𝒞"

"ab𝒞d".codePointAt( 2 ).toString( 16 );		// "1d49e"
```

El método de prototipo de cadena `normalize(..)` se usa para realizar normalizaciones Unicode que combinan caracteres con "marcas combinantes" adyacentes o descomponen caracteres combinados.

Generalmente, la normalización no creará un efecto visible en los contenidos de la cadena, pero cambiará los contenidos de la cadena, lo que puede afectar cómo se reporta la propiedad `length`, así como cómo se comporta el acceso a caracteres por posición:

```js
var s1 = "e\u0301";
s1.length;							// 2

var s2 = s1.normalize();
s2.length;							// 1
s2 === "\xE9";						// true
```

`normalize(..)` toma un argumento opcional que especifica la forma de normalización a usar. Este argumento debe ser uno de los siguientes cuatro valores: `"NFC"` (por defecto), `"NFD"`, `"NFKC"`, o `"NFKD"`.

**Nota:** Las formas de normalización y sus efectos en cadenas están más allá del alcance de lo que discutiremos aquí. Ver "Unicode Normalization Forms" (http://www.unicode.org/reports/tr15/) para más información.

### Función Estática `String.raw(..)`

La utilidad `String.raw(..)` se proporciona como una función de etiqueta integrada para usar con literales de cadena de plantilla (ver Capítulo 2) para obtener el valor de cadena sin procesar, sin ningún procesamiento de secuencias de escape.

Esta función casi nunca se llamará manualmente, sino que se usará con literales de plantilla etiquetados:

```js
var str = "bc";

String.raw`\ta${str}d\xE9`;
// "\tabcd\xE9", no "	abcdé"
```

En la cadena resultante, `\` y `t` son caracteres crudos separados, no el único carácter de secuencia de escape `\t`. Lo mismo es cierto con la secuencia de escape Unicode.

### Función de Prototipo `repeat(..)`

En lenguajes como Python y Ruby, puedes repetir una cadena así:

```js
"foo" * 3;							// "foofoofoo"
```

Eso no funciona en JS, porque la multiplicación `*` solo está definida para números, y por lo tanto `"foo"` se convierte al número `NaN`.

Sin embargo, ES6 define un método de prototipo de cadena `repeat(..)` para lograr la tarea:

```js
"foo".repeat( 3 );					// "foofoofoo"
```

### Funciones de Inspección de Cadenas

Además de `String#indexOf(..)` y `String#lastIndexOf(..)` de antes de ES6, tres nuevos métodos para búsqueda/inspección se han añadido: `startsWith(..)`, `endsWith(..)`, e `includes(..)`.

```js
var palindrome = "step on no pets";

palindrome.startsWith( "step on" );	// true
palindrome.startsWith( "on", 5 );	// true

palindrome.endsWith( "no pets" );	// true
palindrome.endsWith( "no", 10 );	// true

palindrome.includes( "on" );		// true
palindrome.includes( "on", 6 );		// false
```

Para todos los métodos de búsqueda/inspección de cadenas, si buscas una cadena vacía `""`, se encontrará ya sea al principio o al final de la cadena.

**Advertencia:** Estos métodos no aceptarán por defecto una expresión regular para la cadena de búsqueda. Ver "Símbolos de Expresión Regular" en el Capítulo 7 para información sobre desactivar la verificación `isRegExp` que se realiza en este primer argumento.
