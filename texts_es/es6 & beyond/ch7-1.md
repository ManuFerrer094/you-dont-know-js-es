# You Don't Know JS: ES6 & Más Allá
# Capítulo 7: Meta Programación

La meta programación es programar donde la operación apunta al comportamiento del programa en sí. En otras palabras, es programar la programación de tu programa. Sí, un trabalenguas, ¿verdad?

Por ejemplo, si investigas la relación entre un objeto `a` y otro `b` -- ¿están vinculados por `[[Prototype]]`? -- usando `a.isPrototypeOf(b)`, esto se conoce comúnmente como introspección, una forma de meta programación. Las macros (que no existen en JS, aún) -- donde el código se modifica a sí mismo en tiempo de compilación -- son otro ejemplo obvio de meta programación. Enumerar las claves de un objeto con un bucle `for..in`, o verificar si un objeto es una *instancia de* un "constructor de clase", son otras tareas comunes de meta programación.

La meta programación se enfoca en una o más de las siguientes: código que se inspecciona a sí mismo, código que se modifica a sí mismo, o código que modifica el comportamiento predeterminado del lenguaje para que otro código se vea afectado.

El objetivo de la meta programación es aprovechar las capacidades intrínsecas del lenguaje para hacer que el resto de tu código sea más descriptivo, expresivo y/o flexible. Debido a la naturaleza *meta* de la meta programación, es algo difícil darle una definición más precisa que esa. La mejor manera de entender la meta programación es verla a través de ejemplos.

ES6 agrega varias nuevas formas/características para la meta programación sobre lo que JS ya tenía.

## Nombres de Funciones

Hay casos donde tu código puede querer introspectarse a sí mismo y preguntar cuál es el nombre de alguna función. Si preguntas cuál es el nombre de una función, la respuesta es sorprendentemente algo ambigua. Considera:

```js
function daz() {
	// ..
}

var obj = {
	foo: function() {
		// ..
	},
	bar: function baz() {
		// ..
	},
	bam: daz,
	zim() {
		// ..
	}
};
```

En el fragmento anterior, "¿cuál es el nombre de `obj.foo()`?" es ligeramente matizado. ¿Es `"foo"`, `""`, o `undefined`? ¿Y qué hay de `obj.bar()` -- se llama `"bar"` o `"baz"`? ¿`obj.bam()` se llama `"bam"` o `"daz"`? ¿Qué hay de `obj.zim()`?

Además, ¿qué pasa con funciones que se pasan como callbacks, como:

```js
function foo(cb) {
	// ¿cuál es el nombre de `cb()` aquí?
}

foo( function(){
	// ¡Soy anónima!
} );
```

Hay bastantes formas en que las funciones pueden expresarse en programas, y no siempre es claro e inequívoco cuál debería ser el "nombre" de esa función.

Más importante aún, necesitamos distinguir si el "nombre" de una función se refiere a su propiedad `name` -- sí, las funciones tienen una propiedad llamada `name` -- o si se refiere al nombre de enlace léxico, como `bar` en `function bar() { .. }`.

El nombre de enlace léxico es lo que usas para cosas como recursión:

```js
function foo(i) {
	if (i < 10) return foo( i * 2 );
	return i;
}
```

La propiedad `name` es lo que usarías para propósitos de meta programación, así que eso es en lo que nos enfocaremos en esta discusión.

La confusión viene porque por defecto, el nombre léxico que una función tiene (si lo tiene) también se establece como su propiedad `name`. En realidad no había un requisito oficial para ese comportamiento por las especificaciones de ES5 (y anteriores). La asignación de la propiedad `name` no era estándar pero aún así bastante confiable. A partir de ES6, se ha estandarizado.

**Consejo:** Si una función tiene un valor `name` asignado, ese es típicamente el nombre usado en los stack traces en las herramientas de desarrollo.

### Inferencias

Pero ¿qué pasa con la propiedad `name` si una función no tiene un nombre léxico?

A partir de ES6, ahora hay reglas de inferencia que pueden determinar un valor sensato de propiedad `name` para asignar a una función incluso si esa función no tiene un nombre léxico para usar.

Considera:

```js
var abc = function() {
	// ..
};

abc.name;				// "abc"
```

Si le hubiéramos dado a la función un nombre léxico como `abc = function def() { .. }`, la propiedad `name` sería por supuesto `"def"`. Pero en ausencia del nombre léxico, intuitivamente el nombre `"abc"` parece apropiado.

Aquí hay otras formas que inferirán un nombre (o no) en ES6:

```js
(function(){ .. });					// name:
(function*(){ .. });				// name:
window.foo = function(){ .. };		// name:

class Awesome {
	constructor() { .. }			// name: Awesome
	funny() { .. }					// name: funny
}

var c = class Awesome { .. };		// name: Awesome

var o = {
	foo() { .. },					// name: foo
	*bar() { .. },					// name: bar
	baz: () => { .. },				// name: baz
	bam: function(){ .. },			// name: bam
	get qux() { .. },				// name: get qux
	set fuz() { .. },				// name: set fuz
	["b" + "iz"]:
		function(){ .. },			// name: biz
	[Symbol( "buz" )]:
		function(){ .. }			// name: [buz]
};

var x = o.foo.bind( o );			// name: bound foo
(function(){ .. }).bind( o );		// name: bound

export default function() { .. }	// name: default

var y = new Function();				// name: anonymous
var GeneratorFunction =
	function*(){}.__proto__.constructor;
var z = new GeneratorFunction();	// name: anonymous
```

La propiedad `name` no es escribible por defecto, pero es configurable, lo que significa que puedes usar `Object.defineProperty(..)` para cambiarla manualmente si lo deseas.

## Meta Propiedades

En la sección "`new.target`" del Capítulo 3, introdujimos un concepto nuevo en JS en ES6: la meta propiedad. Como el nombre sugiere, las meta propiedades están destinadas a proporcionar meta información especial en forma de acceso a propiedad que de otra manera no habría sido posible.

En el caso de `new.target`, la palabra clave `new` sirve como el contexto para un acceso a propiedad. Claramente `new` en sí mismo no es un objeto, lo que hace que esta capacidad sea especial. Sin embargo, cuando `new.target` se usa dentro de una llamada de constructor (una función/método invocado con `new`), `new` se convierte en un contexto virtual, para que `new.target` pueda referirse al constructor objetivo que `new` invocó.

Este es un ejemplo claro de una operación de meta programación, ya que la intención es determinar desde dentro de una llamada de constructor cuál fue el objetivo `new` original, generalmente para propósitos de introspección (examinando tipado/estructura) o acceso a propiedades estáticas.

Por ejemplo, es posible que desees tener un comportamiento diferente en un constructor dependiendo de si se invoca directamente o se invoca a través de una clase hija:

```js
class Parent {
	constructor() {
		if (new.target === Parent) {
			console.log( "Parent instantiated" );
		}
		else {
			console.log( "A child instantiated" );
		}
	}
}

class Child extends Parent {}

var a = new Parent();
// Parent instantiated

var b = new Child();
// A child instantiated
```

Hay una ligera sutileza aquí, que es que el `constructor()` dentro de la definición de la clase `Parent` en realidad recibe el nombre léxico de la clase (`Parent`), aunque la sintaxis implica que la clase es una entidad separada del constructor.

**Advertencia:** Como con todas las técnicas de meta programación, ten cuidado de crear código que sea demasiado inteligente para que tu yo futuro u otros que mantengan tu código lo entiendan. Usa estos trucos con precaución.

## Símbolos Bien Conocidos

En la sección "Symbols" del Capítulo 2, cubrimos el nuevo tipo primitivo `symbol` de ES6. Además de los símbolos que puedes definir en tu propio programa, JS predefine una serie de símbolos integrados, referidos como *Símbolos Bien Conocidos* (WKS).

Estos valores de símbolo se definen principalmente para exponer meta propiedades especiales que se están exponiendo a tus programas JS para darte más control sobre el comportamiento de JS.

Los introduciremos brevemente y discutiremos su propósito.

### `Symbol.iterator`

En los Capítulos 2 y 3, introdujimos y usamos el símbolo `@@iterator`, usado automáticamente por los spreads `...` y los bucles `for..of`. También vimos `@@iterator` definido en las nuevas colecciones de ES6 como se define en el Capítulo 5.

`Symbol.iterator` representa la ubicación especial (propiedad) en cualquier objeto donde los mecanismos del lenguaje buscan automáticamente un método que construirá una instancia de iterador para consumir los valores de ese objeto. Muchos objetos vienen con uno predeterminado definido.

Sin embargo, podemos definir nuestra propia lógica de iterador para cualquier valor de objeto estableciendo la propiedad `Symbol.iterator`, incluso si eso está anulando el iterador predeterminado. El aspecto de meta programación es que estamos definiendo comportamiento que otras partes de JS (es decir, operadores y construcciones de bucle) usan cuando procesan un valor de objeto que definimos.

Considera:

```js
var arr = [4,5,6,7,8,9];

for (var v of arr) {
	console.log( v );
}
// 4 5 6 7 8 9

// define iterador que solo produce valores
// de índices impares
arr[Symbol.iterator] = function*() {
	var idx = 1;
	do {
		yield this[idx];
	} while ((idx += 2) < this.length);
};

for (var v of arr) {
	console.log( v );
}
// 5 7 9
```

### `Symbol.toStringTag` y `Symbol.hasInstance`

Una de las tareas más comunes de meta programación es introspeccionar un valor para descubrir de qué *tipo* es, usualmente para decidir qué operaciones son apropiadas para realizar sobre él. Con objetos, las dos técnicas de inspección más comunes son `toString()` e `instanceof`.

Considera:

```js
function Foo() {}

var a = new Foo();

a.toString();				// [object Object]
a instanceof Foo;			// true
```

A partir de ES6, puedes controlar el comportamiento de estas operaciones:

```js
function Foo(greeting) {
	this.greeting = greeting;
}

Foo.prototype[Symbol.toStringTag] = "Foo";

Object.defineProperty( Foo, Symbol.hasInstance, {
	value: function(inst) {
		return inst.greeting == "hello";
	}
} );

var a = new Foo( "hello" ),
	b = new Foo( "world" );

b[Symbol.toStringTag] = "cool";

a.toString();				// [object Foo]
String( b );				// [object cool]

a instanceof Foo;			// true
b instanceof Foo;			// false
```

El símbolo `@@toStringTag` en el prototipo (o en la instancia misma) especifica un valor de cadena para usar en la stringificación `[object ___]`.

El símbolo `@@hasInstance` es un método en la función constructora que recibe el valor del objeto instancia y te permite decidir retornando `true` o `false` si el valor debe considerarse una instancia o no.

**Nota:** Para establecer `@@hasInstance` en una función, debes usar `Object.defineProperty(..)`, ya que el predeterminado en `Function.prototype` es `writable: false`. Consulta el título *this & Object Prototypes* de esta serie para más información.

### `Symbol.species`

En "Clases" en el Capítulo 3, introdujimos el símbolo `@@species`, que controla qué constructor es usado por los métodos integrados de una clase que necesita generar nuevas instancias.

El ejemplo más común es cuando se crea una subclase de `Array` y se quiere definir qué constructor (`Array(..)` o tu subclase) deberían usar los métodos heredados como `slice(..)`. Por defecto, `slice(..)` llamado en una instancia de una subclase de `Array` produciría una nueva instancia de esa subclase, que es francamente lo que frecuentemente querrás.

Sin embargo, puedes meta programar anulando la definición predeterminada de `@@species` de una clase:

```js
class Cool {
	// delegar `@@species` al constructor derivado
	static get [Symbol.species]() { return this; }

	again() {
		return new this.constructor[Symbol.species]();
	}
}

class Fun extends Cool {}

class Awesome extends Cool {
	// forzar `@@species` a ser el constructor padre
	static get [Symbol.species]() { return Cool; }
}

var a = new Fun(),
	b = new Awesome(),
	c = a.again(),
	d = b.again();

c instanceof Fun;			// true
d instanceof Awesome;		// false
d instanceof Cool;			// true
```

La configuración de `Symbol.species` tiene como valor predeterminado en los constructores nativos integrados el comportamiento `return this` como se ilustra en el fragmento anterior en la definición de `Cool`. No tiene valor predeterminado en clases de usuario, pero como se muestra ese comportamiento es fácil de emular.

Si necesitas definir métodos que generan nuevas instancias, usa la meta programación del patrón `new this.constructor[Symbol.species](..)` en lugar de escribir directamente `new this.constructor(..)` o `new XYZ(..)`. Las clases derivadas entonces podrán personalizar `Symbol.species` para controlar qué constructor produce esas instancias.

### `Symbol.toPrimitive`

En el título *Types & Grammar* de esta serie, discutimos la operación de coerción abstracta `ToPrimitive`, que se usa cuando un objeto debe ser convertido a un valor primitivo para alguna operación (como la comparación `==` o la adición `+`). Antes de ES6, no había forma de controlar este comportamiento.

A partir de ES6, el símbolo `@@toPrimitive` como propiedad en cualquier valor de objeto puede personalizar esa coerción `ToPrimitive` especificando un método.

Considera:

```js
var arr = [1,2,3,4,5];

arr + 10;				// 1,2,3,4,510

arr[Symbol.toPrimitive] = function(hint) {
	if (hint == "default" || hint == "number") {
		// sumar todos los números
		return this.reduce( function(acc,curr){
			return acc + curr;
		}, 0 );
	}
};

arr + 10;				// 25
```

El método `Symbol.toPrimitive` recibirá una *pista* de `"string"`, `"number"`, o `"default"` (que debería interpretarse como `"number"`), dependiendo de qué tipo espera la operación que invoca `ToPrimitive`. En el fragmento anterior, la operación aditiva `+` no tiene pista (se pasa `"default"`). Una operación multiplicativa `*` daría la pista `"number"` y un `String(arr)` daría la pista `"string"`.

**Advertencia:** El operador `==` invocará la operación `ToPrimitive` sin pista -- el método `@@toPrimitive`, si existe, es llamado con la pista `"default"` -- en un objeto si el otro valor siendo comparado no es un objeto. Sin embargo, si ambos valores de comparación son objetos, el comportamiento de `==` es idéntico a `===`, que es que las referencias mismas se comparan directamente. En este caso, `@@toPrimitive` no es invocado en absoluto. Consulta el título *Types & Grammar* de esta serie para más información sobre coerción y las operaciones abstractas.

### Símbolos de Expresiones Regulares

Hay cuatro símbolos bien conocidos que pueden ser anulados para objetos de expresiones regulares, que controlan cómo esas expresiones regulares son usadas por las cuatro funciones correspondientes de `String.prototype` del mismo nombre:

* `@@match`: El valor `Symbol.match` de una expresión regular es el método usado para emparejar todo o parte de un valor de cadena con la expresión regular dada. Es usado por `String.prototype.match(..)` si le pasas una expresión regular para el emparejamiento de patrones.

   El algoritmo predeterminado para emparejar está establecido en la sección 21.2.5.6 de la especificación ES6 (http://www.ecma-international.org/ecma-262/6.0/#sec-regexp.prototype-@@match). Podrías anular este algoritmo predeterminado y proporcionar características extra de regex, como aserciones de look-behind.

   `Symbol.match` también es usado por la operación abstracta `isRegExp` (ver la nota en "Funciones de Inspección de Cadenas" en el Capítulo 6) para determinar si un objeto está destinado a ser usado como una expresión regular. Para forzar que esta verificación falle para un objeto para que no sea tratado como una expresión regular, establece el valor de `Symbol.match` a `false` (o algo falsy).
* `@@replace`: El valor `Symbol.replace` de una expresión regular es el método usado por `String.prototype.replace(..)` para reemplazar dentro de una cadena una o todas las ocurrencias de secuencias de caracteres que coincidan con el patrón de expresión regular dado.

   El algoritmo predeterminado para reemplazar está establecido en la sección 21.2.5.8 de la especificación ES6 (http://www.ecma-international.org/ecma-262/6.0/#sec-regexp.prototype-@@replace).

   Un uso interesante para anular el algoritmo predeterminado es proporcionar opciones adicionales de argumento `replacer`, como soportar `"abaca".replace(/a/g,[1,2,3])` produciendo `"1b2c3"` consumiendo el iterable para valores de reemplazo sucesivos.
* `@@search`: El valor `Symbol.search` de una expresión regular es el método usado por `String.prototype.search(..)` para buscar una sub-cadena dentro de otra cadena como coincidencia con la expresión regular dada.

   El algoritmo predeterminado para buscar está establecido en la sección 21.2.5.9 de la especificación ES6 (http://www.ecma-international.org/ecma-262/6.0/#sec-regexp.prototype-@@search).
* `@@split`: El valor `Symbol.split` de una expresión regular es el método usado por `String.prototype.split(..)` para dividir una cadena en sub-cadenas en la(s) ubicación(es) del delimitador como coincidencia con la expresión regular dada.

   El algoritmo predeterminado para dividir está establecido en la sección 21.2.5.11 de la especificación ES6 (http://www.ecma-international.org/ecma-262/6.0/#sec-regexp.prototype-@@split).

Anular los algoritmos integrados de expresiones regulares no es para los débiles de corazón. JS viene con un motor de expresiones regulares altamente optimizado, así que tu propio código de usuario probablemente será mucho más lento. Este tipo de meta programación es genial y poderoso, pero solo debería usarse en casos donde sea realmente necesario o beneficioso.

### `Symbol.isConcatSpreadable`

El símbolo `@@isConcatSpreadable` puede definirse como una propiedad booleana (`Symbol.isConcatSpreadable`) en cualquier objeto (como un array u otro iterable) para indicar si debería *expandirse* si se pasa a un `concat(..)` de array.

Considera:

```js
var a = [1,2,3],
	b = [4,5,6];

b[Symbol.isConcatSpreadable] = false;

[].concat( a, b );		// [1,2,3,[4,5,6]]
```

### `Symbol.unscopables`

El símbolo `@@unscopables` puede definirse como una propiedad de objeto (`Symbol.unscopables`) en cualquier objeto para indicar qué propiedades pueden y no pueden ser expuestas como variables léxicas en una sentencia `with`.

Considera:

```js
var o = { a:1, b:2, c:3 },
	a = 10, b = 20, c = 30;

o[Symbol.unscopables] = {
	a: false,
	b: true,
	c: false
};

with (o) {
	console.log( a, b, c );		// 1 20 3
}
```

Un `true` en el objeto `@@unscopables` indica que la propiedad debería ser *no-scopable*, y por lo tanto filtrada de las variables de ámbito léxico. `false` significa que está bien incluirla en las variables de ámbito léxico.

**Advertencia:** La sentencia `with` está completamente prohibida en modo `strict`, y como tal debería considerarse obsoleta del lenguaje. No la uses. Consulta el título *Scope & Closures* de esta serie para más información. Debido a que `with` debería evitarse, el símbolo `@@unscopables` también es discutible.

## Proxies

Una de las características de meta programación más obviamente agregadas a ES6 es la funcionalidad de `Proxy`.

Un proxy es un tipo especial de objeto que creas que "envuelve" -- o se sienta frente a -- otro objeto normal. Puedes registrar manejadores especiales (también conocidos como *trampas*) en el objeto proxy que son llamados cuando varias operaciones se realizan contra el proxy. Estos manejadores tienen la oportunidad de realizar lógica extra además de *reenviar* las operaciones al objeto objetivo/envuelto original.

Un ejemplo del tipo de manejador de *trampa* que puedes definir en un proxy es `get` que intercepta la operación `[[Get]]` -- realizada cuando intentas acceder a una propiedad en un objeto. Considera:

```js
var obj = { a: 1 },
	handlers = {
		get(target,key,context) {
			// nota: target === obj,
			// context === pobj
			console.log( "accessing: ", key );
			return Reflect.get(
				target, key, context
			);
		}
	},
	pobj = new Proxy( obj, handlers );

obj.a;
// 1

pobj.a;
// accessing: a
// 1
```

Declaramos un manejador `get(..)` como un método nombrado en el objeto *handler* (segundo argumento de `Proxy(..)`), que recibe una referencia al objeto *objetivo* (`obj`), el nombre de la propiedad *clave* (`"a"`), y el `self`/receptor/proxy (`pobj`).

Después de la sentencia de rastreo `console.log(..)`, "reenviamos" la operación a `obj` vía `Reflect.get(..)`. Cubriremos la API `Reflect` en la siguiente sección, pero nota que cada trampa de proxy disponible tiene una función `Reflect` correspondiente del mismo nombre.

Estos mapeos son simétricos a propósito. Los manejadores de proxy cada uno interceptan cuando una tarea respectiva de meta programación se realiza, y las utilidades `Reflect` cada una realizan la tarea respectiva de meta programación en un objeto. Casi con certeza usarás tanto `Proxy` como `Reflect` en conjunto.

Aquí hay una lista de manejadores que puedes definir en un proxy para un objeto/función *objetivo*, y cómo/cuándo se activan:

* `get(..)`: vía `[[Get]]`, se accede a una propiedad en el proxy (`Reflect.get(..)`, operador de propiedad `.`, u operador de propiedad `[ .. ]`)
* `set(..)`: vía `[[Set]]`, se establece un valor de propiedad en el proxy (`Reflect.set(..)`, el operador de asignación `=`, o asignación por desestructuración si apunta a una propiedad de objeto)
* `deleteProperty(..)`: vía `[[Delete]]`, se elimina una propiedad del proxy (`Reflect.deleteProperty(..)` o `delete`)
* `apply(..)` (si el *objetivo* es una función): vía `[[Call]]`, el proxy se invoca como una función/método normal (`Reflect.apply(..)`, `call(..)`, `apply(..)`, o el operador de llamada `(..)`)
* `construct(..)` (si el *objetivo* es una función constructora): vía `[[Construct]]`, el proxy se invoca como una función constructora (`Reflect.construct(..)` o `new`)
* `getOwnPropertyDescriptor(..)`: vía `[[GetOwnProperty]]`, se recupera un descriptor de propiedad del proxy (`Object.getOwnPropertyDescriptor(..)` o `Reflect.getOwnPropertyDescriptor(..)`)
* `defineProperty(..)`: vía `[[DefineOwnProperty]]`, se establece un descriptor de propiedad en el proxy (`Object.defineProperty(..)` o `Reflect.defineProperty(..)`)
* `getPrototypeOf(..)`: vía `[[GetPrototypeOf]]`, se recupera el `[[Prototype]]` del proxy (`Object.getPrototypeOf(..)`, `Reflect.getPrototypeOf(..)`, `__proto__`, `Object#isPrototypeOf(..)`, o `instanceof`)
* `setPrototypeOf(..)`: vía `[[SetPrototypeOf]]`, se establece el `[[Prototype]]` del proxy (`Object.setPrototypeOf(..)`, `Reflect.setPrototypeOf(..)`, o `__proto__`)
* `preventExtensions(..)`: vía `[[PreventExtensions]]`, el proxy se hace no-extensible (`Object.preventExtensions(..)` o `Reflect.preventExtensions(..)`)
* `isExtensible(..)`: vía `[[IsExtensible]]`, se sondea la extensibilidad del proxy (`Object.isExtensible(..)` o `Reflect.isExtensible(..)`)
* `ownKeys(..)`: vía `[[OwnPropertyKeys]]`, se recupera el conjunto de propiedades propias y/o propiedades de símbolo propias del proxy (`Object.keys(..)`, `Object.getOwnPropertyNames(..)`, `Object.getOwnSymbolProperties(..)`, `Reflect.ownKeys(..)`, o `JSON.stringify(..)`)
* `enumerate(..)`: vía `[[Enumerate]]`, se solicita un iterador para las propiedades enumerables propias y "heredadas" del proxy (`Reflect.enumerate(..)` o `for..in`)
* `has(..)`: vía `[[HasProperty]]`, se sondea el proxy para ver si tiene una propiedad propia o "heredada" (`Reflect.has(..)`, `Object#hasOwnProperty(..)`, o `"prop" in obj`)

**Consejo:** Para más información sobre cada una de estas tareas de meta programación, consulta la sección "API `Reflect`" más adelante en este capítulo.

Además de las notaciones en la lista anterior sobre acciones que activarán las varias trampas, algunas trampas son activadas indirectamente por la acción predeterminada de otra trampa. Por ejemplo:

```js
var handlers = {
		getOwnPropertyDescriptor(target,prop) {
			console.log(
				"getOwnPropertyDescriptor"
			);
			return Object.getOwnPropertyDescriptor(
				target, prop
			);
		},
		defineProperty(target,prop,desc){
			console.log( "defineProperty" );
			return Object.defineProperty(
				target, prop, desc
			);
		}
	},
	proxy = new Proxy( {}, handlers );

proxy.a = 2;
// getOwnPropertyDescriptor
// defineProperty
```

Los manejadores `getOwnPropertyDescriptor(..)` y `defineProperty(..)` son activados por los pasos del manejador predeterminado `set(..)` al establecer un valor de propiedad (ya sea añadiendo uno nuevo o actualizando). Si también defines tu propio manejador `set(..)`, puedes o no hacer las llamadas correspondientes contra `context` (no `target`!) lo cual activaría estas trampas del proxy.

### Limitaciones de Proxy

Estos manejadores de meta programación atrapan un amplio rango de operaciones fundamentales que puedes realizar contra un objeto. Sin embargo, hay algunas operaciones que no son (aún, al menos) disponibles para interceptar.

Por ejemplo, ninguna de estas operaciones son atrapadas y reenviadas desde el proxy `pobj` al objetivo `obj`:

```js
var obj = { a:1, b:2 },
	handlers = { .. },
	pobj = new Proxy( obj, handlers );

typeof obj;
String( obj );
obj + "";
obj == pobj;
obj === pobj
```

Quizás en el futuro, más de estas operaciones fundamentales subyacentes en el lenguaje serán interceptables, dándonos aún más poder para extender JavaScript desde dentro de sí mismo.

**Advertencia:** Hay ciertos *invariantes* -- comportamientos que no pueden ser anulados -- que se aplican al uso de manejadores de proxy. Por ejemplo, el resultado del manejador `isExtensible(..)` siempre se convierte a un `boolean`. Estos invariantes restringen algo de tu capacidad para personalizar comportamientos con proxies, pero lo hacen solo para prevenir que crees comportamiento extraño e inusual (o inconsistente). Las condiciones para estos invariantes son complicadas así que no entraremos completamente en ellas aquí, pero esta publicación (http://www.2ality.com/2014/12/es6-proxies.html#invariants) hace un gran trabajo cubriéndolas.

### Proxies Revocables

Un proxy regular siempre atrapa para el objeto objetivo, y no puede ser modificado después de la creación -- siempre que se mantenga una referencia al proxy, el proxying sigue siendo posible. Sin embargo, puede haber casos donde quieras crear un proxy que pueda ser deshabilitado cuando quieras dejar de permitir que haga proxy. La solución es crear un *proxy revocable*:

```js
var obj = { a: 1 },
	handlers = {
		get(target,key,context) {
			// nota: target === obj,
			// context === pobj
			console.log( "accessing: ", key );
			return target[key];
		}
	},
	{ proxy: pobj, revoke: prevoke } =
		Proxy.revocable( obj, handlers );

pobj.a;
// accessing: a
// 1

// después:
prevoke();

pobj.a;
// TypeError
```

Un proxy revocable se crea con `Proxy.revocable(..)`, que es una función regular, no un constructor como `Proxy(..)`. De lo contrario, toma los mismos dos argumentos: *objetivo* y *manejadores*.

El valor de retorno de `Proxy.revocable(..)` no es el proxy en sí como con `new Proxy(..)`. En su lugar, es un objeto con dos propiedades: *proxy* y *revoke* -- usamos desestructuración de objetos (ver "Desestructuración" en el Capítulo 2) para asignar estas propiedades a las variables `pobj` y `prevoke()`, respectivamente.

Una vez que un proxy revocable es revocado, cualquier intento de accederlo (activar cualquiera de sus trampas) lanzará un `TypeError`.

Un ejemplo de usar un proxy revocable podría ser entregar un proxy a otra parte en tu aplicación que gestiona datos en tu modelo, en lugar de darles una referencia al objeto del modelo real mismo. Si tu objeto del modelo cambia o es reemplazado, quieres invalidar el proxy que entregaste para que la otra parte sepa (¡a través de los errores!) que debe solicitar una referencia actualizada al modelo.

### Usando Proxies

Los beneficios de meta programación de estos manejadores de Proxy deberían ser obvios. Podemos interceptar casi completamente (y por lo tanto anular) el comportamiento de los objetos, lo que significa que podemos extender el comportamiento de objetos más allá del JS base de formas muy poderosas. Veremos algunos patrones de ejemplo para explorar las posibilidades.

#### Proxy Primero, Proxy Último

Como mencionamos antes, típicamente piensas en un proxy como "envolviendo" el objeto objetivo. En ese sentido, el proxy se convierte en el objeto principal con el que el código interactúa, y el objeto objetivo real permanece oculto/protegido.

Podrías hacer esto porque quieres pasar el objeto a algún lugar que no puede ser completamente "confiable", y así necesitas aplicar reglas especiales alrededor de su acceso en lugar de pasar el objeto mismo.

Considera:

```js
var messages = [],
	handlers = {
		get(target,key) {
			// ¿valor de cadena?
			if (typeof target[key] == "string") {
				// filtrar puntuación
				return target[key]
					.replace( /[^\w]/g, "" );
			}

			// pasar todo lo demás
			return target[key];
		},
		set(target,key,val) {
			// solo establecer cadenas únicas, en minúsculas
			if (typeof val == "string") {
				val = val.toLowerCase();
				if (target.indexOf( val ) == -1) {
					target.push(val);
				}
			}
			return true;
		}
	},
	messages_proxy =
		new Proxy( messages, handlers );

// en otro lugar:
messages_proxy.push(
	"heLLo...", 42, "wOrlD!!", "WoRld!!"
);

messages_proxy.forEach( function(val){
	console.log(val);
} );
// hello world

messages.forEach( function(val){
	console.log(val);
} );
// hello... world!!
```

Llamo a esto diseño *proxy primero*, ya que interactuamos primero (principalmente, enteramente) con el proxy.

Aplicamos algunas reglas especiales al interactuar con `messages_proxy` que no se aplican para `messages` en sí. Solo añadimos elementos si el valor es una cadena y también es único; también convertimos el valor a minúsculas. Cuando recuperamos valores de `messages_proxy`, filtramos cualquier puntuación en las cadenas.

Alternativamente, podemos invertir completamente este patrón, donde el objetivo interactúa con el proxy en lugar de que el proxy interactúe con el objetivo. Así, el código realmente solo interactúa con el objeto principal. La forma más fácil de lograr este fallback es tener el objeto proxy en la cadena `[[Prototype]]` del objeto principal.

Considera:

```js
var handlers = {
		get(target,key,context) {
			return function() {
				context.speak(key + "!");
			};
		}
	},
	catchall = new Proxy( {}, handlers ),
	greeter = {
		speak(who = "someone") {
			console.log( "hello", who );
		}
	};

// configurar `greeter` para recurrir a `catchall`
Object.setPrototypeOf( greeter, catchall );

greeter.speak();				// hello someone
greeter.speak( "world" );		// hello world

greeter.everyone();				// hello everyone!
```

Interactuamos directamente con `greeter` en lugar de `catchall`. Cuando llamamos a `speak(..)`, se encuentra en `greeter` y se usa directamente. Pero cuando intentamos acceder a un método como `everyone()`, esa función no existe en `greeter`.

El comportamiento predeterminado de propiedades de objeto es verificar hacia arriba en la cadena `[[Prototype]]` (consulta el título *this & Object Prototypes* de esta serie), así que se consulta `catchall` para una propiedad `everyone`. El manejador `get()` del proxy entonces se activa y devuelve una función que llama a `speak(..)` con el nombre de la propiedad que se está accediendo (`"everyone"`).

Llamo a este patrón *proxy último*, ya que el proxy se usa solo como último recurso.

#### "No Existe Esa Propiedad/Método"

Una queja común sobre JS es que los objetos no son por defecto muy defensivos en la situación donde intentas acceder o establecer una propiedad que no existe ya. Podrías desear predefinir todas las propiedades/métodos para un objeto, y que se lance un error si un nombre de propiedad inexistente se usa posteriormente.

Podemos lograr esto con un proxy, ya sea en diseño *proxy primero* o *proxy último*. Consideremos ambos.

```js
var obj = {
		a: 1,
		foo() {
			console.log( "a:", this.a );
		}
	},
	handlers = {
		get(target,key,context) {
			if (Reflect.has( target, key )) {
				return Reflect.get(
					target, key, context
				);
			}
			else {
				throw "No such property/method!";
			}
		},
		set(target,key,val,context) {
			if (Reflect.has( target, key )) {
				return Reflect.set(
					target, key, val, context
				);
			}
			else {
				throw "No such property/method!";
			}
		}
	},
	pobj = new Proxy( obj, handlers );

pobj.a = 3;
pobj.foo();			// a: 3

pobj.b = 4;			// Error: No such property/method!
pobj.bar();			// Error: No such property/method!
```

Para ambos `get(..)` y `set(..)`, solo reenviamos la operación si la propiedad del objeto objetivo ya existe; de lo contrario se lanza un error. El objeto proxy (`pobj`) es el objeto principal con el que el código debería interactuar, ya que intercepta estas acciones para proporcionar las protecciones.

Ahora, consideremos invertir con diseño *proxy último*:

```js
var handlers = {
		get() {
			throw "No such property/method!";
		},
		set() {
			throw "No such property/method!";
		}
	},
	pobj = new Proxy( {}, handlers ),
	obj = {
		a: 1,
		foo() {
			console.log( "a:", this.a );
		}
	};

// configurar `obj` para recurrir a `pobj`
Object.setPrototypeOf( obj, pobj );

obj.a = 3;
obj.foo();			// a: 3

obj.b = 4;			// Error: No such property/method!
obj.bar();			// Error: No such property/method!
```

El diseño *proxy último* aquí es bastante más simple con respecto a cómo se definen los manejadores. En lugar de necesitar interceptar las operaciones `[[Get]]` y `[[Set]]` y solo reenviarlas si la propiedad del objetivo existe, en su lugar nos basamos en el hecho de que si `[[Get]]` o `[[Set]]` llegan a nuestro fallback `pobj`, la acción ya ha recorrido toda la cadena `[[Prototype]]` y no encontró una propiedad coincidente. Somos libres en ese punto de lanzar incondicionalmente el error. Genial, ¿verdad?

#### Hackeando la Cadena `[[Prototype]]` con Proxy

La operación `[[Get]]` es el canal principal por el cual se invoca el mecanismo `[[Prototype]]`. Cuando una propiedad no se encuentra en el objeto inmediato, `[[Get]]` automáticamente delega la operación al objeto `[[Prototype]]`.

Eso significa que puedes usar la trampa `get(..)` de un proxy para emular o extender la noción de este mecanismo `[[Prototype]]`.

El primer hack que consideraremos es crear dos objetos que estén circularmente vinculados vía `[[Prototype]]` (o, ¡al menos lo parece!). No puedes realmente crear una cadena `[[Prototype]]` circular real, ya que el motor lanzará un error. ¡Pero un proxy puede simularlo!

Considera:

```js
var handlers = {
		get(target,key,context) {
			if (Reflect.has( target, key )) {
				return Reflect.get(
					target, key, context
				);
			}
			// simular `[[Prototype]]` circular
			else {
				return Reflect.get(
					target[
						Symbol.for( "[[Prototype]]" )
					],
					key,
					context
				);
			}
		}
	},
	obj1 = new Proxy(
		{
			name: "obj-1",
			foo() {
				console.log( "foo:", this.name );
			}
		},
		handlers
	),
	obj2 = Object.assign(
		Object.create( obj1 ),
		{
			name: "obj-2",
			bar() {
				console.log( "bar:", this.name );
				this.foo();
			}
		}
	);

// simular vínculo `[[Prototype]]` circular
obj1[ Symbol.for( "[[Prototype]]" ) ] = obj2;

obj1.bar();
// bar: obj-1 <-- a través del proxy simulando [[Prototype]]
// foo: obj-1 <-- el contexto `this` sigue preservado

obj2.foo();
// foo: obj-2 <-- a través de [[Prototype]]
```

**Nota:** No necesitamos hacer proxy/reenvío de `[[Set]]` en este ejemplo, así que mantuvimos las cosas más simples. Para ser completamente compatible con la emulación de `[[Prototype]]`, querrías implementar un manejador `set(..)` que busque en la cadena `[[Prototype]]` una propiedad coincidente y respete su comportamiento de descriptor (por ejemplo, set, writable). Consulta el título *this & Object Prototypes* de esta serie.

En el fragmento anterior, `obj2` está vinculado por `[[Prototype]]` a `obj1` en virtud de la sentencia `Object.create(..)`. Pero para crear el vínculo inverso (circular), creamos una propiedad en `obj1` en la ubicación del símbolo `Symbol.for("[[Prototype]]")` (ver "Symbols" en el Capítulo 2). Este símbolo puede parecer algo especial/mágico, pero no lo es. Simplemente me permite un gancho convenientemente nombrado que semánticamente parece relacionado con la tarea que estoy realizando.

Luego, el manejador `get(..)` del proxy primero verifica si una `key` solicitada está en el proxy. Si no, la operación se delega manualmente a la referencia de objeto almacenada en la ubicación `Symbol.for("[[Prototype]]")` del `target`.

Una ventaja importante de este patrón es que las definiciones de `obj1` y `obj2` en su mayoría no son intrusivas por la configuración de esta relación circular entre ellos. Aunque el fragmento anterior tiene todos los pasos entrelazados por brevedad, si miras de cerca, la lógica del manejador del proxy es completamente genérica (no sabe sobre `obj1` u `obj2` específicamente). Así que, esa lógica podría extraerse en un helper simple que los conecte, como un `setCircularPrototypeOf(..)` por ejemplo. Lo dejaremos como ejercicio para el lector.

Ahora que hemos visto cómo podemos usar `get(..)` para emular un vínculo `[[Prototype]]`, empujemos el hackeo un poco más lejos. En lugar de un `[[Prototype]]` circular, ¿qué tal múltiples vínculos `[[Prototype]]` (también conocido como "herencia múltiple")? Esto resulta ser bastante sencillo:

```js
var obj1 = {
		name: "obj-1",
		foo() {
			console.log( "obj1.foo:", this.name );
		},
	},
	obj2 = {
		name: "obj-2",
		foo() {
			console.log( "obj2.foo:", this.name );
		},
		bar() {
			console.log( "obj2.bar:", this.name );
		}
	},
	handlers = {
		get(target,key,context) {
			if (Reflect.has( target, key )) {
				return Reflect.get(
					target, key, context
				);
			}
			// simular múltiples `[[Prototype]]`
			else {
				for (var P of target[
					Symbol.for( "[[Prototype]]" )
				]) {
					if (Reflect.has( P, key )) {
						return Reflect.get(
							P, key, context
						);
					}
				}
			}
		}
	},
	obj3 = new Proxy(
		{
			name: "obj-3",
			baz() {
				this.foo();
				this.bar();
			}
		},
		handlers
	);

// simular múltiples vínculos `[[Prototype]]`
obj3[ Symbol.for( "[[Prototype]]" ) ] = [
	obj1, obj2
];

obj3.baz();
// obj1.foo: obj-3
// obj2.bar: obj-3
```

**Nota:** Como se mencionó en la nota después del ejemplo anterior de `[[Prototype]]` circular, no implementamos el manejador `set(..)`, pero sería necesario para una solución completa que emule acciones `[[Set]]` como los `[[Prototype]]`s normales se comportan.

`obj3` está configurado para delegar múltiplemente tanto a `obj1` como a `obj2`. En `obj3.baz()`, la llamada `this.foo()` termina extrayendo `foo()` de `obj1` (primero en llegar, primero en servirse, aunque también hay un `foo()` en `obj2`). Si reordenáramos el vínculo como `obj2, obj1`, se habría encontrado y usado `obj2.foo()`.

Pero como está, la llamada `this.bar()` no encuentra un `bar()` en `obj1`, así que pasa a verificar `obj2`, donde encuentra una coincidencia.

`obj1` y `obj2` representan dos cadenas `[[Prototype]]` paralelas de `obj3`. `obj1` y/o `obj2` podrían ellos mismos tener delegación `[[Prototype]]` normal a otros objetos, o cualquiera podría ser en sí mismo un proxy (como lo es `obj3`) que puede delegar múltiplemente.

Al igual que con el ejemplo anterior de `[[Prototype]]` circular, las definiciones de `obj1`, `obj2` y `obj3` son casi completamente separadas de la lógica genérica del proxy que maneja la delegación múltiple. Sería trivial definir una utilidad como `setPrototypesOf(..)` (¡nota la "s"!) que tome un objeto principal y una lista de objetos para simular el vínculo múltiple de `[[Prototype]]`. De nuevo, lo dejaremos como ejercicio para el lector.

Esperemos que el poder de los proxies se esté volviendo más claro después de estos varios ejemplos. Hay muchas otras tareas poderosas de meta programación que los proxies habilitan.
