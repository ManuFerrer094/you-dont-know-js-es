## API `Reflect`

El objeto `Reflect` es un objeto plano (como `Math`), no una función/constructor como los otros nativos integrados.

Contiene funciones estáticas que corresponden a varias tareas de meta programación que puedes controlar. Estas funciones corresponden uno a uno con los métodos manejadores (*trampas*) que los Proxies pueden definir.

Algunas de las funciones se verán familiares como funciones de los mismos nombres en `Object`:

* `Reflect.getOwnPropertyDescriptor(..)`
* `Reflect.defineProperty(..)`
* `Reflect.getPrototypeOf(..)`
* `Reflect.setPrototypeOf(..)`
* `Reflect.preventExtensions(..)`
* `Reflect.isExtensible(..)`

Estas utilidades en general se comportan igual que sus contrapartes `Object.*`. Sin embargo, una diferencia es que las contrapartes `Object.*` intentan convertir su primer argumento (el objeto objetivo) a un objeto si no lo es ya. Los métodos `Reflect.*` simplemente lanzan un error en ese caso.

Las claves de un objeto pueden ser accedidas/inspeccionadas usando estas utilidades:

* `Reflect.ownKeys(..)`: Devuelve la lista de todas las claves propias (no "heredadas"), como lo devuelven tanto `Object.getOwnPropertyNames(..)` como `Object.getOwnPropertySymbols(..)`. Consulta la sección "Orden de Enumeración de Propiedades" para información sobre el orden de las claves.
* `Reflect.enumerate(..)`: Devuelve un iterador que produce el conjunto de todas las claves que no son símbolos (propias y "heredadas") que son *enumerables* (consulta el título *this & Object Prototypes* de esta serie). Esencialmente, este conjunto de claves es el mismo que procesan los bucles `for..in`. Consulta la sección "Orden de Enumeración de Propiedades" para información sobre el orden de las claves.
* `Reflect.has(..)`: Esencialmente lo mismo que el operador `in` para verificar si una propiedad está en un objeto o en su cadena `[[Prototype]]`. Por ejemplo, `Reflect.has(o,"foo")` esencialmente realiza `"foo" in o`.

Las llamadas a funciones e invocaciones de constructores pueden realizarse manualmente, separadas de la sintaxis normal (por ejemplo, `(..)` y `new`) usando estas utilidades:

* `Reflect.apply(..)`: Por ejemplo, `Reflect.apply(foo,thisObj,[42,"bar"])` llama a la función `foo(..)` con `thisObj` como su `this`, y pasa los argumentos `42` y `"bar"`.
* `Reflect.construct(..)`: Por ejemplo, `Reflect.construct(foo,[42,"bar"])` esencialmente llama a `new foo(42,"bar")`.

El acceso, establecimiento y eliminación de propiedades de objetos puede realizarse manualmente usando estas utilidades:

* `Reflect.get(..)`: Por ejemplo, `Reflect.get(o,"foo")` recupera `o.foo`.
* `Reflect.set(..)`: Por ejemplo, `Reflect.set(o,"foo",42)` esencialmente realiza `o.foo = 42`.
* `Reflect.deleteProperty(..)`: Por ejemplo, `Reflect.deleteProperty(o,"foo")` esencialmente realiza `delete o.foo`.

Las capacidades de meta programación de `Reflect` te dan equivalentes programáticos para emular varias características sintácticas, exponiendo operaciones abstractas que antes eran solo internas. Por ejemplo, puedes usar estas capacidades para extender características y APIs para *lenguajes de dominio específico* (DSLs).

### Orden de Propiedades

Antes de ES6, el orden usado para listar las claves/propiedades de un objeto dependía de la implementación y no estaba definido por la especificación. Generalmente, la mayoría de los motores las enumeraban en orden de creación, aunque se animaba fuertemente a los desarrolladores a nunca depender de este ordenamiento.

A partir de ES6, el orden para listar propiedades propias ahora está definido (especificación ES6, sección 9.1.12) por el algoritmo `[[OwnPropertyKeys]]`, que produce todas las propiedades propias (cadenas o símbolos), independientemente de la enumerabilidad. Este ordenamiento solo está garantizado para `Reflect.ownKeys(..)` (y por extensión, `Object.getOwnPropertyNames(..)` y `Object.getOwnPropertySymbols(..)`).

El ordenamiento es:

1. Primero, enumerar cualquier propiedad propia que sean índices enteros, en orden numérico ascendente.
2. Después, enumerar el resto de los nombres de propiedades de cadena propias en orden de creación.
3. Finalmente, enumerar las propiedades de símbolo propias en orden de creación.

Considera:

```js
var o = {};

o[Symbol("c")] = "yay";
o[2] = true;
o[1] = true;
o.b = "awesome";
o.a = "cool";

Reflect.ownKeys( o );				// [1,2,"b","a",Symbol(c)]
Object.getOwnPropertyNames( o );	// [1,2,"b","a"]
Object.getOwnPropertySymbols( o );	// [Symbol(c)]
```

Por otro lado, el algoritmo `[[Enumerate]]` (especificación ES6, sección 9.1.11) produce solo propiedades enumerables, tanto del objeto objetivo como de su cadena `[[Prototype]]`. Es usado tanto por `Reflect.enumerate(..)` como por `for..in`. El ordenamiento observable depende de la implementación y no está controlado por la especificación.

En contraste, `Object.keys(..)` invoca el algoritmo `[[OwnPropertyKeys]]` para obtener una lista de todas las claves propias. Sin embargo, filtra las propiedades no-enumerables y luego reordena la lista para coincidir con el comportamiento heredado dependiente de la implementación, específicamente con `JSON.stringify(..)` y `for..in`. Así, por extensión el ordenamiento *también* coincide con el de `Reflect.enumerate(..)`.

En otras palabras, los cuatro mecanismos (`Reflect.enumerate(..)`, `Object.keys(..)`, `for..in`, y `JSON.stringify(..)`) coincidirán con el mismo ordenamiento dependiente de la implementación, aunque técnicamente llegan ahí de formas diferentes.

Las implementaciones tienen permitido hacer coincidir estos cuatro con el ordenamiento de `[[OwnPropertyKeys]]`, pero no están obligadas a hacerlo. Sin embargo, probablemente observarás el siguiente comportamiento de ordenamiento de ellos:

```js
var o = { a: 1, b: 2 };
var p = Object.create( o );
p.c = 3;
p.d = 4;

for (var prop of Reflect.enumerate( p )) {
	console.log( prop );
}
// c d a b

for (var prop in p) {
	console.log( prop );
}
// c d a b

JSON.stringify( p );
// {"c":3,"d":4}

Object.keys( p );
// ["c","d"]
```

Resumiendo todo esto: a partir de ES6, `Reflect.ownKeys(..)`, `Object.getOwnPropertyNames(..)`, y `Object.getOwnPropertySymbols(..)` todos tienen un ordenamiento predecible y confiable garantizado por la especificación. Así que es seguro construir código que dependa de este ordenamiento.

`Reflect.enumerate(..)`, `Object.keys(..)`, y `for..in` (así como `JSON.stringify(..)` por extensión) continúan compartiendo un ordenamiento observable entre sí, como siempre lo han hecho. Pero ese ordenamiento no será necesariamente el mismo que el de `Reflect.ownKeys(..)`. Aún se debe tener cuidado al depender de su ordenamiento dependiente de la implementación.

## Pruebas de Características

¿Qué es una prueba de características? Es una prueba que ejecutas para determinar si una característica está disponible o no. A veces, la prueba no es solo de existencia, sino de conformidad con el comportamiento especificado -- las características pueden existir pero tener errores.

Esta es una técnica de meta programación, para probar el entorno en el que tu programa se ejecuta y luego determinar cómo debería comportarse tu programa.

El uso más común de pruebas de características en JS es verificar la existencia de una API y si no está presente, definir un polyfill (ver Capítulo 1). Por ejemplo:

```js
if (!Number.isNaN) {
	Number.isNaN = function(x) {
		return x !== x;
	};
}
```

La sentencia `if` en este fragmento es meta programación: estamos sondeando nuestro programa y su entorno de ejecución para determinar si y cómo debemos proceder.

Pero ¿qué hay de probar características que involucran nueva sintaxis?

Podrías intentar algo como:

```js
try {
	a = () => {};
	ARROW_FUNCS_ENABLED = true;
}
catch (err) {
	ARROW_FUNCS_ENABLED = false;
}
```

Desafortunadamente, esto no funciona, porque nuestros programas JS son compilados. Así, el motor se atorará con la sintaxis `() => {}` si aún no soporta funciones flecha de ES6. Tener un error de sintaxis en tu programa previene que se ejecute, lo que previene que tu programa responda de manera diferente posteriormente si la característica es soportada o no.

Para meta programar con pruebas de características alrededor de características relacionadas con la sintaxis, necesitamos una forma de aislar la prueba del paso de compilación inicial por el que pasa nuestro programa. Por ejemplo, si pudiéramos almacenar el código para la prueba en una cadena, entonces el motor JS no intentaría por defecto compilar el contenido de esa cadena, hasta que se lo pidiéramos.

¿Tu mente acaba de saltar a usar `eval(..)`?

No tan rápido. Consulta el título *Scope & Closures* de esta serie para saber por qué `eval(..)` es una mala idea. Pero hay otra opción con menos desventajas: el constructor `Function(..)`.

Considera:

```js
try {
	new Function( "( () => {} )" );
	ARROW_FUNCS_ENABLED = true;
}
catch (err) {
	ARROW_FUNCS_ENABLED = false;
}
```

OK, ahora estamos meta programando determinando si una característica como las funciones flecha *puede* compilarse en el motor actual o no. Entonces podrías preguntarte, ¿qué haríamos con esta información?

Con verificaciones de existencia para APIs, y definiendo polyfills de API de respaldo, hay un camino claro para qué hacer con el éxito o fracaso de la prueba. Pero ¿qué podemos hacer con la información de que `ARROW_FUNCS_ENABLED` sea `true` o `false`?

Debido a que la sintaxis no puede aparecer en un archivo si el motor no soporta esa característica, no puedes simplemente tener diferentes funciones definidas en el archivo con y sin la sintaxis en cuestión.

Lo que puedes hacer es usar la prueba para determinar cuál de un conjunto de archivos JS debes cargar. Por ejemplo, si tuvieras un conjunto de estas pruebas de características en un bootstrapper para tu aplicación JS, podría entonces probar el entorno para determinar si tu código ES6 puede cargarse y ejecutarse directamente, o si necesitas cargar una versión transpilada de tu código (ver Capítulo 1).

Esta técnica se llama *entrega dividida*.

Reconoce la realidad de que tus programas JS escritos en ES6 a veces podrán ejecutarse completamente "nativamente" en navegadores ES6+, pero otras veces necesitarán transpilación para ejecutarse en navegadores pre-ES6. Si siempre cargas y usas el código transpilado, incluso en los nuevos entornos compatibles con ES6, estás ejecutando código subóptimo al menos parte del tiempo. Esto no es ideal.

La entrega dividida es más complicada y sofisticada, pero representa un enfoque más maduro y robusto para cerrar la brecha entre el código que escribes y el soporte de características en los navegadores donde tus programas deben ejecutarse.

## Optimización de Llamadas en Cola (TCO)

Normalmente, cuando se hace una llamada a función desde dentro de otra función, se asigna un segundo *marco de pila* para gestionar separadamente las variables/estado de esa otra invocación de función. No solo esta asignación cuesta algo de tiempo de procesamiento, sino que también ocupa algo de memoria extra.

Una cadena de pila de llamadas típicamente tiene como máximo 10-15 saltos de una función a otra y otra. En esos escenarios, el uso de memoria probablemente no sea ningún tipo de problema práctico.

Sin embargo, cuando consideras la programación recursiva (una función llamándose a sí misma repetidamente) -- o recursión mutua con dos o más funciones llamándose entre sí -- la pila de llamadas podría fácilmente tener cientos, miles o más niveles de profundidad. Probablemente puedes ver los problemas que eso podría causar, si el uso de memoria crece sin límites.

Los motores de JavaScript tienen que establecer un límite arbitrario para prevenir que tales técnicas de programación colapsen al agotar la memoria del navegador y del dispositivo. Es por eso que obtenemos el frustrante "RangeError: Maximum call stack size exceeded" lanzado si se alcanza el límite.

**Advertencia:** El límite de profundidad de la pila de llamadas no está controlado por la especificación. Depende de la implementación, y variará entre navegadores y dispositivos. Nunca deberías codificar con suposiciones fuertes de límites observables exactos, ya que pueden muy bien cambiar de versión en versión.

Ciertos patrones de llamadas a funciones, llamados *llamadas en cola*, pueden optimizarse de una forma que evite la asignación extra de marcos de pila. Si la asignación extra puede evitarse, no hay razón para limitar arbitrariamente la profundidad de la pila de llamadas, así que los motores pueden dejarlas ejecutarse sin límites.

Una llamada en cola es una sentencia `return` con una llamada a función, donde no hay nada que hacer después de la llamada excepto retornar su valor.

Esta optimización solo puede aplicarse en modo `strict`. ¡Otra razón más para siempre escribir todo tu código como `strict`!

Aquí hay una llamada a función que *no* está en posición de cola:

```js
"use strict";

function foo(x) {
	return x * 2;
}

function bar(x) {
	// no es una llamada en cola
	return 1 + foo( x );
}

bar( 10 );				// 21
```

`1 + ..` tiene que realizarse después de que la llamada `foo(x)` se complete, así que el estado de esa invocación de `bar(..)` necesita preservarse.

Pero el siguiente fragmento demuestra llamadas a `foo(..)` y `bar(..)` donde ambas *están* en posición de cola, ya que son lo último que sucede en su ruta de código (aparte del `return`):

```js
"use strict";

function foo(x) {
	return x * 2;
}

function bar(x) {
	x = x + 1;
	if (x > 10) {
		return foo( x );
	}
	else {
		return bar( x + 1 );
	}
}

bar( 5 );				// 24
bar( 15 );				// 32
```

En este programa, `bar(..)` es claramente recursivo, pero `foo(..)` es solo una llamada a función regular. En ambos casos, las llamadas a funciones están en *posición de cola apropiada*. El `x + 1` se evalúa antes de la llamada a `bar(..)`, y cuando esa llamada termina, todo lo que sucede es el `return`.

Las Llamadas en Cola Apropiadas (PTC) de estas formas pueden optimizarse -- llamada optimización de llamadas en cola (TCO) -- para que la asignación extra de marco de pila sea innecesaria. En lugar de crear un nuevo marco de pila para la siguiente llamada a función, el motor simplemente reutiliza el marco de pila existente. Eso funciona porque una función no necesita preservar ninguno del estado actual, ya que nada sucede con ese estado después de la PTC.

TCO significa que prácticamente no hay límite para qué tan profunda puede ser la pila de llamadas. Ese truco mejora ligeramente las llamadas a funciones regulares en programas normales, pero más importante aún abre la puerta a usar recursión para la expresión de programas incluso si la pila de llamadas podría tener decenas de miles de llamadas de profundidad.

Ya no estamos relegados a simplemente teorizar sobre la recursión para la resolución de problemas, ¡sino que podemos realmente usarla en programas JavaScript reales!

A partir de ES6, todas las PTC deberían ser optimizables de esta manera, ya sea recursión o no.

### Reescritura de Llamadas en Cola

El inconveniente, sin embargo, es que solo las PTC pueden optimizarse; las no-PTC seguirán funcionando por supuesto, pero causarán asignación de marcos de pila como siempre lo hicieron. Tendrás que ser cuidadoso al estructurar tus funciones con PTC si esperas que las optimizaciones se activen.

Si tienes una función que no está escrita con PTC, podrías encontrar la necesidad de reorganizar manualmente tu código para ser elegible para TCO.

Considera:

```js
"use strict";

function foo(x) {
	if (x <= 1) return 1;
	return (x / 2) + foo( x - 1 );
}

foo( 123456 );			// RangeError
```

La llamada a `foo(x-1)` no es una PTC porque su resultado tiene que sumarse a `(x / 2)` antes de hacer `return`.

Sin embargo, para hacer este código elegible para TCO en un motor ES6, podemos reescribirlo de la siguiente manera:

```js
"use strict";

var foo = (function(){
	function _foo(acc,x) {
		if (x <= 1) return acc;
		return _foo( (x / 2) + acc, x - 1 );
	}

	return function(x) {
		return _foo( 1, x );
	};
})();

foo( 123456 );			// 3810376848.5
```

Si ejecutas el fragmento anterior en un motor ES6 que implementa TCO, obtendrás la respuesta `3810376848.5` como se muestra. Sin embargo, seguirá fallando con un `RangeError` en motores sin TCO.

### Optimizaciones Sin TCO

Hay otras técnicas para reescribir el código de forma que la pila de llamadas no crezca con cada llamada.

Una de esas técnicas se llama *trampolining* (trampolín), que consiste en tener cada resultado parcial representado como una función que devuelve otro resultado parcial como función o el resultado final. Luego simplemente iteras hasta que dejes de obtener una función, y tendrás el resultado. Considera:

```js
"use strict";

function trampoline( res ) {
	while (typeof res == "function") {
		res = res();
	}
	return res;
}

var foo = (function(){
	function _foo(acc,x) {
		if (x <= 1) return acc;
		return function partial(){
			return _foo( (x / 2) + acc, x - 1 );
		};
	}

	return function(x) {
		return trampoline( _foo( 1, x ) );
	};
})();

foo( 123456 );			// 3810376848.5
```

Esta reelaboración requirió cambios mínimos para sacar la recursión del bucle en `trampoline(..)`:

1. Primero, envolvimos la línea `return _foo ..` en la expresión de función `return partial() { ..`.
2. Luego envolvimos la llamada `_foo(1,x)` en la llamada `trampoline(..)`.

La razón por la que esta técnica no sufre la limitación de la pila de llamadas es que cada una de esas funciones internas `partial(..)` simplemente se devuelve al bucle `while` en `trampoline(..)`, que la ejecuta y luego itera de nuevo el bucle. En otras palabras, `partial(..)` no se llama a sí misma recursivamente, solo devuelve otra función. La profundidad de la pila permanece constante, así que puede ejecutarse tanto como necesite.

El trampolining expresado de esta manera usa el closure que la función interna `partial()` tiene sobre las variables `x` y `acc` para mantener el estado de iteración en iteración. La ventaja es que la lógica del bucle se extrae en una función utilitaria `trampoline(..)` reutilizable, de la cual muchas bibliotecas proveen versiones. Puedes reutilizar `trampoline(..)` múltiples veces en tu programa con diferentes algoritmos trampolineados.

Por supuesto, si realmente quisieras optimizar profundamente (y la reutilización no fuera una preocupación), podrías descartar el estado del closure e incorporar el seguimiento del estado de `acc` en el ámbito de una sola función junto con un bucle. Esta técnica se llama generalmente *desenrollado de recursión*:

```js
"use strict";

function foo(x) {
	var acc = 1;
	while (x > 1) {
		acc = (x / 2) + acc;
		x = x - 1;
	}
	return acc;
}

foo( 123456 );			// 3810376848.5
```

Esta expresión del algoritmo es más simple de leer, y probablemente tendrá el mejor rendimiento (estrictamente hablando) de las varias formas que hemos explorado. Eso puede parecer un claro ganador, y podrías preguntarte por qué intentarías los otros enfoques.

Hay algunas razones por las que podrías no querer siempre desenrollar manualmente tus recursiones:

* En lugar de sacar la lógica del trampolining (bucle) para reutilización, la hemos incorporado en línea. Esto funciona genial cuando solo hay un ejemplo a considerar, pero tan pronto como tengas media docena o más de estos en tu programa, hay una buena probabilidad de que quieras algo de reutilización para mantener las cosas más cortas y manejables.
* El ejemplo aquí es deliberadamente lo suficientemente simple para ilustrar las diferentes formas. En la práctica, hay muchas más complicaciones en algoritmos recursivos, como la recursión mutua (más de una sola función llamándose a sí misma).

   Cuanto más profundo vayas en esta madriguera, más manuales e intrincadas son las optimizaciones de *desenrollado*. Rápidamente perderás todo el valor percibido de legibilidad. La ventaja principal de la recursión, incluso en la forma PTC, es que preserva la legibilidad del algoritmo, y descarga la optimización de rendimiento al motor.

Si escribes tus algoritmos con PTC, el motor ES6 aplicará TCO para dejar que tu código se ejecute en profundidad de pila constante (reutilizando marcos de pila). Obtienes la legibilidad de la recursión con la mayoría de los beneficios de rendimiento y sin limitaciones de longitud de ejecución.

### ¿Meta?

¿Qué tiene que ver TCO con la meta programación?

Como cubrimos en la sección "Pruebas de Características" antes, puedes determinar en tiempo de ejecución qué características soporta un motor. Esto incluye TCO, aunque determinarlo es bastante por fuerza bruta. Considera:

```js
"use strict";

try {
	(function foo(x){
		if (x < 5E5) return foo( x + 1 );
	})( 1 );

	TCO_ENABLED = true;
}
catch (err) {
	TCO_ENABLED = false;
}
```

En un motor sin TCO, el bucle recursivo eventualmente fallará, lanzando una excepción atrapada por el `try..catch`. De lo contrario, el bucle se completa fácilmente gracias a TCO.

Horrible, ¿verdad?

Pero ¿cómo podría la meta programación alrededor de la característica TCO (o más bien, la falta de ella) beneficiar nuestro código? La respuesta simple es que podrías usar una prueba de característica así para decidir cargar una versión del código de tu aplicación que usa recursión, o una alternativa que haya sido convertida/transpilada para no necesitar recursión.

#### Código Auto-Ajustable

Pero aquí hay otra forma de ver el problema:

```js
"use strict";

function foo(x) {
	function _foo() {
		if (x > 1) {
			acc = acc + (x / 2);
			x = x - 1;
			return _foo();
		}
	}

	var acc = 1;

	while (x > 1) {
		try {
			_foo();
		}
		catch (err) { }
	}

	return acc;
}

foo( 123456 );			// 3810376848.5
```

Este algoritmo funciona intentando hacer tanto del trabajo con recursión como sea posible, pero manteniendo registro del progreso a través de variables de ámbito `x` y `acc`. Si el problema completo puede resolverse con recursión sin un error, genial. Si el motor mata la recursión en algún punto, simplemente atrapamos eso con el `try..catch` y luego intentamos de nuevo, continuando donde lo dejamos.

Considero esto una forma de meta programación en que estás sondeando durante el tiempo de ejecución la capacidad del motor para completar totalmente (recursivamente) la tarea, y trabajando alrededor de cualquier limitación del motor (sin-TCO) que pueda restringirte.

A primera vista (¡o incluso segunda!), mi apuesta es que este código te parece mucho más feo comparado con algunas de las versiones anteriores. También se ejecuta bastante más lento (en ejecuciones más grandes en un entorno sin TCO).

La ventaja principal, aparte de que puede completar cualquier tarea de cualquier tamaño incluso en motores sin TCO, es que esta "solución" a la limitación de la pila de recursión es mucho más flexible que las técnicas de trampolining o desenrollado manual que se mostraron anteriormente.

Esencialmente, `_foo()` en este caso es una especie de sustituto para prácticamente cualquier tarea recursiva, incluso recursión mutua. El resto es el código plantilla que debería funcionar para casi cualquier algoritmo.

La única "trampa" es que para poder reanudar en caso de que se alcance el límite de recursión, el estado de la recursión debe estar en variables de ámbito que existan fuera de la(s) función(es) recursiva(s). Hicimos eso dejando `x` y `acc` fuera de la función `_foo()`, en lugar de pasarlos como argumentos a `_foo()` como antes.

Casi cualquier algoritmo recursivo puede adaptarse para funcionar de esta manera. Eso significa que es la forma más ampliamente aplicable de aprovechar TCO con recursión en tus programas, con reescritura mínima.

Este enfoque aún usa una PTC, lo que significa que este código *mejorará progresivamente* de ejecutarse usando el bucle muchas veces (lotes de recursión) en un navegador más antiguo a aprovechar completamente la recursión con TCO en un entorno ES6+. ¡Creo que eso es bastante genial!

## Revisión

La meta programación es cuando giras la lógica de tu programa para enfocarse en sí mismo (o su entorno de ejecución), ya sea para inspeccionar su propia estructura o para modificarla. El valor principal de la meta programación es extender los mecanismos normales del lenguaje para proporcionar capacidades adicionales.

Antes de ES6, JavaScript ya tenía bastante capacidad de meta programación, pero ES6 aumenta eso significativamente con varias nuevas características.

Desde las inferencias de nombres de funciones para funciones anónimas hasta las meta propiedades que te dan información sobre cosas como cómo se invocó un constructor, puedes inspeccionar la estructura del programa mientras se ejecuta más que nunca antes. Los Símbolos Bien Conocidos te permiten anular comportamientos intrínsecos, como la coerción de un objeto a un valor primitivo. Los Proxies pueden interceptar y personalizar varias operaciones de bajo nivel en objetos, y `Reflect` proporciona utilidades para emularlas.

Las pruebas de características, incluso para comportamientos semánticos sutiles como la Optimización de Llamadas en Cola, cambian el enfoque de la meta programación de tu programa a las capacidades del motor JS mismo. Al saber más sobre lo que el entorno puede hacer, tus programas pueden ajustarse a sí mismos al mejor ajuste mientras se ejecutan.

¿Deberías hacer meta programación? Mi consejo es: primero enfócate en aprender cómo funcionan realmente las mecánicas centrales del lenguaje. ¡Pero una vez que sepas completamente lo que JS puede hacer por sí mismo, es hora de empezar a aprovechar estas poderosas capacidades de meta programación para llevar el lenguaje más lejos!
