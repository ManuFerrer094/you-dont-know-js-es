## Bucles `for..of`

Uniéndose a los bucles `for` y `for..in` del JavaScript con el que todos estamos familiarizados, ES6 añade un bucle `for..of`, que itera sobre el conjunto de valores producidos por un *iterador*.

El valor sobre el que iteras con `for..of` debe ser un *iterable*, o debe ser un valor que pueda ser convertido/encapsulado a un objeto (ver el título *Types & Grammar* de esta serie) que sea un iterable. Un iterable es simplemente un objeto que es capaz de producir un iterador, que el bucle entonces usa.

Comparemos `for..of` con `for..in` para ilustrar la diferencia:

```js
var a = ["a","b","c","d","e"];

for (var idx in a) {
	console.log( idx );
}
// 0 1 2 3 4

for (var val of a) {
	console.log( val );
}
// "a" "b" "c" "d" "e"
```

Como puedes ver, `for..in` itera sobre las claves/índices en el array `a`, mientras que `for..of` itera sobre los valores en `a`.

Aquí está la versión pre-ES6 del `for..of` del fragmento anterior:

```js
var a = ["a","b","c","d","e"],
	k = Object.keys( a );

for (var val, i = 0; i < k.length; i++) {
	val = a[ k[i] ];
	console.log( val );
}
// "a" "b" "c" "d" "e"
```

Y aquí está el equivalente ES6 pero sin `for..of`, que también da un vistazo a la iteración manual de un iterador (ver "Iteradores" en el Capítulo 3):

```js
var a = ["a","b","c","d","e"];

for (var val, ret, it = a[Symbol.iterator]();
	(ret = it.next()) && !ret.done;
) {
	val = ret.value;
	console.log( val );
}
// "a" "b" "c" "d" "e"
```

Bajo el capó, el bucle `for..of` le pide al iterable un iterador (usando el `Symbol.iterator` incorporado; ver "Símbolos Bien Conocidos" en el Capítulo 7), luego llama repetidamente al iterador y asigna su valor producido a la variable de iteración del bucle.

Los valores incorporados estándar en JavaScript que son iterables por defecto (o los proporcionan) incluyen:

* Arrays
* Strings
* Generators (ver Capítulo 3)
* Colecciones / TypedArrays (ver Capítulo 5)

**Advertencia:** Los objetos simples no son adecuados por defecto para iterar con `for..of`. Esto es porque no tienen un iterador por defecto, lo cual es intencional, no un error. Sin embargo, no profundizaremos más en esos razonamientos matizados aquí. En "Iteradores" en el Capítulo 3, veremos cómo definir iteradores para nuestros propios objetos, lo que permite que `for..of` itere sobre cualquier objeto para obtener un conjunto de valores que definamos.

Así es como se itera sobre los caracteres en una cadena primitiva:

```js
for (var c of "hello") {
	console.log( c );
}
// "h" "e" "l" "l" "o"
```

El valor primitivo de cadena `"hello"` se convierte/encapsula a su equivalente objeto envoltorio `String`, que es un iterable por defecto.

En `for (XYZ of ABC)..`, la cláusula `XYZ` puede ser tanto una expresión de asignación como una declaración, idéntica a esa misma cláusula en los bucles `for` y `for..in`. Así que puedes hacer cosas como estas:

```js
var o = {};

for (o.a of [1,2,3]) {
	console.log( o.a );
}
// 1 2 3

for ({x: o.a} of [ {x: 1}, {x: 2}, {x: 3} ]) {
  console.log( o.a );
}
// 1 2 3
```

Los bucles `for..of` pueden detenerse prematuramente, al igual que otros bucles, con `break`, `continue`, `return` (si están en una función), y excepciones lanzadas. En cualquiera de estos casos, la función `return(..)` del iterador se llama automáticamente (si existe una) para permitir que el iterador realice tareas de limpieza, si es necesario.

**Nota:** Ver "Iteradores" en el Capítulo 3 para una cobertura más completa sobre iterables e iteradores.

## Expresiones Regulares

Enfrentémoslo: las expresiones regulares no han cambiado mucho en JS en mucho tiempo. Así que es bueno que finalmente hayan aprendido un par de trucos nuevos en ES6. Cubriremos brevemente las adiciones aquí, pero el tema general de las expresiones regulares es tan denso que necesitarás recurrir a capítulos/libros dedicados a él (¡de los cuales hay muchos!) si necesitas un repaso.

### Bandera Unicode

Cubriremos el tema de Unicode en más detalle en "Unicode" más adelante en este capítulo. Aquí, solo miraremos brevemente la nueva bandera `u` para expresiones regulares de ES6+, que activa la coincidencia Unicode para esa expresión.

Las cadenas de JavaScript típicamente se interpretan como secuencias de caracteres de 16 bits, que corresponden a los caracteres en el *Plano Multilingüe Básico (BMP)* (http://en.wikipedia.org/wiki/Plane_%28Unicode%29). Pero hay muchos caracteres UTF-16 que caen fuera de este rango, y por lo tanto las cadenas pueden tener estos caracteres multibyte en ellas.

Antes de ES6, las expresiones regulares solo podían coincidir basándose en caracteres BMP, lo que significa que esos caracteres extendidos se trataban como dos caracteres separados para propósitos de coincidencia. Esto frecuentemente no es ideal.

Así que, a partir de ES6, la bandera `u` le dice a una expresión regular que procese una cadena con la interpretación de caracteres Unicode (UTF-16), de modo que tal carácter extendido será coincidido como una sola entidad.

**Advertencia:** A pesar de la implicación del nombre, "UTF-16" no significa estrictamente 16 bits. El Unicode moderno usa 21 bits, y estándares como UTF-8 y UTF-16 se refieren aproximadamente a cuántos bits se usan en la representación de un carácter.

Un ejemplo (directamente de la especificación ES6): 𝄞 (el símbolo musical clave de sol) es el punto Unicode U+1D11E (0x1D11E).

Si este carácter aparece en un patrón de expresión regular (como `/𝄞/`), la interpretación BMP estándar sería que son dos caracteres separados (0xD834 y 0xDD1E) con los que coincidir. Pero el nuevo modo consciente de Unicode de ES6 significa que `/𝄞/u` (o la forma escapada Unicode `/\u{1D11E}/u`) coincidirá con `"𝄞"` en una cadena como un solo carácter coincidido.

Podrías preguntarte por qué esto importa. En modo BMP no Unicode, el patrón se trata como dos caracteres separados, pero aún encontraría la coincidencia en una cadena con el carácter `"𝄞"` en ella, como puedes ver si lo intentas:

```js
/𝄞/.test( "𝄞-clef" );			// true
```

La longitud de la coincidencia es lo que importa. Por ejemplo:

```js
/^.-clef/ .test( "𝄞-clef" );		// false
/^.-clef/u.test( "𝄞-clef" );		// true
```

El `^.-clef` en el patrón dice coincidir solo un único carácter al principio antes del texto normal `"-clef"`. En modo BMP estándar, la coincidencia falla (dos caracteres), pero con el modo Unicode activado con la bandera `u`, la coincidencia tiene éxito (un carácter).

También es importante notar que `u` hace que los cuantificadores como `+` y `*` se apliquen al punto de código Unicode completo como un solo carácter, no solo al *sustituto inferior* (también conocido como la mitad derecha del símbolo) del carácter. Lo mismo aplica para caracteres Unicode apareciendo en clases de caracteres, como `/[💩-💫]/u`.

**Nota:** Hay muchos más detalles minuciosos sobre el comportamiento de `u` en expresiones regulares, que Mathias Bynens (https://twitter.com/mathias) ha escrito extensamente sobre (https://mathiasbynens.be/notes/es6-unicode-regex).

### Bandera Sticky

Otro modo de bandera añadido en ES6 a las expresiones regulares es `y`, que frecuentemente se llama "modo sticky". *Sticky* esencialmente significa que la expresión regular tiene un ancla virtual en su inicio que la mantiene enraizada a coincidir solo en la posición indicada por la propiedad `lastIndex` de la expresión regular.

Para ilustrar, consideremos dos expresiones regulares, la primera sin modo sticky y la segunda con él:

```js
var re1 = /foo/,
	str = "++foo++";

re1.lastIndex;			// 0
re1.test( str );		// true
re1.lastIndex;			// 0 -- no actualizado

re1.lastIndex = 4;
re1.test( str );		// true -- ignoró `lastIndex`
re1.lastIndex;			// 4 -- no actualizado
```

Tres cosas a observar sobre este fragmento:

* `test(..)` no presta atención al valor de `lastIndex`, y siempre simplemente realiza su coincidencia desde el principio de la cadena de entrada.
* Porque nuestro patrón no tiene un ancla `^` de inicio de entrada, la búsqueda de `"foo"` es libre de avanzar por toda la cadena buscando una coincidencia.
* `lastIndex` no es actualizado por `test(..)`.

Ahora, intentemos con una expresión regular en modo sticky:

```js
var re2 = /foo/y,		// <-- nota la bandera sticky `y`
	str = "++foo++";

re2.lastIndex;			// 0
re2.test( str );		// false -- "foo" no encontrado en `0`
re2.lastIndex;			// 0

re2.lastIndex = 2;
re2.test( str );		// true
re2.lastIndex;			// 5 -- actualizado a después de la coincidencia anterior

re2.test( str );		// false
re2.lastIndex;			// 0 -- reiniciado después del fallo de coincidencia anterior
```

Y así nuestras nuevas observaciones sobre el modo sticky:

* `test(..)` usa `lastIndex` como la posición exacta y única en `str` donde buscar para hacer una coincidencia. No hay avance para buscar la coincidencia -- o está ahí en la posición de `lastIndex` o no.
* Si se hace una coincidencia, `test(..)` actualiza `lastIndex` para apuntar al carácter inmediatamente después de la coincidencia. Si una coincidencia falla, `test(..)` reinicia `lastIndex` de vuelta a `0`.

Los patrones normales no sticky que no están de otra manera enraizados con `^` al inicio de la entrada son libres de avanzar en la cadena de entrada buscando una coincidencia. Pero el modo sticky restringe el patrón a coincidir solo en la posición de `lastIndex`.

Como sugerí al inicio de esta sección, otra manera de ver esto es que `y` implica un ancla virtual al principio del patrón que es relativa (es decir, restringe el inicio de la coincidencia) a exactamente la posición de `lastIndex`.

**Advertencia:** En la literatura previa sobre el tema, se ha afirmado alternativamente que este comportamiento es como si `y` implicara un `^` (inicio de entrada) en el patrón. Esto es inexacto. Lo explicaremos con más detalle en "Sticky Anclado" más adelante.

#### Posicionamiento Sticky

Puede parecer extrañamente limitante que para usar `y` para coincidencias repetidas, tengas que asegurar manualmente que `lastIndex` esté en la posición exacta correcta, ya que no tiene capacidad de avance para coincidir.

Aquí hay un escenario posible: si sabes que la coincidencia que te importa siempre va a estar en una posición que es múltiplo de un número (por ejemplo, `0`, `10`, `20`, etc.), puedes simplemente construir un patrón limitado que coincida con lo que te importa, pero luego establecer manualmente `lastIndex` cada vez antes de coincidir a esas posiciones fijas.

Considera:

```js
var re = /f../y,
	str = "foo       far       fad";

str.match( re );		// ["foo"]

re.lastIndex = 10;
str.match( re );		// ["far"]

re.lastIndex = 20;
str.match( re );		// ["fad"]
```

Sin embargo, si estás parseando una cadena que no está formateada en posiciones fijas como esa, determinar qué establecer como `lastIndex` antes de cada coincidencia probablemente será insostenible.

Hay un matiz salvador a considerar aquí. `y` requiere que `lastIndex` esté en la posición exacta para que ocurra una coincidencia. Pero no requiere estrictamente que *tú* establezcas `lastIndex` manualmente.

En cambio, puedes construir tus expresiones de tal manera que capturen en cada coincidencia principal todo lo que esté antes y después de lo que te importa, hasta justo antes de la siguiente cosa que te importará coincidir.

Porque `lastIndex` se establecerá al siguiente carácter más allá del final de una coincidencia, si has coincidido con todo hasta ese punto, `lastIndex` siempre estará en la posición correcta para que el patrón `y` comience desde la próxima vez.

**Advertencia:** Si no puedes predecir la estructura de la cadena de entrada de una manera suficientemente patronada como esa, esta técnica puede no ser adecuada y puede que no puedas usar `y`.

Tener entrada de cadena estructurada es probablemente el escenario más práctico donde `y` será capaz de realizar coincidencias repetidas a lo largo de una cadena. Considera:

```js
var re = /\d+\.\s(.*?)(?:\s|$)/y
	str = "1. foo 2. bar 3. baz";

str.match( re );		// [ "1. foo ", "foo" ]

re.lastIndex;			// 7 -- ¡posición correcta!
str.match( re );		// [ "2. bar ", "bar" ]

re.lastIndex;			// 14 -- ¡posición correcta!
str.match( re );		// ["3. baz", "baz"]
```

Esto funciona porque sabía algo de antemano sobre la estructura de la cadena de entrada: siempre hay un prefijo numeral como `"1. "` antes de la coincidencia deseada (`"foo"`, etc.), y ya sea un espacio después, o el final de la cadena (ancla `$`). Así que la expresión regular que construí captura todo eso en cada coincidencia principal, y luego uso un grupo de coincidencia `( )` para que lo que realmente me importa se separe por conveniencia.

Después de la primera coincidencia (`"1. foo "`), el `lastIndex` es `7`, que ya es la posición necesaria para comenzar la siguiente coincidencia, para `"2. bar "`, y así sucesivamente.

Si vas a usar el modo sticky `y` para coincidencias repetidas, probablemente querrás buscar oportunidades para que `lastIndex` se posicione automáticamente como acabamos de demostrar.

#### Sticky Versus Global

Algunos lectores pueden saber que puedes emular algo como esta coincidencia relativa a `lastIndex` con la bandera de coincidencia global `g` y el método `exec(..)`, así:

```js
var re = /o+./g,		// <-- ¡mira, `g`!
	str = "foot book more";

re.exec( str );			// ["oot"]
re.lastIndex;			// 4

re.exec( str );			// ["ook"]
re.lastIndex;			// 9

re.exec( str );			// ["or"]
re.lastIndex;			// 13

re.exec( str );			// null -- ¡no más coincidencias!
re.lastIndex;			// 0 -- ¡empieza de nuevo ahora!
```

Aunque es cierto que las coincidencias de patrón `g` con `exec(..)` comienzan su coincidencia desde el valor actual de `lastIndex`, y también actualizan `lastIndex` después de cada coincidencia (o fallo), esto no es lo mismo que el comportamiento de `y`.

Nota en el fragmento anterior que `"ook"`, ubicado en la posición `6`, fue coincidido y encontrado por la segunda llamada `exec(..)`, aunque en ese momento, `lastIndex` era `4` (del final de la coincidencia anterior). ¿Por qué? Porque como dijimos antes, las coincidencias no sticky son libres de avanzar en su coincidencia. Una expresión en modo sticky habría fallado aquí, porque no se le permitiría avanzar.

Además del comportamiento de avance de coincidencia posiblemente no deseado, otra desventaja de solo usar `g` en lugar de `y` es que `g` cambia el comportamiento de algunos métodos de coincidencia, como `str.match(re)`.

Considera:

```js
var re = /o+./g,		// <-- ¡mira, `g`!
	str = "foot book more";

str.match( re );		// ["oot","ook","or"]
```

¿Ves cómo todas las coincidencias se devolvieron de una vez? A veces eso está bien, pero a veces no es lo que quieres.

La bandera sticky `y` te dará coincidencia progresiva de una a la vez con utilidades como `test(..)` y `match(..)`. Solo asegúrate de que `lastIndex` esté siempre en la posición correcta para cada coincidencia.

#### Sticky Anclado

Como advertimos antes, es inexacto pensar en el modo sticky como que implica que un patrón comienza con `^`. El ancla `^` tiene un significado distinto en expresiones regulares, que *no es alterado* por el modo sticky. `^` es un ancla que *siempre* se refiere al principio de la entrada, y *no es* de ninguna manera relativa a `lastIndex`.

Además de documentación pobre/inexacta sobre este tema, la confusión se refuerza desafortunadamente aún más porque un experimento pre-ES6 más antiguo con el modo sticky en Firefox *sí* hacía `^` relativo a `lastIndex`, así que ese comportamiento ha estado alrededor por años.

ES6 eligió no hacerlo de esa manera. `^` en un patrón significa inicio de entrada de manera absoluta y únicamente.

Como consecuencia, un patrón como `/^foo/y` siempre y solo encontrará una coincidencia `"foo"` al principio de una cadena, *si se le permite coincidir allí*. Si `lastIndex` no es `0`, la coincidencia fallará. Considera:

```js
var re = /^foo/y,
	str = "foo";

re.test( str );			// true
re.test( str );			// false
re.lastIndex;			// 0 -- reiniciado después del fallo

re.lastIndex = 1;
re.test( str );			// false -- falló por posicionamiento
re.lastIndex;			// 0 -- reiniciado después del fallo
```

En resumen: `y` más `^` más `lastIndex > 0` es una combinación incompatible que siempre causará una coincidencia fallida.

**Nota:** Aunque `y` no altera el significado de `^` de ninguna manera, el modo multilínea `m` *sí* lo hace, de modo que `^` significa inicio de entrada *o* inicio de texto después de una nueva línea. Entonces, si combinas las banderas `y` y `m` juntas para un patrón, puedes encontrar múltiples coincidencias enraizadas en `^` en una cadena. Pero recuerda: porque es sticky `y`, tendrás que asegurarte de que `lastIndex` apunte a la posición de nueva línea correcta (probablemente coincidiendo hasta el final de la línea) cada vez subsiguiente, o no se harán coincidencias posteriores.

### `flags` de Expresión Regular

Antes de ES6, si querías examinar un objeto de expresión regular para ver qué banderas tenía aplicadas, necesitabas parsearlas -- irónicamente, probablemente con otra expresión regular -- del contenido de la propiedad `source`, como:

```js
var re = /foo/ig;

re.toString();			// "/foo/ig"

var flags = re.toString().match( /\/([gim]*)$/ )[1];

flags;					// "ig"
```

A partir de ES6, ahora puedes obtener estos valores directamente, con la nueva propiedad `flags`:

```js
var re = /foo/ig;

re.flags;				// "gi"
```

Es un pequeño matiz, pero la especificación ES6 establece que las banderas de la expresión se listen en este orden: `"gimuy"`, independientemente del orden en que se especificó el patrón original. Esa es la razón de la diferencia entre `/ig` y `"gi"`.

No, el orden de las banderas especificadas o listadas no importa.

Otro ajuste de ES6 es que el constructor `RegExp(..)` ahora es consciente de `flags` si le pasas una expresión regular existente:

```js
var re1 = /foo*/y;
re1.source;							// "foo*"
re1.flags;							// "y"

var re2 = new RegExp( re1 );
re2.source;							// "foo*"
re2.flags;							// "y"

var re3 = new RegExp( re1, "ig" );
re3.source;							// "foo*"
re3.flags;							// "gi"
```

Antes de ES6, la construcción de `re3` lanzaría un error, pero a partir de ES6 puedes sobrescribir las banderas al duplicar.

## Extensiones de Literales Numéricos

Antes de ES5, los literales numéricos se veían así -- la forma octal no estaba oficialmente especificada, solo se permitía como una extensión sobre la que los navegadores habían llegado a un acuerdo de facto:

```js
var dec = 42,
	oct = 052,
	hex = 0x2a;
```

**Nota:** Aunque estés especificando un número en diferentes bases, el valor matemático del número es lo que se almacena, y la interpretación de salida por defecto es siempre base-10. Las tres variables en el fragmento anterior tienen todas el valor `42` almacenado en ellas.

Para ilustrar aún más que `052` era una extensión de forma no estándar, considera:

```js
Number( "42" );				// 42
Number( "052" );			// 52
Number( "0x2a" );			// 42
```

ES5 continuó permitiendo la forma octal extendida por navegadores (incluyendo tales inconsistencias), excepto que en modo estricto, la forma literal octal (`052`) no está permitida. Esta restricción se hizo principalmente porque muchos desarrolladores tenían la costumbre (de otros lenguajes) de prefijar aparentemente de manera inocente números en base 10 con `0`s para propósitos de alineación de código, y luego encontrarse con el hecho accidental de que habían cambiado el valor del número por completo.

ES6 continúa el legado de cambios/variaciones a cómo los literales numéricos fuera de los números base-10 pueden representarse. Ahora hay una forma octal oficial, una forma hexadecimal enmendada, y una forma binaria completamente nueva. Por razones de compatibilidad web, la antigua forma octal `052` continuará siendo legal (aunque no especificada) en modo no estricto, pero realmente nunca debería usarse más.

Aquí están las nuevas formas de literales numéricos de ES6:

```js
var dec = 42,
	oct = 0o52,			// o `0O52` :(
	hex = 0x2a,			// o `0X2a` :/
	bin = 0b101010;		// o `0B101010` :/
```

La única forma decimal permitida es base-10. Octal, hexadecimal y binario son todas formas enteras.

Y las representaciones de cadena de estas formas son todas capaces de ser convertidas a su equivalente numérico:

```js
Number( "42" );			// 42
Number( "0o52" );		// 42
Number( "0x2a" );		// 42
Number( "0b101010" );	// 42
```

Aunque no es estrictamente nuevo en ES6, es un hecho poco conocido que puedes ir en la dirección opuesta de conversión (bueno, más o menos):

```js
var a = 42;

a.toString();			// "42" -- también `a.toString( 10 )`
a.toString( 8 );		// "52"
a.toString( 16 );		// "2a"
a.toString( 2 );		// "101010"
```

De hecho, puedes representar un número de esta manera en cualquier base del `2` al `36`, aunque sería raro que fueras más allá de las bases estándar: 2, 8, 10 y 16.

## Unicode

Déjame decir que esta sección no es un recurso exhaustivo de todo-lo-que-siempre-quisiste-saber-sobre-Unicode. Quiero cubrir lo que necesitas saber que está *cambiando* para Unicode en ES6, pero no iremos mucho más profundo que eso. Mathias Bynens (http://twitter.com/mathias) ha escrito/hablado extensiva y brillantemente sobre JS y Unicode (ver https://mathiasbynens.be/notes/javascript-unicode y http://fluentconf.com/javascript-html-2015/public/content/2015/02/18-javascript-loves-unicode).

Los caracteres Unicode que van desde `0x0000` hasta `0xFFFF` contienen todos los caracteres impresos estándar (en varios idiomas) que probablemente hayas visto o con los que hayas interactuado. Este grupo de caracteres se llama el *Plano Multilingüe Básico (BMP)*. El BMP incluso contiene símbolos divertidos como este genial muñeco de nieve: ☃ (U+2603).

Hay muchos otros caracteres Unicode extendidos más allá de este conjunto BMP, que van hasta `0x10FFFF`. Estos símbolos frecuentemente se denominan símbolos *astrales*, ya que ese es el nombre dado al conjunto de 16 *planos* (es decir, capas/agrupaciones) de caracteres más allá del BMP. Ejemplos de símbolos astrales incluyen 𝄞 (U+1D11E) y 💩 (U+1F4A9).

Antes de ES6, las cadenas de JavaScript podían especificar caracteres Unicode usando escapado Unicode, como:

```js
var snowman = "\u2603";
console.log( snowman );			// "☃"
```

Sin embargo, el escapado Unicode `\uXXXX` solo soporta cuatro caracteres hexadecimales, así que solo puedes representar el conjunto de caracteres BMP de esta manera. Para representar un carácter astral usando escapado Unicode antes de ES6, necesitas usar un *par sustituto* -- básicamente dos caracteres escapados Unicode especialmente calculados lado a lado, que JS interpreta juntos como un solo carácter astral:

```js
var gclef = "\uD834\uDD1E";
console.log( gclef );			// "𝄞"
```

A partir de ES6, ahora tenemos una nueva forma de escapado Unicode (en cadenas y expresiones regulares), llamada escapado de *punto de código* Unicode:

```js
var gclef = "\u{1D11E}";
console.log( gclef );			// "𝄞"
```

Como puedes ver, la diferencia es la presencia de los `{ }` en la secuencia de escape, lo que permite que contenga cualquier número de caracteres hexadecimales. Porque solo necesitas seis para representar el valor de punto de código más alto posible en Unicode (es decir, 0x10FFFF), esto es suficiente.

### Operaciones de Cadena Conscientes de Unicode

Por defecto, las operaciones y métodos de cadena de JavaScript no son sensibles a los símbolos astrales en los valores de cadena. Así que tratan cada carácter BMP individualmente, incluso las dos mitades sustituto que componen un carácter astral individual. Considera:

```js
var snowman = "☃";
snowman.length;					// 1

var gclef = "𝄞";
gclef.length;					// 2
```

Entonces, ¿cómo calculamos con precisión la longitud de tal cadena? En este escenario, el siguiente truco funcionará:

```js
var gclef = "𝄞";

[...gclef].length;				// 1
Array.from( gclef ).length;		// 1
```

Recuerda de la sección "Bucles `for..of`" anteriormente en este capítulo que las cadenas ES6 tienen iteradores incorporados. Este iterador resulta ser consciente de Unicode, lo que significa que automáticamente dará como resultado un símbolo astral como un solo valor. Aprovechamos eso usando el operador spread `...` en un literal de array, que crea un array de los símbolos de la cadena. Luego simplemente inspeccionamos la longitud de ese array resultante. `Array.from(..)` de ES6 hace básicamente lo mismo que `[...XYZ]`, pero cubriremos esa utilidad en detalle en el Capítulo 6.

**Advertencia:** Debe notarse que construir y agotar un iterador solo para obtener la longitud de una cadena es bastante costoso en rendimiento, relativamente hablando, comparado con lo que una utilidad/propiedad nativa teóricamente optimizada haría.

Desafortunadamente, la respuesta completa no es tan simple o directa. Además de los pares sustitutos (de los que el iterador de cadena se encarga), hay puntos de código Unicode especiales que se comportan de maneras especiales diferentes, lo cual es mucho más difícil de tener en cuenta. Por ejemplo, hay un conjunto de puntos de código que modifican el carácter adyacente anterior, conocidos como *Marcas Diacríticas de Combinación*.

Considera estas dos salidas de cadena:

```js
console.log( s1 );				// "é"
console.log( s2 );				// "é"
```

Se ven iguales, ¡pero no lo son! Así es como creamos `s1` y `s2`:

```js
var s1 = "\xE9",
	s2 = "e\u0301";
```

Como probablemente puedas adivinar, nuestro truco de `length` anterior no funciona con `s2`:

```js
[...s1].length;					// 1
[...s2].length;					// 2
```

Entonces, ¿qué podemos hacer? En este caso, podemos realizar una *normalización Unicode* sobre el valor antes de consultar su longitud, usando la utilidad de ES6 `String#normalize(..)` (que cubriremos más en el Capítulo 6):

```js
var s1 = "\xE9",
	s2 = "e\u0301";

s1.normalize().length;			// 1
s2.normalize().length;			// 1

s1 === s2;						// false
s1 === s2.normalize();			// true
```

Esencialmente, `normalize(..)` toma una secuencia como `"e\u0301"` y la normaliza a `"\xE9"`. La normalización puede incluso combinar múltiples marcas de combinación adyacentes si hay un carácter Unicode adecuado al que se combinan:

```js
var s1 = "o\u0302\u0300",
	s2 = s1.normalize(),
	s3 = "ồ";

s1.length;						// 3
s2.length;						// 1
s3.length;						// 1

s2 === s3;						// true
```

Desafortunadamente, la normalización tampoco es completamente perfecta aquí. Si tienes múltiples marcas de combinación modificando un solo carácter, puede que no obtengas el conteo de longitud que esperarías, porque puede que no haya un solo carácter normalizado definido que represente la combinación de todas las marcas. Por ejemplo:

```js
var s1 = "e\u0301\u0330";

console.log( s1 );				// "ḛ́"

s1.normalize().length;			// 2
```

Cuanto más profundices en esta madriguera, más te darás cuenta de que es difícil obtener una definición precisa para "longitud". Lo que vemos visualmente renderizado como un solo carácter -- más precisamente llamado un *grafema* -- no siempre se relaciona estrictamente con un solo "carácter" en el sentido de procesamiento del programa.

**Consejo:** Si quieres ver qué tan profunda va esta madriguera, consulta el algoritmo "Grapheme Cluster Boundaries" (http://www.Unicode.org/reports/tr29/#Grapheme_Cluster_Boundaries).

### Posicionamiento de Caracteres

Similar a las complicaciones de longitud, ¿qué significa realmente preguntar "cuál es el carácter en la posición 2"? La respuesta ingenua pre-ES6 viene de `charAt(..)`, que no respetará la atomicidad de un carácter astral, ni tendrá en cuenta las marcas de combinación.

Considera:

```js
var s1 = "abc\u0301d",
	s2 = "ab\u0107d",
	s3 = "ab\u{1d49e}d";

console.log( s1 );				// "abćd"
console.log( s2 );				// "abćd"
console.log( s3 );				// "ab𝒞d"

s1.charAt( 2 );					// "c"
s2.charAt( 2 );					// "ć"
s3.charAt( 2 );					// "" <-- sustituto no imprimible
s3.charAt( 3 );					// "" <-- sustituto no imprimible
```

Entonces, ¿nos da ES6 una versión de `charAt(..)` consciente de Unicode? Desafortunadamente, no. Al momento de escribir esto, hay una propuesta para tal utilidad que está bajo consideración para post-ES6.

Pero con lo que exploramos en la sección anterior (¡y por supuesto con las limitaciones allí mencionadas!), podemos improvisar una respuesta ES6:

```js
var s1 = "abc\u0301d",
	s2 = "ab\u0107d",
	s3 = "ab\u{1d49e}d";

[...s1.normalize()][2];			// "ć"
[...s2.normalize()][2];			// "ć"
[...s3.normalize()][2];			// "𝒞"
```

**Advertencia:** Recordatorio de una advertencia anterior: construir y agotar un iterador cada vez que quieras acceder a un solo carácter es... no muy ideal, en términos de rendimiento. Esperemos que obtengamos una utilidad incorporada y optimizada para esto pronto, post-ES6.

¿Qué hay de una versión consciente de Unicode de la utilidad `charCodeAt(..)`? ES6 nos da `codePointAt(..)`:

```js
var s1 = "abc\u0301d",
	s2 = "ab\u0107d",
	s3 = "ab\u{1d49e}d";

s1.normalize().codePointAt( 2 ).toString( 16 );
// "107"

s2.normalize().codePointAt( 2 ).toString( 16 );
// "107"

s3.normalize().codePointAt( 2 ).toString( 16 );
// "1d49e"
```

¿Y en la otra dirección? Una versión consciente de Unicode de `String.fromCharCode(..)` es `String.fromCodePoint(..)` de ES6:

```js
String.fromCodePoint( 0x107 );		// "ć"

String.fromCodePoint( 0x1d49e );	// "𝒞"
```

Entonces espera, ¿podemos simplemente combinar `String.fromCodePoint(..)` y `codePointAt(..)` para obtener una mejor versión de un `charAt(..)` consciente de Unicode de antes? ¡Sí!

```js
var s1 = "abc\u0301d",
	s2 = "ab\u0107d",
	s3 = "ab\u{1d49e}d";

String.fromCodePoint( s1.normalize().codePointAt( 2 ) );
// "ć"

String.fromCodePoint( s2.normalize().codePointAt( 2 ) );
// "ć"

String.fromCodePoint( s3.normalize().codePointAt( 2 ) );
// "𝒞"
```

Hay bastantes otros métodos de cadena que no hemos abordado aquí, incluyendo `toUpperCase()`, `toLowerCase()`, `substring(..)`, `indexOf(..)`, `slice(..)`, y una docena más. Ninguno de estos ha sido cambiado o aumentado para tener consciencia completa de Unicode, así que deberías tener mucho cuidado -- ¡probablemente simplemente evítalos! -- cuando trabajes con cadenas que contengan símbolos astrales.

También hay varios métodos de cadena que usan expresiones regulares para su comportamiento, como `replace(..)` y `match(..)`. Afortunadamente, ES6 trae consciencia Unicode a las expresiones regulares, como cubrimos en "Bandera Unicode" anteriormente en este capítulo.

OK, ¡ahí lo tenemos! El soporte de cadenas Unicode de JavaScript es significativamente mejor comparado con pre-ES6 (aunque todavía no es perfecto) con las diversas adiciones que acabamos de cubrir.

### Nombres de Identificadores Unicode

Unicode también puede ser usado en nombres de identificadores (variables, propiedades, etc.). Antes de ES6, podías hacer esto con escapados Unicode, como:

```js
var \u03A9 = 42;

// lo mismo que: var Ω = 42;
```

A partir de ES6, también puedes usar la sintaxis de escapado de punto de código explicada anteriormente:

```js
var \u{2B400} = 42;

// lo mismo que: var 𫐀 = 42;
```

Hay un conjunto complejo de reglas sobre exactamente qué caracteres Unicode están permitidos. Además, algunos solo están permitidos si no son el primer carácter del nombre del identificador.

**Nota:** Mathias Bynens tiene una gran publicación (https://mathiasbynens.be/notes/javascript-identifiers-es6) sobre todos los detalles minuciosos.

Las razones para usar tales caracteres inusuales en nombres de identificadores son bastante raras y académicas. Típicamente no te servirá mejor escribir código que dependa de estas capacidades esotéricas.

## Symbols

Con ES6, por primera vez en bastante tiempo, un nuevo tipo primitivo ha sido añadido a JavaScript: el `symbol`. A diferencia de los otros tipos primitivos, sin embargo, los symbols no tienen una forma literal.

Así es como creas un symbol:

```js
var sym = Symbol( "some optional description" );

typeof sym;		// "symbol"
```

Algunas cosas a notar:

* No puedes y no deberías usar `new` con `Symbol(..)`. No es un constructor, ni estás produciendo un objeto.
* El parámetro pasado a `Symbol(..)` es opcional. Si se pasa, debería ser una cadena que dé una descripción amigable para el propósito del symbol.
* La salida de `typeof` es un nuevo valor (`"symbol"`) que es la manera principal de identificar un symbol.

La descripción, si se proporciona, se usa únicamente para la representación de stringificación del symbol:

```js
sym.toString();		// "Symbol(some optional description)"
```

Similar a cómo los valores primitivos de cadena no son instancias de `String`, los symbols tampoco son instancias de `Symbol`. Si, por alguna razón, quieres construir una forma de objeto envoltorio encapsulado de un valor symbol, puedes hacer lo siguiente:

```js
sym instanceof Symbol;		// false

var symObj = Object( sym );
symObj instanceof Symbol;	// true

symObj.valueOf() === sym;	// true
```

**Nota:** `symObj` en este fragmento es intercambiable con `sym`; cualquiera de las formas puede usarse en todos los lugares donde se utilizan symbols. No hay mucha razón para usar la forma de objeto envoltorio encapsulado (`symObj`) en lugar de la forma primitiva (`sym`). Siguiendo consejos similares para otros primitivos, probablemente sea mejor preferir `sym` sobre `symObj`.

El valor interno de un symbol en sí -- referido como su `name` -- está oculto del código y no puede obtenerse. Puedes pensar en este valor de symbol como un valor de cadena automáticamente generado, único (dentro de tu aplicación).

Pero si el valor está oculto e inobtenible, ¿cuál es el punto de tener un symbol en absoluto?

El punto principal de un symbol es crear un valor similar a una cadena que no pueda colisionar con ningún otro valor. Así que, por ejemplo, considera usar un symbol como una constante representando un nombre de evento:

```js
const EVT_LOGIN = Symbol( "event.login" );
```

Luego usarías `EVT_LOGIN` en lugar de un literal de cadena genérico como `"event.login"`:

```js
evthub.listen( EVT_LOGIN, function(data){
	// ..
} );
```

El beneficio aquí es que `EVT_LOGIN` contiene un valor que no puede ser duplicado (accidentalmente o de otra manera) por ningún otro valor, así que es imposible que haya confusión sobre qué evento se está despachando o manejando.

**Nota:** Bajo el capó, la utilidad `evthub` asumida en el fragmento anterior casi con certeza estaría usando el valor del symbol del argumento `EVT_LOGIN` directamente como la propiedad/clave en algún objeto interno (hash) que rastrea manejadores de eventos. Si `evthub` necesitara usar el valor del symbol como una cadena real, necesitaría convertir explícitamente con `String(..)` o `toString()`, ya que la coerción implícita de cadena de symbols no está permitida.

Puedes usar un symbol directamente como un nombre/clave de propiedad en un objeto, como una propiedad especial que quieres tratar como oculta o meta en uso. Es importante saber que aunque tienes la intención de tratarla como tal, *en realidad* no es una propiedad oculta o intocable.

Considera este módulo que implementa el comportamiento del patrón *singleton* -- es decir, solo permite que se cree una vez:

```js
const INSTANCE = Symbol( "instance" );

function HappyFace() {
	if (HappyFace[INSTANCE]) return HappyFace[INSTANCE];

	function smile() { .. }

	return HappyFace[INSTANCE] = {
		smile: smile
	};
}

var me = HappyFace(),
	you = HappyFace();

me === you;			// true
```

El valor del symbol `INSTANCE` aquí es una propiedad especial, casi oculta, tipo meta almacenada estáticamente en el objeto función `HappyFace()`.

Alternativamente podría haber sido una propiedad simple y normal como `__instance`, y el comportamiento habría sido idéntico. El uso de un symbol simplemente mejora el estilo de metaprogramación, manteniendo esta propiedad `INSTANCE` separada de cualquier otra propiedad normal.

### Registro de Symbols

Una ligera desventaja de usar symbols como en los últimos ejemplos es que las variables `EVT_LOGIN` e `INSTANCE` tenían que estar almacenadas en un alcance externo (quizás incluso el alcance global), o de otra manera almacenadas de alguna forma en una ubicación públicamente accesible, para que todas las partes del código que necesiten usar los symbols puedan acceder a ellos.

Para ayudar en la organización del código con acceso a estos symbols, puedes crear valores de symbol con el *registro global de symbols*. Por ejemplo:

```js
const EVT_LOGIN = Symbol.for( "event.login" );

console.log( EVT_LOGIN );		// Symbol(event.login)
```

Y:

```js
function HappyFace() {
	const INSTANCE = Symbol.for( "instance" );

	if (HappyFace[INSTANCE]) return HappyFace[INSTANCE];

	// ..

	return HappyFace[INSTANCE] = { .. };
}
```

`Symbol.for(..)` busca en el registro global de symbols para ver si ya hay un symbol almacenado con el texto de descripción proporcionado, y lo devuelve si es así. Si no, crea uno para devolver. En otras palabras, el registro global de symbols trata los valores de symbols, por texto de descripción, como singletons en sí mismos.

Pero eso también significa que cualquier parte de tu aplicación puede recuperar el symbol del registro usando `Symbol.for(..)`, siempre que se use el nombre de descripción correspondiente.

Irónicamente, los symbols están básicamente destinados a reemplazar el uso de *cadenas mágicas* (valores de cadena arbitrarios con significado especial) en tu aplicación. ¡Pero tú precisamente usas valores de cadena de descripción *mágicos* para identificarlos/localizarlos de manera única en el registro global de symbols!

Para evitar colisiones accidentales, probablemente querrás hacer las descripciones de tus symbols bastante únicas. Una manera fácil de hacer eso es incluir información de prefijo/contexto/espacio de nombres en ellas.

Por ejemplo, considera una utilidad como la siguiente:

```js
function extractValues(str) {
	var key = Symbol.for( "extractValues.parse" ),
		re = extractValues[key] ||
			/[^=&]+?=([^&]+?)(?=&|$)/g,
		values = [], match;

	while (match = re.exec( str )) {
		values.push( match[1] );
	}

	return values;
}
```

Usamos el valor de cadena mágica `"extractValues.parse"` porque es bastante improbable que cualquier otro symbol en el registro colisione con esa descripción.

Si un usuario de esta utilidad quiere sobrescribir la expresión regular de parseo, también puede usar el registro de symbols:

```js
extractValues[Symbol.for( "extractValues.parse" )] =
	/..some pattern../g;

extractValues( "..some string.." );
```

Aparte de la asistencia que el registro de symbols proporciona para almacenar globalmente estos valores, todo lo que estamos viendo aquí podría haberse hecho simplemente usando la cadena mágica `"extractValues.parse"` como la clave, en lugar del symbol. Las mejoras existen más al nivel de metaprogramación que al nivel funcional.

Puedes tener ocasión de usar un valor de symbol que ha sido almacenado en el registro para buscar bajo qué texto de descripción (clave) está almacenado. Por ejemplo, puedes necesitar señalar a otra parte de tu aplicación cómo localizar un symbol en el registro porque no puedes pasar el valor del symbol en sí.

Puedes recuperar el texto de descripción (clave) de un symbol registrado usando `Symbol.keyFor(..)`:

```js
var s = Symbol.for( "something cool" );

var desc = Symbol.keyFor( s );
console.log( desc );			// "something cool"

// obtener el symbol del registro de nuevo
var s2 = Symbol.for( desc );

s2 === s;						// true
```

### Symbols como Propiedades de Objeto

Si un symbol se usa como propiedad/clave de un objeto, se almacena de una manera especial para que la propiedad no aparezca en una enumeración normal de las propiedades del objeto:

```js
var o = {
	foo: 42,
	[ Symbol( "bar" ) ]: "hello world",
	baz: true
};

Object.getOwnPropertyNames( o );	// [ "foo","baz" ]
```

Para recuperar las propiedades symbol de un objeto:

```js
Object.getOwnPropertySymbols( o );	// [ Symbol(bar) ]
```

Esto deja claro que una propiedad symbol no está realmente oculta ni es inaccesible, ya que siempre puedes verla en la lista de `Object.getOwnPropertySymbols(..)`.

#### Symbols Incorporados

ES6 viene con un número de symbols incorporados predefinidos que exponen varios meta comportamientos en valores de objetos JavaScript. Sin embargo, estos symbols *no* están registrados en el registro global de symbols, como uno podría esperar.

En cambio, están almacenados como propiedades en el objeto función `Symbol`. Por ejemplo, en la sección "`for..of`" anteriormente en este capítulo, introdujimos el valor `Symbol.iterator`:

```js
var a = [1,2,3];

a[Symbol.iterator];			// native function
```

La especificación usa la notación de prefijo `@@` para referirse a los symbols incorporados, los más comunes siendo: `@@iterator`, `@@toStringTag`, `@@toPrimitive`. Varios otros también están definidos, aunque probablemente no se usarán tan frecuentemente.

**Nota:** Ver "Símbolos Bien Conocidos" en el Capítulo 7 para información detallada sobre cómo estos symbols incorporados se usan para propósitos de metaprogramación.
