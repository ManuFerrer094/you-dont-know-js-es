## Generators

Todas las funciones se ejecutan hasta completarse, ¿verdad? En otras palabras, una vez que una función comienza a ejecutarse, termina antes de que cualquier otra cosa pueda interrumpirla.

Al menos así ha sido durante toda la historia de JavaScript hasta este punto. A partir de ES6, se introduce una nueva forma algo exótica de función, llamada generator. Un generator puede pausarse a sí mismo a mitad de ejecución, y puede reanudarse inmediatamente o en un momento posterior. Así que claramente no mantiene la garantía de ejecución-hasta-completar que las funciones normales sí tienen.

Además, cada ciclo de pausa/reanudación a mitad de ejecución es una oportunidad para paso de mensajes bidireccional, donde el generator puede devolver un valor, y el código controlador que lo reanuda puede enviar un valor de vuelta.

Al igual que con los iteradores en la sección anterior, hay múltiples maneras de pensar qué es un generator, o más bien para qué es más útil. No hay una sola respuesta correcta, pero intentaremos considerar varios ángulos.

**Nota:** Ver el título *Async & Performance* de esta serie para más información sobre generators, y también ver el Capítulo 4 de este título actual.

### Sintaxis

La función generator se declara con esta nueva sintaxis:

```js
function *foo() {
	// ..
}
```

La posición del `*` no es funcionalmente relevante. La misma declaración podría escribirse como cualquiera de las siguientes:

```js
function *foo()  { .. }
function* foo()  { .. }
function * foo() { .. }
function*foo()   { .. }
..
```

La *única* diferencia aquí es preferencia estilística. La mayoría de la otra literatura parece preferir `function* foo(..) { .. }`. Yo prefiero `function *foo(..) { .. }`, así que es como las presentaré para el resto de este título.

Mi razón es puramente didáctica por naturaleza. En este texto, cuando me refiero a una función generator, usaré `*foo(..)`, en oposición a `foo(..)` para una función normal. Observo que `*foo(..)` coincide más cercanamente con el posicionamiento de `*` de `function *foo(..) { .. }`.

Además, como vimos en el Capítulo 2 con métodos concisos, hay una forma concisa de generator en literales de objeto:

```js
var a = {
	*foo() { .. }
};
```

Diría que con generators concisos, `*foo() { .. }` es bastante más natural que `* foo() { .. }`. Así que eso argumenta aún más a favor de mantener la consistencia con `*foo()`.

La consistencia facilita la comprensión y el aprendizaje.

#### Ejecutar un Generator

Aunque un generator se declara con `*`, aún lo ejecutas como una función normal:

```js
foo();
```

Todavía puedes pasarle argumentos, como en:

```js
function *foo(x,y) {
	// ..
}

foo( 5, 10 );
```

La diferencia principal es que ejecutar un generator, como `foo(5,10)`, en realidad no ejecuta el código en el generator. En cambio, produce un iterador que controlará el generator para ejecutar su código.

Volveremos a esto más adelante en "Control del Iterador", pero brevemente:

```js
function *foo() {
	// ..
}

var it = foo();

// para iniciar/avanzar `*foo()`, llama
// `it.next(..)`
```

#### `yield`

Los generators también tienen una nueva palabra clave que puedes usar dentro de ellos, para señalar el punto de pausa: `yield`. Considera:

```js
function *foo() {
	var x = 10;
	var y = 20;

	yield;

	var z = x + y;
}
```

En este generator `*foo()`, las operaciones en las primeras dos líneas se ejecutarían al principio, luego `yield` pausaría el generator. Si se reanuda y cuando se reanude, se ejecutaría la última línea de `*foo()`. `yield` puede aparecer cualquier número de veces (¡o ninguna, técnicamente!) en un generator.

Incluso puedes poner `yield` dentro de un bucle, y puede representar un punto de pausa repetido. De hecho, un bucle que nunca se completa simplemente significa un generator que nunca se completa, lo cual es completamente válido, y a veces completamente lo que necesitas.

`yield` no es solo un punto de pausa. Es una expresión que envía un valor al pausar el generator. Aquí hay un bucle `while..true` en un generator que por cada iteración produce (`yield`) un nuevo número aleatorio:

```js
function *foo() {
	while (true) {
		yield Math.random();
	}
}
```

La expresión `yield ..` no solo envía un valor -- `yield` sin un valor es lo mismo que `yield undefined` -- sino que también recibe (es decir, es reemplazada por) el valor eventual de reanudación. Considera:

```js
function *foo() {
	var x = yield 10;
	console.log( x );
}
```

Este generator primero producirá (`yield`) el valor `10` al pausarse. Cuando reanudes el generator -- usando el `it.next(..)` al que nos referimos antes -- cualquier valor (si lo hay) con el que lo reanudes reemplazará/completará toda la expresión `yield 10`, lo que significa que ese valor será asignado a la variable `x`.

Una expresión `yield ..` puede aparecer en cualquier lugar donde una expresión normal puede. Por ejemplo:

```js
function *foo() {
	var arr = [ yield 1, yield 2, yield 3 ];
	console.log( arr, yield 4 );
}
```

`*foo()` aquí tiene cuatro expresiones `yield ..`. Cada `yield` resulta en que el generator se pause para esperar un valor de reanudación que luego se usa en los varios contextos de expresión.

`yield` técnicamente no es un operador, aunque cuando se usa como `yield 1` ciertamente lo parece. Porque `yield` puede usarse por sí solo como en `var x = yield;`, pensarlo como un operador a veces puede ser confuso.

Técnicamente, `yield ..` tiene la misma "precedencia de expresión" -- similar conceptualmente a la precedencia de operadores -- que una expresión de asignación como `a = 3`. Eso significa que `yield ..` puede aparecer básicamente en cualquier lugar donde `a = 3` pueda aparecer válidamente.

Ilustremos la simetría:

```js
var a, b;

a = 3;					// válido
b = 2 + a = 3;			// inválido
b = 2 + (a = 3);		// válido

yield 3;				// válido
a = 2 + yield 3;		// inválido
a = 2 + (yield 3);		// válido
```

**Nota:** Si lo piensas, tiene una especie de sentido conceptual que una expresión `yield ..` se comporte de manera similar a una expresión de asignación. Cuando una expresión `yield` pausada se reanuda, se completa/reemplaza por el valor de reanudación de una manera que no es terriblemente distinta a ser "asignada" ese valor.

La conclusión: si necesitas que `yield ..` aparezca en una posición donde una asignación como `a = 3` no sería en sí permitida, necesita estar envuelta en un `( )`.

Debido a la baja precedencia de la palabra clave `yield`, casi cualquier expresión después de un `yield ..` será computada primero antes de ser enviada con `yield`. Solo el operador spread `...` y el operador coma `,` tienen menor precedencia, lo que significa que se enlazarían después de que el `yield` haya sido evaluado.

Así que al igual que con múltiples operadores en sentencias normales, otro caso donde `( )` podría ser necesario es para sobrescribir (elevar) la baja precedencia de `yield`, como la diferencia entre estas expresiones:

```js
yield 2 + 3;			// lo mismo que `yield (2 + 3)`

(yield 2) + 3;			// `yield 2` primero, luego `+ 3`
```

Al igual que la asignación `=`, `yield` también es "asociativo a la derecha", lo que significa que múltiples expresiones `yield` en sucesión se tratan como si estuvieran agrupadas con `( .. )` de derecha a izquierda. Así que, `yield yield yield 3` se trata como `yield (yield (yield 3))`. Una interpretación "asociativa a la izquierda" como `((yield) yield) yield 3` no tendría sentido.

Al igual que con operadores, es buena idea usar agrupación `( .. )`, incluso si no es estrictamente necesaria, para desambiguar tu intención si `yield` se combina con otros operadores u otros `yield`s.

**Nota:** Ver el título *Types & Grammar* de esta serie para más información sobre precedencia de operadores y asociatividad.

#### `yield *`

De la misma manera que el `*` hace que una declaración `function` se convierta en una declaración de generator `function *`, un `*` hace que `yield` se convierta en `yield *`, que es un mecanismo muy diferente, llamado *delegación de yield*. Gramaticalmente, `yield *..` se comportará igual que un `yield ..`, como se discutió en la sección anterior.

`yield * ..` requiere un iterable; luego invoca el iterador de ese iterable, y delega el control del generator anfitrión a ese iterador hasta que se agota. Considera:

```js
function *foo() {
	yield *[1,2,3];
}
```

**Nota:** Al igual que con la posición del `*` en la declaración de un generator (discutido antes), el posicionamiento del `*` en expresiones `yield *` es estilísticamente a tu elección. La mayoría de la otra literatura prefiere `yield* ..`, pero yo prefiero `yield *..`, por razones muy simétricas como ya se discutió.

El valor `[1,2,3]` produce un iterador que recorrerá sus valores, así que el generator `*foo()` producirá esos valores al ser consumido. Otra manera de ilustrar el comportamiento es en la delegación de yield a otro generator:

```js
function *foo() {
	yield 1;
	yield 2;
	yield 3;
}

function *bar() {
	yield *foo();
}
```

El iterador producido cuando `*bar()` llama a `*foo()` es delegado a través de `yield *`, lo que significa que cualquier valor(es) que `*foo()` produce será(n) producido(s) por `*bar()`.

Mientras que con `yield ..` el valor de completitud de la expresión viene de reanudar el generator con `it.next(..)`, el valor de completitud de la expresión `yield *..` viene del valor de retorno (si lo hay) del iterador al que se delegó.

Los iteradores incorporados generalmente no tienen valores de retorno, como cubrimos al final de la sección "Bucle de Iterador" anteriormente en este capítulo. Pero si defines tu propio iterador personalizado (o generator), puedes diseñarlo para hacer `return` de un valor, que `yield *..` capturaría:

```js
function *foo() {
	yield 1;
	yield 2;
	yield 3;
	return 4;
}

function *bar() {
	var x = yield *foo();
	console.log( "x:", x );
}

for (var v of bar()) {
	console.log( v );
}
// 1 2 3
// x: 4
```

Mientras que los valores `1`, `2` y `3` son producidos (`yield`) fuera de `*foo()` y luego fuera de `*bar()`, el valor `4` retornado de `*foo()` es el valor de completitud de la expresión `yield *foo()`, que luego se asigna a `x`.

Porque `yield *` puede llamar a otro generator (mediante delegación a su iterador), también puede hacer una especie de recursión de generator llamándose a sí mismo:

```js
function *foo(x) {
	if (x < 3) {
		x = yield *foo( x + 1 );
	}
	return x * 2;
}

foo( 1 );
```

El resultado de `foo(1)` y luego llamar al `next()` del iterador para ejecutarlo a través de sus pasos recursivos será `24`. La primera ejecución de `*foo(..)` tiene `x` con valor `1`, que es `x < 3`. `x + 1` se pasa recursivamente a `*foo(..)`, así que `x` es entonces `2`. Una llamada recursiva más resulta en un `x` de `3`.

Ahora, porque `x < 3` falla, la recursión se detiene, y `return 3 * 2` da `6` de vuelta a la expresión `yield *..` de la llamada anterior, que luego se asigna a `x`. Otro `return 6 * 2` devuelve `12` de vuelta al `x` de la llamada anterior. Finalmente `12 * 2`, o `24`, es retornado de la ejecución completada del generator `*foo(..)`.

### Control del Iterador

Antes, introdujimos brevemente el concepto de que los generators son controlados por iteradores. Profundicemos completamente en eso ahora.

Recuerda el `*foo(..)` recursivo de la sección anterior. Así es como lo ejecutaríamos:

```js
function *foo(x) {
	if (x < 3) {
		x = yield *foo( x + 1 );
	}
	return x * 2;
}

var it = foo( 1 );
it.next();				// { value: 24, done: true }
```

En este caso, el generator realmente nunca pausa, ya que no hay expresión `yield ..`. En cambio, `yield *` simplemente mantiene el paso de iteración actual en marcha a través de la llamada recursiva. Así que, solo una llamada a la función `next()` del iterador ejecuta completamente el generator.

Ahora consideremos un generator que tendrá múltiples pasos y por lo tanto múltiples valores producidos:

```js
function *foo() {
	yield 1;
	yield 2;
	yield 3;
}
```

Ya sabemos que podemos consumir un iterador, incluso uno adjunto a un generator como `*foo()`, con un bucle `for..of`:

```js
for (var v of foo()) {
	console.log( v );
}
// 1 2 3
```

**Nota:** El bucle `for..of` requiere un iterable. Una referencia a función generator (como `foo`) por sí sola no es un iterable; debes ejecutarla con `foo()` para obtener el iterador (que es también un iterable, como explicamos anteriormente en este capítulo). Teóricamente podrías extender el `GeneratorPrototype` (el prototipo de todas las funciones generator) con una función `Symbol.iterator` que esencialmente solo hace `return this()`. Eso haría la referencia `foo` en sí un iterable, lo que significa que `for (var v of foo) { .. }` (nota que no hay `()` en `foo`) funcionaría.

En cambio, iteremos el generator manualmente:

```js
function *foo() {
	yield 1;
	yield 2;
	yield 3;
}

var it = foo();

it.next();				// { value: 1, done: false }
it.next();				// { value: 2, done: false }
it.next();				// { value: 3, done: false }

it.next();				// { value: undefined, done: true }
```

Si miras de cerca, hay tres sentencias `yield` y cuatro llamadas `next()`. Eso puede parecer una falta de correspondencia extraña. De hecho, siempre habrá una llamada `next()` más que expresiones `yield`, asumiendo que todas se evalúan y el generator se ejecuta completamente hasta el final.

Pero si lo miras desde la perspectiva opuesta (de adentro hacia afuera en lugar de afuera hacia adentro), la correspondencia entre `yield` y `next()` tiene más sentido.

Recuerda que la expresión `yield ..` será completada por el valor con el que reanudas el generator. Eso significa que el argumento que pasas a `next(..)` completa cualquier expresión `yield ..` que esté actualmente pausada esperando una completitud.

Ilustremos esta perspectiva de esta manera:

```js
function *foo() {
	var x = yield 1;
	var y = yield 2;
	var z = yield 3;
	console.log( x, y, z );
}
```

En este fragmento, cada `yield ..` está enviando un valor (`1`, `2`, `3`), pero más directamente, está pausando el generator para esperar un valor. En otras palabras, es casi como preguntar, "¿Qué valor debería usar aquí? Esperaré la respuesta."

Ahora, así es como controlamos `*foo()` para iniciarlo:

```js
var it = foo();

it.next();				// { value: 1, done: false }
```

Esa primera llamada `next()` está iniciando el generator desde su estado pausado inicial, y ejecutándolo hasta el primer `yield`. En el momento en que llamas ese primer `next()`, no hay expresión `yield ..` esperando una completitud. Si pasaras un valor a esa primera llamada `next()`, actualmente simplemente se descartaría, porque ningún `yield` está esperando recibir tal valor.

**Nota:** Una propuesta temprana para el periodo "beyond ES6" *permitiría* acceder a un valor pasado a una llamada `next(..)` inicial a través de una meta propiedad separada (ver Capítulo 7) dentro del generator.

Ahora, respondamos la pregunta actualmente pendiente, "¿Qué valor debería asignar a `x`?" La responderemos enviando un valor a la *siguiente* llamada `next(..)`:

```js
it.next( "foo" );		// { value: 2, done: false }
```

Ahora, `x` tendrá el valor `"foo"`, pero también hemos hecho una nueva pregunta, "¿Qué valor debería asignar a `y`?" Y respondemos:

```js
it.next( "bar" );		// { value: 3, done: false }
```

Respuesta dada, otra pregunta hecha. Respuesta final:

```js
it.next( "baz" );		// "foo" "bar" "baz"
						// { value: undefined, done: true }
```

Ahora debería ser más claro cómo cada "pregunta" `yield ..` es respondida por la *siguiente* llamada `next(..)`, y así la llamada "extra" `next()` que observamos es siempre simplemente la inicial que arranca todo.

Pongamos todos esos pasos juntos:

```js
var it = foo();

// iniciar el generator
it.next();				// { value: 1, done: false }

// responder primera pregunta
it.next( "foo" );		// { value: 2, done: false }

// responder segunda pregunta
it.next( "bar" );		// { value: 3, done: false }

// responder tercera pregunta
it.next( "baz" );		// "foo" "bar" "baz"
						// { value: undefined, done: true }
```

Puedes pensar en un generator como un productor de valores, en cuyo caso cada iteración es simplemente producir un valor a ser consumido.

Pero en un sentido más general, quizás es apropiado pensar en los generators como ejecución de código controlada y progresiva, muy similar al ejemplo de cola `tasks` de la sección anterior "Iteradores Personalizados".

**Nota:** Esa perspectiva es exactamente la motivación de cómo revisitaremos los generators en el Capítulo 4. Específicamente, no hay razón por la que `next(..)` tenga que ser llamado inmediatamente después de que el `next(..)` anterior termine. Mientras el contexto de ejecución interno del generator está pausado, el resto del programa continúa sin bloquearse, incluyendo la capacidad de que acciones asíncronas controlen cuándo se reanuda el generator.

### Completitud Temprana

Como cubrimos antes en este capítulo, el iterador adjunto a un generator soporta los métodos opcionales `return(..)` y `throw(..)`. Ambos tienen el efecto de abortar un generator pausado inmediatamente.

Considera:

```js
function *foo() {
	yield 1;
	yield 2;
	yield 3;
}

var it = foo();

it.next();				// { value: 1, done: false }

it.return( 42 );		// { value: 42, done: true }

it.next();				// { value: undefined, done: true }
```

`return(x)` es algo así como forzar a que un `return x` sea procesado exactamente en ese momento, de modo que obtienes el valor especificado de vuelta inmediatamente. Una vez que un generator se completa, ya sea normalmente o tempranamente como se muestra, ya no procesa más código ni devuelve más valores.

Además de que `return(..)` sea llamable manualmente, también se llama automáticamente al final de la iteración por cualquiera de las construcciones de ES6 que consumen iteradores, como el bucle `for..of` y el operador spread `...`.

El propósito de esta capacidad es para que el generator pueda ser notificado si el código controlador ya no va a iterar sobre él, de modo que pueda quizás hacer tareas de limpieza (liberar recursos, reiniciar estado, etc.). Idéntico a un patrón de limpieza de función normal, la manera principal de lograr esto es usar una cláusula `finally`:

```js
function *foo() {
	try {
		yield 1;
		yield 2;
		yield 3;
	}
	finally {
		console.log( "cleanup!" );
	}
}

for (var v of foo()) {
	console.log( v );
}
// 1 2 3
// cleanup!

var it = foo();

it.next();				// { value: 1, done: false }
it.return( 42 );		// cleanup!
						// { value: 42, done: true }
```

**Advertencia:** ¡No pongas una sentencia `yield` dentro de la cláusula `finally`! Es válido y legal, pero es una idea realmente terrible. Actúa en cierto sentido como diferir la completitud de la llamada `return(..)` que hiciste, ya que cualquier expresión `yield ..` en la cláusula `finally` se respeta para pausar y enviar mensajes; no obtienes inmediatamente un generator completado como se espera. Básicamente no hay buena razón para optar por esa loca *mala parte*, ¡así que evita hacerlo!

Además de que el fragmento anterior muestra cómo `return(..)` aborta el generator mientras aún dispara la cláusula `finally`, también demuestra que un generator produce un iterador completamente nuevo cada vez que se llama. De hecho, puedes usar múltiples iteradores adjuntos al mismo generator de manera concurrente:

```js
function *foo() {
	yield 1;
	yield 2;
	yield 3;
}

var it1 = foo();
it1.next();				// { value: 1, done: false }
it1.next();				// { value: 2, done: false }

var it2 = foo();
it2.next();				// { value: 1, done: false }

it1.next();				// { value: 3, done: false }

it2.next();				// { value: 2, done: false }
it2.next();				// { value: 3, done: false }

it2.next();				// { value: undefined, done: true }
it1.next();				// { value: undefined, done: true }
```

#### Aborto Temprano

En lugar de llamar a `return(..)`, puedes llamar a `throw(..)`. Al igual que `return(x)` es esencialmente inyectar un `return x` en el generator en su punto de pausa actual, llamar a `throw(x)` es esencialmente como inyectar un `throw x` en el punto de pausa.

Aparte del comportamiento de excepción (cubriremos lo que eso significa para las cláusulas `try` en la siguiente sección), `throw(..)` produce el mismo tipo de completitud temprana que aborta la ejecución del generator en su punto de pausa actual. Por ejemplo:

```js
function *foo() {
	yield 1;
	yield 2;
	yield 3;
}

var it = foo();

it.next();				// { value: 1, done: false }

try {
	it.throw( "Oops!" );
}
catch (err) {
	console.log( err );	// Exception: Oops!
}

it.next();				// { value: undefined, done: true }
```

Porque `throw(..)` básicamente inyecta un `throw ..` en reemplazo de la línea `yield 1` del generator, y nada maneja esta excepción, inmediatamente se propaga de vuelta al código que llama, que la maneja con un `try..catch`.

A diferencia de `return(..)`, el método `throw(..)` del iterador nunca se llama automáticamente.

Por supuesto, aunque no se muestra en el fragmento anterior, si una cláusula `try..finally` estuviera esperando dentro del generator cuando llamas a `throw(..)`, la cláusula `finally` recibiría una oportunidad de completarse antes de que la excepción se propague de vuelta al código que llama.

### Manejo de Errores

Como ya hemos insinuado, el manejo de errores con generators puede expresarse con `try..catch`, que funciona en ambas direcciones, entrante y saliente:

```js
function *foo() {
	try {
		yield 1;
	}
	catch (err) {
		console.log( err );
	}

	yield 2;

	throw "Hello!";
}

var it = foo();

it.next();				// { value: 1, done: false }

try {
	it.throw( "Hi!" );	// Hi!
						// { value: 2, done: false }
	it.next();

	console.log( "never gets here" );
}
catch (err) {
	console.log( err );	// Hello!
}
```

Los errores también pueden propagarse en ambas direcciones a través de la delegación `yield *`:

```js
function *foo() {
	try {
		yield 1;
	}
	catch (err) {
		console.log( err );
	}

	yield 2;

	throw "foo: e2";
}

function *bar() {
	try {
		yield *foo();

		console.log( "never gets here" );
	}
	catch (err) {
		console.log( err );
	}
}

var it = bar();

try {
	it.next();			// { value: 1, done: false }

	it.throw( "e1" );	// e1
						// { value: 2, done: false }

	it.next();			// foo: e2
						// { value: undefined, done: true }
}
catch (err) {
	console.log( "never gets here" );
}

it.next();				// { value: undefined, done: true }
```

Cuando `*foo()` llama a `yield 1`, el valor `1` pasa a través de `*bar()` sin cambios, como ya hemos visto.

Pero lo más interesante de este fragmento es que cuando `*foo()` llama a `throw "foo: e2"`, este error se propaga a `*bar()` y es inmediatamente capturado por el bloque `try..catch` de `*bar()`. El error no pasa a través de `*bar()` como lo hizo el valor `1`.

El `catch` de `*bar()` luego hace una salida normal de `err` (`"foo: e2"`) y luego `*bar()` termina normalmente, lo cual es por qué el resultado de iterador `{ value: undefined, done: true }` vuelve de `it.next()`.

Si `*bar()` no tuviera un `try..catch` alrededor de la expresión `yield *..`, el error por supuesto se propagaría completamente hacia afuera, y en el camino aún completaría (abortaría) `*bar()`.

### Transpilar un Generator

¿Es posible representar las capacidades de un generator antes de ES6? Resulta que sí, y hay varias herramientas excelentes que lo hacen, incluyendo más notablemente la herramienta Regenerator de Facebook (https://facebook.github.io/regenerator/).

Pero solo para entender mejor los generators, intentemos convertir manualmente. Básicamente, vamos a crear una máquina de estados simple basada en clausuras.

Mantendremos nuestro generator fuente muy simple:

```js
function *foo() {
	var x = yield 42;
	console.log( x );
}
```

Para empezar, necesitaremos una función llamada `foo()` que podamos ejecutar, la cual necesita devolver un iterador:

```js
function foo() {
	// ..

	return {
		next: function(v) {
			// ..
		}

		// saltaremos `return(..)` y `throw(..)`
	};
}
```

Ahora, necesitamos alguna variable interna para rastrear dónde estamos en los pasos de la lógica de nuestro "generator". La llamaremos `state`. Habrá tres estados: `0` inicialmente, `1` mientras se espera cumplir la expresión `yield`, y `2` una vez que el generator está completo.

Cada vez que se llama a `next(..)`, necesitamos procesar el siguiente paso, y luego incrementar `state`. Por conveniencia, pondremos cada paso en una cláusula `case` de una sentencia `switch`, y la mantendremos en una función interna llamada `nextState(..)` que `next(..)` puede llamar. También, porque `x` es una variable a lo largo del alcance general del "generator", necesita vivir fuera de la función `nextState(..)`.

Aquí está todo junto (obviamente algo simplificado, para mantener la ilustración conceptual más clara):

```js
function foo() {
	function nextState(v) {
		switch (state) {
			case 0:
				state++;

				// la expresión `yield`
				return 42;
			case 1:
				state++;

				// expresión `yield` cumplida
				x = v;
				console.log( x );

				// el `return` implícito
				return undefined;

			// no hay necesidad de manejar el estado `2`
		}
	}

	var state = 0, x;

	return {
		next: function(v) {
			var ret = nextState( v );

			return { value: ret, done: (state == 2) };
		}

		// saltaremos `return(..)` y `throw(..)`
	};
}
```

Y finalmente, probemos nuestro "generator" pre-ES6:

```js
var it = foo();

it.next();				// { value: 42, done: false }

it.next( 10 );			// 10
						// { value: undefined, done: true }
```

Nada mal, ¿eh? Con suerte este ejercicio solidifica en tu mente que los generators en realidad son solo sintaxis simple para lógica de máquina de estados. Eso los hace ampliamente aplicables.

### Usos de Generators

Entonces, ahora que entendemos mucho más profundamente cómo funcionan los generators, ¿para qué son útiles?

Hemos visto dos patrones principales:

* *Producir una serie de valores:* Este uso puede ser simple (por ejemplo, cadenas aleatorias o números incrementados), o puede representar acceso a datos más estructurado (por ejemplo, iterar sobre filas devueltas de una consulta a base de datos).

   De cualquier manera, usamos el iterador para controlar un generator de modo que alguna lógica pueda ser invocada por cada llamada a `next(..)`. Los iteradores normales sobre estructuras de datos simplemente extraen valores sin ninguna lógica de control.
* *Cola de tareas a realizar en serie:* Este uso frecuentemente representa control de flujo para los pasos en un algoritmo, donde cada paso requiere recuperación de datos de alguna fuente externa. El cumplimiento de cada pieza de datos puede ser inmediato, o puede estar retrasado asíncronamente.

   Desde la perspectiva del código dentro del generator, los detalles de sincronía o asincronía en un punto `yield` son totalmente opacos. Además, estos detalles son intencionalmente abstraídos, como para no oscurecer la expresión natural secuencial de pasos con tales complicaciones de implementación. La abstracción también significa que las implementaciones pueden ser intercambiadas/refactorizadas frecuentemente sin tocar el código en el generator en absoluto.

Cuando los generators se ven a la luz de estos usos, se convierten en mucho más que solo una sintaxis diferente o más agradable para una máquina de estados manual. Son una herramienta de abstracción poderosa para organizar y controlar la producción y consumo ordenado de datos.
