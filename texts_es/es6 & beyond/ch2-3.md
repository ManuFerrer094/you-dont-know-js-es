## Extensiones de Literales de Objeto

ES6 añade una cantidad de extensiones de conveniencia importantes al humilde literal de objeto `{ .. }`.

### Propiedades Concisas

Ciertamente estás familiarizado con declarar literales de objeto en esta forma:

```js
var x = 2, y = 3,
	o = {
		x: x,
		y: y
	};
```

Si siempre has sentido redundante decir `x: x` por todos lados, hay buenas noticias. Si necesitas definir una propiedad que tiene el mismo nombre que un identificador léxico, puedes acortarlo de `x: x` a `x`. Considera:

```js
var x = 2, y = 3,
	o = {
		x,
		y
	};
```

### Métodos Concisos

En un espíritu similar a las propiedades concisas que acabamos de examinar, las funciones adjuntas a propiedades en literales de objeto también tienen una forma concisa, por conveniencia.

La forma antigua:

```js
var o = {
	x: function(){
		// ..
	},
	y: function(){
		// ..
	}
}
```

Y a partir de ES6:

```js
var o = {
	x() {
		// ..
	},
	y() {
		// ..
	}
}
```

**Advertencia:** Aunque `x() { .. }` parece ser solo una abreviatura de `x: function(){ .. }`, los métodos concisos tienen comportamientos especiales que sus contrapartes más antiguas no tienen; específicamente, la permisión de `super` (ver "Objeto `super`" más adelante en este capítulo).

Los generators (ver Capítulo 4) también tienen una forma de método conciso:

```js
var o = {
	*foo() { .. }
};
```

#### Concisamente Sin Nombre

Aunque esa abreviatura de conveniencia es bastante atractiva, hay una trampa sutil de la cual estar consciente. Para ilustrar, examinemos código pre-ES6 como el siguiente, que podrías intentar refactorizar para usar métodos concisos:

```js
function runSomething(o) {
	var x = Math.random(),
		y = Math.random();

	return o.something( x, y );
}

runSomething( {
	something: function something(x,y) {
		if (x > y) {
			// llamar recursivamente con `x`
			// e `y` intercambiados
			return something( y, x );
		}

		return y - x;
	}
} );
```

Este código obviamente tonto simplemente genera dos números aleatorios y resta el menor del mayor. Pero lo importante aquí no es lo que hace, sino más bien cómo está definido. Enfoquémonos en el literal de objeto y la definición de función, como vemos aquí:

```js
runSomething( {
	something: function something(x,y) {
		// ..
	}
} );
```

¿Por qué decimos tanto `something:` como `function something`? ¿No es eso redundante? En realidad, no, ambos son necesarios para diferentes propósitos. La propiedad `something` es cómo podemos llamar `o.something(..)`, algo así como su nombre público. Pero el segundo `something` es un nombre léxico para referirse a la función desde dentro de sí misma, para propósitos de recursión.

¿Puedes ver por qué la línea `return something(y,x)` necesita el nombre `something` para referirse a la función? No hay un nombre léxico para el objeto, de modo que pudiera haber dicho `return o.something(y,x)` o algo por el estilo.

Esa es en realidad una práctica bastante común cuando el literal de objeto sí tiene un nombre identificador, como:

```js
var controller = {
	makeRequest: function(..){
		// ..
		controller.makeRequest(..);
	}
};
```

¿Es esta una buena idea? Quizás sí, quizás no. Estás asumiendo que el nombre `controller` siempre apuntará al objeto en cuestión. Pero bien puede que no -- la función `makeRequest(..)` no controla el código externo y por lo tanto no puede forzar que ese sea el caso. Esto podría volver a morderte.

Otros prefieren usar `this` para definir tales cosas:

```js
var controller = {
	makeRequest: function(..){
		// ..
		this.makeRequest(..);
	}
};
```

Eso se ve bien, y debería funcionar si siempre invocas el método como `controller.makeRequest(..)`. Pero ahora tienes una trampa de enlace de `this` si haces algo como:

```js
btn.addEventListener( "click", controller.makeRequest, false );
```

Por supuesto, puedes resolver eso pasando `controller.makeRequest.bind(controller)` como la referencia del manejador para enlazar el evento. Pero puaj -- no es muy atractivo.

¿O qué pasa si tu llamada interna `this.makeRequest(..)` necesita ser hecha desde una función anidada? Tendrás otro peligro de enlace de `this`, que la gente a menudo resuelve con el hack `var self = this`, como:

```js
var controller = {
	makeRequest: function(..){
		var self = this;

		btn.addEventListener( "click", function(){
			// ..
			self.makeRequest(..);
		}, false );
	}
};
```

Más puaj.

**Nota:** Para más información sobre reglas de enlace de `this` y trampas, ver los Capítulos 1-2 del título *this & Object Prototypes* de esta serie.

OK, ¿qué tiene que ver todo esto con los métodos concisos? Recuerda nuestra definición del método `something(..)`:

```js
runSomething( {
	something: function something(x,y) {
		// ..
	}
} );
```

El segundo `something` aquí proporciona un identificador léxico súper conveniente que siempre apuntará a la función en sí, dándonos la referencia perfecta para recursión, enlace/desenlace de eventos, y demás -- sin andar jugando con `this` o tratando de usar una referencia de objeto no confiable.

¡Genial!

Entonces, ahora intentamos refactorizar esa referencia de función a esta forma de método conciso de ES6:

```js
runSomething( {
	something(x,y) {
		if (x > y) {
			return something( y, x );
		}

		return y - x;
	}
} );
```

Parece bien a primera vista, excepto que este código se romperá. La llamada `return something(..)` no encontrará un identificador `something`, así que obtendrás un `ReferenceError`. Ups. ¿Pero por qué?

El fragmento ES6 anterior se interpreta como significando:

```js
runSomething( {
	something: function(x,y){
		if (x > y) {
			return something( y, x );
		}

		return y - x;
	}
} );
```

Mira de cerca. ¿Ves el problema? La definición de método conciso implica `something: function(x,y)`. ¿Ves cómo el segundo `something` del que dependíamos ha sido omitido? En otras palabras, los métodos concisos implican expresiones de función anónimas.

Sí, puaj.

**Nota:** Podrías estar tentado a pensar que las funciones flecha `=>` son una buena solución aquí, pero son igualmente insuficientes, ya que también son expresiones de función anónimas. Las cubriremos en "Funciones Flecha" más adelante en este capítulo.

La noticia parcialmente redentora es que nuestro método conciso `something(x,y)` no será totalmente anónimo. Ver "Nombres de Funciones" en el Capítulo 7 para información sobre las reglas de inferencia de nombres de funciones de ES6. Eso no nos ayudará para nuestra recursión, pero ayuda con la depuración al menos.

Entonces, ¿qué nos queda por concluir sobre los métodos concisos? Son cortos y dulces, y una buena conveniencia. Pero solo deberías usarlos si nunca vas a necesitar que hagan recursión o enlace/desenlace de eventos. De lo contrario, quédate con tus antiguas definiciones de métodos `something: function something(..)`.

Muchos de tus métodos probablemente se beneficiarán de las definiciones de métodos concisos, ¡así que esas son buenas noticias! Solo ten cuidado con los pocos donde hay un peligro de pérdida de nombre.

#### Getter/Setter de ES5

Técnicamente, ES5 definió formas de literales getter/setter, pero no parecían usarse mucho, principalmente debido a la falta de transpiladores para manejar esa nueva sintaxis (la única nueva sintaxis principal añadida en ES5, realmente). Así que aunque no es una nueva característica de ES6, refrescaremos brevemente esa forma, ya que probablemente será mucho más útil con ES6 en adelante.

Considera:

```js
var o = {
	__id: 10,
	get id() { return this.__id++; },
	set id(v) { this.__id = v; }
}

o.id;			// 10
o.id;			// 11
o.id = 20;
o.id;			// 20

// y:
o.__id;			// 21
o.__id;			// 21 -- ¡todavía!
```

Estas formas de literales getter y setter también están presentes en clases; ver Capítulo 3.

**Advertencia:** Puede no ser obvio, pero el setter literal debe tener exactamente un parámetro declarado; omitirlo o listar otros es sintaxis ilegal. El único parámetro requerido *puede* usar desestructuración y valores por defecto (por ejemplo, `set id({ id: v = 0 }) { .. }`), pero el gather/rest `...` no está permitido (`set id(...v) { .. }`).

### Nombres de Propiedad Computados

Probablemente has estado en una situación como el siguiente fragmento, donde tienes uno o más nombres de propiedad que vienen de algún tipo de expresión y por lo tanto no pueden ponerse en el literal de objeto:

```js
var prefix = "user_";

var o = {
	baz: function(..){ .. }
};

o[ prefix + "foo" ] = function(..){ .. };
o[ prefix + "bar" ] = function(..){ .. };
..
```

ES6 añade una sintaxis a la definición de literal de objeto que te permite especificar una expresión que debería ser computada, cuyo resultado es el nombre de propiedad asignado. Considera:

```js
var prefix = "user_";

var o = {
	baz: function(..){ .. },
	[ prefix + "foo" ]: function(..){ .. },
	[ prefix + "bar" ]: function(..){ .. }
	..
};
```

Cualquier expresión válida puede aparecer dentro del `[ .. ]` que se sitúa en la posición del nombre de propiedad de la definición del literal de objeto.

Probablemente el uso más común de nombres de propiedad computados será con `Symbol`s (que cubrimos en "Symbols" más adelante en este capítulo), como:

```js
var o = {
	[Symbol.toStringTag]: "really cool thing",
	..
};
```

`Symbol.toStringTag` es un valor especial incorporado, que evaluamos con la sintaxis `[ .. ]`, para que podamos asignar el valor `"really cool thing"` al nombre de propiedad especial.

Los nombres de propiedad computados también pueden aparecer como el nombre de un método conciso o un generator conciso:

```js
var o = {
	["f" + "oo"]() { .. }	// método conciso computado
	*["b" + "ar"]() { .. }	// generator conciso computado
};
```

### Establecer `[[Prototype]]`

No cubriremos prototipos en detalle aquí, así que para más información, ver el título *this & Object Prototypes* de esta serie.

A veces será útil asignar el `[[Prototype]]` de un objeto al mismo tiempo que declaras su literal de objeto. Lo siguiente ha sido una extensión no estándar en muchos motores JS durante un tiempo, pero está estandarizado a partir de ES6:

```js
var o1 = {
	// ..
};

var o2 = {
	__proto__: o1,
	// ..
};
```

`o2` se declara con un literal de objeto normal, pero también está enlazado por `[[Prototype]]` a `o1`. El nombre de propiedad `__proto__` aquí también puede ser un string `"__proto__"`, pero nota que *no puede* ser el resultado de un nombre de propiedad computado (ver la sección anterior).

`__proto__` es controversial, por decir lo menos. Es una extensión propietaria de décadas de antigüedad de JS que finalmente se estandariza, algo a regañadientes parece, en ES6. Muchos desarrolladores sienten que nunca debería usarse. De hecho, está en el "Anexo B" de ES6, que es la sección que lista cosas que JS siente que tiene que estandarizar solo por razones de compatibilidad.

**Advertencia:** Aunque estoy respaldando estrechamente `__proto__` como una clave en una definición de literal de objeto, definitivamente no respaldo usarlo en su forma de propiedad de objeto, como `o.__proto__`. Esa forma es tanto un getter como un setter (de nuevo por razones de compatibilidad), pero definitivamente hay mejores opciones. Ver el título *this & Object Prototypes* de esta serie para más información.

Para establecer el `[[Prototype]]` de un objeto existente, puedes usar la utilidad de ES6 `Object.setPrototypeOf(..)`. Considera:

```js
var o1 = {
	// ..
};

var o2 = {
	// ..
};

Object.setPrototypeOf( o2, o1 );
```

**Nota:** Discutiremos `Object` de nuevo en el Capítulo 6. "Función Estática `Object.setPrototypeOf(..)`" proporciona detalles adicionales sobre `Object.setPrototypeOf(..)`. También ver "Función Estática `Object.assign(..)`" para otra forma que relaciona `o2` prototípicamente a `o1`.

### Objeto `super`

`super` típicamente se piensa como algo solo relacionado con clases. Sin embargo, debido a la naturaleza de objetos-sin-clases-con-prototipos de JS, `super` es igualmente efectivo, y casi igual en comportamiento, con los métodos concisos de objetos simples.

Considera:

```js
var o1 = {
	foo() {
		console.log( "o1:foo" );
	}
};

var o2 = {
	foo() {
		super.foo();
		console.log( "o2:foo" );
	}
};

Object.setPrototypeOf( o2, o1 );

o2.foo();		// o1:foo
				// o2:foo
```

**Advertencia:** `super` solo está permitido en métodos concisos, no en propiedades de expresión de función regular. También solo está permitido en forma `super.XXX` (para acceso a propiedad/método), no en forma `super()`.

La referencia `super` en el método `o2.foo()` está bloqueada estáticamente a `o2`, y específicamente al `[[Prototype]]` de `o2`. `super` aquí sería básicamente `Object.getPrototypeOf(o2)` -- se resuelve a `o1` por supuesto -- que es cómo encuentra y llama a `o1.foo()`.

Para detalles completos sobre `super`, ver "Clases" en el Capítulo 3.

## Literales de Plantilla

Al comienzo mismo de esta sección, voy a tener que señalar que el nombre de esta característica de ES6 es terriblemente... engañoso, dependiendo de tus experiencias con lo que la palabra *plantilla* significa.

Muchos desarrolladores piensan en las plantillas como piezas de texto renderizables reutilizables, como la capacidad proporcionada por la mayoría de los motores de plantillas (Mustache, Handlebars, etc.). El uso de ES6 de la palabra *plantilla* implicaría algo similar, como una manera de declarar literales de plantilla inline que pueden ser re-renderizados. Sin embargo, esa no es en absoluto la manera correcta de pensar sobre esta característica.

Así que, antes de continuar, estoy renombrando a lo que debería haberse llamado: *literales de cadena interpolada* (o *interpoliterales* para abreviar).

Ya eres bien consciente de declarar literales de cadena con delimitadores `"` o `'`, y también sabes que estas no son *cadenas inteligentes* (como algunos lenguajes tienen), donde los contenidos serían parseados para expresiones de interpolación.

Sin embargo, ES6 introduce un nuevo tipo de literal de cadena, usando el acento grave `` ` `` como delimitador. Estos literales de cadena permiten que expresiones básicas de interpolación de cadenas sean incrustadas, que luego son automáticamente parseadas y evaluadas.

Aquí está la forma antigua pre-ES6:

```js
var name = "Kyle";

var greeting = "Hello " + name + "!";

console.log( greeting );			// "Hello Kyle!"
console.log( typeof greeting );		// "string"
```

Ahora, considera la nueva forma ES6:

```js
var name = "Kyle";

var greeting = `Hello ${name}!`;

console.log( greeting );			// "Hello Kyle!"
console.log( typeof greeting );		// "string"
```

Como puedes ver, usamos `` `..` `` alrededor de una serie de caracteres, que se interpretan como un literal de cadena, pero cualquier expresión de la forma `${..}` se parsea y evalúa inline inmediatamente. El término elegante para tal parseo y evaluación es *interpolación* (mucho más preciso que plantilla).

El resultado de la expresión de literal de cadena interpolada es simplemente una cadena normal y corriente, asignada a la variable `greeting`.

**Advertencia:** `typeof greeting == "string"` ilustra por qué es importante no pensar en estas entidades como valores de plantilla especiales, ya que no puedes asignar la forma no evaluada del literal a algo y reutilizarlo. El literal de cadena `` `..` `` es más como un IIFE en el sentido de que se evalúa automáticamente inline. El resultado de un literal de cadena `` `..` `` es, simplemente, solo una cadena.

Un beneficio realmente bueno de los literales de cadena interpolada es que se les permite dividirse en múltiples líneas:

```js
var text =
`Now is the time for all good men
to come to the aid of their
country!`;

console.log( text );
// Now is the time for all good men
// to come to the aid of their
// country!
```

Los saltos de línea (nuevas líneas) en el literal de cadena interpolada se preservaron en el valor de la cadena.

A menos que aparezcan como secuencias de escape explícitas en el valor literal, el valor del carácter de retorno de carro `\r` (punto de código `U+000D`) o el valor de la secuencia de retorno de carro + salto de línea `\r\n` (puntos de código `U+000D` y `U+000A`) se normalizan ambos a un carácter de salto de línea `\n` (punto de código `U+000A`). Sin embargo, no te preocupes; esta normalización es rara y probablemente solo ocurriría si copias y pegas texto en tu archivo JS.

### Expresiones Interpoladas

Cualquier expresión válida está permitida dentro de `${..}` en un literal de cadena interpolada, incluyendo llamadas a funciones, llamadas a expresiones de función inline, ¡e incluso otros literales de cadena interpolada!

Considera:

```js
function upper(s) {
	return s.toUpperCase();
}

var who = "reader";

var text =
`A very ${upper( "warm" )} welcome
to all of you ${upper( `${who}s` )}!`;

console.log( text );
// A very WARM welcome
// to all of you READERS!
```

Aquí, el literal de cadena interpolada interno `` `${who}s` `` fue una conveniencia un poco más agradable para nosotros al combinar la variable `who` con la cadena `"s"`, en oposición a `who + "s"`. Habrá casos en que anidar literales de cadena interpolada es útil, pero ten cuidado si te encuentras haciendo ese tipo de cosas a menudo, o si te encuentras anidando varios niveles de profundidad.

Si ese es el caso, lo más probable es que tu producción de valores de cadena podría beneficiarse de algunas abstracciones.

**Advertencia:** Como palabra de precaución, ten mucho cuidado con la legibilidad de tu código con tal nuevo poder encontrado. Al igual que con las expresiones de valor por defecto y las expresiones de asignación de desestructuración, solo porque *puedas* hacer algo no significa que *debas* hacerlo. Nunca te excedas tanto con los nuevos trucos de ES6 que tu código se vuelva más listo que tú o los otros miembros de tu equipo.

#### Alcance de Expresiones

Una nota rápida sobre el alcance que se usa para resolver variables en expresiones. Mencioné antes que un literal de cadena interpolada es algo así como un IIFE, y resulta que pensar en ello así también explica el comportamiento de alcance.

Considera:

```js
function foo(str) {
	var name = "foo";
	console.log( str );
}

function bar() {
	var name = "bar";
	foo( `Hello from ${name}!` );
}

var name = "global";

bar();					// "Hello from bar!"
```

En el momento en que el literal de cadena `` `..` `` se expresa, dentro de la función `bar()`, el alcance disponible para él encuentra la variable `name` de `bar()` con valor `"bar"`. Ni el `name` global ni el `name` de `foo(..)` importan. En otras palabras, un literal de cadena interpolada es simplemente de alcance léxico donde aparece, no de alcance dinámico de ninguna manera.

### Literales de Cadena Etiquetados

De nuevo, renombrando la característica por cordura: *literales de cadena etiquetados*.

Para ser honesto, este es uno de los trucos más geniales que ES6 ofrece. Puede parecer un poco extraño, y quizás no tan generalmente práctico al principio. Pero una vez que has pasado algo de tiempo con ello, los literales de cadena etiquetados pueden simplemente sorprenderte en su utilidad.

Por ejemplo:

```js
function foo(strings, ...values) {
	console.log( strings );
	console.log( values );
}

var desc = "awesome";

foo`Everything is ${desc}!`;
// [ "Everything is ", "!"]
// [ "awesome" ]
```

Tomemos un momento para considerar lo que está pasando en el fragmento anterior. Primero, lo más chocante que salta a la vista es ``foo`Everything...`;``. Eso no se parece a nada que hayamos visto antes. ¿Qué es?

Es esencialmente un tipo especial de llamada a función que no necesita los `( .. )`. La *etiqueta* -- la parte `foo` antes del literal de cadena `` `..` `` -- es un valor de función que debería ser llamado. En realidad, puede ser cualquier expresión que resulte en una función, incluso una llamada a función que devuelve otra función, como:

```js
function bar() {
	return function foo(strings, ...values) {
		console.log( strings );
		console.log( values );
	}
}

var desc = "awesome";

bar()`Everything is ${desc}!`;
// [ "Everything is ", "!"]
// [ "awesome" ]
```

Pero, ¿qué se pasa a la función `foo(..)` cuando se invoca como etiqueta para un literal de cadena?

El primer argumento -- que llamamos `strings` -- es un array de todas las cadenas simples (lo que está entre cualquier expresión interpolada). Obtenemos dos valores en el array `strings`: `"Everything is "` y `"!"`.

Por conveniencia en nuestro ejemplo, luego reunimos todos los argumentos subsiguientes en un array llamado `values` usando el operador `...` gather/rest (ver la sección "Spread/Rest" anteriormente en este capítulo), aunque por supuesto podrías haberlos dejado como parámetros nombrados individuales siguiendo al parámetro `strings`.

Los argumentos reunidos en nuestro array `values` son los resultados de las expresiones de interpolación ya evaluadas encontradas en el literal de cadena. Así que obviamente el único elemento en `values` en nuestro ejemplo es `"awesome"`.

Puedes pensar en estos dos arrays como: los valores en `values` son los separadores si fueras a empotrarlos entre los valores en `strings`, y luego si unieras todo junto, obtendrías el valor de cadena interpolada completo.

Un literal de cadena etiquetado es como un paso de procesamiento después de que las expresiones de interpolación se evalúan pero antes de que el valor final de cadena se compile, permitiéndote más control sobre la generación de la cadena a partir del literal.

Típicamente, la función de etiqueta de literal de cadena (`foo(..)` en los fragmentos anteriores) debería computar un valor de cadena apropiado y devolverlo, para que puedas usar el literal de cadena etiquetado como un valor al igual que los literales de cadena no etiquetados:

```js
function tag(strings, ...values) {
	return strings.reduce( function(s,v,idx){
		return s + (idx > 0 ? values[idx-1] : "") + v;
	}, "" );
}

var desc = "awesome";

var text = tag`Everything is ${desc}!`;

console.log( text );			// Everything is awesome!
```

En este fragmento, `tag(..)` es una operación de paso directo, en que no realiza ninguna modificación especial, sino que simplemente usa `reduce(..)` para recorrer y empalmar/intercalar `strings` y `values` juntos de la misma manera que un literal de cadena no etiquetado habría hecho.

Entonces, ¿cuáles son algunos usos prácticos? Hay muchos avanzados que están más allá de nuestro alcance para discutir aquí. Pero aquí hay una idea simple que formatea números como dólares estadounidenses (algo así como localización básica):

```js
function dollabillsyall(strings, ...values) {
	return strings.reduce( function(s,v,idx){
		if (idx > 0) {
			if (typeof values[idx-1] == "number") {
				// mira, ¡también usando literales
				// de cadena interpolada!
				s += `$${values[idx-1].toFixed( 2 )}`;
			}
			else {
				s += values[idx-1];
			}
		}

		return s + v;
	}, "" );
}

var amt1 = 11.99,
	amt2 = amt1 * 1.08,
	name = "Kyle";

var text = dollabillsyall
`Thanks for your purchase, ${name}! Your
product cost was ${amt1}, which with tax
comes out to ${amt2}.`

console.log( text );
// Thanks for your purchase, Kyle! Your
// product cost was $11.99, which with tax
// comes out to $12.95.
```

Si se encuentra un valor `number` en el array `values`, ponemos `"$"` delante y lo formateamos a dos posiciones decimales con `toFixed(2)`. De lo contrario, dejamos pasar el valor sin tocar.

#### Cadenas Crudas

En los fragmentos anteriores, nuestras funciones de etiqueta reciben el primer argumento que llamamos `strings`, que es un array. Pero hay un bit adicional de datos incluido: las versiones crudas sin procesar de todas las cadenas. Puedes acceder a esos valores de cadena crudos usando la propiedad `.raw`, así:

```js
function showraw(strings, ...values) {
	console.log( strings );
	console.log( strings.raw );
}

showraw`Hello\nWorld`;
// [ "Hello
// World" ]
// [ "Hello\nWorld" ]
```

La versión cruda del valor preserva la secuencia escapada cruda `\n` (el `\` y la `n` son caracteres separados), mientras que la versión procesada la considera un único carácter de nueva línea. Sin embargo, la normalización de fin de línea mencionada anteriormente se aplica a ambos valores.

ES6 viene con una función incorporada que puede ser usada como etiqueta de literal de cadena: `String.raw(..)`. Simplemente pasa las versiones crudas de los valores de `strings`:

```js
console.log( `Hello\nWorld` );
// Hello
// World

console.log( String.raw`Hello\nWorld` );
// Hello\nWorld

String.raw`Hello\nWorld`.length;
// 12
```

¡Otros usos para etiquetas de literales de cadena incluyen procesamiento especial para internacionalización, localización, y más!

## Funciones Flecha

Hemos tocado las complicaciones de enlace de `this` con funciones anteriormente en este capítulo, y se cubren extensamente en el título *this & Object Prototypes* de esta serie. Es importante entender las frustraciones que la programación basada en `this` con funciones normales trae, porque esa es la motivación principal para la nueva característica de funciones flecha `=>` de ES6.

Primero ilustremos cómo se ve una función flecha, comparada con funciones normales:

```js
function foo(x,y) {
	return x + y;
}

// versus

var foo = (x,y) => x + y;
```

La definición de función flecha consiste en una lista de parámetros (de cero o más parámetros, y `( .. )` alrededor si no hay exactamente un parámetro), seguido por el marcador `=>`, seguido por un cuerpo de función.

Así que, en el fragmento anterior, la función flecha es solo la parte `(x,y) => x + y`, y esa referencia de función resulta estar asignada a la variable `foo`.

El cuerpo solo necesita estar encerrado por `{ .. }` si hay más de una expresión, o si el cuerpo consiste en una sentencia que no es expresión. Si solo hay una expresión, y omites los `{ .. }` circundantes, hay un `return` implícito delante de la expresión, como se ilustra en el fragmento anterior.

Aquí hay algunas otras variaciones de funciones flecha a considerar:

```js
var f1 = () => 12;
var f2 = x => x * 2;
var f3 = (x,y) => {
	var z = x * 2 + y;
	y++;
	x *= 3;
	return (x + y + z) / 2;
};
```

Las funciones flecha son *siempre* expresiones de función; no hay declaración de función flecha. También debería estar claro que son expresiones de función anónimas -- no tienen una referencia nombrada para propósitos de recursión o enlace/desenlace de eventos -- aunque "Nombres de Funciones" en el Capítulo 7 describirá las reglas de inferencia de nombres de funciones de ES6 para propósitos de depuración.

**Nota:** Todas las capacidades de parámetros de funciones normales están disponibles para las funciones flecha, incluyendo valores por defecto, desestructuración, parámetros rest, y demás.

Las funciones flecha tienen una sintaxis agradable y más corta, lo que las hace en la superficie muy atractivas para escribir código más conciso. De hecho, casi toda la literatura sobre ES6 (aparte de los títulos en esta serie) parece adoptar inmediata y exclusivamente la función flecha como "la nueva función."

Es revelador que casi todos los ejemplos en la discusión de funciones flecha son utilidades cortas de una sola sentencia, como las que se pasan como callbacks a varias utilidades. Por ejemplo:

```js
var a = [1,2,3,4,5];

a = a.map( v => v * 2 );

console.log( a );				// [2,4,6,8,10]
```

En esos casos, donde tienes tales expresiones de función inline, y encajan en el patrón de computar un cálculo rápido en una sola sentencia y devolver ese resultado, las funciones flecha de hecho se ven como una alternativa atractiva y ligera a la más verbosa palabra clave y sintaxis `function`.

¡La mayoría de la gente tiende a *quedarse maravillada* con ejemplos concisos y bonitos como ese, como imagino que acabas de hacer!

Sin embargo, te advertiría que me parecería algo una mala aplicación de esta característica usar la sintaxis de función flecha con funciones normales de múltiples sentencias, especialmente aquellas que de otra manera se expresarían naturalmente como declaraciones de función.

Recuerda la función de etiqueta de literal de cadena `dollabillsyall(..)` de antes en este capítulo -- cambiémosla para usar la sintaxis `=>`:

```js
var dollabillsyall = (strings, ...values) =>
	strings.reduce( (s,v,idx) => {
		if (idx > 0) {
			if (typeof values[idx-1] == "number") {
				// mira, ¡también usando literales
				// de cadena interpolada!
				s += `$${values[idx-1].toFixed( 2 )}`;
			}
			else {
				s += values[idx-1];
			}
		}

		return s + v;
	}, "" );
```

En este ejemplo, las únicas modificaciones que hice fueron la eliminación de `function`, `return`, y algunos `{ .. }`, y luego la inserción de `=>` y un `var`. ¿Es esto una mejora significativa en la legibilidad del código? Meh.

De hecho argumentaría que la falta de `return` y `{ .. }` externo oscurece parcialmente el hecho de que la llamada `reduce(..)` es la única sentencia en la función `dollabillsyall(..)` y que su resultado es el resultado previsto de la llamada. También, el ojo entrenado que está tan acostumbrado a buscar la palabra `function` en el código para encontrar límites de alcance ahora necesita buscar el marcador `=>`, que definitivamente puede ser más difícil de encontrar en lo denso del código.

Aunque no es una regla rígida, diría que las ganancias de legibilidad de la conversión a función flecha `=>` son inversamente proporcionales a la longitud de la función que se convierte. Cuanto más larga la función, menos `=>` ayuda; cuanto más corta la función, más `=>` puede brillar.

Creo que es probablemente más sensato y razonable adoptar `=>` para los lugares en el código donde sí necesitas expresiones de función inline cortas, pero dejar tus funciones principales de longitud normal como están.

### No Solo Sintaxis Más Corta, Sino `this`

La mayor parte de la atención popular hacia `=>` ha sido en ahorrar esas preciosas pulsaciones de tecla al eliminar `function`, `return`, y `{ .. }` de tu código.

Pero hay un gran detalle que hemos omitido hasta ahora. Dije al principio de la sección que las funciones `=>` están estrechamente relacionadas con el comportamiento de enlace de `this`. De hecho, las funciones flecha `=>` están *diseñadas principalmente* para alterar el comportamiento de `this` de una manera específica, resolviendo un punto de dolor particular y común con la codificación consciente de `this`.

El ahorro de pulsaciones de tecla es una pista falsa, un espectáculo secundario engañoso en el mejor de los casos.

Revisitemos otro ejemplo de antes en este capítulo:

```js
var controller = {
	makeRequest: function(..){
		var self = this;

		btn.addEventListener( "click", function(){
			// ..
			self.makeRequest(..);
		}, false );
	}
};
```

Usamos el hack `var self = this`, y luego referenciamos `self.makeRequest(..)`, porque dentro de la función callback que estamos pasando a `addEventListener(..)`, el enlace de `this` no será el mismo que en `makeRequest(..)` en sí. En otras palabras, porque los enlaces de `this` son dinámicos, recurrimos a la predictibilidad del alcance léxico vía la variable `self`.

Aquí finalmente podemos ver la característica de diseño principal de las funciones flecha `=>`. Dentro de las funciones flecha, el enlace de `this` no es dinámico, sino léxico. En el fragmento anterior, si usamos una función flecha para el callback, `this` será predeciblemente lo que queríamos que fuera.

Considera:

```js
var controller = {
	makeRequest: function(..){
		btn.addEventListener( "click", () => {
			// ..
			this.makeRequest(..);
		}, false );
	}
};
```

El `this` léxico en el callback de función flecha del fragmento anterior ahora apunta al mismo valor que en la función `makeRequest(..)` envolvente. En otras palabras, `=>` es un sustituto sintáctico de `var self = this`.

En casos donde `var self = this` (o, alternativamente, una llamada `.bind(this)` de función) normalmente sería útil, las funciones flecha `=>` son una alternativa más agradable operando sobre el mismo principio. Suena genial, ¿verdad?

No tan simple.

Si `=>` reemplaza `var self = this` o `.bind(this)` y ayuda, adivina qué pasa si usas `=>` con una función consciente de `this` que *no* necesita `var self = this` para funcionar. Podrías adivinar que va a estropear las cosas. Sí.

Considera:

```js
var controller = {
	makeRequest: (..) => {
		// ..
		this.helper(..);
	},
	helper: (..) => {
		// ..
	}
};

controller.makeRequest(..);
```

Aunque invocamos como `controller.makeRequest(..)`, la referencia `this.helper` falla, porque `this` aquí no apunta a `controller` como normalmente lo haría. ¿A dónde apunta? Hereda léxicamente `this` del alcance circundante. En este fragmento anterior, ese es el alcance global, donde `this` apunta al objeto global. Ugh.

Además del `this` léxico, las funciones flecha también tienen `arguments` léxicos -- no tienen su propio array `arguments` sino que heredan de su padre -- así como `super` léxico y `new.target` (ver "Clases" en el Capítulo 3).

Así que ahora podemos concluir un conjunto de reglas más matizado para cuándo `=>` es apropiado y cuándo no:

* Si tienes una expresión de función inline corta de una sola sentencia, donde la única sentencia es un `return` de algún valor computado, *y* esa función no hace ya una referencia a `this` dentro de ella, *y* no hay auto-referencia (recursión, enlace/desenlace de eventos), *y* no esperas razonablemente que la función sea así alguna vez, probablemente puedes refactorizarla de manera segura a ser una función flecha `=>`.
* Si tienes una expresión de función interna que depende de un hack `var self = this` o una llamada `.bind(this)` en la función envolvente para asegurar el enlace apropiado de `this`, esa expresión de función interna probablemente puede convertirse de manera segura en una función flecha `=>`.
* Si tienes una expresión de función interna que depende de algo como `var args = Array.prototype.slice.call(arguments)` en la función envolvente para hacer una copia léxica de `arguments`, esa expresión de función interna probablemente puede convertirse de manera segura en una función flecha `=>`.
* Para todo lo demás -- declaraciones de función normales, expresiones de función de múltiples sentencias más largas, funciones que necesitan un identificador de nombre léxico para auto-referencia (recursión, etc.), y cualquier otra función que no encaje en las características anteriores -- probablemente deberías evitar la sintaxis de función `=>`.

En resumen: `=>` se trata del enlace léxico de `this`, `arguments` y `super`. Estas son características intencionales diseñadas para arreglar algunos problemas comunes, no bugs, peculiaridades o errores en ES6.

No creas ningún hype de que `=>` es principalmente, o incluso mayormente, sobre menos pulsaciones de tecla. Ya sea que ahorres pulsaciones de tecla o las desperdicies, deberías saber exactamente lo que estás haciendo intencionalmente con cada carácter tecleado.

**Consejo:** Si tienes una función que por cualquiera de estas razones articuladas no es una buena candidata para una función flecha `=>`, pero está siendo declarada como parte de un literal de objeto, recuerda de "Métodos Concisos" anteriormente en este capítulo que hay otra opción para sintaxis de función más corta.
