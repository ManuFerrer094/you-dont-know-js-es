## Clases

Desde casi el comienzo de JavaScript, la sintaxis y los patrones de desarrollo se han esforzado (léase: luchado) por colocar una fachada de soporte para desarrollo orientado a clases. Con cosas como `new` e `instanceof` y una propiedad `.constructor`, ¿quién no podría sino ser tentado a pensar que JS tenía clases escondidas en algún lugar dentro de su sistema de prototipos?

Por supuesto, las "clases" de JS no son ni de cerca lo mismo que las clases clásicas. Las diferencias están bien documentadas, así que no insistiré más en ese punto aquí.

**Nota:** Para aprender más sobre los patrones usados en JS para simular "clases", y una vista alternativa de los prototipos llamada "delegación", ver la segunda mitad del título *this & Object Prototypes* de esta serie.

### `class`

Aunque el mecanismo de prototipos de JS no funciona como las clases tradicionales, eso no detiene la fuerte marea de demanda sobre el lenguaje para extender el azúcar sintáctico de modo que expresar "clases" se vea más como clases reales. Introducimos la palabra clave `class` de ES6 y su mecanismo asociado.

Esta característica es el resultado de un debate altamente contencioso y prolongado, y representa un compromiso de un subconjunto menor de varias vistas fuertemente opuestas sobre cómo abordar las clases en JS. La mayoría de los desarrolladores que quieren clases completas en JS encontrarán partes de la nueva sintaxis bastante atractivas, pero encontrarán partes importantes aún faltantes. No te preocupes sin embargo. TC39 ya está trabajando en características adicionales para aumentar las clases en el período post-ES6.

En el corazón del nuevo mecanismo de clases de ES6 está la palabra clave `class`, que identifica un *bloque* donde los contenidos definen los miembros del prototipo de una función. Considera:

```js
class Foo {
	constructor(a,b) {
		this.x = a;
		this.y = b;
	}

	gimmeXY() {
		return this.x * this.y;
	}
}
```

Algunas cosas a notar:

* `class Foo` implica crear una función (especial) con el nombre `Foo`, muy similar a lo que hacías pre-ES6.
* `constructor(..)` identifica la firma de esa función `Foo(..)`, así como el contenido de su cuerpo.
* Los métodos de clase usan la misma sintaxis de "método conciso" disponible para literales de objeto, como se discutió en el Capítulo 2. Esto también incluye la forma concisa de generador como se discutió anteriormente en este capítulo, así como la sintaxis getter/setter de ES5. Sin embargo, los métodos de clase no son enumerables mientras que los métodos de objeto lo son por defecto.
* A diferencia de los literales de objeto, ¡no hay comas separando miembros en el cuerpo de un `class`! De hecho, ni siquiera están permitidas.

La definición de sintaxis `class` en el fragmento anterior puede pensarse aproximadamente como este equivalente pre-ES6, que probablemente se verá bastante familiar para aquellos que han hecho codificación estilo prototipo antes:

```js
function Foo(a,b) {
	this.x = a;
	this.y = b;
}

Foo.prototype.gimmeXY = function() {
	return this.x * this.y;
}
```

Tanto en la forma pre-ES6 como en la nueva forma `class` de ES6, esta "clase" ahora puede ser instanciada y usada justo como esperarías:

```js
var f = new Foo( 5, 15 );

f.x;						// 5
f.y;						// 15
f.gimmeXY();				// 75
```

¡Precaución! Aunque `class Foo` parece mucho a `function Foo()`, hay diferencias importantes:

* Una llamada `Foo(..)` de `class Foo` *debe* hacerse con `new`, ya que la opción pre-ES6 de `Foo.call( obj )` *no* funcionará.
* Mientras que `function Foo` se "eleva" (ver el título *Scope & Closures* de esta serie), `class Foo` no; la cláusula `extends ..` especifica una expresión que no puede ser "elevada". Así que, debes declarar un `class` antes de poder instanciarlo.
* `class Foo` en el alcance global de nivel superior crea un identificador léxico `Foo` en ese alcance, pero a diferencia de `function Foo` no crea una propiedad del objeto global con ese nombre.

El operador establecido `instanceof` todavía funciona con clases ES6, porque `class` simplemente crea una función constructora del mismo nombre. Sin embargo, ES6 introduce una manera de personalizar cómo funciona `instanceof`, usando `Symbol.hasInstance` (ver "Símbolos Bien Conocidos" en el Capítulo 7).

Otra manera de pensar en `class`, que encuentro más conveniente, es como un *macro* que se usa para poblar automáticamente un objeto `prototype`. Opcionalmente, también conecta la relación `[[Prototype]]` si se usa `extends` (ver la siguiente sección).

Un `class` de ES6 no es realmente una entidad en sí misma, sino un metaconcepto que envuelve otras entidades concretas, como funciones y propiedades, y las une.

**Consejo:** Además de la forma de declaración, un `class` también puede ser una expresión, como en: `var x = class Y { .. }`. Esto es principalmente útil para pasar una definición de clase (técnicamente, el constructor mismo) como argumento de función o asignarlo a una propiedad de objeto.

### `extends` y `super`

Las clases ES6 también tienen azúcar sintáctico para establecer el enlace de delegación `[[Prototype]]` entre dos prototipos de función — comúnmente mal etiquetado "herencia" o confusamente etiquetado "herencia prototípica" — usando la terminología familiar orientada a clases `extends`:

```js
class Bar extends Foo {
	constructor(a,b,c) {
		super( a, b );
		this.z = c;
	}

	gimmeXYZ() {
		return super.gimmeXY() * this.z;
	}
}

var b = new Bar( 5, 15, 25 );

b.x;						// 5
b.y;						// 15
b.z;						// 25
b.gimmeXYZ();				// 1875
```

Una adición nueva significativa es `super`, que en realidad es algo no directamente posible pre-ES6 (sin algunos desafortunados compromisos con hacks). En el constructor, `super` automáticamente se refiere al "constructor padre", que en el ejemplo anterior es `Foo(..)`. En un método, se refiere al "objeto padre", de modo que puedes entonces hacer un acceso a propiedad/método desde él, como `super.gimmeXY()`.

`Bar extends Foo` por supuesto significa enlazar el `[[Prototype]]` de `Bar.prototype` a `Foo.prototype`. Así que, `super` en un método como `gimmeXYZ()` específicamente significa `Foo.prototype`, mientras que `super` significa `Foo` cuando se usa en el constructor de `Bar`.

**Nota:** `super` no está limitado a declaraciones `class`. También funciona en literales de objeto, de manera muy similar a como estamos discutiendo aquí. Ver "`super` en Objetos" en el Capítulo 2 para más información.

#### Aquí Hay Dragones con `super`

No es insignificante notar que `super` se comporta diferentemente dependiendo de dónde aparece. En justicia, la mayoría del tiempo, eso no será un problema. Pero sorpresas aguardan si te desvías de una norma estrecha.

Puede haber casos donde en el constructor quieras referenciar a `Foo.prototype`, como para acceder directamente a una de sus propiedades/métodos. Sin embargo, `super` en el constructor no puede usarse de esa manera; `super.prototype` no funcionará. `super(..)` significa aproximadamente llamar a `new Foo(..)`, pero no es en realidad una referencia utilizable a `Foo` en sí.

Simétricamente, podrías querer referenciar la función `Foo(..)` desde dentro de un método no-constructor. `super.constructor` apuntará a la función `Foo(..)`, pero ten cuidado porque esta función solo *puede* ser invocada con `new`. `new super.constructor(..)` sería válido, pero no sería terriblemente útil en la mayoría de los casos, porque no puedes hacer que esa llamada use o referencie el objeto de contexto `this` actual, que es probablemente lo que querrías.

También, `super` parece que podría estar determinado por el contexto de una función igual que `this` — es decir, que ambos se enlazarían dinámicamente. Sin embargo, `super` no es dinámico como lo es `this`. Cuando un constructor o método hace una referencia `super` dentro de él en el momento de la declaración (en el cuerpo del `class`), ese `super` está estáticamente enlazado a esa jerarquía de clases específica, y no puede ser sobrescrito (al menos en ES6).

¿Qué significa eso? Significa que si tienes el hábito de tomar un método de una "clase" y "pedir prestado" para otra clase sobrescribiendo su `this`, digamos con `call(..)` o `apply(..)`, eso puede muy bien crear sorpresas si el método que estás tomando prestado tiene un `super` en él. Considera esta jerarquía de clases:

```js
class ParentA {
	constructor() { this.id = "a"; }
	foo() { console.log( "ParentA:", this.id ); }
}

class ParentB {
	constructor() { this.id = "b"; }
	foo() { console.log( "ParentB:", this.id ); }
}

class ChildA extends ParentA {
	foo() {
		super.foo();
		console.log( "ChildA:", this.id );
	}
}

class ChildB extends ParentB {
	foo() {
		super.foo();
		console.log( "ChildB:", this.id );
	}
}

var a = new ChildA();
a.foo();					// ParentA: a
							// ChildA: a
var b = new ChildB();		// ParentB: b
b.foo();					// ChildB: b
```

Todo parece bastante natural y esperado en el fragmento anterior. Sin embargo, si intentas pedir prestado `b.foo()` y usarlo en el contexto de `a` — en virtud del enlace dinámico de `this`, tal préstamo es bastante común y se usa de muchas maneras diferentes, incluyendo mixins más notablemente — puedes encontrar este resultado como una sorpresa fea:

```js
// pedir prestado `b.foo()` para usar en contexto `a`
b.foo.call( a );			// ParentB: a
							// ChildB: a
```

Como puedes ver, la referencia `this.id` fue re-enlazada dinámicamente de modo que `: a` se reporta en ambos casos en vez de `: b`. Pero la referencia `super.foo()` de `b.foo()` no fue re-enlazada dinámicamente, así que todavía reportó `ParentB` en vez del esperado `ParentA`.

Porque `b.foo()` referencia `super`, está estáticamente enlazado a la jerarquía `ChildB`/`ParentB` y no puede usarse contra la jerarquía `ChildA`/`ParentA`. No hay solución ES6 para esta limitación.

`super` parece funcionar intuitivamente si tienes una jerarquía de clases estática sin polinización cruzada. Pero en toda justicia, uno de los principales beneficios de hacer codificación consciente de `this` es exactamente ese tipo de flexibilidad. Simplemente, `class` + `super` requiere que evites tales técnicas.

La elección se reduce a estrechar tu diseño de objetos a estas jerarquías estáticas — `class`, `extends`, y `super` serán bastante agradables — o abandonar todos los intentos de "simular" clases y en su lugar abrazar objetos dinámicos y flexibles sin clases y delegación `[[Prototype]]` (ver el título *this & Object Prototypes* de esta serie).

#### Constructor de Subclase

Los constructores no son requeridos para clases o subclases; un constructor por defecto se sustituye en ambos casos si se omite. Sin embargo, el constructor sustituido por defecto es diferente para una clase directa versus una clase extendida.

Específicamente, el constructor por defecto de subclase automáticamente llama al constructor padre, y pasa cualquier argumento. En otras palabras, podrías pensar del constructor por defecto de subclase algo como esto:

```js
constructor(...args) {
	super(...args);
}
```

Este es un detalle importante a notar. No todos los lenguajes con clases tienen el constructor de subclase llamando automáticamente al constructor padre. C++ lo hace, pero Java no. Pero más importante, en clases pre-ES6, tal llamada automática al "constructor padre" no ocurre. Ten cuidado al convertir a `class` de ES6 si has estado dependiendo de que tales llamadas *no* ocurran.

Otra quizás sorprendente desviación/limitación de los constructores de subclase ES6: en un constructor de una subclase, no puedes acceder a `this` hasta que se haya llamado a `super(..)`. La razón es matizada y complicada, pero se reduce al hecho de que el constructor padre es en realidad el que crea/inicializa el `this` de tu instancia. Pre-ES6, funciona al revés; el objeto `this` es creado por el "constructor de subclase", y luego llamas al "constructor padre" con el contexto del `this` de la "subclase".

Ilustremos. Esto funciona pre-ES6:

```js
function Foo() {
	this.a = 1;
}

function Bar() {
	this.b = 2;
	Foo.call( this );
}

// `Bar` "extiende" `Foo`
Bar.prototype = Object.create( Foo.prototype );
```

Pero este equivalente ES6 no está permitido:

```js
class Foo {
	constructor() { this.a = 1; }
}

class Bar extends Foo {
	constructor() {
		this.b = 2;			// no permitido antes de `super()`
		super();			// para arreglar intercambia estas dos sentencias
	}
}
```

En este caso, la solución es simple. Solo intercambia las dos sentencias en el constructor de la subclase `Bar`. Sin embargo, si has estado dependiendo pre-ES6 de poder omitir llamar al "constructor padre", ten cuidado porque eso ya no será permitido.

#### Extendiendo Nativos con `extend`

Uno de los beneficios más aclamados del nuevo diseño de `class` y `extend` es la capacidad de (¡finalmente!) crear subclases de los nativos integrados, como `Array`. Considera:

```js
class MyCoolArray extends Array {
	first() { return this[0]; }
	last() { return this[this.length - 1]; }
}

var a = new MyCoolArray( 1, 2, 3 );

a.length;					// 3
a;							// [1,2,3]

a.first();					// 1
a.last();					// 3
```

Antes de ES6, una "subclase" falsa de `Array` usando creación manual de objetos y enlace a `Array.prototype` solo funcionaba parcialmente. Se perdía los comportamientos especiales de un array real, como la propiedad `length` que se actualiza automáticamente. ¡Las subclases ES6 deberían funcionar completamente con comportamientos "heredados" y aumentados como se espera!

Otra limitación común de "subclases" pre-ES6 es con el objeto `Error`, al crear "subclases" personalizadas de error. Cuando objetos `Error` genuinos son creados, automáticamente capturan información especial de `stack`, incluyendo el número de línea y archivo donde el error es creado. Las "subclases" personalizadas de error pre-ES6 no tienen tal comportamiento especial, lo que limita severamente su utilidad.

ES6 al rescate:

```js
class Oops extends Error {
	constructor(reason) {
		super(reason);
		this.oops = reason;
	}
}

// después:
var ouch = new Oops( "I messed up!" );
throw ouch;
```

El objeto de error personalizado `ouch` en este fragmento anterior se comportará como cualquier otro objeto de error genuino, incluyendo la captura de `stack`. ¡Esa es una gran mejora!

### `new.target`

ES6 introduce un nuevo concepto llamado *meta propiedad* (ver Capítulo 7), en la forma de `new.target`.

Si eso parece extraño, lo es; emparejar una palabra clave con un `.` y un nombre de propiedad es definitivamente un patrón fuera de lo ordinario para JS.

`new.target` es un nuevo valor "mágico" disponible en todas las funciones, aunque en funciones normales siempre será `undefined`. En cualquier constructor, `new.target` siempre apunta al constructor que `new` realmente invocó directamente, incluso si el constructor está en una clase padre y fue delegado por una llamada `super(..)` desde un constructor hijo. Considera:

```js
class Foo {
	constructor() {
		console.log( "Foo: ", new.target.name );
	}
}

class Bar extends Foo {
	constructor() {
		super();
		console.log( "Bar: ", new.target.name );
	}
	baz() {
		console.log( "baz: ", new.target );
	}
}

var a = new Foo();
// Foo: Foo

var b = new Bar();
// Foo: Bar   <-- respeta el lugar de llamada `new`
// Bar: Bar

b.baz();
// baz: undefined
```

La meta propiedad `new.target` no tiene mucho propósito en constructores de clase, excepto acceder a una propiedad/método estático (ver la siguiente sección).

Si `new.target` es `undefined`, sabes que la función no fue llamada con `new`. Puedes entonces forzar una invocación con `new` si es necesario.

### `static`

Cuando una subclase `Bar` extiende una clase padre `Foo`, ya observamos que `Bar.prototype` está enlazado vía `[[Prototype]]` a `Foo.prototype`. Pero adicionalmente, `Bar()` está enlazado vía `[[Prototype]]` a `Foo()`. Esa parte puede no tener un razonamiento tan obvio.

Sin embargo, es bastante útil en el caso donde declaras métodos `static` (no solo propiedades) para una clase, ya que estos se añaden directamente al objeto función de esa clase, no al objeto `prototype` del objeto función. Considera:

```js
class Foo {
	static cool() { console.log( "cool" ); }
	wow() { console.log( "wow" ); }
}

class Bar extends Foo {
	static awesome() {
		super.cool();
		console.log( "awesome" );
	}
	neat() {
		super.wow();
		console.log( "neat" );
	}
}

Foo.cool();					// "cool"
Bar.cool();					// "cool"
Bar.awesome();				// "cool"
							// "awesome"

var b = new Bar();
b.neat();					// "wow"
							// "neat"

b.awesome;					// undefined
b.cool;						// undefined
```

Ten cuidado de no confundirte pensando que los miembros `static` están en la cadena de prototipos de la clase. En realidad están en la cadena dual/paralela entre los constructores de función.

#### Getter Constructor `Symbol.species`

Un lugar donde `static` puede ser útil es en establecer el getter `Symbol.species` (conocido internamente en la especificación como `@@species`) para una clase derivada (hija). Esta capacidad permite a una clase hija señalar a una clase padre qué constructor debe ser usado — cuando no se pretende usar el constructor de la clase hija en sí — si algún método de la clase padre necesita crear una nueva instancia.

Por ejemplo, muchos métodos en `Array` crean y devuelven una nueva instancia de `Array`. Si defines una clase derivada de `Array`, pero quieres que esos métodos continúen creando instancias reales de `Array` en vez de tu clase derivada, esto funciona:

```js
class MyCoolArray extends Array {
	// forzar `species` a ser constructor padre
	static get [Symbol.species]() { return Array; }
}

var a = new MyCoolArray( 1, 2, 3 ),
	b = a.map( function(v){ return v * 2; } );

b instanceof MyCoolArray;	// false
b instanceof Array;			// true
```

Para ilustrar cómo un método de clase padre puede usar la declaración species de un hijo algo así como `Array#map(..)` está haciendo, considera:

```js
class Foo {
	// diferir `species` al constructor derivado
	static get [Symbol.species]() { return this; }
	spawn() {
		return new this.constructor[Symbol.species]();
	}
}

class Bar extends Foo {
	// forzar `species` a ser constructor padre
	static get [Symbol.species]() { return Foo; }
}

var a = new Foo();
var b = a.spawn();
b instanceof Foo;					// true

var x = new Bar();
var y = x.spawn();
y instanceof Bar;					// false
y instanceof Foo;					// true
```

El `Symbol.species` de la clase padre hace `return this` para diferir a cualquier clase derivada, como normalmente esperarías. `Bar` entonces lo sobrescribe para declarar manualmente que `Foo` sea usado para tal creación de instancias. Por supuesto, una clase derivada todavía puede crear instancias de sí misma usando `new this.constructor(..)`.

## Revisión

ES6 introduce varias nuevas características que ayudan en la organización de código:

* Los Iteradores proporcionan acceso secuencial a datos u operaciones. Pueden ser consumidos por nuevas características del lenguaje como `for..of` y `...`.
* Los Generadores son funciones con capacidad de pausa/reanudación local controladas por un iterador. Pueden ser usados para generar programáticamente (e interactivamente, a través del paso de mensajes `yield`/`next(..)`) valores para ser consumidos vía iteración.
* Los Módulos permiten encapsulamiento privado de detalles de implementación con una API exportada públicamente. Las definiciones de módulo son basadas en archivos, instancias singleton, y resueltas estáticamente en tiempo de compilación.
* Las Clases proporcionan sintaxis más limpia alrededor de codificación basada en prototipos. La adición de `super` también resuelve problemas complicados con referencias relativas en la cadena `[[Prototype]]`.

Estas nuevas herramientas deberían ser tu primera parada cuando intentes mejorar la arquitectura de tus proyectos JS adoptando ES6.
