## Desestructuración

ES6 introduce una nueva característica sintáctica llamada *desestructuración*, que puede ser un poco menos confusa si en su lugar piensas en ella como *asignación estructurada*. Para entender este significado, considera:

```js
function foo() {
	return [1,2,3];
}

var tmp = foo(),
	a = tmp[0], b = tmp[1], c = tmp[2];

console.log( a, b, c );				// 1 2 3
```

Como puedes ver, creamos una asignación manual de los valores en el array que `foo()` devuelve a las variables individuales `a`, `b` y `c`, y para hacerlo (desafortunadamente) necesitamos la variable `tmp`.

De manera similar, podemos hacer lo siguiente con objetos:

```js
function bar() {
	return {
		x: 4,
		y: 5,
		z: 6
	};
}

var tmp = bar(),
	x = tmp.x, y = tmp.y, z = tmp.z;

console.log( x, y, z );				// 4 5 6
```

El valor de la propiedad `tmp.x` se asigna a la variable `x`, e igualmente para `tmp.y` a `y` y `tmp.z` a `z`.

Asignar manualmente valores indexados de un array o propiedades de un objeto puede pensarse como *asignación estructurada*. ES6 añade una sintaxis dedicada para la *desestructuración*, específicamente *desestructuración de arrays* y *desestructuración de objetos*. Esta sintaxis elimina la necesidad de la variable `tmp` en los fragmentos anteriores, haciéndolos mucho más limpios. Considera:

```js
var [ a, b, c ] = foo();
var { x: x, y: y, z: z } = bar();

console.log( a, b, c );				// 1 2 3
console.log( x, y, z );				// 4 5 6
```

Probablemente estés más acostumbrado a ver sintaxis como `[a,b,c]` en el lado derecho de una asignación `=`, como el valor que se asigna.

La desestructuración simétricamente invierte ese patrón, de modo que `[a,b,c]` en el lado izquierdo de la asignación `=` se trata como una especie de "patrón" para descomponer el valor del array del lado derecho en asignaciones de variables separadas.

De manera similar, `{ x: x, y: y, z: z }` especifica un "patrón" para descomponer el valor del objeto de `bar()` en asignaciones de variables separadas.

### Patrón de Asignación de Propiedades de Objeto

Profundicemos en esa sintaxis `{ x: x, .. }` del fragmento anterior. Si el nombre de la propiedad que se está coincidiendo es el mismo que la variable que quieres declarar, puedes realmente acortar la sintaxis:

```js
var { x, y, z } = bar();

console.log( x, y, z );				// 4 5 6
```

Bastante cool, ¿verdad?

Pero, ¿`{ x, .. }` está omitiendo la parte `x: ` o la parte `: x`? En realidad estamos omitiendo la parte `x: ` cuando usamos la sintaxis más corta. Eso puede no parecer un detalle importante, pero entenderás su importancia en solo un momento.

Si puedes escribir la forma más corta, ¿por qué escribirías alguna vez la forma más larga? Porque esa forma más larga en realidad te permite asignar una propiedad a un nombre de variable diferente, lo cual a veces puede ser bastante útil:

```js
var { x: bam, y: baz, z: bap } = bar();

console.log( bam, baz, bap );		// 4 5 6
console.log( x, y, z );				// ReferenceError
```

Hay una peculiaridad sutil pero súper importante que entender sobre esta variación de la forma de desestructuración de objetos. Para ilustrar por qué puede ser una trampa de la cual necesitas tener cuidado, consideremos el "patrón" de cómo los literales de objeto normales se especifican:

```js
var X = 10, Y = 20;

var o = { a: X, b: Y };

console.log( o.a, o.b );			// 10 20
```

En `{ a: X, b: Y }`, sabemos que `a` es la propiedad del objeto, y `X` es el valor fuente que se le asigna. En otras palabras, el patrón sintáctico es `destino: fuente`, o más obviamente, `alias-propiedad: valor`. Intuitivamente entendemos esto porque es lo mismo que la asignación `=`, donde el patrón es `destino = fuente`.

Sin embargo, cuando usas asignación de desestructuración de objetos -- es decir, poniendo la sintaxis de aspecto de literal de objeto `{ .. }` en el lado izquierdo del operador `=` -- inviertes ese patrón `destino: fuente`.

Recuerda:

```js
var { x: bam, y: baz, z: bap } = bar();
```

El patrón sintáctico aquí es `fuente: destino` (o `valor: alias-variable`). `x: bam` significa que la propiedad `x` es el valor fuente y `bam` es la variable destino a la cual asignar. En otras palabras, los literales de objeto son `destino <-- fuente`, y las asignaciones de desestructuración de objetos son `fuente --> destino`. ¿Ves cómo está invertido?

Hay otra manera de pensar sobre esta sintaxis, sin embargo, que puede ayudar a aliviar la confusión. Considera:

```js
var aa = 10, bb = 20;

var o = { x: aa, y: bb };
var     { x: AA, y: BB } = o;

console.log( AA, BB );				// 10 20
```

En la línea `{ x: aa, y: bb }`, `x` e `y` representan las propiedades del objeto. En la línea `{ x: AA, y: BB }`, `x` e `y` *también* representan las propiedades del objeto.

¿Recuerdas cómo antes afirmé que `{ x, .. }` omitía la parte `x: `? En esas dos líneas, si borras las partes `x: ` e `y: ` en ese fragmento, te quedas solo con `aa, bb` y `AA, BB`, que en efecto -- solo conceptualmente, no realmente -- son asignaciones de `aa` a `AA` y de `bb` a `BB`.

Así que, esa simetría puede ayudar a explicar por qué el patrón sintáctico fue intencionalmente invertido para esta característica de ES6.

**Nota:** Habría preferido que la sintaxis fuera `{ AA: x , BB: y }` para la asignación de desestructuración, ya que eso habría preservado la consistencia del patrón más familiar `destino: fuente` para ambos usos. Pero bueno, tengo que entrenar mi cerebro para la inversión, como algunos lectores también tendrán que hacer.

### No Solo Declaraciones

Hasta ahora, hemos usado asignación de desestructuración con declaraciones `var` (por supuesto, también podrían usar `let` y `const`), pero la desestructuración es una operación de asignación general, no solo una declaración.

Considera:

```js
var a, b, c, x, y, z;

[a,b,c] = foo();
( { x, y, z } = bar() );

console.log( a, b, c );				// 1 2 3
console.log( x, y, z );				// 4 5 6
```

Las variables ya pueden estar declaradas, y entonces la desestructuración solo hace asignaciones, exactamente como ya hemos visto.

**Nota:** Para la forma de desestructuración de objetos específicamente, al omitir un declarador `var`/`let`/`const`, tuvimos que rodear toda la expresión de asignación en `( )`, porque de lo contrario el `{ .. }` en el lado izquierdo como primer elemento en la sentencia se toma como una sentencia de bloque en lugar de un objeto.

De hecho, las expresiones de asignación (`a`, `y`, etc.) no necesitan ser realmente solo identificadores de variables. Cualquier cosa que sea una expresión de asignación válida está permitida. Por ejemplo:

```js
var o = {};

[o.a, o.b, o.c] = foo();
( { x: o.x, y: o.y, z: o.z } = bar() );

console.log( o.a, o.b, o.c );		// 1 2 3
console.log( o.x, o.y, o.z );		// 4 5 6
```

Incluso puedes usar expresiones de propiedades computadas en la desestructuración. Considera:

```js
var which = "x",
	o = {};

( { [which]: o[which] } = bar() );

console.log( o.x );					// 4
```

La parte `[which]:` es la propiedad computada, que resulta en `x` -- la propiedad a desestructurar del objeto en cuestión como la fuente de la asignación. La parte `o[which]` es solo una referencia normal de clave de objeto, que equivale a `o.x` como el destino de la asignación.

Puedes usar las asignaciones generales para crear mapeos/transformaciones de objetos, como:

```js
var o1 = { a: 1, b: 2, c: 3 },
	o2 = {};

( { a: o2.x, b: o2.y, c: o2.z } = o1 );

console.log( o2.x, o2.y, o2.z );	// 1 2 3
```

O puedes mapear un objeto a un array, como:

```js
var o1 = { a: 1, b: 2, c: 3 },
	a2 = [];

( { a: a2[0], b: a2[1], c: a2[2] } = o1 );

console.log( a2 );					// [1,2,3]
```

O al revés:

```js
var a1 = [ 1, 2, 3 ],
	o2 = {};

[ o2.a, o2.b, o2.c ] = a1;

console.log( o2.a, o2.b, o2.c );	// 1 2 3
```

O podrías reordenar un array a otro:

```js
var a1 = [ 1, 2, 3 ],
	a2 = [];

[ a2[2], a2[0], a2[1] ] = a1;

console.log( a2 );					// [2,3,1]
```

Incluso puedes resolver la tarea tradicional de "intercambiar dos variables" sin una variable temporal:

```js
var x = 10, y = 20;

[ y, x ] = [ x, y ];

console.log( x, y );				// 20 10
```

**Advertencia:** Ten cuidado: no deberías mezclar declaración con asignación a menos que quieras que todas las expresiones de asignación *también* se traten como declaraciones. De lo contrario, obtendrás errores de sintaxis. Por eso en el ejemplo anterior tuve que hacer `var a2 = []` separado de la asignación de desestructuración `[ a2[0], .. ] = ..`. No tendría sentido intentar `var [ a2[0], .. ] = ..`, porque `a2[0]` no es un identificador de declaración válido; tampoco podría crear implícitamente una declaración `var a2 = []` para usar.

### Asignaciones Repetidas

La forma de desestructuración de objetos permite que una propiedad fuente (conteniendo cualquier tipo de valor) se liste múltiples veces. Por ejemplo:

```js
var { a: X, a: Y } = { a: 1 };

X;	// 1
Y;	// 1
```

Eso también significa que puedes tanto desestructurar una propiedad de sub-objeto/array como también capturar el valor del sub-objeto/array en sí. Considera:

```js
var { a: { x: X, x: Y }, a } = { a: { x: 1 } };

X;	// 1
Y;	// 1
a;	// { x: 1 }

( { a: X, a: Y, a: [ Z ] } = { a: [ 1 ] } );

X.push( 2 );
Y[0] = 10;

X;	// [10,2]
Y;	// [10,2]
Z;	// 1
```

Una palabra de precaución sobre la desestructuración: puede ser tentador listar asignaciones de desestructuración todas en una sola línea como se ha hecho hasta ahora en nuestra discusión. Sin embargo, es mucho mejor idea expandir los patrones de asignación de desestructuración en múltiples líneas, usando indentación apropiada -- muy parecido a como lo harías en JSON o con un valor de literal de objeto -- por el bien de la legibilidad.

```js
// más difícil de leer:
var { a: { b: [ c, d ], e: { f } }, g } = obj;

// mejor:
var {
	a: {
		b: [ c, d ],
		e: { f }
	},
	g
} = obj;
```

Recuerda: **el propósito de la desestructuración no es solo menos escritura, sino legibilidad más declarativa.**

#### Expresiones de Asignación de Desestructuración

La expresión de asignación con desestructuración de objeto o array tiene como valor de finalización el valor completo del objeto/array del lado derecho. Considera:

```js
var o = { a:1, b:2, c:3 },
	a, b, c, p;

p = { a, b, c } = o;

console.log( a, b, c );			// 1 2 3
p === o;						// true
```

En el fragmento anterior, a `p` se le asignó la referencia del objeto `o`, no uno de los valores `a`, `b` o `c`. Lo mismo es verdad para la desestructuración de arrays:

```js
var o = [1,2,3],
	a, b, c, p;

p = [ a, b, c ] = o;

console.log( a, b, c );			// 1 2 3
p === o;						// true
```

Al llevar el valor del objeto/array como finalización, puedes encadenar expresiones de asignación de desestructuración juntas:

```js
var o = { a:1, b:2, c:3 },
	p = [4,5,6],
	a, b, c, x, y, z;

( {a} = {b,c} = o );
[x,y] = [z] = p;

console.log( a, b, c );			// 1 2 3
console.log( x, y, z );			// 4 5 4
```

### Demasiados, Muy Pocos, Justo Suficientes

Con tanto la asignación de desestructuración de arrays como la de objetos, no tienes que asignar todos los valores que están presentes. Por ejemplo:

```js
var [,b] = foo();
var { x, z } = bar();

console.log( b, x, z );				// 2 4 6
```

Los valores `1` y `3` que regresaron de `foo()` se descartan, al igual que el valor `5` de `bar()`.

De manera similar, si intentas asignar más valores de los que están presentes en el valor que estás desestructurando/descomponiendo, obtienes un fallback elegante a `undefined`, como esperarías:

```js
var [,,c,d] = foo();
var { w, z } = bar();

console.log( c, z );				// 3 6
console.log( d, w );				// undefined undefined
```

Este comportamiento sigue simétricamente del principio "`undefined` es faltante" establecido anteriormente.

Examinamos el operador `...` anteriormente en este capítulo, y vimos que a veces puede usarse para esparcir un valor de array en sus valores separados, y a veces puede usarse para hacer lo opuesto: reunir un conjunto de valores juntos en un array.

Además del uso gather/rest en declaraciones de funciones, `...` puede realizar el mismo comportamiento en asignaciones de desestructuración. Para ilustrar, recordemos un fragmento de antes en este capítulo:

```js
var a = [2,3,4];
var b = [ 1, ...a, 5 ];

console.log( b );					// [1,2,3,4,5]
```

Aquí vemos que `...a` está esparciendo `a`, porque aparece en la posición de valor del array `[ .. ]`. Si `...a` aparece en una posición de desestructuración de array, realiza el comportamiento de reunión:

```js
var a = [2,3,4];
var [ b, ...c ] = a;

console.log( b, c );				// 2 [3,4]
```

La asignación de desestructuración `var [ .. ] = a` esparce `a` para ser asignado al patrón descrito dentro del `[ .. ]`. La primera parte nombra `b` para el primer valor en `a` (`2`). Pero luego `...c` reúne el resto de los valores (`3` y `4`) en un array y lo llama `c`.

**Nota:** Hemos visto cómo `...` funciona con arrays, pero ¿qué hay de los objetos? No es una característica de ES6, pero ver el Capítulo 8 para discusión de una posible característica "más allá de ES6" donde `...` funciona con esparcir o reunir objetos.

### Asignación de Valor por Defecto

Ambas formas de desestructuración pueden ofrecer una opción de valor por defecto para una asignación, usando la sintaxis `=` similar a los valores de argumento de función por defecto discutidos anteriormente.

Considera:

```js
var [ a = 3, b = 6, c = 9, d = 12 ] = foo();
var { x = 5, y = 10, z = 15, w = 20 } = bar();

console.log( a, b, c, d );			// 1 2 3 12
console.log( x, y, z, w );			// 4 5 6 20
```

Puedes combinar la asignación de valor por defecto con la sintaxis de expresión de asignación alternativa cubierta anteriormente. Por ejemplo:

```js
var { x, y, z, w: WW = 20 } = bar();

console.log( x, y, z, WW );			// 4 5 6 20
```

Ten cuidado de confundirte a ti mismo (o a otros desarrolladores que lean tu código) si usas un objeto o array como el valor por defecto en una desestructuración. Puedes crear código realmente difícil de entender:

```js
var x = 200, y = 300, z = 100;
var o1 = { x: { y: 42 }, z: { y: z } };

( { y: x = { y: y } } = o1 );
( { z: y = { y: z } } = o1 );
( { x: z = { y: x } } = o1 );
```

¿Puedes decir de ese fragmento qué valores tienen `x`, `y` y `z` al final? Toma un momento de reflexión, me imagino. Terminaré con el suspenso:

```js
console.log( x.y, y.y, z.y );		// 300 100 42
```

La conclusión aquí: la desestructuración es genial y puede ser muy útil, pero también es una espada afilada que puede causar daño (al cerebro de alguien) si se usa imprudentemente.

### Desestructuración Anidada

Si los valores que estás desestructurando tienen objetos o arrays anidados, puedes desestructurar esos valores anidados también:

```js
var a1 = [ 1, [2, 3, 4], 5 ];
var o1 = { x: { y: { z: 6 } } };

var [ a, [ b, c, d ], e ] = a1;
var { x: { y: { z: w } } } = o1;

console.log( a, b, c, d, e );		// 1 2 3 4 5
console.log( w );					// 6
```

La desestructuración anidada puede ser una forma simple de aplanar espacios de nombres de objetos. Por ejemplo:

```js
var App = {
	model: {
		User: function(){ .. }
	}
};

// en lugar de:
// var User = App.model.User;

var { model: { User } } = App;
```

### Desestructuración de Parámetros

En el siguiente fragmento, ¿puedes identificar la asignación?

```js
function foo(x) {
	console.log( x );
}

foo( 42 );
```

La asignación está algo oculta: `42` (el argumento) se asigna a `x` (el parámetro) cuando `foo(42)` se ejecuta. Si el emparejamiento parámetro/argumento es una asignación, entonces tiene sentido que sea una asignación que podría desestructurarse, ¿verdad? ¡Por supuesto!

Considera desestructuración de arrays para parámetros:

```js
function foo( [ x, y ] ) {
	console.log( x, y );
}

foo( [ 1, 2 ] );					// 1 2
foo( [ 1 ] );						// 1 undefined
foo( [] );							// undefined undefined
```

La desestructuración de objetos para parámetros también funciona:

```js
function foo( { x, y } ) {
	console.log( x, y );
}

foo( { y: 1, x: 2 } );				// 2 1
foo( { y: 42 } );					// undefined 42
foo( {} );							// undefined undefined
```

Esta técnica es una aproximación de argumentos nombrados (¡una característica largamente solicitada para JS!), en que las propiedades del objeto se mapean a los parámetros desestructurados del mismo nombre. Eso también significa que obtenemos parámetros opcionales (en cualquier posición) gratis, como puedes ver que omitir el "parámetro" `x` funcionó como esperaríamos.

Por supuesto, todas las variaciones de desestructuración discutidas anteriormente están disponibles para nosotros con desestructuración de parámetros, incluyendo desestructuración anidada, valores por defecto, y más. La desestructuración también se mezcla bien con otras capacidades de parámetros de función de ES6, como valores de parámetros por defecto y parámetros rest/gather.

Considera estas ilustraciones rápidas (ciertamente no exhaustivas de las posibles variaciones):

```js
function f1([ x=2, y=3, z ]) { .. }
function f2([ x, y, ...z], w) { .. }
function f3([ x, y, ...z], ...w) { .. }

function f4({ x: X, y }) { .. }
function f5({ x: X = 10, y = 20 }) { .. }
function f6({ x = 10 } = {}, { y } = { y: 10 }) { .. }
```

Tomemos un ejemplo de este fragmento y examinémoslo, con propósitos ilustrativos:

```js
function f3([ x, y, ...z], ...w) {
	console.log( x, y, z, w );
}

f3( [] );							// undefined undefined [] []
f3( [1,2,3,4], 5, 6 );				// 1 2 [3,4] [5,6]
```

Hay dos operadores `...` en uso aquí, y ambos están reuniendo valores en arrays (`z` y `w`), aunque `...z` reúne del resto de los valores sobrantes en el primer argumento array, mientras que `...w` reúne del resto de los argumentos principales sobrantes después del primero.

#### Valores por Defecto de Desestructuración + Valores por Defecto de Parámetros

Hay un punto sutil al que deberías prestar particular atención -- la diferencia en comportamiento entre un valor por defecto de desestructuración y un valor por defecto de parámetro de función. Por ejemplo:

```js
function f6({ x = 10 } = {}, { y } = { y: 10 }) {
	console.log( x, y );
}

f6();								// 10 10
```

Al principio, parecería que hemos declarado un valor por defecto de `10` para ambos parámetros `x` e `y`, pero de dos maneras diferentes. Sin embargo, estos dos enfoques diferentes se comportarán de manera diferente en ciertos casos, y la diferencia es terriblemente sutil.

Considera:

```js
f6( {}, {} );						// 10 undefined
```

Espera, ¿por qué pasó eso? Es bastante claro que el parámetro nombrado `x` tiene por defecto `10` si no se pasa como una propiedad del mismo nombre en el objeto del primer argumento.

Pero ¿qué hay de que `y` sea `undefined`? El valor `{ y: 10 }` es un objeto como valor por defecto de parámetro de función, no un valor por defecto de desestructuración. Como tal, solo aplica si el segundo argumento no se pasa en absoluto, o se pasa como `undefined`.

En el fragmento anterior, *estamos* pasando un segundo argumento (`{}`), así que el valor por defecto `{ y: 10 }` no se usa, y la desestructuración `{ y }` ocurre contra el valor de objeto vacío `{}` pasado.

Ahora, compara `{ y } = { y: 10 }` con `{ x = 10 } = {}`.

Para el uso de la forma de `x`, si el primer argumento de función se omite o es `undefined`, el objeto vacío `{}` por defecto aplica. Entonces, cualquier valor que esté en la posición del primer argumento -- ya sea el `{}` por defecto o lo que sea que hayas pasado -- se desestructura con `{ x = 10 }`, que verifica si se encuentra una propiedad `x`, y si no se encuentra (o es `undefined`), el valor por defecto `10` se aplica al parámetro nombrado `x`.

Respira profundo. Vuelve a leer los últimos párrafos un par de veces. Revisemos vía código:

```js
function f6({ x = 10 } = {}, { y } = { y: 10 }) {
	console.log( x, y );
}

f6();								// 10 10
f6( undefined, undefined );			// 10 10
f6( {}, undefined );				// 10 10

f6( {}, {} );						// 10 undefined
f6( undefined, {} );				// 10 undefined

f6( { x: 2 }, { y: 3 } );			// 2 3
```

Generalmente parecería que el comportamiento por defecto del parámetro `x` es probablemente el caso más deseable y sensato comparado con el de `y`. Como tal, es importante entender por qué y cómo la forma `{ x = 10 } = {}` es diferente de la forma `{ y } = { y: 10 }`.

Si eso todavía está un poco borroso, vuelve atrás y léelo de nuevo, y juega con esto tú mismo. Tu yo futuro te agradecerá por tomarte el tiempo para aclarar este detalle de matiz de trampa muy sutil.

#### Valores por Defecto Anidados: Desestructurado y Reestructurado

Aunque al principio puede ser difícil de comprender, emerge un modismo interesante para establecer valores por defecto para las propiedades de un objeto anidado: usar desestructuración de objetos junto con lo que yo llamaría *reestructuración*.

Considera un conjunto de valores por defecto en una estructura de objeto anidado, como la siguiente:

```js
// tomado de: http://es-discourse.com/t/partial-default-arguments/120/7

var defaults = {
	options: {
		remove: true,
		enable: false,
		instance: {}
	},
	log: {
		warn: true,
		error: true
	}
};
```

Ahora, digamos que tienes un objeto llamado `config`, que tiene algunas de estas aplicadas, pero quizás no todas, y te gustaría establecer todos los valores por defecto en este objeto en los lugares faltantes, pero sin sobreescribir configuraciones específicas ya presentes:

```js
var config = {
	options: {
		remove: false,
		instance: null
	}
};
```

Por supuesto puedes hacerlo manualmente, como podrías haber hecho en el pasado:

```js
config.options = config.options || {};
config.options.remove = (config.options.remove !== undefined) ?
	config.options.remove : defaults.options.remove;
config.options.enable = (config.options.enable !== undefined) ?
	config.options.enable : defaults.options.enable;
...
```

Puaj.

Otros pueden preferir el enfoque de asignar-sobreescribir para esta tarea. Podrías estar tentado por la utilidad `Object.assign(..)` de ES6 (ver Capítulo 6) para clonar las propiedades primero de `defaults` y luego sobreescritas con las propiedades clonadas de `config`, así:

```js
config = Object.assign( {}, defaults, config );
```

Eso se ve mucho más bonito, ¿eh? ¡Pero hay un problema mayor! `Object.assign(..)` es superficial, lo que significa que cuando copia `defaults.options`, solo copia esa referencia de objeto, no clona profundamente las propiedades de ese objeto a un objeto `config.options`. `Object.assign(..)` necesitaría ser aplicado (en cierta forma "recursivamente") en todos los niveles del árbol de tu objeto para obtener la clonación profunda que esperas.

**Nota:** Muchas librerías/frameworks de utilidades JS proporcionan su propia opción para clonación profunda de un objeto, pero esos enfoques y sus trampas están más allá de nuestro alcance para discutir aquí.

Así que examinemos si la desestructuración de objetos de ES6 con valores por defecto puede ayudar en algo:

```js
config.options = config.options || {};
config.log = config.log || {};
({
	options: {
		remove: config.options.remove = defaults.options.remove,
		enable: config.options.enable = defaults.options.enable,
		instance: config.options.instance = defaults.options.instance
	} = {},
	log: {
		warn: config.log.warn = defaults.log.warn,
		error: config.log.error = defaults.log.error
	} = {}
} = config);
```

No tan bonito como la falsa promesa de `Object.assign(..)` (siendo que solo es superficial), pero es mejor que el enfoque manual por bastante, creo. Sin embargo, es desafortunadamente verboso y repetitivo.

El enfoque del fragmento anterior funciona porque estoy hackeando los mecanismos de desestructuración y valores por defecto para hacer las verificaciones `=== undefined` y decisiones de asignación de propiedades por mí. Es un truco en que estoy desestructurando `config` (ver el `= config` al final del fragmento), pero estoy reasignando todos los valores desestructurados de vuelta a `config`, con las referencias de asignación `config.options.enable`.

Aún demasiado, sin embargo. Veamos si podemos hacer algo mejor.

El siguiente truco funciona mejor si sabes que todas las diversas propiedades que estás desestructurando tienen nombres únicos. Aún puedes hacerlo incluso si ese no es el caso, pero no es tan bonito -- tendrás que hacer la desestructuración en etapas, o crear variables locales únicas como alias temporales.

Si desestructuramos completamente todas las propiedades en variables de nivel superior, podemos entonces inmediatamente reestructurar para reconstituir la estructura de objeto anidado original.

Pero todas esas variables temporales colgando contaminarían el alcance. Así que, usemos alcance de bloque (ver "Declaraciones con Alcance de Bloque" anteriormente en este capítulo) con un bloque envolvente general `{ }`:

```js
// fusionar `defaults` en `config`
{
	// desestructurar (con asignaciones de valor por defecto)
	let {
		options: {
			remove = defaults.options.remove,
			enable = defaults.options.enable,
			instance = defaults.options.instance
		} = {},
		log: {
			warn = defaults.log.warn,
			error = defaults.log.error
		} = {}
	} = config;

	// reestructurar
	config = {
		options: { remove, enable, instance },
		log: { warn, error }
	};
}
```

¿Eso parece bastante mejor, eh?

**Nota:** También podrías lograr el cierre de alcance con un IIFE de función flecha en lugar del bloque general `{ }` y declaraciones `let`. Tu desestructuración de asignaciones/valores por defecto estaría en la lista de parámetros y tu reestructuración sería la sentencia `return` en el cuerpo de la función.

La sintaxis `{ warn, error }` en la parte de reestructuración puede parecerte nueva; eso se llama "propiedades concisas" y lo cubrimos en la siguiente sección.
