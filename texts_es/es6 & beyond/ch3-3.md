## Módulos

No creo que sea una exageración sugerir que el patrón de organización de código más importante en todo JavaScript es, y siempre ha sido, el módulo. Para mí mismo, y creo que para una gran parte de la comunidad, el patrón de módulos impulsa la gran mayoría del código.

### La Manera Antigua

El patrón de módulo tradicional está basado en una función exterior con variables y funciones internas, y una "API pública" retornada con métodos que tienen clausura sobre los datos y capacidades internos. Frecuentemente se expresa así:

```js
function Hello(name) {
	function greeting() {
		console.log( "Hello " + name + "!" );
	}

	// API pública
	return {
		greeting: greeting
	};
}

var me = Hello( "Kyle" );
me.greeting();			// Hello Kyle!
```

Este módulo `Hello(..)` puede producir múltiples instancias al ser llamado subsecuentemente. A veces, un módulo solo se necesita como un singleton (es decir, solo necesita una instancia), en cuyo caso una ligera variación del fragmento anterior, usando un IIFE, es común:

```js
var me = (function Hello(name){
	function greeting() {
		console.log( "Hello " + name + "!" );
	}

	// API pública
	return {
		greeting: greeting
	};
})( "Kyle" );

me.greeting();			// Hello Kyle!
```

Este patrón está probado y comprobado. También es lo suficientemente flexible para tener una gran variedad de variaciones para diferentes escenarios.

Uno de los más comunes es la Definición de Módulo Asíncrono (AMD), y otro es la Definición de Módulo Universal (UMD). No cubriremos los detalles de estos patrones y técnicas aquí, pero están explicados extensamente en muchos lugares en línea.

### Avanzando

A partir de ES6, ya no necesitamos depender de la función envolvente y la clausura para proporcionarnos soporte de módulos. Los módulos ES6 tienen soporte sintáctico y funcional de primera clase.

Antes de entrar en la sintaxis específica, es importante entender algunas diferencias conceptuales bastante significativas con los módulos ES6 comparados con cómo puedes haber manejado módulos en el pasado:

* ES6 usa módulos basados en archivos, lo que significa un módulo por archivo. En este momento, no hay una manera estandarizada de combinar múltiples módulos en un solo archivo.

   Eso significa que si vas a cargar módulos ES6 directamente en una aplicación web de navegador, los estarás cargando individualmente, no como un gran paquete en un solo archivo como ha sido común en esfuerzos de optimización de rendimiento.

   Se espera que la llegada contemporánea de HTTP/2 mitigue significativamente cualquier preocupación de rendimiento, ya que opera sobre una conexión de socket persistente y por lo tanto puede cargar muy eficientemente muchos archivos más pequeños en paralelo e intercalados entre sí.
* La API de un módulo ES6 es estática. Es decir, defines estáticamente cuáles son todas las exportaciones de nivel superior en la API pública de tu módulo, y estas no pueden enmendarse después.

   Algunos usos están acostumbrados a poder proporcionar definiciones de API dinámicas, donde métodos pueden añadirse/removerse/reemplazarse en respuesta a condiciones de tiempo de ejecución. O estos usos tendrán que cambiar para ajustarse a las APIs estáticas de ES6, o tendrán que restringir los cambios dinámicos a propiedades/métodos de un objeto de segundo nivel.
* Los módulos ES6 son singletons. Es decir, solo hay una instancia del módulo, que mantiene su estado. Cada vez que importas ese módulo en otro módulo, obtienes una referencia a la única instancia centralizada. Si quieres poder producir múltiples instancias de módulo, tu módulo necesitará proporcionar algún tipo de factory para hacerlo.
* Las propiedades y métodos que expones en la API pública de un módulo no son solo asignaciones normales de valores o referencias. Son enlaces reales (casi como punteros) a los identificadores en la definición interna de tu módulo.

   En módulos pre-ES6, si ponías una propiedad en tu API pública que contenía un valor primitivo como un número o cadena, esa asignación de propiedad era por copia de valor, y cualquier actualización interna de una variable correspondiente sería separada y no afectaría la copia pública en el objeto API.

   Con ES6, exportar una variable privada local, incluso si actualmente contiene una cadena/número/etc. primitivo, exporta un enlace a la variable. Si el módulo cambia el valor de la variable, el enlace de importación externo ahora se resuelve a ese nuevo valor.
* Importar un módulo es lo mismo que solicitar estáticamente que se cargue (si no se ha cargado ya). Si estás en un navegador, eso implica una carga bloqueante por la red. Si estás en un servidor (es decir, Node.js), es una carga bloqueante desde el sistema de archivos.

   Sin embargo, no entres en pánico por las implicaciones de rendimiento. Porque los módulos ES6 tienen definiciones estáticas, los requerimientos de importación pueden ser escaneados estáticamente, y las cargas sucederán de manera preventiva, incluso antes de que hayas usado el módulo.

   ES6 en realidad no especifica ni maneja la mecánica de cómo funcionan estas solicitudes de carga. Hay una noción separada de un Cargador de Módulos, donde cada entorno de alojamiento (navegador, Node.js, etc.) proporciona un Cargador por defecto apropiado para el entorno. La importación de un módulo usa un valor de cadena para representar dónde obtener el módulo (URL, ruta de archivo, etc.), pero este valor es opaco en tu programa y solo significativo para el Cargador en sí.

   Puedes definir tu propio Cargador personalizado si quieres un control más granular del que el Cargador por defecto ofrece -- que básicamente es ninguno, ya que está totalmente oculto del código de tu programa.

Como puedes ver, los módulos ES6 servirán el caso de uso general de organizar código con encapsulamiento, controlar APIs públicas, y referenciar importaciones de dependencias. Pero tienen una manera muy particular de hacerlo, y eso puede o no ajustarse muy cercanamente a cómo ya has estado haciendo módulos durante años.

#### CommonJS

Hay una sintaxis de módulo similar, pero no totalmente compatible, llamada CommonJS, que es familiar para aquellos en el ecosistema de Node.js.

A falta de una manera más diplomática de decir esto, a largo plazo, los módulos ES6 esencialmente están destinados a reemplazar todos los formatos y estándares previos para módulos, incluso CommonJS, ya que están construidos sobre soporte sintáctico en el lenguaje. Esto, con el tiempo, inevitablemente ganará como el enfoque superior, si no por otra razón que por la ubicuidad.

Sin embargo, enfrentamos un camino bastante largo para llegar a ese punto. Hay literalmente cientos de miles de módulos estilo CommonJS en el mundo JavaScript del lado del servidor, y 10 veces más módulos de varios formatos estándar (UMD, AMD, ad hoc) en el mundo del navegador. Tomará muchos años para que las transiciones hagan un progreso significativo.

Mientras tanto, los transpiladores/convertidores de módulos serán una necesidad absoluta. Más vale que te acostumbres a esa nueva realidad. Ya sea que escribas en módulos regulares, AMD, UMD, CommonJS, o ES6, estas herramientas tendrán que parsear y convertir a un formato que sea adecuado para cualquier entorno en el que tu código se ejecute.

Para Node.js, eso probablemente significa (por ahora) que el objetivo es CommonJS. Para el navegador, probablemente es UMD o AMD. Espera mucho cambio en esto durante los próximos años mientras estas herramientas maduran y emergen las mejores prácticas.

De aquí en adelante, mi mejor consejo sobre módulos es este: cualquiera que sea el formato al que hayas estado religiosamente ligado con fuerte afinidad, también desarrolla una apreciación y comprensión de los módulos ES6, tal como son, y deja que tus otras tendencias de módulos se desvanezcan. *Son* el futuro de los módulos en JS, incluso si esa realidad está un poco lejos.

### La Nueva Manera

Las dos principales palabras clave nuevas que habilitan los módulos ES6 son `import` y `export`. Hay mucho matiz en la sintaxis, así que echemos un vistazo más profundo.

**Advertencia:** Un detalle importante que es fácil pasar por alto: tanto `import` como `export` deben aparecer siempre en el alcance de nivel superior de su uso respectivo. Por ejemplo, no puedes poner ni un `import` ni un `export` dentro de un condicional `if`; deben aparecer fuera de todos los bloques y funciones.

#### Exportar Miembros de API con `export`

La palabra clave `export` se pone delante de una declaración, o se usa como un operador (de algún tipo) con una lista especial de enlaces a exportar. Considera:

```js
export function foo() {
	// ..
}

export var awesome = 42;

var bar = [1,2,3];
export { bar };
```

Otra manera de expresar las mismas exportaciones:

```js
function foo() {
	// ..
}

var awesome = 42;
var bar = [1,2,3];

export { foo, awesome, bar };
```

Todas estas se llaman *exportaciones nombradas*, ya que en efecto estás exportando los enlaces de nombre de las variables/funciones/etc.

Todo lo que no *etiquetes* con `export` permanece privado dentro del alcance del módulo. Es decir, aunque algo como `var bar = ..` parece estar declarando en el alcance global de nivel superior, el alcance de nivel superior en realidad es el módulo en sí; no hay alcance global en los módulos.

**Nota:** Los módulos *sí* tienen acceso a `window` y todos los "globales" que cuelgan de él, pero no como alcance léxico de nivel superior. Sin embargo, realmente deberías mantenerte alejado de los globales en tus módulos si es posible.

También puedes "renombrar" (es decir, crear un alias para) un miembro del módulo durante la exportación nombrada:

```js
function foo() { .. }

export { foo as bar };
```

Cuando este módulo se importa, solo el nombre de miembro `bar` está disponible para importar; `foo` permanece oculto dentro del módulo.

Las exportaciones de módulo no son simplemente asignaciones normales de valores o referencias, como estás acostumbrado con el operador de asignación `=`. En realidad, cuando exportas algo, estás exportando un enlace (algo así como un puntero) a esa cosa (variable, etc.).

Dentro de tu módulo, si cambias el valor de una variable a la cual ya exportaste un enlace, incluso si ya ha sido importada (ver la siguiente sección), el enlace importado se resolverá al valor actual (actualizado).

Considera:

```js
var awesome = 42;
export { awesome };

// después
awesome = 100;
```

Cuando este módulo se importa, independientemente de si eso es antes o después de la asignación `awesome = 100`, una vez que esa asignación ha ocurrido, el enlace importado se resuelve al valor `100`, no `42`.

Eso es porque el enlace es, en esencia, una referencia a, o un puntero a, la variable `awesome` en sí, en lugar de una copia de su valor. Este es un concepto mayormente sin precedentes para JS introducido con los enlaces de módulos ES6.

Aunque claramente puedes usar `export` múltiples veces dentro de la definición de un módulo, ES6 definitivamente prefiere el enfoque de que un módulo tenga una sola exportación, que se conoce como *exportación por defecto*. En las palabras de algunos miembros del comité TC39, eres "recompensado con sintaxis de `import` más simple" si sigues ese patrón, y conversamente "penalizado" con sintaxis más verbosa si no lo haces.

Una exportación por defecto establece un enlace de exportación particular como el predeterminado al importar el módulo. El nombre del enlace es literalmente `default`. Como verás más adelante, al importar enlaces de módulo también puedes renombrarlos, como comúnmente harás con una exportación por defecto.

Solo puede haber un `default` por definición de módulo. Cubriremos `import` en la siguiente sección, y verás cómo la sintaxis de `import` es más concisa si el módulo tiene una exportación por defecto.

Hay un matiz sutil en la sintaxis de exportación por defecto al que debes prestar mucha atención. Compara estos dos fragmentos:

```js
function foo(..) {
	// ..
}

export default foo;
```

Y este:

```js
function foo(..) {
	// ..
}

export { foo as default };
```

En el primer fragmento, estás exportando un enlace al valor de la expresión de función en ese momento, *no* al identificador `foo`. En otras palabras, `export default ..` toma una expresión. Si después asignas `foo` a un valor diferente dentro de tu módulo, la importación del módulo aún revela la función exportada originalmente, no el nuevo valor.

Por cierto, el primer fragmento también podría haberse escrito como:

```js
export default function foo(..) {
	// ..
}
```

**Advertencia:** Aunque la parte `function foo..` aquí es técnicamente una expresión de función, para los propósitos del alcance interno del módulo, se trata como una declaración de función, en que el nombre `foo` está enlazado en el alcance de nivel superior del módulo (frecuentemente llamado "hoisting"). Lo mismo es cierto para `export default class Foo..`. Sin embargo, aunque *puedes* hacer `export var foo = ..`, actualmente no puedes hacer `export default var foo = ..` (o `let` o `const`), en un caso frustrante de inconsistencia. Al momento de escribir esto, ya hay discusión de añadir esa capacidad pronto, post-ES6, por consistencia.

Recuerda el segundo fragmento de nuevo:

```js
function foo(..) {
	// ..
}

export { foo as default };
```

En esta versión de la exportación del módulo, el enlace de exportación por defecto es en realidad al identificador `foo` en lugar de a su valor, así que obtienes el comportamiento de enlace descrito previamente (es decir, si después cambias el valor de `foo`, el valor visto en el lado de importación también se actualizará).

Ten mucho cuidado con esta trampa sutil en la sintaxis de exportación por defecto, especialmente si tu lógica requiere que los valores de exportación se actualicen. Si nunca planeas actualizar el valor de una exportación por defecto, `export default ..` está bien. Si planeas actualizar el valor, debes usar `export { .. as default }`. De cualquier manera, ¡asegúrate de comentar tu código para explicar tu intención!

Porque solo puede haber un `default` por módulo, podrías estar tentado a diseñar tu módulo con una exportación por defecto de un objeto con todos tus métodos de API en él, como:

```js
export default {
	foo() { .. },
	bar() { .. },
	..
};
```

Ese patrón parece mapear cercanamente a cómo muchos desarrolladores ya han estructurado sus módulos pre-ES6, así que parece un enfoque natural. Desafortunadamente, tiene algunas desventajas y es oficialmente desaconsejado.

En particular, el motor JS no puede analizar estáticamente los contenidos de un objeto simple, lo que significa que no puede hacer algunas optimizaciones para el rendimiento de `import` estático. La ventaja de tener cada miembro individual y explícitamente exportado es que el motor *puede* hacer el análisis estático y la optimización.

Si tu API ya tiene más de un miembro, parece que estos principios -- una exportación por defecto por módulo, y todos los miembros de API como exportaciones nombradas -- están en conflicto, ¿no? Pero *puedes* tener una sola exportación por defecto así como otras exportaciones nombradas; no son mutuamente excluyentes.

Así que, en lugar de este patrón (desaconsejado):

```js
export default function foo() { .. }

foo.bar = function() { .. };
foo.baz = function() { .. };
```

Puedes hacer:

```js
export default function foo() { .. }

export function bar() { .. }
export function baz() { .. }
```

**Nota:** En el fragmento anterior, usé el nombre `foo` para la función que `default` etiqueta. Ese nombre `foo`, sin embargo, se ignora para propósitos de exportación -- `default` es en realidad el nombre exportado. Cuando importes este enlace por defecto, puedes darle cualquier nombre que quieras, como verás en la siguiente sección.

Alternativamente, algunos preferirán:

```js
function foo() { .. }
function bar() { .. }
function baz() { .. }

export { foo as default, bar, baz, .. };
```

Los efectos de mezclar exportaciones por defecto y nombradas serán más claros cuando cubramos `import` en breve. Pero esencialmente significa que la forma de importación por defecto más concisa solo recuperaría la función `foo()`. El usuario podría adicionalmente listar manualmente `bar` y `baz` como importaciones nombradas, si las quiere.

Probablemente puedes imaginar lo tedioso que va a ser para los consumidores de tu módulo si tienes muchos enlaces de exportación nombrados. Hay una forma de importación con comodín donde importas todas las exportaciones de un módulo dentro de un solo objeto de espacio de nombres, pero no hay manera de importar con comodín a enlaces de nivel superior.

De nuevo, el mecanismo de módulos ES6 está intencionalmente diseñado para desalentar módulos con muchas exportaciones; relativamente hablando, se desea que tales enfoques sean un poco más difíciles, como una especie de ingeniería social para alentar el diseño de módulos simples en favor de diseño de módulos grandes/complejos.

Probablemente recomendaría que no mezcles exportación por defecto con exportaciones nombradas, especialmente si tienes una API grande y refactorizar a módulos separados no es práctico o deseado. En ese caso, simplemente usa todas exportaciones nombradas, y documenta que los consumidores de tu módulo probablemente deberían usar el enfoque `import * as ..` (importación de espacio de nombres, discutido en la siguiente sección) para traer toda la API de una vez en un solo espacio de nombres.

Mencionamos esto antes, pero volvamos a ello con más detalle. Aparte de la forma `export default ...` que exporta un enlace de valor de expresión, todas las otras formas de exportación exportan enlaces a identificadores locales. Para esos enlaces, si cambias el valor de una variable dentro de un módulo después de exportar, el enlace importado externamente accederá al valor actualizado:

```js
var foo = 42;
export { foo as default };

export var bar = "hello world";

foo = 10;
bar = "cool";
```

Cuando importas este módulo, las exportaciones `default` y `bar` estarán enlazadas a las variables locales `foo` y `bar`, lo que significa que revelarán los valores actualizados `10` y `"cool"`. Los valores en el momento de la exportación son irrelevantes. Los valores en el momento de la importación son irrelevantes. Los enlaces son vínculos vivos, así que todo lo que importa es cuál es el valor actual cuando accedes al enlace.

**Advertencia:** Los enlaces bidireccionales no están permitidos. Si importas un `foo` de un módulo, e intentas cambiar el valor de tu variable `foo` importada, ¡se lanzará un error! Lo revisaremos en la siguiente sección.

También puedes re-exportar las exportaciones de otro módulo, como:

```js
export { foo, bar } from "baz";
export { foo as FOO, bar as BAR } from "baz";
export * from "baz";
```

Esas formas son similares a primero importar del módulo `"baz"` y luego listar sus miembros explícitamente para exportación desde tu módulo. Sin embargo, en estas formas, los miembros del módulo `"baz"` nunca se importan al alcance local de tu módulo; son una especie de paso directo sin tocar.

#### Importar Miembros de API con `import`

Para importar un módulo, sin sorprender, usas la sentencia `import`. Así como `export` tiene varias variaciones matizadas, también las tiene `import`, así que pasa suficiente tiempo considerando los siguientes temas y experimentando con tus opciones.

Si quieres importar ciertos miembros nombrados específicos de la API de un módulo a tu alcance de nivel superior, usas esta sintaxis:

```js
import { foo, bar, baz } from "foo";
```

**Advertencia:** La sintaxis `{ .. }` aquí puede parecerse a un literal de objeto, o incluso a una sintaxis de desestructuración de objetos. Sin embargo, su forma es especial solo para módulos, así que ten cuidado de no confundirla con otros patrones `{ .. }` en otros lugares.

La cadena `"foo"` se llama un *especificador de módulo*. Porque el objetivo completo es sintaxis estáticamente analizable, el especificador de módulo debe ser un literal de cadena; no puede ser una variable que contenga el valor de cadena.

Desde la perspectiva de tu código ES6 y el motor JS en sí, los contenidos de este literal de cadena son completamente opacos y sin significado. El cargador de módulos interpretará esta cadena como una instrucción de dónde encontrar el módulo deseado, ya sea como una ruta URL o una ruta del sistema de archivos local.

Los identificadores `foo`, `bar` y `baz` listados deben coincidir con exportaciones nombradas en la API del módulo (el análisis estático y la aserción de errores aplican). Se enlazan como identificadores de nivel superior en tu alcance actual:

```js
import { foo } from "foo";

foo();
```

Puedes renombrar los identificadores enlazados importados, como:

```js
import { foo as theFooFunc } from "foo";

theFooFunc();
```

Si el módulo tiene solo una exportación por defecto que quieres importar y enlazar a un identificador, puedes optar por omitir la sintaxis circundante `{ .. }` para ese enlace. El `import` en este caso preferido obtiene la más agradable y concisa de las formas de sintaxis de `import`:

```js
import foo from "foo";

// o:
import { default as foo } from "foo";
```

**Nota:** Como se explicó en la sección anterior, la palabra clave `default` en el `export` de un módulo especifica una exportación nombrada donde el nombre es en realidad `default`, como se ilustra con la segunda opción de sintaxis más verbosa. El renombramiento de `default` a, en este caso, `foo`, es explícito en la última sintaxis e idéntico pero implícito en la primera.

También puedes importar una exportación por defecto junto con otras exportaciones nombradas, si el módulo tiene tal definición. Recuerda esta definición de módulo de antes:

```js
export default function foo() { .. }

export function bar() { .. }
export function baz() { .. }
```

Para importar la exportación por defecto de ese módulo y sus dos exportaciones nombradas:

```js
import FOOFN, { bar, baz as BAZ } from "foo";

FOOFN();
bar();
BAZ();
```

El enfoque fuertemente sugerido por la filosofía de módulos de ES6 es que solo importes los enlaces específicos de un módulo que necesitas. Si un módulo proporciona 10 métodos de API, pero solo necesitas dos de ellos, algunos creen que es un desperdicio traer el conjunto completo de enlaces de API.

Un beneficio, además de que el código sea más explícito, es que las importaciones estrechas hacen que el análisis estático y la detección de errores (usar accidentalmente el nombre de enlace equivocado, por ejemplo) sean más robustos.

Por supuesto, esa es solo la posición estándar influenciada por la filosofía de diseño de ES6; no hay nada que requiera adherirse a ese enfoque.

Muchos desarrolladores se apresurarían a señalar que tales enfoques pueden ser más tediosos, requiriendo que regularmente revises y actualices tu(s) sentencia(s) `import` cada vez que te das cuenta de que necesitas algo más de un módulo. El intercambio es a cambio de conveniencia.

En esa perspectiva, la preferencia podría ser importar todo del módulo en un solo espacio de nombres, en lugar de importar miembros individuales, cada uno directamente al alcance. Afortunadamente, la sentencia `import` tiene una variación de sintaxis que puede soportar este estilo de consumo de módulo, llamada *importación de espacio de nombres*.

Considera un módulo `"foo"` exportado como:

```js
export function bar() { .. }
export var x = 42;
export function baz() { .. }
```

Puedes importar toda esa API a un solo enlace de espacio de nombres de módulo:

```js
import * as foo from "foo";

foo.bar();
foo.x;			// 42
foo.baz();
```

**Nota:** La cláusula `* as ..` requiere el comodín `*`. En otras palabras, no puedes hacer algo como `import { bar, x } as foo from "foo"` para traer solo parte de la API pero aún enlazar al espacio de nombres `foo`. Me habría gustado algo así, pero para ES6 es todo o nada con la importación de espacio de nombres.

Si el módulo que estás importando con `* as ..` tiene una exportación por defecto, se nombra `default` en el espacio de nombres especificado. Puedes adicionalmente nombrar la importación por defecto fuera del enlace de espacio de nombres, como un identificador de nivel superior. Considera un módulo `"world"` exportado como:

```js
export default function foo() { .. }
export function bar() { .. }
export function baz() { .. }
```

Y este `import`:

```js
import foofn, * as hello from "world";

foofn();
hello.default();
hello.bar();
hello.baz();
```

Aunque esta sintaxis es válida, puede ser bastante confuso que un método del módulo (la exportación por defecto) esté enlazado al nivel superior de tu alcance, mientras que el resto de las exportaciones nombradas (y una llamada `default`) están enlazadas como propiedades en un identificador de espacio de nombres con nombre diferente (`hello`).

Como mencioné antes, mi sugerencia sería evitar diseñar las exportaciones de tu módulo de esta manera, para reducir las posibilidades de que los usuarios de tu módulo sufran estas peculiaridades extrañas.

Todos los enlaces importados son inmutables y/o de solo lectura. Considera la importación anterior; todos estos intentos de asignación subsiguientes lanzarán `TypeError`s:

```js
import foofn, * as hello from "world";

foofn = 42;			// (runtime) TypeError!
hello.default = 42;	// (runtime) TypeError!
hello.bar = 42;		// (runtime) TypeError!
hello.baz = 42;		// (runtime) TypeError!
```

Recuerda antes en la sección "Exportar Miembros de API con `export`" que hablamos sobre cómo los enlaces `bar` y `baz` están enlazados a los identificadores reales dentro del módulo `"world"`. Eso significa que si el módulo cambia esos valores, `hello.bar` y `hello.baz` ahora referencian los valores actualizados.

Pero la naturaleza inmutable/de solo lectura de tus enlaces importados locales impone que no puedas cambiarlos desde los enlaces importados, de ahí los `TypeError`s. Eso es bastante importante, porque sin esas protecciones, tus cambios terminarían afectando a todos los demás consumidores del módulo (recuerda: singleton), lo que podría crear algunos efectos secundarios muy sorprendentes.

Además, aunque un módulo *puede* cambiar sus miembros de API desde adentro, deberías ser muy cauteloso de diseñar tus módulos intencionalmente de esa manera. Los módulos ES6 están *destinados* a ser estáticos, así que las desviaciones de ese principio deberían ser raras y deberían estar cuidadosa y verbosamente documentadas.

**Advertencia:** Hay filosofías de diseño de módulos donde realmente pretendes dejar que un consumidor cambie el valor de una propiedad en tu API, o las APIs de módulo están diseñadas para ser "extendidas" teniendo otros "plugins" que añaden al espacio de nombres de la API. Como acabamos de afirmar, las APIs de módulos ES6 deberían pensarse y diseñarse como estáticas e inmutables, lo que restringe y desalienta fuertemente estos patrones alternativos de diseño de módulos. Puedes sortear estas limitaciones exportando un objeto simple, que por supuesto puede entonces ser cambiado a voluntad. Pero ten cuidado y piensa dos veces antes de ir por ese camino.

Las declaraciones que ocurren como resultado de un `import` son "elevadas" (ver el título *Scope & Closures* de esta serie). Considera:

```js
foo();

import { foo } from "foo";
```

`foo()` puede ejecutarse porque no solo la resolución estática de la sentencia `import ..` determinó qué es `foo` durante la compilación, sino que también "elevó" la declaración al tope del alcance del módulo, haciéndola disponible a lo largo del módulo.

Finalmente, la forma más básica del `import` se ve así:

```js
import "foo";
```

Esta forma en realidad no importa ninguno de los enlaces del módulo a tu alcance. Carga (si no está ya cargado), compila (si no está ya compilado), y evalúa (si no se ha ejecutado ya) el módulo `"foo"`.

En general, ese tipo de importación probablemente no va a ser terriblemente útil. Puede haber casos nicho donde la definición de un módulo tiene efectos secundarios (como asignar cosas al objeto `window`/global). También podrías imaginar usar `import "foo"` como una especie de precarga para un módulo que pueda necesitarse después.

### Dependencia Circular de Módulos

A importa B. B importa A. ¿Cómo funciona esto realmente?

Diré de entrada que diseñar sistemas con dependencia circular intencional es generalmente algo que trato de evitar. Dicho esto, reconozco que hay razones por las que la gente hace esto y puede resolver algunas situaciones de diseño complicadas.

Consideremos cómo ES6 maneja esto. Primero, módulo `"A"`:

```js
import bar from "B";

export default function foo(x) {
	if (x > 10) return bar( x - 1 );
	return x * 2;
}
```

Ahora, módulo `"B"`:

```js
import foo from "A";

export default function bar(y) {
	if (y > 5) return foo( y / 2 );
	return y * 3;
}
```

Estas dos funciones, `foo(..)` y `bar(..)`, funcionarían como declaraciones de función estándar si estuvieran en el mismo alcance, porque las declaraciones se "elevan" al alcance completo y por lo tanto están disponibles entre sí independientemente del orden de autoría.

Con módulos, tienes declaraciones en alcances completamente diferentes, así que ES6 tiene que hacer trabajo extra para ayudar a que estas referencias circulares funcionen.

En un sentido conceptual aproximado, así es como las dependencias circulares de `import` se validan y resuelven:

* Si el módulo `"A"` se carga primero, el primer paso es escanear el archivo y analizar todas las exportaciones, para poder registrar todos esos enlaces disponibles para importación. Luego procesa el `import .. from "B"`, que señala que necesita ir a buscar `"B"`.
* Una vez que el motor carga `"B"`, hace el mismo análisis de sus enlaces de exportación. Cuando ve el `import .. from "A"`, ya conoce la API de `"A"`, así que puede verificar que la importación es válida. Ahora que conoce la API de `"B"`, también puede validar el `import .. from "B"` en el módulo `"A"` que está esperando.

En esencia, las importaciones mutuas, junto con la verificación estática que se hace para validar ambas sentencias `import`, virtualmente componen los dos alcances de módulo separados (a través de los enlaces), de modo que `foo(..)` puede llamar a `bar(..)` y viceversa. Esto es simétrico a si hubieran sido originalmente declarados en el mismo alcance.

Ahora intentemos usar los dos módulos juntos. Primero, probaremos `foo(..)`:

```js
import foo from "foo";
foo( 25 );				// 11
```

O podemos probar `bar(..)`:

```js
import bar from "bar";
bar( 25 );				// 11.5
```

Para el momento en que las llamadas `foo(25)` o `bar(25)` se ejecutan, todo el análisis/compilación de todos los módulos se ha completado. Eso significa que `foo(..)` internamente conoce directamente a `bar(..)` y `bar(..)` internamente conoce directamente a `foo(..)`.

Si todo lo que necesitamos es interactuar con `foo(..)`, entonces solo necesitamos importar el módulo `"foo"`. De manera similar con `bar(..)` y el módulo `"bar"`.

Por supuesto, *podemos* importar y usar ambos si queremos:

```js
import foo from "foo";
import bar from "bar";

foo( 25 );				// 11
bar( 25 );				// 11.5
```

La semántica de carga estática de la sentencia `import` significa que un `"foo"` y `"bar"` que dependen mutuamente entre sí a través de `import` asegurarán que ambos estén cargados, parseados y compilados antes de que cualquiera de ellos se ejecute. Así que su dependencia circular se resuelve estáticamente y esto funciona como esperarías.

### Carga de Módulos

Afirmamos al principio de esta sección "Módulos" que la sentencia `import` usa un mecanismo separado, proporcionado por el entorno de alojamiento (navegador, Node.js, etc.), para realmente resolver la cadena de especificador de módulo en alguna instrucción útil para encontrar y cargar el módulo deseado. Ese mecanismo es el *Cargador de Módulos* del sistema.

El cargador de módulos por defecto proporcionado por el entorno interpretará un especificador de módulo como una URL si está en el navegador, y (generalmente) como una ruta del sistema de archivos local si está en un servidor como Node.js. El comportamiento por defecto es asumir que el archivo cargado está escrito en el formato de módulo estándar de ES6.

Además, podrás cargar un módulo en el navegador a través de una etiqueta HTML, similar a cómo se cargan los programas de script actuales. Al momento de escribir esto, no está completamente claro si esta etiqueta será `<script type="module">` o `<module>`. ES6 no controla esa decisión, pero las discusiones en los organismos de estándares apropiados ya están bien avanzadas en paralelo con ES6.

Sea cual sea la apariencia de la etiqueta, puedes estar seguro de que bajo el capó usará el cargador por defecto (o uno personalizado que hayas pre-especificado, como discutiremos en la siguiente sección).

Igual que la etiqueta que usarás en el markup, el cargador de módulos en sí no está especificado por ES6. Es un estándar separado y paralelo (http://whatwg.github.io/loader/) controlado actualmente por el grupo de estándares de navegadores WHATWG.

Al momento de escribir esto, las siguientes discusiones reflejan un primer paso en el diseño de la API, y las cosas probablemente cambiarán.

#### Cargar Módulos Fuera de Módulos

Un uso para interactuar directamente con el cargador de módulos es si un no-módulo necesita cargar un módulo. Considera:

```js
// script normal cargado en el navegador vía `<script>`,
// `import` es ilegal aquí

Reflect.Loader.import( "foo" ) // devuelve una promesa para `"foo"`
.then( function(foo){
	foo.bar();
} );
```

La utilidad `Reflect.Loader.import(..)` importa el módulo completo al parámetro nombrado (como un espacio de nombres), igual que la importación de espacio de nombres `import * as foo ..` que discutimos antes.

**Nota:** La utilidad `Reflect.Loader.import(..)` devuelve una promesa que se cumple una vez que el módulo está listo. Para importar múltiples módulos, puedes componer promesas de múltiples llamadas `Reflect.Loader.import(..)` usando `Promise.all([ .. ])`. Para más información sobre Promesas, ver "Promesas" en el Capítulo 4.

También puedes usar `Reflect.Loader.import(..)` en un módulo real para cargar dinámica/condicionalmente un módulo, donde `import` en sí no funcionaría. Podrías, por ejemplo, elegir cargar un módulo que contenga un polyfill para alguna característica de ES7+ si una prueba de características revela que no está definida por el motor actual.

Por razones de rendimiento, querrás evitar la carga dinámica siempre que sea posible, ya que obstaculiza la capacidad del motor JS de disparar búsquedas tempranas desde su análisis estático.

#### Carga Personalizada

Otro uso para interactuar directamente con el cargador de módulos es si quieres personalizar su comportamiento a través de configuración o incluso redefinición.

Al momento de escribir esto, se está desarrollando un polyfill para la API del cargador de módulos (https://github.com/ModuleLoader/es6-module-loader). Aunque los detalles son escasos y altamente sujetos a cambio, podemos explorar qué posibilidades podrían eventualmente materializarse.

La llamada `Reflect.Loader.import(..)` puede soportar un segundo argumento para especificar varias opciones para personalizar la tarea de importación/carga. Por ejemplo:

```js
Reflect.Loader.import( "foo", { address: "/path/to/foo.js" } )
.then( function(foo){
	// ..
} )
```

También se espera que se proporcione una personalización (a través de algún medio) para engancharse al proceso de carga de un módulo, donde una traducción/transpilación podría ocurrir después de la carga pero antes de que el motor compile el módulo.

Por ejemplo, podrías cargar algo que no sea ya un formato de módulo compatible con ES6 (por ejemplo, CoffeeScript, TypeScript, CommonJS, AMD). Tu paso de traducción podría entonces convertirlo a un módulo compatible con ES6 para que el motor lo procese.
